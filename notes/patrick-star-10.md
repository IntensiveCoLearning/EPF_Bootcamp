## EPF 学习打卡｜State 与 Storage 的本质理解

## 今天学习了以太坊中的 State、Storage 以及状态树结构，这一部分让我从“交易执行流程”的理解，进一步深入到“链上数据是如何存在与组织的”。

首先，以太坊的本质可以理解为一个全局状态机。所谓 State，就是当前整个网络中所有账户的状态集合，包括每个账户的余额、交易计数，以及合约中的所有存储变量。每一笔交易的执行，本质上都是对这个全局状态的一次更新。

在账户模型中，每个账户包含几个关键字段，其中最重要的是余额、nonce 以及与合约相关的存储和代码信息。特别是对于合约账户，其内部数据并不是简单存储在某个变量中，而是通过存储结构进行组织和管理。

进一步来看，合约中的变量并不是直接存放在某个简单的数组或映射中，而是存储在链上的 Storage 中。Storage 是持久化的链上数据，一旦写入，就会成为全局状态的一部分，并被所有节点共同维护。与之相比，memory 只是执行过程中的临时数据，在函数执行结束后就会消失，不会进入链上状态。

这一点让我理解了为什么 storage 的读写成本远高于 memory。因为 storage 的写入不仅仅是一次简单的数据修改，而是会引发全局状态的变化，并需要所有节点执行相同的操作来保持一致性。这种设计保证了去中心化系统中的数据一致，但也带来了更高的资源成本。

在更底层的结构上，以太坊并不是通过普通的键值存储来管理状态，而是使用一种特殊的数据结构来组织所有账户信息。这个结构的核心特点是将所有数据通过哈希进行连接，使得任何一个数据的变化都会影响到最终的整体结果。

每个区块中都会包含一个 stateRoot，它可以看作是当前整个状态的唯一标识。只要状态中的任意一部分发生改变，这个根值就会随之变化。这种设计使得节点可以快速验证状态是否一致，同时也提供了防篡改的能力。

从执行角度来看，当一笔交易被处理时，EVM 会读取当前状态，根据交易输入执行逻辑，然后生成新的状态。这意味着交易并不是简单的函数调用，而是一个从旧状态到新状态的确定性转换过程。

通过这一课，我逐渐形成了一个更清晰的整体认知。以太坊并不是在执行一段段代码，而是在不断更新一个全局共享的状态。所有交易的执行结果，最终都会反映在这个状态上。

此外，我也开始理解到为什么 Gas 与存储操作密切相关。由于写入 storage 会改变全局状态，并影响整个状态结构，因此其成本远高于其他操作。这也解释了为什么在合约设计中需要尽量减少不必要的存储写入。

总结来说，State 是以太坊世界的全部数据，Storage 是状态的重要组成部分，而状态树结构则保证了这些数据在去中心化环境下的可验证性与一致性。交易的本质是对状态的一次确定性更新，而整个系统则是在不断进行这样的状态演化。

今天的学习让我从“交易如何执行”进一步过渡到“数据如何存在”，也为后续理解更底层的数据结构和执行机制打下了基础。

## EPF 学习打卡 ｜Finality 与 Reorg 的深入理解

## 今天继续深入学习了以太坊中的最终性与链重组机制，相比之前的理解，这一部分让我开始真正意识到区块链系统的时间维度和状态确定性问题。

首先，一个非常关键的认知是，上链并不等于最终确定。交易被打包进区块，只代表它在当前时刻被某条链接受，但在共识尚未完全收敛之前，这个结果仍然可能被替换。这说明区块链在短时间内并不是一个绝对确定的系统，而是一个逐步收敛的过程。

链重组的本质是网络在不同时间点对“哪一条链是正确链”的判断发生了变化。在实际运行中，由于网络传播存在延迟，不同节点可能先接收到不同的区块。当多个验证者在相近时间提出区块时，网络中会短暂地出现分叉状态。随着后续区块的产生以及更多节点参与投票，系统会根据共识规则选择一条更优的链作为主链，而其他分支则被丢弃。这一过程就是链重组。

从更底层的角度来看，链重组并不是异常情况，而是去中心化系统中信息不同步所导致的自然现象。每个节点都是基于自己当前所见的链来判断状态，因此在短时间内，全网并不存在一个完全统一的“真实状态”。只有在经过多轮区块扩展和验证者投票之后，这种分歧才会逐渐消失。

以太坊通过权益证明机制中的投票过程来实现最终性。验证者会对区块进行确认，当一个区块被足够多的验证者认可后，它就进入最终确定状态。一旦达到这个阶段，除非发生极端的系统级问题，该区块不会再被替换。这种最终性并不是瞬间达成的，而是需要经过多个时间周期逐步完成。

从时间维度来看，可以将区块状态划分为三个阶段。第一个阶段是刚被打包的区块，此时仅仅是被某个验证者提出，风险最高。第二个阶段是经过若干后续区块确认，此时被替换的概率已经降低，但仍然存在不确定性。第三个阶段是达到最终性，此时区块被全网共识锁定，可以视为不可逆。

这一机制揭示了区块链系统的一个核心特征，即它是一个最终一致性系统，而不是强一致性系统。系统在短时间内允许存在多个可能的状态分支，但通过共识机制逐步收敛到唯一结果。

从工程角度来看，这种不确定性直接影响系统设计。开发者不能简单地将“交易已上链”作为判断依据，而必须考虑其确认深度。例如，对于低风险操作，可以接受较少的确认以提高响应速度；而对于高价值操作，则需要等待更多确认甚至最终性，以降低风险。

进一步来看，链重组带来的问题不仅是交易回滚，还包括状态层面的连锁影响。如果一个系统在区块尚未稳定时基于其状态执行了后续逻辑，那么一旦发生重组，这些逻辑就可能建立在错误的状态之上，从而导致结果偏差。这说明，在不稳定状态上做决策，本质上是一种风险行为。

这一点也让我对区块链的理解从“执行正确性”进一步扩展到“状态可信性”。在传统系统中，一旦操作完成，结果通常是立即确定的，而在区块链中，结果的可信程度是随着时间逐步提高的。

从更抽象的角度来看，可以将以太坊视为一个不断收敛的状态机。每个新区块都是对当前状态的一次候选更新，而最终性则是这个状态更新被永久确认的过程。在最终性达成之前，所有状态都处于一种“可能被修正”的状态。

总结来说，最终性决定了状态何时可以被视为不可逆，而链重组揭示了在达到最终性之前系统存在的动态不确定性。理解这一点，不仅是理解共识机制的关键，也是从应用开发走向协议层思维的重要一步。

今天的学习让我意识到，在区块链系统中，时间和确定性是分离的。一个状态的出现并不代表它已经稳定，而真正可靠的状态，是经过共识不断验证后逐步形成的结果。

# EPF 学习打卡｜MEV 本质与链上执行的博弈逻辑

**今天学习了 MEV 这一核心概念，这是从“交易如何执行”进一步走向“交易为何这样执行”的关键一课，也让我开始真正从经济博弈的角度理解以太坊。**

**首先，MEV 可以理解为在区块构建过程中，通过控制交易顺序所提取的额外价值。它并不是直接从用户账户中扣除资产，而是利用交易之间的相互影响，通过调整执行顺序来获取利润。**

**MEV 之所以存在，是因为以太坊同时具备三个条件。第一，交易在被打包之前是公开的，mempool 中的交易可以被所有人观察到，这意味着用户的交易意图是可见的。第二，交易的执行顺序会直接影响最终结果，尤其是在涉及价格变化或状态变化的场景中。第三，区块构建者在打包交易时拥有排序和选择的权力，并且其目标是最大化收益，而不是保证公平性。**

**在这样的机制下，任何可以预测“某笔交易将会改变状态”的行为，都会产生套利空间。攻击者可以在目标交易执行之前或之后插入自己的交易，从而围绕这笔交易构建一个有利的执行顺序。**

**最常见的 MEV 形式包括抢跑、尾随以及三明治攻击。抢跑指的是在目标交易之前插入自己的交易，提前利用即将发生的状态变化获利。尾随则是在目标交易之后执行，从其结果中获利。三明治攻击是将用户交易夹在中间，通过前后两笔交易同时获利，这在交易所场景中尤为常见。**

**通过这些例子可以看出，MEV 的核心并不是单一交易本身，而是交易之间的相对顺序。谁能够更早执行，谁能够围绕某笔交易构建顺序，谁就拥有更大的获利空间。因此，Gas 在这里不仅是执行成本，更是一种争夺排序权的工具。**

**进一步来看，MEV 并不是一个偶然现象，而是区块链结构本身的自然结果。只要交易是公开的，只要执行顺序可以被影响，只要区块空间是有限的，那么 MEV 就不可避免地存在。因此，它更接近于一种系统性特征，而不是简单的漏洞。**

**在实际运行中，MEV 已经形成了一条完整的生态链。包括负责发现机会的搜索者、负责构建区块的构建者，以及最终出块的验证者。这些角色之间通过经济激励协作，使得区块构建逐渐演变为一个收益最大化的过程。**

**这一点让我意识到，以太坊并不是一个单纯的执行系统，而是一个复杂的市场。交易执行并不是按照时间顺序进行，而是在一个竞争环境中，根据出价和潜在收益进行排序。**

**结合我目前在做的自动化执行和 Reactive 相关方向，这一课带来了非常重要的启发。任何基于事件触发的自动化系统，只要其执行条件和行为是可以被观察或预测的，就有可能被利用。例如，当系统监听某个事件并触发交易时，攻击者可以提前布局，在执行前后调整交易顺序，从而获取额外收益。**

**此外，自动化系统往往存在一定的执行延迟，例如事件检测、网络传播以及区块打包等环节，这些时间窗口都会被放大为潜在的 MEV 空间。这意味着，即使逻辑完全正确，如果没有考虑排序和博弈因素，系统依然可能在实际运行中出现偏差。**

**因此，在设计链上自动化系统时，不能只关注功能实现，还必须考虑交易的可见性、执行顺序以及潜在的博弈行为。例如通过限制滑点、优化执行条件、减少可预测性等方式，降低被利用的风险。**

**总结来看，MEV 的本质是围绕交易顺序展开的价值提取行为，它揭示了区块链系统中“执行”背后的经济动机。理解 MEV，不仅是理解攻击方式，更是理解整个系统如何运作的重要一步。**

**今天的学习让我意识到，从应用层走向协议层，本质上是在从“写代码”转向“理解规则与博弈”，而 MEV 正是这个转变过程中最关键的概念之一。**

# EPF 学习打卡 ｜Gas 的本质理解

# 今天重点学习了以太坊中的 Gas 机制，这一部分相比第一课更加底层，也让我对“交易为什么能被执行、为什么有的交易会失败”有了更本质的理解。

首先，Gas 并不是简单的手续费，而是 EVM 执行过程中对计算资源的一种计量单位。以太坊上的每一笔交易都会被全网节点执行，因此会消耗计算、存储和带宽等资源。Gas 的存在，本质上是为了衡量资源消耗，并限制滥用网络的行为。

Gas 本身不是钱，它只是一个单位。真正支付的费用是由 Gas Used 乘以价格决定的。也就是说，Gas 表示消耗了多少资源，而价格决定了这些资源值多少钱。

在执行过程中，不同操作消耗的 Gas 不同。例如简单转账消耗较少，而涉及合约执行或存储写入的操作消耗更多。这让我意识到，智能合约并不是普通程序，而是一个每一步执行都需要付费的系统，因此在设计合约时需要考虑执行成本。

另一个重要概念是 Gas Limit。它表示用户愿意为这笔交易最多提供多少 Gas，相当于执行的预算上限。如果执行过程中 Gas 不够，就会发生 Out of Gas，导致交易失败，但已经消耗的 Gas 不会退还。这一点体现了网络对资源使用的真实计费机制。

在更深层理解中，我认识到 Gas 的核心作用不仅仅是计量资源，更重要的是参与区块空间的竞争。由于每个区块的容量有限，validator 在打包交易时，会优先选择能带来更高收益的交易。因此，交易并不是按照进入 mempool 的时间排序，而是按照收益进行排序。

这让我对 Gas 有了一个更本质的认知：它实际上是用户为进入区块空间所出的价格，是一种市场竞价机制。在 EIP-1559 之后，费用被拆分为基础费用和小费，其中基础费用由系统决定并销毁，而小费则直接激励 validator 优先打包交易。

通过这一课，我意识到区块链中的执行并不是一个简单的“调用函数”过程，而是一个资源受限且存在竞争的系统。交易的执行顺序、是否能及时上链，都与 Gas 密切相关。

结合我目前在做的自动化执行方向，这一点尤为重要。如果自动化交易的 Gas 设置不合理，可能会导致执行延迟、被其他交易插队，甚至完全无法执行。这说明在设计链上自动化系统时，不仅要考虑逻辑正确性，还必须考虑执行成本和交易排序机制。

总结来说，Gas 既是执行资源的计量方式，也是区块空间竞争中的定价工具，而交易的执行本质上是一个基于价格的竞争过程，而不是简单的时间顺序。

今天的学习让我从“理解交易流程”进一步过渡到“理解交易为什么会这样运行”，对协议层的认识更深入了一步。

**一笔交易在以太坊中的完整生命周期（从应用层走向协议层的第一步）**

今天系统学习了一笔交易在以太坊中的完整生命周期

## **一、整体流程梳理**

一笔交易从发起到最终改变链上状态，大致经历如下过程：

用户发起 → 钱包构造交易 → 私钥签名 → 广播到节点 → 进入 mempool → P2P 网络扩散 → Validator 选择交易 → 打包区块 → 共识确认 → EVM 执行 → 状态更新 → 产生日志（Event）

这一流程本质上可以抽象为：

> **签名 → 传播 → 排队 → 打包 → 共识 → 执行 → 状态变更 → 日志输出**

## **二、分阶段理解关键机制**

### **1.钱包与交易签名（Transaction Creation）**

在应用层调用合约（如 writeContract）时，本质是构造一笔交易：

-   to：目标合约地址
    
-   data：函数调用编码（calldata）
    
-   gas / nonce：执行约束参数
    

随后由钱包使用私钥进行签名。

签名的本质不是“加密”，而是：

> **证明这笔交易的发送者身份，并保证交易不可篡改**

### **2\. mempool（交易池）**

交易被发送到节点后，不会立即上链，而是进入 mempool：

-   所有节点本地维护一份待打包交易集合
    
-   交易在此阶段仍未确认
    
-   用户可以通过提高 gas fee 替换交易（Replace-By-Fee）
    

可以类比为：

> “尚未被矿工/验证者接单的订单池”

### **3.P2P 网络传播**

以太坊是去中心化网络，没有中心服务器：

-   交易通过节点之间广播逐步扩散
    
-   每个节点都会校验交易合法性后再转发
    

本质是：

> **一种去中心化的 gossip 传播机制**

### **4\. Validator 与区块打包（PoS）**

在 PoS 机制下：

-   Validator（质押 ETH 的节点）负责出块
    
-   从 mempool 中选择交易（通常优先 gas 高的）
    
-   构建新区块并广播
    

本质是：

> **对有限区块空间的资源竞争（Block Space Auction）**

### **5\. 共识机制与 reorg 风险**

新区块并不是立即“最终确定”的：

-   网络中可能同时出现多个候选区块
    
-   通过投票（如 LMD-GHOST + Casper FFG）确定主链
    
-   在短时间内可能发生链重组（reorg）
    

关键理解：

> **“上链 ≠ 最终确定”**

这也是协议层与应用层最大的认知差异之一。

### **6.EVM 执行（Execution Layer）**

当区块被接受后：

-   所有节点执行区块中的交易
    
-   EVM 解析 calldata 并执行合约代码
    
-   更新链上状态（账户余额 / 存储变量）
    

本质是：

**全网对同一笔交易进行确定性执行**

### **7.状态更新（State Transition）**

交易执行后：

-   修改全局状态（state）
    
-   写入状态树（State Trie）
    

可以理解为：

> **一次全局状态机的状态转移（State Transition）**

### **8\. Event / Log（日志机制）**

在合约执行过程中：

```
emit ValueChanged(100);
```

会产生 Event：

-   被写入区块的 log 中
    
-   不属于状态（不会影响 state）
    
-   可被前端或自动化系统监听
    

关键认知：

**Event 是执行结果的“日志”，而不是状态本身**

* * *

## 三，理解

### **1\. 从“调用函数”到“状态变更请求”**

过去我理解：

> 调用合约函数 = 执行代码

现在理解为：

> **提交一笔交易，请求全网执行一次状态变更**

### **2\. 以太坊本质是状态机**

以太坊可以抽象为：

**一个通过共识维护的全球状态机**

每一笔交易都是：

> **state → state’ 的转换过程**

### **3\. Event 不等于真实世界**

这点对我当前做的 Reactive / 自动化系统非常关键：

-   Event 可以被伪造（恶意 emit）
    
-   Event 可能被 reorg 回滚
    
-   Event 只是“执行痕迹”，不是最终状态

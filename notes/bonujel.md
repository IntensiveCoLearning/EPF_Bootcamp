先看 **Transaction Intrinsic Validity**。网页把“交易的内在有效性”写成一组前置约束，意思是这笔交易在真正执行任何字节码之前，必须先满足这些条件：发送者账户不能是空账户，而且发送者不能是合约账户；交易 nonce 必须小于 264−12^{64}-1264−1，并且要和链上发送者当前 nonce 精确匹配；输入数据大小不能超过最大代码大小的两倍；交易的 intrinsic gas 不能超过交易自己的 gas limit；发送者余额必须足以覆盖“最大 gas 费用 + 转账 value”；交易愿意支付的 gas 单价必须不低于当前区块的 base fee；如果是 EIP-1559 交易，则 `maxFeePerGas` 还必须大于等于 `maxPriorityFeePerGas`；此外，这笔交易在当前区块里也必须还能塞得下，不能让区块累计 gas 超出 block gas limit。这里的含义很明确：执行层先判断“这笔交易在当前前态、当前区块环境下是否可接受”，只有通过，后面的状态修改才会开始。

**Gas Accounting** 这一部分，核心是在讲：以太坊里“gas 是怎么被预扣、怎么在执行中消耗、怎么在区块层面累计、以及最后怎么结算”的。它先从**交易级别的内在 gas（intrinsic gas）讲起，再延伸到区块级累计 gas、receipt、state root 的承诺校验**，最后用普通转账和 blob 交易两个例子把前面的规则串起来。**intrinsic gas 是一笔交易开始执行前必须先支付的最低 gas 成本**，它对应的是“让 EVM 接受这笔交易并为它建立执行上下文”所必需的最小开销；这个成本会先从交易的 `T_gasLimit` 中扣掉，然后剩余 gas 才真正进入 EVM 供后续执行使用。

第一块内容是 **Intrinsic Gas Calculation**。它把内在 gas g0g\_0g0​ 拆成几部分。第一部分与 **初始化代码（initialization code）** 有关：如果这是合约创建交易，那么 calldata 会被当作 `T_initializationCode`，并按 32 字节为单位做归一化计费。第二部分是 **CALLDATA 本身的字节成本**：无论是普通交易的 `inputData`，还是创建交易里的 `initializationCode`，都会逐字节收费；其中 **零字节按 4 gas 计费，非零字节按 16 gas 计费**。第三部分是 **合约创建附加费**：如果 `T_to = ∅`，也就是合约创建交易，还要额外收一笔 `G_txcreate`，网页列出的值是 **32000 gas**。第四部分是 **所有交易都要支付的基础交易成本** `G_transaction`，其值是 **21000 gas**。第五部分则是 **access list 的附加成本**：每个访问列表地址要收 `G_accesslistaddress`，每个预声明的 storage slot 还要按 `G_accessliststorage` 继续收费。网页想传达的重点是：交易在真正执行任何 opcode 之前，就已经因为“它是什么类型、携带了多少输入数据、是不是创建合约、是不是带 access list”而决定了一笔不可避免的起始成本。

这部分后面还专门解释了 calldata 收费的意义。`T_inputData` 和 `T_initializationCode` 本质上共同构成交易的 CallData 参数；如果 `T_to ≠ 0_Bytes`，这些数据就会被视为传给合约入口点的输入。之所以对零字节和非零字节使用不同价格，是因为它们会影响区块体大小、传播延迟和处理成本。这套 4/16 的定价逻辑原本是在 PoW 时代为了在**出块速度、链增长速度和网络延迟**之间做平衡而设计出来的；把它进一步适配到 PoS 时代，仍然是一个值得继续研究和优化的方向。也就是说，这里不只是给了一个常数表，而是在强调：**intrinsic gas 的一部分本质上是在给“链上数据占用”和“网络传播负担”定价**。

随后把 gas accounting 放回到整个执行流程里理解。它前面在环境初始化部分已经定义了 `I_gasLimit` 和 `I_gasPrice` 这类执行环境变量，其中 `I_gasLimit` 表示当前交易在考虑区块内已用 gas 后还可用的最大 gas，`I_gasPrice` 表示当前消息调用在 gas accounting 上使用的有效 gas 价格。接着在 message call 的参数映射里，网页把 `availableGas_g` 对应到 `I_gas`，把 `effectiveGasPrice_p` 对应到 `I_gasPrice`。这其实是在说明：**gas accounting 并不是交易执行之外的附加逻辑，而是直接进入了 EVM 调用语义本身**；消息调用时，EVM 总是带着“你还剩多少 gas、这单位 gas 按什么价格计费”这样的上下文一起工作。

接着给出 **Execution Result** 的 gas 语义。它说明一次消息调用会产出一个五元组 (σ′,g′,A′,z,o)(σ′, g′, A′, z, o)(σ′,g′,A′,z,o)。如果执行失败，比如出现 OOG 或其他异常终止，那么状态会回滚，执行状态码 zzz 为 0，而且 **剩余 gas 记为 0**，等于这次调用可用 gas 全部耗尽；如果执行成功，则 z=1z = 1z=1，未消耗完的 gas g′′g''g′′ 会作为剩余 gas 返回。与此同时，执行过程中产生的 logs 和 refunds 之类的 substate，只有在执行成功时才会真正提交。这里网页实际上是在讲一个很关键的 gas accounting 原则：**以太坊不是简单地“先执行，再算钱”，而是在执行语义内部就把异常路径、gas 消耗、剩余 gas 返还和 substate 是否提交绑定在一起了**。

到了区块层面，网页又引入了 **Gas Accumulation and Receipts**。它定义 R\[n\]R\[n\]R\[n\] 为执行完第 nnn 笔交易之后的**累计 gas 使用量**，并用递归公式说明：前 nnn 笔交易的累计 gas，等于第 nnn 笔交易自身消耗的 gas，再加上前 n−1n-1n−1 笔交易的累计 gas。这里的 Υg(σ\[n−1\],BT\[n\])\\Upsilon^g(\\sigma\[n-1\], B\_T\[n\])Υg(σ\[n−1\],BT​\[n\]) 就是“在状态 σ\[n−1\]\\sigma\[n-1\]σ\[n−1\] 下执行区块中第 nnn 笔交易所消耗的 gas”。receipt 里记录的不只是某笔交易自己的执行结果，还包含一个**截至当前交易为止的累计 gas 值**，而这个累计值又直接参与区块头承诺的验证。

因此下一步就是 **Commitment Verification**。交易 receipt 的有序序列，以及最终世界状态，都会分别通过 Merkle–Patricia Trie 做承诺；区块要想有效，至少要满足三件事：一是**计算出来的 receipts root 必须和区块头中的对应字段一致**，二是**最终 state root 必须和区块头一致**，三是**累计 gas 使用量必须满足区块 gas limit 约束，即 R\[n\]≤BHlR\[n\] \\le B\_{H\_l}R\[n\]≤BHl​​**。这段内容非常关键，因为它说明 gas accounting 到最后并不是停留在“算出一个费用数字”而已，而是直接变成了区块有效性条件的一部分。网页还特别强调，**区块有效性是原子性的**：块级状态转换函数只在“重建、执行、累计、承诺校验”全部通过时才接受这个块，不存在“区块里一部分交易先算通过、部分接受”的说法。

最后通过两个例子把 gas accounting 的实际结算方式讲得更直观。第一个例子是 **Simple ETH Transfer**。它说，普通 ETH 转账可以来自三种交易类型：Type 0（legacy）、Type 1（legacy + access list）和 Type 2（EIP-1559）。发送者自己定义的参数里，所有类型都有 `gasLimit`；Type 0/1 直接给出 `gasPrice`；Type 2 则给出 `maxFeePerGas` 和 `maxPriorityFeePerGas`。而协议侧会给 Type 2 额外提供区块级的 `baseFee`。在进入执行前，会先做一笔**upfront reservation**：Type 0/1 预留 `gasLimit × gasPrice`，Type 2 预留 `gasLimit × maxFeePerGas`。执行结束后，真正的实际费用再按 `gasUsed × effectiveGasPrice` 计算。对于 Type 0/1，`effectiveGasPrice = gasPrice`，整笔 gas 费用都给 proposer；对于 Type 2，`effectiveGasPrice = baseFee + min(maxPriorityFeePerGas, maxFeePerGas − baseFee)`，其中 **base fee 被烧毁，tip 给 proposer**。如果预留金额大于实际花费，差额就按 `reserved − actualCost` 退还给发送者。网页在这里想说明的是：**EIP-1559 交易不是直接按 maxFeePerGas 全额扣光，而是先按上限预留、执行后再按 effectiveGasPrice 精算，多余部分退回**。

第二个例子是 **Blob Transaction**。Type 3（EIP-4844）交易除了普通 EVM gas 之外，还要额外支付一套单独的 **blob gas fee**。这一部分只围绕 blob 来看：交易参数包括 `blobVersionedHashes`、`totalBlobGas = GasPerBlob × numberOfBlobs`，以及发送者愿意支付的最高 blob gas 单价 `maxFeePerBlobGas`；区块参数则提供动态变化的 `blobGasPrice`。在这一步想强调的是：**blob 交易有两套 gas accounting 并行存在**——一套是普通执行 gas，和传统交易一样；另一套是专门给 blob 数据空间定价的 blob gas，它有自己的价格变量和自己的上限约束

**“区块执行过程”**的主旨是：一个区块并不是一上来就直接执行，执行之前先做头部校验；只有这些较便宜、较快的检查通过之后，区块才会进入真正的执行阶段。这样设计的好处是，如果 payload 在头部层面就已经不合法，状态转换函数可以尽早向共识层返回 **“Invalid Payload”**，避免继续进入区块执行和交易执行这种开销更高的路径。

接下来把执行阶段拆成了一组非常具体的初始化步骤。第一步，把 `blobGasUsed` 初始化为 0，表示当前区块还没有消耗任何 blob gas。第二步，把 `gasAvailable` 设为区块头中的 `H_gasLimit`，也就是把整个区块可用的执行 gas 上限先装载进执行上下文。第三步，再初始化一些配套的数据结构，包括 **receipts trie**、**withdrawals trie**，以及一个保存区块日志的元组结构；这些对象后面都会随着交易执行和提款处理逐步被填充。这里的重点是：区块执行不是“直接跑交易”，而是先把整块执行所需的计量器、输出容器和日志容器全部准备好。

然后是 Cancun / Deneb 时代一个很关键的新步骤：访问 **Beacon Block Roots Contract Code**。这个机制来自 **EIP-4788**，通过执行层里预先约定的 **BEACON ROOTS ADDRESS**，把信标链的区块根引入 EVM，使 EVM 可以以最小信任方式读取共识层数据。它特别提到，这种能力可用于质押池、再质押、跨链桥以及某些 MEV 缓解场景。换句话说，从这个阶段开始，执行层不只是“执行用户交易”，它还会先插入一段与共识层状态同步相关的系统级逻辑。

对应地，下一步就是构造一笔 **系统事务消息**。这条消息以 **System Address** 为调用者，以 **BEACON ROOTS ADDRESS** 为目标地址，并携带 `H_parentBeaconBlockRoot` 以及刚刚取到的合约代码。可以把它解释成 Duncan 中引入的一种 **system contract**：它和传统 precompile 不同，不是无状态的固定函数，而是一个有状态的合约；同时，只有系统地址才能向里面写数据。随后执行层会搭建 VM 环境，处理这次消息调用，把 `H_parentBeaconBlockRoot` 写入合约存储，供之后的交易通过 slot 时间戳等信息去读取。执行完后，还会删除这一步中触碰到的空账户，以保持状态整洁。

在系统级消息完成之后，才正式进入 **区块内交易处理**。交易首先会被解码，然后加入交易 trie，再进入执行流程。每笔交易的执行又进一步包括几步核心动作：先利用签名分量 `T_v`、`T_r`、`T_s` 恢复发送者地址；再验证交易本身的内在合法性；接着计算 **effective gas price**；随后初始化该交易自己的执行环境；最后才在虚拟机中真正执行这笔交易，包括基于当前状态做校验、进行 gas 计费，并在执行成功时把状态变更落地。这里其实体现了以太坊非常典型的顺序：**先确定交易是谁发的、是否合法、该怎么计费，再让 EVM 真正动状态。**

用户交易处理完后，还要处理 **validator withdrawals**。这一部分来自 **EIP-4895**。处理方式是：遍历区块中每一个 withdrawal，把它们加入 withdrawals trie；然后把提款金额从 **Gwei 转成 Wei**，并把对应余额记到指定地址；如果某些提款相关账户在处理后变成空账户，还会顺带删除，以继续维持状态整洁。也就是说，在今天的以太坊里，区块执行不仅包含普通交易执行，还显式包含 beacon chain 已经验证过的提款入账逻辑。

这一节后面还专门给出了 **Environment initialization**，也就是交易 / 消息执行时 VM 环境里到底有哪些变量。网页列出的环境变量包括：`I_caller` 和 `I_origin`，分别对应当前代码执行的发起者和原始交易发送者；`I_blockHashes`，即最近 255 个区块哈希；`I_coinbase`，当前区块受益人地址；`I_number`，当前区块号；`I_gasLimit`，当前交易在考虑区块已消耗 gas 后还能用多少 gas；`I_baseFeePerGas`，区块基础费；`I_gasPrice`，当前交易的有效 gas 单价；`I_time`，区块时间戳；`I_prevRandao`，来自信标链的随机性值；`I_state`，当前世界状态；`I_chainId`，链标识；`I_traces`，执行跟踪占位；`I_excessBlobGas`，从父区块继承并计算出的 excess blob gas；以及 `I_blobVersionedHashes`，当前 blob 交易所附带 blob 的 versioned hash 列表。网页想表达的是：交易在 EVM 中执行时，并不是只拿到“代码 + 输入”，而是被放进一个非常完整的执行环境里，这个环境同时携带了链上下文、区块上下文、费用上下文和 blob 相关上下文。

如果把这一节总结成一条清晰主线，可以概括为：**区块执行是一个“先系统同步、再逐笔交易、最后处理提款”的流水线。** 先做头部验证以便快速失败；进入执行后先初始化区块级 gas 与输出容器；再运行 EIP-4788 对应的系统消息，把父 beacon block root 写入系统合约；然后顺序执行区块内所有交易，构建交易 trie、回执 trie 与日志；最后处理 EIP-4895 的 validator withdrawals，并清理空账户。整个过程中，EVM 不是在一个抽象真空里运行，而是在一个显式初始化的环境对象里运行，这个环境把 block hash、coinbase、timestamp、base fee、prevRandao、chain id、blob gas 等信息全部传给执行逻辑。

在 EIP-1559 的机制下，如果网络长期处于高负载，`base fee` 和 `gas limit` 从长期看会呈现出完全不同的行为。首先强调的是 **基础费的爆炸式敏感性**：因为每个区块的基础费最多可以按 12.5% 的幅度上调，而这种上调是逐块复利累积的，所以一旦长期持续满载，基础费会增长得非常夸张。在持续满载的情况下，大约 200 个区块，也就是四十分钟左右，发一笔交易的底价就可能高到 1 ETH；再往后推，不到 2000 个区块，价格就会逼近数学意义上的极端高位。它想表达的重点不是某个精确数字本身，而是：**基础费的长期响应是指数型的，持续拥堵会迅速把价格推得非常高**。

和基础费不同，**Gas Limit 的理论行为是“无上界增长”**。网页的意思是，Gas 上限并不像基础费那样会因为某种内部公式自己爆炸，它本质上是一个可被逐步调高的容量参数。从纯数学模型上看，它没有硬性的理论天花板，因此可以随着网络需求不断向上调整，以容纳更大的计算吞吐。这里的潜台词是：**基础费承担的是“价格抑制需求”的职责，而 gas limit 承担的是“容量适应需求”的职责**，两者虽然都跟拥堵有关，但调节方式完全不同。

在现实里基础费不可能真的无穷上涨，因为一旦价格高到一定程度，用户就会退出，需求自然下降；与此同时，如果节点逐步提高 gas limit，那么目标 gas 使用量也会随之上升，网络可容纳的交易量会变大。于是系统不会无限走向“天价费用”，而是会在“高价格抑制需求”和“更大容量释放供给”之间形成一个新的动态平衡点。换句话说，这一节的核心结论是：**长期满载会先把 base fee 以复利方式迅速推高，但真实市场会通过需求收缩和容量调整把系统重新拉回到某种平衡状态**。

**Dynamics of Blob Gas Price** 可以译成：**“Blob Gas 价格的动态变化”**。这一节讨论的是 EIP-4844，也就是 Proto-Danksharding 引入的 Blob 费用模型。 Blob 的基本容量设定：把 Blob 视为以太坊专门给 L2 提供的数据通道，每个区块的目标是 3 个 Blob，最大是 6 个 Blob。只要 L2 上传的数据量没有超过目标值，也就是不超过 3 个 Blob，那么 Blob 的基础价格就维持在一个极低的地板价，也就是 1 Wei。：**Blob 费用不是一开始就灵敏波动的，而是在低负载区间几乎保持常量，给 L2 一个非常便宜的数据发布通道**。

**“延迟计费机制”**。它的意思是，如果某一个区块突然塞满了 6 个 Blob，也就是相对目标超载了 3 个，系统并不会立刻把 Blob 基础费抬高。相反，它会先把这部分超出的 Blob Gas 记到一个内部账本里，也就是 `Accumulated Excess Gas`，可以理解成“累积超额 Blob Gas”。只要这个累计值还没有积累到足够高，价格仍然会停留在 1 Wei 的低位。这个设计说明 Blob 费用模型并不是“单块超载，单块立刻猛涨”，而更像是一种 **对持续性拥堵才作出反应的记账式调节机制**。

当 L2 的数据需求长期持续爆满，也就是连续很多区块都把 Blob 塞到最大值时，那个“累积超额”会不断滚大。此时 Blob 基础费不再只是线性上调，而是按一个以自然常数 eee 为底的指数函数增长。

它要表达的是，一旦累积超额突破阈值，Blob 价格就会出现明显的指数级上升。这种设计带有很强的拥堵惩罚和抗滥用属性：当网络上出现大量低价值、持续性的 Blob 数据占用时，系统会迅速提高价格，把这些边际价值较低的数据挤出市场，从而避免 Blob 通道被廉价垃圾数据长期塞满。网页把这理解成一种明显的 **防 DOS 式经济调节机制**

EIP-1559 之下，**一个区块的 gas 使用量会怎样影响下一个区块的 base fee（基础费）**。这是一种“阶梯状的线性推进”：也就是说，整体趋势近似线性，但因为底层计算使用整数和取整，曲线不会是光滑直线，而是一级一级的台阶。特别强调，在中点也就是目标 gas 使用量附近，台阶会显得更宽，这表示**当区块接近目标容量时，base fee 对小幅波动不那么敏感，因此更稳定**。这和 EIP-1559 的设计目标一致：网络希望把平均区块使用量拉回到目标值附近，而不是让价格对每一点轻微拥堵都剧烈反应。网页中的图把目标点写成 15,000，这更像是做图时的归一化或缩放表示；在以太坊主网上，目标区块大小在机制上始终是 gas limit 的一半。

接着讲 **最大上行变化**。它把这部分称为“Maximum upward change ~12.5%”，意思是：如果上一个区块明显超过目标容量，那么下一个区块的 base fee 会上涨，而且**单个区块的最大涨幅被硬性限制在 12.5%**。这个上限来自 EIP-1559 里的 `BaseFeeChangeDenominator = 8`，也就是每个区块最多按八分之一的幅度调整。换句话说，如果父区块满到达到最大区块容量，下一个区块的 base fee 最多只能涨到原来的 1.125 倍，而不会无限制跳涨。网页把这看成一种“物理极限”式的限速器，用来抑制单块拥堵带来的价格暴冲。官方文档和 EIP-1559 本身都明确写了这一点：base fee 是根据**父区块 gas used 与目标 gas**的偏离程度线性调整，但单块调整幅度上限就是 12.5%。

然后谈到 **最大下行变化**，写成了 “Maximum downward change ~10%”。按图里自己的说法，它认为下跌幅度之所以看起来小于 12.5%，是因为图里假定区块 gas 消耗不可能降到绝对 0，而是设置了一个非零下限，所以在那张具体图的参数下，曲线底部对应的跌幅大约只有 10%。但从 **EIP-1559 的正式机制** 来说，若父区块真的是空块，base fee 的单块最大跌幅同样可以达到 **12.5%**；只有当父区块 gas used 低于目标、但不是 0 时，跌幅才会小于这个上限。所以这一节网页里“约 10%”更适合理解为**图形分析中的一个场景结果**，而不是协议层面的普遍常数。

网页最后一个观察点是 **目标值附近的灵敏触发器**，它写成 “The 1% increase near target”。它的意思是：即便区块 gas 使用量只是刚刚碰到目标值附近，或者略微高于目标，base fee 也可能出现一个非常小的上调，比如图上示意的 1% 左右。这说明该函数并不是在目标点附近完全“僵住不动”，而是保留了一点弹性。这种设计可以理解成一种“缓冲区”：它既避免了费用在目标值附近乱跳，又不会让系统在接近拥堵时完全丧失调节能力。用更协议化的话说，就是 **base fee 的调节幅度与“父区块偏离目标值的程度”成比例**，所以靠近目标时变动自然较小，远离目标时变动才会更明显。

用一张图帮助你建立对 **EIP-1559 base fee 更新曲线** 的直觉。第一，这条曲线整体上是“围绕目标 gas 使用量进行线性反馈调节”的：超过目标就涨，低于目标就跌，目标点是平衡点。第二，因为实现里使用整数和取整，图形不会是连续光滑曲线，而是台阶状，这也是网页所说的“step-like linear progression”。第三，系统对单块变化做了硬限制，标准参数下最大单块涨跌幅都是 12.5%，这保证了费用不会因为一两个块突然剧烈失控。第四，目标点附近的变动故意被设计得更温和，从而让 base fee 在正常波动范围内更平滑、更可预测。网页想传达的核心不是某个精确数字本身，而是这样一个机制特征：**以太坊不是按拍卖那样瞬时重定价，而是通过一个有阻尼、有限速、围绕目标点回归的离散反馈系统，逐块调节基础费。**

以太坊协议把一些**区块级参数**——例如 gas used、gas limit 以及 base fee per gas ——定义为**自然数** N 。这个设计并不是随意的，而是在区块链底层经济模型中嵌入了一层很直观的逻辑。自然数从 0 开始、向上无限延伸，它们为这些参数的理解和操作提供了一个直接而清晰的框架。与实数不同，自然数是离散的、精确的，不存在任意两个值之间“无限可分”的中间状态，因此非常适合区块链这种对精度要求极高的场景。这样的定义使得处理这些参数的函数更容易推理，也更容易对交易成本和网络容量做精确计算与预测。

自然数的一个重要特点是它们可以被理解为由 0 和不断加 1 构造出来的对象，因此“增加”或“减少”都可以用非常原子的方式来表达。网页把这种性质称为自然数的“原子性”：以 0 和 Successor（后继，+1）为基础构件，能够在 Ethereum 的逻辑中构造更稳健、也更容易证明正确性的规则。与之相对，若使用实数或小数，就会引入无限可分性，而这会把协议运算带到更复杂、也更难界定的计算语义里。网页的意思是：执行层经济参数之所以选自然数，不只是“因为不会有小数 gas”，更深一层是因为**离散数值更适合构造可计算、可验证、可证明的协议规则**。

然后话题延伸到**交易级参数**。像 `maxPriorityFeePerGas`、`maxFeePerGas`、`gasLimit`、`gasPrice` 这些交易参数，并不是放在无限的 N 里，而是放在一个**有界的自然数集合** N256​ 里，也就是受 256 位上限约束的非负整数集合。这个上限大约是 2的256次方，数量级约为 10的77次方。网页想表达的是，这样的设计在两件事之间做了平衡：一方面，数值空间足够大，几乎不会在正常使用中碰到上界；另一方面，它又不是无界的，从实现和安全角度仍然处于机器字长可管理、可约束的范围内。

它其实在讲三个层次。第一层是**协议语义层**：gas、base fee 这些量被建模成自然数，意味着它们天然就是离散资源，而不是连续变量。第二层是**推理与证明层**：自然数体系更适合写出清晰、可形式化验证的规则，因为它基于“0 与后继”这种非常简单的构造方式。第三层是**工程实现层**：区块级参数可放在无界的 N 中做数学建模，而交易字段在实现时又进一步限制在 N256​ 中，使协议既有数学表达上的干净性，也有实现上的安全边界。

再说得更直白一点，这一节的核心观点不是“自然数很基础”这么简单，而是：**Ethereum 的费用模型之所以稳定、可实现、可验证，很大程度上依赖于它把关键经济变量都放在离散整数系统里处理，而不是放在连续数系统里处理。** 这样做能避免小数与无限精度带来的歧义、复杂度和潜在漏洞，同时让协议规则更容易被客户端一致地执行

# 区块头验证

先拿到链上最近的区块头，也就是**父区块头**。然后基于父区块头去计算当前区块应有的 `excess_blob_gas`，并检查当前区块头里写的这个值是否正确。这一步是 Cancun / EIP-4844 之后新增的重要校验，因为 blob gas 的状态不是孤立存在的，而是由父区块延续下来的，所以当前块头里的 `excess_blob_gas` 必须和父块推导结果一致；否则，这个块在执行层上就应该被判为无效。

接下来是**区块头本身的合法性检查**。这里检查的不是“这个头看起来像不像区块头”，而是它是否满足执行层当前 fork 规则下的 header validity。进入 PoS 时代以后，这里面有一些字段变成了强约束常量：比如 `ommersHash` 必须对应空 ommer 列表，`difficulty` 必须为 0，`nonce` 必须为 0，而且客户端必须按照 post-Merge 的区块头和执行环境规则去验证 payload。换句话说，今天执行层里的 block header validation，已经不再是 PoW 时代那套“验证挖矿结果”的逻辑了，而是“验证这个 execution payload 在当前 fork 规则下是否是一个合法区块”。

另外，`ommers` 字段还要额外检查为空。这一点本质上也是 Merge 之后规则的一部分：PoS 时代主网不再接受 uncle / ommer 机制，所以 block body 里的 `ommers` 必须为空，header 里的 `ommersHash` 也必须是空列表的固定哈希。之后才进入真正的**区块执行**阶段，也就是顺序执行区块里的交易，统计执行后的 `gasUsed`，计算交易 Trie 根、回执 Trie 根、日志布隆过滤器，以及执行完成后的状态。这里虽然标题叫 Block Header Validation，但它并不是只做“静态头字段检查”，而是把**执行区块并回填验证 header 承诺值**也算进了这一步里。因为以太坊区块头不是单纯的元数据，它本身就承诺了执行结果。

再往后要把执行得到的结果与区块头里的字段逐项比对。至少包括 `state_root`，而且实际上也包括 `gas_used`、`receipts_root`、`logs_bloom` 这类执行产物。只有当“本地重新执行算出来的值”和“区块头里声明的值”一致时，这个块才算真的通过验证。这个点很重要，因为它说明执行层验证区块不是“信 header”，而是“重算 execution result，再验证 header 没撒谎”。如果这些值对不上，客户端就应该抛出 `Invalid Block`。执行规范测试里近年的测试更新还专门补了 `gas_used` 通过 `header_verify` 检查的用例，说明这确实是 header validation 的一部分，而不是可有可无的附加检查。

最后，如果所有检查都通过，就把区块追加到区块链；然后丢弃过旧区块，只保留最近 255 个。这一段更偏“规范文档中的过程化描述”：它表达的是，状态转换函数成功后，这个块会进入本地链视图，而客户端只需要保留足够近的历史块来支撑后续状态转换和最近区块哈希语义。网页同时也明确了失败路径：任何一步校验不通过，都直接报 `Invalid Block`；全部通过则正常返回。严格说，这里的“Block Header Validation”其实不只是 header 的语法检查，而是一个更完整的流程：**先用父块验证当前块头的派生字段和 fork 规则，再执行区块体，最后用执行结果反向验证区块头中的承诺字段。**

# 两个核心问题：1.是否可以将该区块追加到区块链末尾？2.状态会因此发生怎样的变化？

执行层把“本地已有的链状态”当作输入，再拿一个“候选区块”进来，跑一次 **区块级状态转换函数**。如果这个区块在协议规则下是有效的，就把它执行并接到本地链尾，同时把状态更新成新状态；如果无效，就直接拒绝。这个思路正对应公式.

图左上那个 **Block**，表示“外部送进执行层的候选区块”。图里把它拆成了 `header / ommers / transactions / withdrawals`。可以把它理解成两层：一层是 **header**，它像一个摘要和承诺，声明“这个区块执行完应该得到什么结果”；另一层是 **body**，里面真正装着要执行的数据，最核心的是交易列表，另外还有`withdrawals`。这里要注意一点：在 Merge 之后的以太坊主网里，`ommer/uncles` 已经被废弃了；从 Paris 开始，`ommers` 必须为空，`ommersHash` 也必须是空列表的固定哈希，同时 `difficulty=0`、`nonce=0` 等字段也都变成固定值的一部分。

图左下那个 **BlockChain**，表示“执行层当前手里维护的本地区块链视图”。图里写了 `blocks: [Block]`、`chain_id` 和一个内嵌的 `State`。这可以理解成：执行层不是只拿一个孤立的旧状态在算，它需要知道 **父区块是谁、当前链 ID 是多少、当前状态是什么**。其中 `chain_id` 很关键，因为交易签名验证和重放保护都和它有关；同一笔交易在不同 `chain_id` 上不应该被当成同一条可重放的交易。

再看 **State**。图里把它画成 `main trie / storage tries / snapshots / created accounts`。这里可以分两层理解。第一层，最核心、最“协议”的部分，是 **world state**：账户主 trie，以及每个合约账户自己的 storage trie。它们共同描述账户 nonce、余额、代码哈希、存储等内容。第二层，是执行客户端为了高效执行和表达某些语义而维护的辅助结构，比如 snapshots 这类缓存/加速视图；而 `created accounts` 这种，更接近“执行过程中的辅助信息”，用于表达某些与“本次执行里创建过哪些账户/合约”有关的规则。也就是说，这个框不是在说“链上只有这些字段”，而是在用 EELS/客户端视角告诉你：执行层真正处理的状态，不只是一个抽象数学符号，而是一组可执行、可索引、可验证的数据结构。

中间那个大椭圆 **STATE TRANSITION FUNCTION**，就是整张图的核心。它表示：执行层拿到“旧链 + 新区块”之后，会在里面做两件事。第一，判断这个区块能不能合法地接到当前执行链上。第二，如果能接，就顺序执行区块里的内容，得到一个新的状态和一组执行产物。EELS 对这一步的描述很直接：`state_transition(chain, block)` 负责执行新块、验证其有效性，并在成功时把它追加到链上。

下面重点说 **“Block valid?” 到底检查什么**。最准确的理解是：它不是一个单独的小 if，而是一整套 **“执行前检查 + 执行区块 + 执行后核对”** 的组合判断。EELS 对这件事的描述也是这个顺序：先校验区块头是否自洽、是否和父区块关系正确；然后执行区块体；最后把执行出来的结果和区块头里写死的承诺值做比对，只要有不一致，就抛 `InvalidBlock`。

第一类是 **执行前的静态/结构性检查**。典型包括：父区块是否存在、当前块和父块的关系是否正确、时间戳是否大于父块、区块头里的字段组合是否讲得通。对于 post-Merge 的以太坊，还要检查一批“现在必须固定”的字段，比如 `ommersHash` 必须等于空列表哈希、`difficulty` 必须是 0、`nonce` 必须是 0、`extraData` 长度不能超过上限；而以前 PoW 时代那些针对 Ethash、difficulty 公式、ommer 列表本身的验证，则被移除了。也就是说，今天的“block valid”里，已经不再有“这块的 PoW 工作量证明对不对”这种检查了。

第二类是 **执行阶段的动态检查**。这一步不是只看 header，而是真把区块“跑一遍”。EELS 的 `apply_body` 会先处理系统交易，再按顺序一笔一笔处理用户交易，然后处理 validator withdrawals，最后汇总执行输出。这里所谓“验证区块”，本质上已经包含了“验证区块里每笔交易在当前前序状态下是否可执行”：例如签名是否匹配、nonce 是否正确、发送方余额和 gas 是否足够、交易在前面交易已经改过状态之后还能不能继续成立。因为以太坊区块内交易是顺序执行的，所以同一笔交易换个顺序、或者放到另一个前态下，结果都可能不同。

第三类是 **执行后的结果核对**。这一步最关键。区块 proposer 在 header 里已经承诺了“我这个块执行完以后，应该得到哪些结果”。执行层把区块跑完后，会把自己真实算出来的结果与 header 对比。共识层缓存的 execution payload header 字段里就包括 `state_root`、`receipts_root`、`gas_used`、`logs_bloom`、`transactions_root`、`withdrawals_root` 等；而执行层这边会验证自己算出的结果是否和这些值一致。只要有一项对不上，这个区块就是无效块。你可以把它理解成：**header 不是拿来参考的，而是拿来验算的**。

如果所有检查都通过，图右边那一步就是 **Append block to blockchain**。EELS 的描述是：区块通过验证后，会被追加到本地链视图中，旧状态被新状态替换；同时，为了继续支持 `BLOCKHASH` 这类依赖最近历史区块哈希的语义，协议上至少需要保留最近 255 个块来继续处理新块。真实客户端通常会保留更多历史，因为它们还要应对 reorg、调试、索引和数据服务等需求

第一，**σ 是数学意义上的“状态”**，它不是 Python 代码里某个具体类实例的简单别名。 黄皮书里的 σ 更像一个抽象的世界状态概念，而 Python 规范里的 `State` 类是工程实现中的数据结构。两者有关联，但不是同一个层面的东西。

第二，以太坊的状态不是“平铺存着”的，而是经过两层折叠得到根哈希。 先是账户内部存储折叠成 `storageRoot`，再是所有非空账户折叠成世界状态 Trie，最终得到 `stateRoot`。 所以区块头里的 `stateRoot`，本质上是“整个执行后世界状态”的加密承诺。

第三，**状态转换函数做的事情可以概括成：校验父子关系 → 执行区块 → 重新计算结果 → 对比区块头承诺**。 如果重新执行后算出来的 `gasUsed`、`receiptsRoot`、`logsBloom`、`stateRoot` 等结果，和区块头声明的一致，这个块才算有效；否则就是无效块。

# 以太坊升级

第一阶段是 **Frontier**。这代表以太坊主网上线，本质上更像一个面向开发者的 beta 版：先把网络跑起来，让大家能部署客户端、试验合约、开始做工具和应用。这个阶段的重点不是功能完备，而是谨慎启动，所以一开始 gas limit 很小，还用了 canary contracts 这种“升级预警机制”来避免客户端分叉后长时间停机。

第二阶段是 **Homestead**。这代表以太坊从“能跑”走向“更稳定、更适合正式使用”。核心变化不是推翻架构，而是做协议层加固：比如修补交易可塑性问题、调整合约创建的 gas 成本和失败语义、改进难度调整算法，以及引入 `DELEGATECALL`，这对后来的代理合约、可升级合约模式都很关键；同时，网络层也开始强调前向兼容，为以后继续升级打基础。

第三阶段是 **The Merge**。这是历史上最关键的一次范式切换：以太坊把共识从 PoW 切到 PoS。更准确地说，是把原来执行和共识混在一起的单层结构，变成了“执行层 + 共识层”的双层结构：执行仍负责交易和状态转换，共识则由已经运行多时的 Beacon Chain 接管。也就是从这一步开始，以太坊真正进入今天这套模块化架构。

# 设计理念

以太坊的协议设计理念，是在“尽量保持底层简单、通用、可组合”的前提下，把复杂性尽可能封装起来，并把高层应用自由留给合约和上层生态，而不是把各种具体功能硬编码进协议本身。

它主要讲了三层意思。第一层是设计哲学：以太坊追求简洁、普适、模块化、非歧视和敏捷演进。简洁不是说系统真的很小，而是希望核心规范尽量可理解；普适是说协议本身尽量不内建太多“专用功能”，而是提供像EVM这样的通用执行平台；模块化是希望很多创新可以作为独立组件存在，既能服务以太坊，也能被别的协议复用；非歧视是说协议不主动限制某类用途，而是更倾向于通过费用机制让使用者自己承担成本；敏捷则意味着协议不是不可变的，只要改动能明显改善安全性或可扩展性，就应当认真考虑。

第二层是协议原则：最核心的是“管理复杂性”。这页反复强调，复杂性不可完全避免，但要尽量把它放在更合适的位置，比如Layer2优先于客户端实现，客户端实现优先于协议规范本身。它还区分了“封装起来的复杂性”和“系统性交织的复杂性”，前者虽然也复杂，但危险小得多，因为它可以被隔离、替换和单独推理。与此相连的另外几个原则是自由、一般化和“我们没有特性”：协议应该尽量提供底层、通用、可组合的原语，而不是不断为流行场景加新特性；很多高层需求，应该由合约或子协议自己实现。

第三层是这些理念如何落到具体技术选型上。比如，以太坊选择账户模型而不是UTXO，是因为账户模型对复杂状态转换和智能合约更自然；选择Merkle Patricia Trie，是为了让状态具备确定性和可验证性；继续研究Verkle树，是为了在状态越来越大时缩小证明、推进无状态化；从RLP转向在信标链中使用SSZ，是因为SSZ更适合Merkle化、索引和轻客户端证明；在共识上追求终局性，则体现为Casper FFG、LMD-GHOST以及它们组合出的Gasper；在网络层上，用Kademlia型DHT做节点发现，再用gossip网络传播区块，则体现了“结构化引导+非结构化扩散”的折中。换句话说，这页不是在单独解释某几个组件，而是在说明：这些组件之所以被选中，是因为它们符合以太坊一贯的设计取向——核心层尽量通用、组合性强、可验证，同时为未来扩展保留空间。

# Ethererum 协议架构

由两大部分组成——执行层和共识层。执行层（EL）负责处理实际的交易和用户交互，它是“全球计算机”执行程序的核心场所。共识层（CL）则提供了权益证明（PoS）共识机制。在实际应用中，这些层级通过各自独立的客户端实现，并通过 API 相互连接。每一层都拥有专属的 P2P 网络，用于处理不同类型的数据。

## 第一张架构图

最外层的 “Ethereum client” 可以理解成你机器上跑的一整套以太坊节点系统。这个系统里面有两个核心进程。上面的 Beacon Node 是共识层节点，它不负责跑 EVM，不直接执行用户交易；它负责 PoS 相关的事，比如接收和传播 beacon block、处理 attestation、维护 fork choice、跟踪 finality。下面的 Execution Engine 是执行层节点，它才是真正执行交易、维护 world state、保存账户/storage、运行 EVM 的地方。用户熟悉的 Geth、Nethermind、Besu，本质上都属于这一层。共识客户端和执行客户端是分开的软件，但必须协同工作。

Beacon APIs 连到 Beacon Node，主要给验证者相关组件或运维系统使用，比如获取当前 slot、提议/证明职责、提交 attestations、查询 beacon 状态。User APIs 连到 Execution Engine，这就是大家平时最熟悉的 JSON-RPC 入口，钱包、DApp、脚本、Web3 SDK 基本都从这里进来，比如 `eth_call`、`eth_sendRawTransaction`、`eth_getBalance` 这类调用。也就是说，普通用户通常主要接触执行层 API，而不是共识层 API。JSON-RPC 作为执行客户端对外接口是以太坊长期保留的用户入口。

中间的 Engine API 不是给外部用户用的，而是共识层和执行层之间的“内部协议”。共识层通过它告诉执行层：“我现在认为哪个 head 是对的”“请基于当前状态帮我构造一个 execution payload”“这是新收到的 payload，你帮我验证并执行一下”。反过来，执行层也会把 payload 的执行结果、状态有效性等反馈给共识层。Merge 之后，这条接口就变成了以太坊节点内最关键的耦合点。

右边两组 p2p 表示以太坊现在不是一个单一网络，而是两张逻辑上独立的网络同时存在。Beacon Node 参加的是共识层 p2p 网络，传播的是 beacon block、attestation、sync committee 等共识消息。Execution Engine 参加的是执行层 p2p 网络，传播的是交易、execution payload 对应的区块数据等执行层内容。也就是说，CL 和 EL 不只是职责分离，连网络传播面也分离了。你的节点实际上同时在两个网络里说话。

## 第二张架构图

上半部分是 Consensus Layer。里面的 PoS 说明它负责权益证明共识。Fork Choice LMD-GHOST + Finality Casper-FFG，表明：LMD-GHOST 负责在存在多个候选链头时选出当前 head，Casper FFG 负责把链上的某些 checkpoint 推到 finalized 状态。前者更像“现在跟谁走”，后者更像“哪些历史已经不可逆”。此外，这一层还管理 Validators、RANDAO、以及 blob 相关的数据可用性职责。Beacon node 本身就是这一层的核心节点实现。Merge 之后，规范上的 canonical chain 头部选择和 finalized block 信息都来自 Beacon Chain 的 fork choice / finality 机制，而不再由旧的 PoW total difficulty 规则决定。

严格说，validator 不是“另一条链”，而是参与 PoS 的角色。验证者会通过 Beacon APIs 跟 beacon node 协作：接收自己在某个 slot/epoch 的职责，进行 block proposal 或 attestation，再把签名后的消息交给 beacon node 广播到共识层网络。所以验证者逻辑上站在共识层这边，而不是执行层这边。Beacon node 本身负责同步 Beacon Chain、做 attestation 处理、维护验证者集合与职责分配等。

下半部分是 Execution Layer。执行层 p2p 网络会接收和传播用户交易；这些交易先进入 mempool；当需要构造区块时，会从 mempool 里挑交易组成 execution payload；EVM 逐笔执行这些交易，对账户余额、nonce、storage、合约代码相关状态进行读写；执行完后更新 state data。你可以把执行层理解成“确定性状态机”：给我一个前状态和一组交易，我算出后状态。以太坊黄皮书本来描述的核心就是这种基于交易的状态转换系统。

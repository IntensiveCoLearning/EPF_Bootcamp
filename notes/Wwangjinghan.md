以太坊协议经多年演进，核心分为两大层，通过 API 对接、各自独立实现与组网，共同支撑网络运行。

一、两大核心层

1.  执行层（EL）
    

-   负责：交易执行、用户交互、智能合约运行
    
-   定位：以太坊 “世界计算机” 的计算与业务层
    
-   职责：处理链上状态变更、执行合约逻辑
    

2.  共识层（CL）
    

-   负责：权益证明（PoS） 共识机制
    
-   定位：网络安全与链序层
    
-   职责：确保全网节点统一链头、维护唯一主链，提供密码经济学安全保障
    

二、实现方式

-   两层各自有独立客户端，通过Engine API / Beacon API 通信
    
-   各层拥有独立 P2P 网络，处理不同类型数据
    
-   客户端内部包含网络、执行、共识、API 等基础功能模块
    

一、以太坊 5 大核心设计哲学

简洁性

1.  协议尽量简单，普通开发者也能读懂实现，靠模块化与清晰规范保持简洁。
    
2.  通用性
    
    无内置功能，只提供图灵完备 EVM，可通过合约实现任意链上逻辑。
    
3.  模块化
    
    组件解耦、可独立替换升级，复杂度封装在内部，对外提供简单接口。
    
4.  无歧视性
    
    不限制使用场景，只通过手续费调控对协议的损害，不禁止特定应用。
    
5.  敏捷性
    
    协议可通过 EIP 持续迭代，不固化规则，优先提升可扩展性与安全性。
    

## 二、核心设计原则

1.  复杂度管理
    
    底层与接口尽量简单，复杂逻辑压到中间层（编译器、序列化、存储等）。
    
2.  自由中立
    
    用户使用自由，不歧视任何合约 / 交易，用庇古税（手续费） 调节资源占用。
    
3.  泛化性
    
    底层指令最小化，可自由组合，不内置高频场景（如锁仓），交由合约实现。
    

## 三、关键技术选型理由

### 1\. 账户模型（而非 UTXO）

-   更简单、更省空间、交易体积更小
    
-   资产完全可互换，适合 DEX 等复杂应用
    
-   缺点：需 nonce 防重放，无法轻易修剪闲置账户
    

### 2\. Merkle Patricia Trie（MPT）

-   确定性、可密码学验证，状态读写效率 O (log n)
    
-   未来计划用 Verkle 树 替代，大幅缩小证明体积，支撑无状态客户端
    

### 3\. 序列化方案

-   RLP：极简、确定性序列化，只处理嵌套结构，不定义数据类型
    
-   SSZ：替代 RLP，支持固定 / 可变类型、默克尔化、快速索引，服务 PoS 共识
    

### 4\. 共识机制（Gasper）

-   Casper FFG：负责区块最终确定性，作恶需烧掉 ≥33% 质押 ETH
    
-   LMD GHOST：分叉选择规则，选权重最大的链
    
-   二者结合 = Gasper，以太坊完整 PoS 协议
    

### 5\. 网络层

-   DHT（discv5）：用于节点发现，构建路由表
    
-   Gossip 协议：用于区块广播，适合高波动节点网络
    
-   架构：结构化 DHT 引导节点 → 非结构化 Gossip 扩散数据

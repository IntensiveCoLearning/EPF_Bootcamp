关于扩容：

一开始认为，扩容就是让以太坊处理更多交易，但 roadmap 里的思路完全不是这样。以太坊没有选择去不断提高 L1 的执行能力，因为那样会导致节点成本上升，最终破坏去中心化。取而代之的是一种更“克制”的设计——让 L1 不再承担所有事情，而是专注做它最擅长的部分。

这个转变的核心就是所谓的 rollup-centric roadmap。在这个模式下，交易的执行被大量转移到 Rollup（L2）上完成，而以太坊主网只负责三件事：共识、安全和数据可用性。也就是说，用户的大部分交易其实不再直接在 L1 上执行，而是在 Rollup 中打包处理，然后把结果和必要数据提交回 L1。这一步的关键不在于“计算”，而在于“数据必须公开”，因为只有这样别人才能验证这些状态变化是正确的。

但问题很快就出现了：如果 Rollup 还是把数据发到 calldata 里，那成本依然很高，这就卡住了扩容的上限。于是就有了像 EIP-4844 这样的核心升级。它引入了 blob 这种新的数据形式，把数据从“永久存储”变成“临时可用”，并且专门为这种数据建立了一套独立的费用市场。这一步非常关键，因为它第一次真正把数据可用性从执行里拆出来，也让 Rollup 的成本大幅下降，从而让扩容在现实中变得可行。

再往后看，像 Danksharding、DAS这些技术，其实是在进一步放大这个思路。数据会变得越来越多，但节点不需要下载全部数据，而是通过采样就能验证整体可用性，这样既提高了吞吐量，又不会让节点负担爆炸。同时像 PBS这样的机制，则是在解决扩容过程中带来的新问题，比如 MEV 集中和区块构建的复杂性。

所以从整体来看，以太坊扩容真正做的事情不是“让一条链更强”，而是把系统拆分成不同层：L1 提供安全和数据，L2 负责执行。这种模块化设计让系统既能扩展，又不会牺牲去中心化。这种思路上的转变比任何具体技术都更重要，因为它意味着扩容不再是线性提升性能，而是通过架构重构来实现规模化。

merkleization本质上是将已经序列化的数据结构进一步转化为一棵Merkle树，通过递归哈希最终得到一个固定长度的根哈希（hash tree root），这个根可以作为整个数据的唯一“指纹”，用于高效比较和验证状态的一致性。具体过程通常是先把序列化后的数据拆分为固定大小的块（在SSZ中一般是32字节），这些块作为叶子节点，两两组合并进行哈希，逐层向上直到得到唯一的根，如果叶子数量不足2的幂则需要填充零值以保持树的结构完整。

这种设计的核心意义在于它不仅避免了直接对大规模数据整体哈希带来的高成本，还允许对未变化的子树进行缓存，从而显著提升性能，同时还能生成Merkle证明，使得只需少量数据就能验证某一部分是否属于整体数据结构，这对于轻节点验证和分布式一致性非常关键。

结合SSZ可以看到，序列化与merkleization是紧密配合的两个步骤：前者将复杂结构转为确定性的字节序列，后者则将这些字节结构映射为可验证的哈希树，从而在保证确定性的同时提供高效的完整性证明机制，这也是以太坊共识层能够在大规模状态下仍然保持高效同步与验证的基础。

# 2026-04-15
<!-- DAILY_CHECKIN_2026-04-15_START -->













在EL中，实体是核心概念，它代表了具有一组属性的对象，每个实体都有唯一标识符（ID）和相关的属性。属性可以是基础类型，如整数或字符串，也可以是更复杂的数据类型，如数组或其他对象。这种结构允许开发者定义非常灵活的数据模型，以适应不同的业务需求。

此外，EL支持集合（Collection）这一数据结构，它表示多个实体的集合。通过集合，开发者可以批量操作数据，比如添加、删除或查询集合中的实体。实体之间的关系也是EL的一个重要特性。通过关系，实体之间可以建立引用或关联，这使得EL能够更好地模拟现实世界中对象间的交互和依赖关系。EL的设计注重灵活性，不仅支持基础的数据结构操作，还提供了丰富的表达式和查询机制，使得开发者能够高效地对数据进行筛选、排序和修改。这种设计方式大大简化了数据操作的复杂性，并提高了系统的可扩展性和可维护性。

在 EPF 体系中，EL architecture 本质上是一种面向方法工程的结构化组织方式，它通过将软件过程中的各种元素（如角色、任务、工件等）进行模块化拆分，并以统一的元模型进行描述，从而实现过程知识的可复用与可扩展。这种架构依托类似于统一方法架构（UMA）的思想，把过程内容与具体实现解耦，使得开发团队可以根据实际需求对流程进行裁剪和组合，而不是依赖固定流程。同时，EPF 中的 Wiki 与 Composer 之间是松耦合关系，通过“harvesting”等机制将过程资产从编辑环境同步到发布环境，进一步体现了内容管理与执行环境分离的设计理念。这种架构不仅支持敏捷和迭代开发方式，还强调过程资产的标准化与持续演进，使软件工程方法能够在不同项目和组织中灵活适配与复用。

# 2026-04-09
<!-- DAILY_CHECKIN_2026-04-09_START -->



















以太坊当前的架构是在 The Merge 之后形成的分层系统，其中最核心的分裂是执行层与共识层的分离，执行层负责交易执行、EVM 计算和状态维护，而共识层负责区块生产、验证者机制以及最终性，两者通过 Engine API 进行通信共同构成链的主干。

但在这之下仍然存在网络层负责节点发现与数据传播，在这之上又逐渐演化出以 Rollup 为代表的执行外包层，使得 L1 更多承担数据可用性、结算与安全锚的职责。

完整的以太坊更像是一个围绕执行与共识分离而组织起来的多层模块化系统

# 2026-04-08
<!-- DAILY_CHECKIN_2026-04-08_START -->




















以太坊在史前时代里，花了大量篇幅介绍互联网的发展、开源软件运动和密码朋克运动，来证明以太坊的出现不是因为vitalik的灵机一动，而是最近这几十年技术文化的大爆发。

互联网让我们实现了信息的自由流动，但产生的价值归于平台。

开源软件运动让我们知道，代码不该是剥削的工具，协作不需要经过某个大公司的批准。

而包括中本聪在内的密码朋克们则在长达数十年的密码战争中，走出了一条不需要中心的路。

所以当大家发现比特币的系统上再也跑不了其他应用的时候，以太坊诞生了，它解决了过去“去中心化系统只能做货币”的问题，给了去中心化系统一个可编程的系统，大家可以创建自己想要的任何应用。

以太坊的出现不是什么幸运的偶然，而是这种文化被中心化系统压制太久后，一次必然的技术反击。它是前人所有关于自由、隐私和自治的梦想，终于找到了一个落地的技术路线。

# 以太坊 Core Development 核心开发流程

## 一、整体概念

在 Ethereum 中，核心开发（Core Development）指：

-   对协议本身的修改与升级
    
-   涉及共识层（CL）与执行层（EL）
    
-   影响全网节点行为
    

其本质是一个**去中心化协作流程**，没有“官方决策者”，而是：

> 通过社区共识 + 开发者协调 + 客户端实现 达成升级

* * *

## 二、核心参与角色

### 1\. Core Developers（核心开发者）

-   来自不同团队：
    
    -   客户端团队（CL / EL）
        
    -   研究员（researchers）
        
-   负责：
    
    -   设计协议
        
    -   实现功能
        
    -   讨论技术细节
        

* * *

### 2\. 客户端团队

负责具体实现，例如：

-   共识层客户端
    
-   执行层客户端
    

特点：

-   多实现（multi-client）
    
-   独立开发，遵循同一规范
    

* * *

### 3\. 研究员（Researchers）

-   设计协议改进
    
-   提供理论基础
    
-   常见来源：
    
    -   Ethereum Foundation
        
    -   独立研究者
        

* * *

### 4\. 社区（Community）

包括：

-   开发者
    
-   节点运营者
    
-   用户
    

作用：

-   反馈意见
    
-   形成社会共识（social consensus）
    

* * *

## 三、核心流程总览

以一个协议升级为例，流程如下：

```text
Idea → EIP → Discussion → Specification → Implementation → Testing → Fork Activation
```

* * *

## 四、EIP 流程（核心机制）

### 1\. 什么是 EIP

Ethereum Improvement Proposal 是：

> 提出协议改进的标准文档

* * *

### 2\. EIP 分类

-   Core：协议级变更（最重要）
    
-   Networking：网络层
    
-   Interface：API 标准
    
-   ERC：应用层标准（如代币）
    

* * *

### 3\. 生命周期

（1）Draft（草案）

-   提出想法
    
-   初步设计
    

（2）Review（评审）

-   社区讨论
    
-   技术质疑
    

（3）Last Call

-   最终意见征集
    

（4）Final / Accepted

-   被接受
    

（5）Implemented

-   客户端实现完成
    

* * *

## 五、核心开发会议（AllCoreDevs）

### 1\. ACD 会议

全称：

-   All Core Devs Call
    

作用：

-   协调升级内容

# EIP-7702 （账户抽象过渡方案）

## 一、背景

在 Ethereum 中，账户分为两类：

-   EOA（Externally Owned Account，外部账户）
    
-   Contract Account（合约账户）
    

长期问题：

-   EOA 功能过于简单（只能发交易）
    
-   无法实现：
    
    -   批量交易
        
    -   Gas 代付
        
    -   自定义验证逻辑
        
    -   社交恢复
        

为解决这些问题，引入账户抽象（Account Abstraction, AA）。

已有方案：

-   EIP-4337（不修改协议层）
    

但 4337 存在：

-   架构复杂（bundler / mempool / entry point）
    
-   gas 开销较大
    
-   生态割裂
    

因此提出：

> EIP-7702：**在协议层让 EOA “临时变成合约”**

* * *

## 二、核心思想

EIP-7702 的本质：

> 允许 EOA 在交易执行期间，**附带一段合约代码进行执行**

换句话说：

-   EOA ≠ 固定逻辑
    
-   EOA 可以“挂载代码”
    

类似：

-   “一次性智能账户”
    

* * *

## 三、关键机制

### 1\. 临时代码（Ephemeral Code）

交易中包含：

-   一段 code（类似合约代码）
    

执行时：

-   EOA 被视为“带代码的账户”
    

执行结束：

-   代码不持久化（不会写入 state）
    

* * *

### 2\. 执行流程

大致流程：

1.  用户签名交易
    
2.  交易中携带：
    
    -   code
        
    -   calldata
        
3.  执行时：
    
    -   将 EOA 视为 contract
        
4.  执行完成：
    
    -   丢弃 code
        

* * *

### 3\. 状态变化

注意：

-   **账户代码本身不会改变**
    
-   只影响：
    
    -   当前交易执行逻辑
        

* * *

## 四、和传统模型对比

### 1\. vs EOA

| 特性 | EOA | EIP-7702 |
| --- | --- | --- |
| 执行逻辑 | 固定（签名） | 可自定义 |
| 批处理 | 不支持 | 支持 |
| Gas 代付 | 不支持 | 支持 |
| 可编程性 | 无 | 有 |

* * *

### 2\. vs 合约账户

| 特性 | 合约账户 | EIP-7702 |
| --- | --- | --- |
| 代码存储 | 永久 | 临时 |
| 部署成本 | 高 | 无需部署 |
| 灵活性 | 高 | 高（但短期） |

* * *

### 3\. vs EIP-4337

| 维度 | 4337 | 7702 |
| --- | --- | --- |
| 层级 | 应用层 | 协议层 |
| 复杂度 | 高 | 较低 |
| 兼容性 | 需新基础设施 | 原生支持 |
| gas | 较高 | 更低 |

* * *

## 六、安全性分析

### 1\. 风险点

（1）恶意代码注入

-   用户签名的 code 可能被篡改
    

解决：

-   code 参与签名
    

* * *

（2）重放攻击

-   相同 code 被重复执行
    

解决：

-   nonce 控制
    

* * *

（3）权限扩大

-   临时代码拥有账户权限
    

需要：

-   严格验证逻辑
    

* * *

# 一、Ethereum Protocol Roadmap 总体思想

Ethereum 的路线图不是一个“固定计划”，而是一个**持续演化的研究方向集合**，核心特点：

-   无中心治理（no authority）
    
-   社区共识驱动（rough consensus）
    
-   持续升级，而非一次性完成（Infinite Garden）
    

## 1\. Infinite Garden（无限花园）

核心理念：

-   Ethereum 没有终点
    
-   目标不是“完成”，而是**长期可持续运行**
    
-   最终状态：**Ossification（协议固化）**
    

演化路径：

```
持续升级 → 性能/安全逐步优化 → 达到稳定 → 只做小修剪
```

* * *

## 2\. Roadmap 六大方向（Vitalik 划分）

Ethereum roadmap 被抽象为六个方向（“The Six Urges”）：

| 模块 | 核心目标 |
| --- | --- |

| The Surge | 扩展性（Rollup + Sharding） |
| The Scourge | MEV + 去中心化 + 抗审查 |
| The Verge | 轻节点验证（Verkle + SNARK） |
| The Purge | 状态/历史压缩 |
| The Splurge | 杂项优化（UX / 安全） |

# Merkleization 与 Hash Tree Root （以太坊共识层）

## 一、核心背景

在 Ethereum 共识层中，所有节点必须：

-   对状态（state）达成一致
    
-   高效验证数据
    
-   支持轻节点（light client）
    

为此，引入：

-   SSZ（Simple Serialize）
    
-   Merkleization（默克尔化）
    

其核心目标是：

> 用一个固定长度的哈希（root）表示复杂状态，并支持局部验证

* * *

## 二、基本概念

### 1\. Merkleization

定义：

-   将序列化后的数据构建为 **Merkle Tree**
    
-   最终得到一个唯一的 root hash
    

本质：

-   数据结构 → 哈希树 → 单一摘要
    

* * *

### 2\. Hash Tree Root

定义：

-   对 SSZ 对象进行 Merkleization 后得到的根哈希
    

特点：

-   唯一标识整个对象
    
-   任意数据变动 → root 改变
    

* * *

## 三、为什么需要 Merkleization

### 1\. 高效一致性

节点无需比较完整状态，只需比较：

-   root hash
    

即可判断是否一致

* * *

### 2\. 轻客户端支持

轻节点不存完整状态，只需：

-   root + proof（Merkle proof）
    

即可验证数据

* * *

### 3\. 可扩展性

支持：

-   局部更新
    
-   局部验证
    
-   状态分片（未来）
    

* * *

## 四、Merkleization 过程（核心流程）

### Step 1：Chunking（分块）

-   将数据划分为 **32 bytes（256 bit）**
    
-   每块作为叶子节点
    

* * *

### Step 2：构建 Merkle Tree

-   两两拼接后 hash：
    

```
H(A + B)
H(C + D)
```

-   递归向上：
    

```
Root = H(H(A+B) + H(C+D))
```

* * *

### Step 3：Padding（填充）

两种 padding：

（1）数据 padding

-   不满 32 bytes → 补 0
    

（2）树 padding

-   叶子数量不是 2^n → 补零 chunk
    

目的：

-   构建**满二叉树（full binary tree）**
    

* * *

## 五、Generalized Index（广义索引）

用于：

-   在树中定位节点
    
-   构造 proof
    

### 规则：

-   root = 1
    
-   左子节点 = 2 \* i
    
-   右子节点 = 2 \* i + 1
    

示例：

```
        1
      /   \
     2     3
    / \   / \
   4  5  6  7
```

* * *

## 六、Merkle Proof 与 Multiproof

### 1\. 单点证明（Merkle Proof）

验证某个叶子：

需要：

-   该叶子
    
-   路径上的“兄弟节点”
    

* * *

### 2\. Multiproof

验证多个节点：

-   使用 generalized index
    
-   合并路径，减少数据量
    

核心思想：

> 只提供“必要的哈希节点”

* * *

## 七、Hash Tree Root 计算规则（SSZ）

### 1\. 基本类型（Basic Types）

流程：

-   序列化 → 32 bytes
    
-   直接作为叶子
    
-   Merkleize
    

* * *

### 2\. List / Vector

流程：

1.  序列化
    
2.  chunking
    
3.  padding
    
4.  Merkleize
    
5.  **mix in length（关键）**
    

* * *

## 八、Mixing in Length（重点）

### 问题：

两个列表：

```
[1,2,3]
[1,2,3,0]
```

如果不加长度：

→ root 相同（严重问题）

* * *

### 解决：

将长度作为额外输入：

```
root = H(merkle_root || length)
```

* * *

### 本质：

保证：

-   内容 + 结构 → 唯一性
    

* * *

## 九、Composite Type（容器类型）

例如：

-   Container（类似 struct）
    

规则：

> container 的 root = 各字段 root 的 Merkleization

* * *

### 示例结构：

```
IndexedAttestation {
    attesting_indices
    data
    signature
}
```

计算：

```
root =
  H(
    root(attesting_indices),
    root(data),
    root(signature)
  )
```

* * *

## 十、嵌套结构处理

### 递归规则：

-   container → 展开字段 → 各自 Merkleize
    
-   list → chunk → mix length
    
-   基本类型 → 直接 hash
    

* * *

## 十一、Signature 特殊处理

BLS 签名：

-   固定 96 bytes
    
-   拆为 3 个 chunk
    
-   Merkleize
    

* * *

## 十二、Summaries vs Expansions

### 1\. Summary（摘要）

-   仅包含 root
    
-   用于快速验证
    

例：

-   BeaconBlockHeader
    

* * *

### 2\. Expansion（完整数据）

-   完整结构
    
-   用于执行和验证细节
    

例：

-   BeaconBlock
    

* * *

### 作用：

| 类型 | 用途 |
| --- | --- |
| Summary | 快速验证 |
| Expansion | 完整执行 |

* * *

## 十三、性能优化

### 1\. 子树缓存（Caching）

-   未变化部分 → 不重新 hash
    
-   只更新变动路径
    

* * *

### 2\. 增量更新

复杂度：

-   从 O(n) → O(log n)
    

* * *

## 十四、核心性质总结

Merkleization 提供：

### 1\. 完整性（Integrity）

-   任意修改 → root 改变
    

### 2\. 可验证性（Verifiability）

-   支持轻节点 proof
    

### 3\. 高效性（Efficiency）

-   局部更新
    
-   子树复用
    

* * *

## 十五、一句话总结

> Merkleization = 把复杂状态变成“可验证的哈希树”，  
> Hash Tree Root = 整个状态的唯一指纹

# 以太坊主流共识层（CL）客户端对比笔记

# Beacon Chain （以太坊共识层核心）

## 一、基本概念

Beacon Chain 是以太坊在向 PoS（Proof of Stake）转型过程中引入的核心链，属于共识层（Consensus Layer, CL）。它负责：

-   管理验证者（validators）
    
-   维护区块共识
    
-   提供最终性（finality）
    
-   协调分片（未来扩展）
    

在 Merge（合并）之后，Beacon Chain 成为整个以太坊的“主链”，执行层（Execution Layer, EL）依附其共识运行。

* * *

## 二、时间结构（Time Structure）

Beacon Chain 采用严格的时间分片结构：

### 1\. Slot（槽）

-   每个 slot = 12 秒
    
-   每个 slot 最多产生一个区块
    
-   每个 slot 会指定一个 proposer（提议者）
    

### 2\. Epoch（纪元）

-   1 epoch = 32 slots（约 6.4 分钟）
    
-   是投票、奖励、最终性判断的基本单位
    

关系：

```
1 epoch = 32 slots
1 slot = 12 seconds
```

* * *

## 三、验证者（Validator）

### 1\. 成为验证者

-   需要质押 32 ETH
    
-   进入激活队列（activation queue）
    

### 2\. 验证者职责

每个 epoch 内，验证者会被分配到不同委员会（committee）执行：

-   提议区块（block proposal）
    
-   进行证明投票（attestation）
    

### 3\. 验证者状态

-   Pending（等待激活）
    
-   Active（活跃）
    
-   Exiting（退出中）
    
-   Slashed（被惩罚）
    

* * *

## 四、区块结构（Beacon Block）

Beacon Block 与传统区块不同，主要包含：

-   Slot
    
-   Proposer index
    
-   Parent root
    
-   State root
    
-   Body
    

### Block Body 包含：

-   Attestations（投票）
    
-   Deposits（质押）
    
-   Slashings（惩罚记录）
    
-   Execution payload（来自执行层的区块数据）
    

* * *

## 五、Attestation（证明/投票）

Attestation 是验证者对链状态的投票，是共识的核心。

每个 attestation 包含：

-   当前 head（链头）
    
-   source checkpoint
    
-   target checkpoint
    

可以理解为：

-   我认为当前哪条链是正确的
    
-   我支持哪个 checkpoint
    

* * *

## 六、Fork Choice Rule（分叉选择规则）

Beacon Chain 使用：

### LMD GHOST（Latest Message Driven GHOST）

核心思想：

-   每个验证者的“最新投票”决定权重
    
-   选择权重最大的链作为 head
    

特点：

-   动态选择最长且“被支持最多”的链
    
-   提供活性（liveness）
    

* * *

## 七、Finality（最终性）

由 Casper FFG 提供。

### 关键概念：

Checkpoint

-   每个 epoch 的第一个 slot
    
-   用于共识确认
    

投票机制：

验证者对 checkpoint 投票：

-   source → target
    

### Finality 条件：

当满足以下条件时：

-   超过 2/3 验证者投票支持某个 checkpoint
    
-   且形成连续的投票链
    

则：

-   target checkpoint 被 justified（已证明）
    
-   上一个 checkpoint 被 finalized（最终确定）
    

### Finalized 的意义：

-   不可回滚（除非 >1/3 验证者被惩罚）
    

* * *

## 八、奖励与惩罚机制

### 1\. 奖励（Rewards）

验证者行为正确可获得奖励：

-   正确投票（attestation）
    
-   及时投票
    
-   提议区块
    

### 2\. 惩罚（Penalties）

轻微违规：

-   离线
    
-   延迟投票
    

严重违规（Slashing）：

-   双重签名（double vote）
    
-   包围投票（surround vote）
    

Slashing 结果：

-   扣除质押
    
-   强制退出网络
    

* * *

## 九、状态管理（State）

Beacon Chain 使用 SSZ（Simple Serialize）进行状态编码，并通过 Merkleization 构建哈希树。

核心状态包括：

-   验证者列表
    
-   余额
    
-   历史投票
    
-   随机数（RANDAO）
    

* * *

## 十、随机性（RANDAO）

用于：

-   随机分配 proposer 和 committee
    

机制：

-   每个验证者提供随机数
    
-   聚合生成全局随机性
    

避免：

-   被单个节点操控
    

* * *

## 十一、同步机制（Syncing）

### 1\. 弱主观性（Weak Subjectivity）

-   新节点不能完全信任创世状态
    
-   需要一个“可信 checkpoint”
    

### 2\. 同步方式：

-   Full sync（全同步）
    
-   Checkpoint sync（快速同步）
    

* * *

## 十二、Beacon Chain 的作用总结

Beacon Chain 本质上做三件事：

1.  选谁出块（proposer selection）
    
2.  让大家投票（attestation）
    
3.  决定哪条链是最终的（finality）
    

* * *

## 十三、整体架构总结（Merge之后）

```
Consensus Layer（Beacon Chain）
        ↓
决定区块顺序 + 最终性
        ↓
Execution Layer（EVM）
        ↓
执行交易 + 更新状态
```

* * *

# Consensus Layer (CL) Architecture

## 一、总体概念

以太坊在转向 PoS（Proof-of-Stake）之后，其共识层（Consensus Layer, CL）采用的是一种**组合式共识协议**：

-   **LMD GHOST**：负责链的推进（liveness）
    
-   **Casper FFG**：负责链的最终性（finality）
    

两者组合形成：

> **Gasper = LMD GHOST + Casper FFG**

这是一个典型的工程折中设计：

-   不追求“绝对安全 + 绝对活性”
    
-   而是在现实网络条件下达到**可用性最优解**
    

* * *

## 二、核心问题：为什么需要 Fork Choice

在分布式系统中，不同节点可能看到不同的链状态，原因包括：

-   网络延迟（latency）
    
-   消息乱序（out-of-order）
    
-   网络分区（partition）
    
-   节点作恶（malicious behavior）
    

因此会产生：

> 同时存在多个“合法链分支”（fork）

目标是：

> 所有诚实节点最终收敛到一条**唯一的 canonical chain**

实现这个目标的机制就是：

> **Fork Choice Rule（分叉选择规则）**

* * *

## 三、Block Tree 结构

区块链不是一条链，而是一个树结构：

```
        A
        |
        B
       / \
      C   D
      |   |
      G   E
           |
           F
```

这个结构称为：

> **Block Tree（区块树）**

特点：

-   每个节点维护本地视角的树
    
-   不同节点的树可能暂时不同
    
-   fork choice 负责从中选出“主链”
    

* * *

## 四、Fork Choice Rule 机制

### 1\. 基本原则

任何 fork choice 必须满足：

1.  只选择**合法区块**
    
2.  所有祖先区块必须合法
    
3.  无效区块及其子孙全部忽略
    

* * *

### 2\. 不同共识的 Fork Choice

（1）PoW：Heaviest Chain Rule

用于：

-   比特币
    
-   旧以太坊（PoW）
    

规则：

> 选择累计工作量最大的链

注意：

-   并非严格“最长链”
    
-   是“累计 difficulty 最大”
    

关键点：

-   PoW **没有使用 GHOST**
    
-   白皮书提到但未实现
    

* * *

（2）Casper FFG（PoS）

规则：

> 选择包含“最高 justified checkpoint”的链

并且：

> **永远不能回滚 finalized block**

作用：

-   提供安全性（Safety）
    
-   防止长回滚
    

* * *

（3）LMD GHOST（PoS）

全称：

> Latest Message Driven Greediest Heaviest Observed SubTree

规则核心：

-   从当前节点开始
    
-   每一步选择“子节点中投票权重最大”的分支
    

权重来源：

-   validator 的最新投票（attestation）
    

特点：

-   动态投票驱动
    
-   局部最优 → 全局路径
    

* * *

### 3\. 统一理解

所有 fork choice 本质：

> 给每个 block 一个“分数”，选最大者作为 head

* * *

## 五、Head Block 与 Canonical Chain

Fork choice 的输出是：

> **Head Block（链头）**

它唯一确定：

> 一条从 Genesis 到 Head 的线性链

即：

> Canonical Chain（规范链）

* * *

## 六、Reorg（链重组）

### 1\. 什么是 Reorg

当 fork choice 结果改变时：

> 节点需要切换主链

这个过程称为：

> **Reorganisation（Reorg）**

* * *

### 2\. Reorg 过程

假设当前链：

```
A ← B ← D ← E ← F   （当前 head = F）
```

后来收到：

```
A ← B ← C ← G
```

如果 G 更优：

执行步骤：

1.  回滚：
    
    -   D, E, F 被移除
        
2.  回到：
    
    -   B
        
3.  应用：
    
    -   C, G
        

新链：

```
A ← B ← C ← G
```

* * *

### 3\. Reorg 特点

-   短 reorg（1~2块）是常态
    
-   长 reorg：
    
    -   可能是攻击
        
    -   或客户端 bug
        

* * *

## 七、安全性 vs 活性

### 1\. Safety（安全性）

定义：

> 不会发生错误状态

例如：

-   不会双花
    
-   不会 finalize 冲突链
    

* * *

### 2\. Liveness（活性）

定义：

> 系统持续推进

例如：

-   不停出块
    
-   不会卡死
    

* * *

### 3\. 不可能三角（CAP 定理）

分布式系统无法同时满足：

-   Consistency（安全）
    
-   Availability（可用）
    
-   Partition tolerance（分区容错）
    

* * *

### 4\. 以太坊的选择

以太坊策略：

> **优先保证 Liveness**

在网络分区时：

-   两边都继续出块
    
-   但无法 finalize
    

极端情况：

-   两条链可能各自 finalize
    
-   形成永久分裂
    

* * *

## 八、Gasper 设计思想

### 1\. LMD GHOST

提供：

-   链持续增长（活性）
    

问题：

-   容易 fork
    
-   不安全
    

* * *

### 2\. Casper FFG

提供：

-   Finality（最终性）
    

机制：

-   checkpoint 投票
    
-   2/3 validator 达成共识
    

* * *

### 3\. 组合效果

| 模块 | 功能 |
| --- | --- |
| LMD GHOST | 推进链 |
| Casper FFG | 锁定历史 |

总结：

> LMD GHOST 让链“动起来”，Casper FFG 让链“稳下来”

一、SSZ 与 Merkleization

SSZ（Simple Serialize）是以太坊共识层专用的序列化格式，目标是“可哈希（hashable）+ 可证明（provable）+ 结构简单”。

1）基本思想  
SSZ不是像RLP那样面向通用编码，而是为“状态树 + Merkle证明”设计的。它有两个核心目标：

-   确定性序列化（同一对象唯一编码）
    
-   能直接映射为Merkle树结构
    

2）数据类型  
SSZ只支持有限类型，保证可预测性：

-   基本类型：uintN、bool
    
-   容器类型：vector（固定长度）、list（可变长度）
    
-   复合类型：container（类似struct）
    

关键区别：

-   vector：长度固定，直接映射树
    
-   list：长度可变，需要额外长度混入（length mix-in）
    

3）序列化规则（SSZ编码）

-   固定长度类型：直接拼接（类似结构体内存布局）
    
-   可变长度类型：使用“偏移量（offset）+ 数据区”结构
    
-   所有偏移是相对整个对象起始位置
    

这点类似ABI编码，但更严格，保证一致性。

4）Merkleization（默克尔化）

SSZ的真正核心不在编码，而在“树摘要（hash\_tree\_root）”。

核心思想：  
把一个SSZ对象转换为一个二叉Merkle树，然后取根哈希。

步骤：

-   将数据拆分为32字节“chunk”
    
-   每个chunk作为叶子节点
    
-   不足32字节右填充0
    
-   构建二叉树（不足补零）
    
-   逐层hash（通常用SHA256）
    
-   得到root
    

5）重要规则

（1）chunk化  
所有数据最终都会变成32字节块，这是统一计算基础。

（2）length mix-in（针对list）  
list的root不是纯Merkle root，而是：  
hash(root || length)

这样避免不同长度但相同前缀的冲突。

（3）container处理  
container类似struct，每个字段先算root，然后这些root组成新的Merkle树。

总结一句话：  
SSZ = 结构化数据 + chunk化 + Merkle树 + root哈希

它直接决定了：

-   状态根（state root）
    
-   区块根（block root）
    
-   各类证明（proof）

-   aggregate\_and\_proof
    
-   sync\_committee\_message
    

3）subnet机制

为了扩展性，attestation不会全网广播，而是分配到不同subnet：

-   每个验证者只监听部分子网
    
-   减少带宽压力
    

4）验证规则

节点收到消息后必须验证：

-   签名正确（BLS）
    
-   slot合法
    
-   未重复
    

否则丢弃

5）DoS防护

-   限速（rate limiting）
    
-   验证前先做轻量检查
    
-   peer评分机制
    

总结一句话：  
CL网络 = libp2p + gossip + 子网分片 + 强验证

四、Weak Subjectivity / Syncing

这是PoS中最容易被忽略但非常关键的安全假设。

1）为什么需要弱主观性

在PoW中：

-   最长链 = 真链（客观）
    

在PoS中：

-   攻击者可以构造“历史分叉”
    
-   新节点无法单靠规则判断哪条链是真
    

因此引入：  
弱主观性（Weak Subjectivity）

2）核心思想

节点需要一个“可信的最近检查点（checkpoint）”，否则可能被欺骗。

这个checkpoint可以来自：

-   官方发布
    
-   可信朋友
    
-   多数客户端一致结果
    

3）Weak Subjectivity Period

定义：  
在多长时间内，不需要更新checkpoint仍然安全。

取决于：

-   验证者退出速度
    
-   slash机制
    
-   活跃验证者数量
    

超过这个时间：  
必须重新获取可信状态

4）同步方式（Syncing）

（1）从genesis同步  
最安全，但极慢

（2）checkpoint sync（常用）

-   从一个可信state root开始
    
-   下载后续区块
    
-   快速进入最新状态
    

（3）snap sync（部分实现）  
类似EL的快速同步

5）攻击模型

如果没有弱主观性：  
攻击者可以：

-   构造旧链
    
-   修改验证者集合
    
-   欺骗新节点
    

这就是“long-range attack”。

6）防御机制

-   弱主观性checkpoint
    
-   slashing（经济惩罚）
    
-   finality（不可回滚）

# **共识层概述**

共识协议旨在解决的主要挑战是再不可靠的基础建设之上构建可靠的分布式系统。

以太坊的共识层目标是确保全球独立节点保持合理同步。每个节点都包存一个包含每个账户状态的分类账簿，这些分类账簿必须完全匹配，不能有差异。节点必须同意并尽快达成一致。 ——可靠的分布式系统

这些节点通常使用消费级硬件，并通过互联网连接进行通信，而这些连接可能速度较慢，会丢包，或意外中断。节点运营者肯能会错误配置他们的软件，或者未能及时更新软件。此外，还可能有许多出于私利而运行恶意节点或篡改通信的不良行为者， ——不可靠的基础设施

**拜占庭容错(BFT)**

分布式系统的一种属性，即使某些组件发生故障或恶意行为，也允许他们正常运行。

BFT在去中心化网络中至关重要，因为节点之间的信任是无法假设的，BFT的系统可以容忍拜占庭错误。对于一个具有拜占庭容错能力的系统来说，尽管存在这些错误，它仍然必须达成共识。

（拜占庭错误：在分布式系统中，某些节点可能出现任意行为的错误，比如发生错误信息，对不同节点发送矛盾的信息，甚至表现得像恶意节点一样。这类错误就叫 “拜占庭错误” ）

**共识层（CL）简洁**

共识是一种用不可靠的组件构建可靠的分布式系统的方法。基于区块链的分布式系统旨在就单一交易历史达成一致。

PoW 和 PoS 不是共识协议，而是启用共识协议，在以太坊中，节点和验证者是共识系统的参与者。Slot (时隙，每个时隙12s）和 epoch (每个纪元有32个时隙) 调节共识时间，区块和证明是共识运转的核心单位。

共识层是确保网络安全、可靠和高效的基础组件。因为PoW存在高能耗和有限的可拓展性（但也可以保持去中心化和安全性），以太坊已转向PoS，更具可持续性和可拓展性的共识机制。

用户将交易发送到节点网络，共识协议确保所有诚信节点最终就单一、一致的交易历史达成一致。这意味着他们就交易顺序及其结果达成一致。

以太坊的共识协议实际是由 `LMD GHOST`和`Casper FFG`两个协议结合在一起（这个组合被成为 Gasper）

**PoW 和 PoS (工作量证明和权益证明)**

PoW 和 PoS 本身都不是共识协议，它们本身只是启用共识协议的机制

PoW 和 PoS 主要用于防御女巫攻击(Sybil attack)

PoW : 靠算力和电力成本抵抗，想要获得更大影响力，就得投入更多矿机和电费

PoS ：靠质押资金抵抗，想要获得更大影响力，就得锁定更多ETH

`fork choice rule`：分叉选择规则，就是当网络里同时出现多条候选链时，节点该按照什么标准选出“当前正确链”

区块链网络里可能会短暂出现分叉，这时系统必须有规则决定哪条是主链，这套规则就是`fork choice rule`

PoW 和 PoS 不会直接告诉我们选哪条链，它们提供的是“每条链的权重怎么计算”

PoW ： 链的权重看累计工作量，也就是哪条链背后消耗的总算力最多，哪条链通常更重，以前被概括为“最长链”，更准确的是“累计难度最高的链”

PoS ：链的权重看有多少质押权益支持它，不是看谁块更多，而是看有多少验证者，多少质押余额对这条链投票或支持。

**区块链**

区块链的基本元素是区块。一个区块包含一组由领导者（区块提议者）组装的交易。块的内容可能因协议而定。

-   一个区块的payload在以太坊执行链上是用户交易的列表
    
-   在合并前的PoS信标链上，区块的有效payload主要是验证者的一组证明
    
-   合并后的信标链区块还包含执行 payload（用户交易）
    
-   在EIP-4844之前，区块现在还包含对不透明数据blob以及用户交易的承诺
    

除了创世区块之外，每个块都建立在父块之上并指向父块，形成区块链

随着节点向其末端添加新块，该链不断增长。这是通过选出一个"leader"来完成的，也就是负责拓展区块链的那个节点。在PoW中，leader 是第一个为其区块解出PoW难题的矿工；在PoS中，leader 是从活跃验证者集合中伪随机选出的。

每一轮里，系统会临时选出一个 leader，也就是 block proposer，负责构造新区块、决定区块里放哪些交易、交易顺序是什么，然后把这个区块广播出去。 但提议者不是想怎么写都行，区块必须符合协议规则，否则其他节点不会接受。

为什么要把交易打包成区块，而不是一笔一笔单独共识？因为单笔交易逐个达成共识，通信和验证成本太高。 所以区块本质上就是“交易批处理”。

在以太坊里：

-   执行层区块大小主要受 gas limit 限制
    
-   信标链区块大小主要受协议常量限制
    

**Merge**

2022 年 9 月 15 日，以太坊完成 The Merge，从工作量证明切换到权益证明。 切换后，不再由矿工靠算力竞争出块，而是由验证者靠质押 ETH 参与共识。

这里一个很重要的点是：

Merge 不是按区块高度触发的，而是按 TTD（Terminal Total Difficulty，终端总难度）触发的。

意思是，只有当 PoW 链累计难度达到预设阈值时，系统才真正切换到 PoS。 这样做比“到第几号区块切换”更安全，因为它能减少恶意分叉或节奏偏差带来的风险。

**Beacon Chain 是 PoS 的核心调度中心**

切到 PoS 后，真正负责协调共识的是 Beacon Chain。 你可以把它理解成以太坊 PoS 的“指挥系统”，它负责：

-   管理验证者
    
-   决定谁在哪个时隙出块
    
-   决定谁在哪个委员会里投票
    
-   收集投票
    
-   推进检查点的 justified 和 finalized
    

验证者要至少质押 32 ETH 才能激活。 他们的主要工作有四个：

-   质押 ETH
    
-   被选中时提议区块
    
-   对别人提议的区块做 attestation
    
-   按协议参与投票，帮助区块链最终确定
    

**Slot 和 Epoch 是 PoS 的时间单位**

-   1 slot = 12 秒
    
-   1 epoch = 32 个 slot = 384 秒 = 6.4 分钟
    

每个 slot 都是一次“可能出块的机会”。 通常每个 slot 会指定一个 proposer，但这个 slot 也可能空掉，没有区块产生。

所以：

-   slot 像“时间格子”
    
-   block 是“这个格子里有没有真的放进去一个块”
    

**Validators、Attestations、Committees**

validator 就是验证者，是 PoS 的参与者。 大多数时候，他们不是出块的人，而是在投票。

attestation 就是验证者发出的见证投票。 它不是随口说“我支持这个块”，而是带权重的正式投票，权重和验证者的有效质押有关。

committee 是委员会。 每个 slot 会有一组验证者被分到同一个 committee，对该 slot 的链头或区块做见证。 这样做有两个目的：

-   分摊验证工作
    
-   便于把很多签名聚合成一个聚合签名，提高效率
    

委员会通常要求至少 128 个验证者，这样更安全。 如果委员会太小，被攻击者控制多数的概率就会上升。

**RANDAO**

以太坊不能让大家提前精确知道“谁会出块、谁会投票”，不然容易被攻击、贿赂或定向干扰。 所以它用 RANDAO 这种伪随机机制来：

-   给每个 slot 选 proposer
    
-   把验证者打散分配到不同 committee
    

也就是说，PoS 不是人工指定谁出块，而是协议用随机过程来选。

**LMD GHOST 和 Casper FFG**

这是以太坊 PoS 最核心的地方。

验证者的投票其实有两层：

-   LMD GHOST：用来选择当前链头，也就是“现在哪条分叉应该被视为主链头部”
    
-   Casper FFG：用来确认检查点，推进 justified 和 finalized
    

可以简单记：

-   LMD GHOST 管“现在跟哪条链走”
    
-   Casper FFG 管“哪些历史已经不可逆”
    

所以 attestation 里通常同时包含这两类信息。

**Checkpoint、Justified、Finalized**

每个 epoch 都会形成一个 checkpoint。 一般来说，它对应那个 epoch 的边界块。

然后验证者会对 checkpoint 投票。 如果一个 checkpoint 获得了不少于总有效质押 2/3 的支持，它就会变成：

-   justified：已获得足够多数背书
    

如果后续 epoch 的 checkpoint 也再次得到足够支持，那么前一个 justified checkpoint 就会变成：

-   finalized：最终确定，正常情况下不能再回滚
    

所以最终性不是“块一出来立刻确定”，而是经过一轮轮投票逐渐锁定。

正常情况下，最终性大约要两个 epoch 左右，也就是十几分钟量级。

**交易确认和最终性**

交易进块之后，并不代表它已经绝对安全。 安全性是分层增长的：

-   先被提议进某个块
    
-   再被 attestation 支持
    
-   再所在 checkpoint 被 justified
    
-   再最终变成 finalized
    

所以很多应用并不是非得等 finality 才继续，而是会根据风险容忍度，选择更早的确认阈值。

**EIP-4844 和 blobs**

这部分讲的是以太坊扩容的新机制。

EIP-4844 引入了 blob，可以理解成：

给以太坊额外加了一层临时数据可用性空间

这些 blob 不是普通合约永久存储，而是短期保存的大块数据，主要给 Layer 2 用。 这样 L2 可以把数据更便宜地挂到以太坊上，从而降低费用、提高吞吐量。

这里还有两个关键词：

-   KZG commitments：用来证明 blob 数据正确
    
-   Trusted Setup：KZG 方案需要的可信初始化
    

对节点运营者来说，blob 最大的现实影响是：

-   临时存储需求明显增加
    
-   客户端会在保留期后剪枝旧 blob
    

**奖励、惩罚、Slashings 是为了约束验证者行为**

PoS 不是“质押了就躺赚”，而是有完整激励约束。

验证者会因为这些事情拿奖励：

-   及时做正确 attestation
    
-   成功提议区块
    
-   包含他人的惩罚证据
    

也会因为这些事情受罚：

-   该投票时没投
    
-   长时间离线
    
-   投票和多数不一致
    
-   严重违规，也就是 slashing
    

可被 slash 的典型行为有四类：

-   一个 slot 提两个块
    
-   同一 slot 对两个不同链头投票
    
-   FFG 双重投票
    
-   FFG 环绕投票
    

slashing 是重罚，不只是少赚，而是真的会损失质押金并被踢出活跃集。

另一个特殊机制叫 inactivity leak。 如果网络长期无法 finality，协议会持续惩罚离线或不参与的验证者，直到活跃验证者重新形成足够大的 2/3 多数，恢复最终性。

**验证者生命周期**

验证者不是永久在线就完了，它有生命周期：

-   存入至少 32 ETH，等待激活
    
-   激活后开始参与提议和投票
    
-   可以主动退出
    
-   退出后还有延迟期
    
-   如果被 slashed，提现等待时间会更长
    

这样设计是为了避免验证者集合变化太快，也为了给惩罚机制留出执行窗口。

# 一、Transaction（交易机制）

### 1\. 交易本质

交易是由外部账户（EOA）发起的、经过密码学签名的指令，通过 JSON-RPC 广播到整个网络，用于触发状态变化。

* * *

### 2\. 交易核心字段

（1）nonce（Tₙ）

表示发送者已发送的交易数量，用途：

-   防止重放攻击  
    相同 nonce 的交易不能重复执行
    
-   决定合约地址  
    合约地址 = hash(发送者地址 + nonce)
    
-   替换交易  
    使用相同 nonce + 更高 gasPrice 可替换 pending 交易
    

* * *

（2）gasPrice（Tₚ）

单位 gas 的价格（单位：Wei）

-   1 ETH = 10¹⁸ Wei
    
-   gasPrice 越高，越容易被打包
    

* * *

（3）gasLimit（Tg）

最大可消耗的 gas：

-   执行过程中耗尽则交易失败（out of gas）
    
-   未用完的 gas 会退回
    

* * *

（4）to（Tt）

决定交易类型：

| to 值 | 交易类型 | 含义 |
| --- | --- | --- |
| 空 | 合约创建 | 创建新合约 |
| 外部账户 | 转账 | 普通 ETH 转账 |
| 合约地址 | 合约调用 | 执行代码 |

* * *

（5）value（Tv）

转账金额（单位 Wei）

-   合约创建时：作为合约初始余额
    

* * *

（6）data / init（Td / Ti）

-   普通调用：函数调用数据
    
-   创建合约：init bytecode
    

* * *

（7）签名（v, r, s）

-   使用 ECDSA 签名
    
-   用于恢复发送者地址
    

* * *

## 二、合约创建机制

### 1\. 两种代码

（1）init code

-   只执行一次
    
-   返回 runtime code
    

（2）runtime code

-   存储在链上
    
-   每次调用执行
    

* * *

### 2\. 执行流程

1.  执行 init code
    
2.  使用 RETURN 返回 runtime code
    
3.  runtime code 写入账户
    

* * *

### 3\. 示例逻辑

runtime code：

```
PUSH1 06
PUSH1 07
MUL
PUSH1 00
SSTORE
```

作用：

将 6 × 7 = 42 存入 storage slot 0

* * *

### 4\. init code 结构

分两步：

1.  CODECOPY（复制 runtime code 到内存）
    
2.  RETURN（返回 runtime code）
    

* * *

### 5\. 合约调用执行

-   发送交易到合约地址
    
-   执行 runtime code
    
-   修改 storage
    

结果：

```
slot[0] = 42 (0x2a)
```

## 一、总体架构

Execution Layer Client（执行层客户端，如 Geth）可以抽象为分层结构：

```
Consensus Layer（CL）
        ↓ Engine API
Execution Engine（执行引擎）
        ↓
EVM（执行环境）
        ↓
State（状态数据库）
        ↓
Transaction Pool（交易池）
        ↓
DevP2P（网络层）
```

同时包含两个对外接口：

-   JSON-RPC：面向用户（钱包、DApp）
    
-   Engine API：面向共识层（CL）
    

* * *

## 二、核心本质

执行层的核心功能可以归结为：

**状态转移函数（State Transition Function, STF）**

```
(state, transactions) → new_state
```

即：给定当前状态和交易集合，计算新的状态。

* * *

## 三、核心模块

### 1\. EVM（Ethereum Virtual Machine）

本质：虚拟化执行引擎（类似 CPU）

作用：

-   执行智能合约字节码
    
-   保证所有节点执行结果一致（确定性）
    

特点：

-   与底层硬件无关
    
-   输入相同，输出必然相同
    

* * *

### 2\. State（全局状态）

Ethereum 采用账户模型（Account-based），与 Bitcoin 的 UTXO 模型不同。

State 包含：

-   账户地址
    
-   余额（balance）
    
-   nonce
    
-   合约代码（code）
    
-   存储（storage）
    
-   Merkle-Patricia Trie 结构
    

核心表示：

```
state_root = hash(global_state)
```

* * *

### 3\. Transactions（交易）

交易是状态变化的唯一驱动：

```
state → state'
```

执行流程：

```
Transaction → EVM → 修改 State
```

* * *

### 4\. Transaction Pool（交易池）

作用：

-   存储待打包交易（mempool）
    
-   按优先级排序
    

类型：

Legacy Pool

-   按 gas price 排序
    
-   使用优先队列（heap）
    

Blob Pool（EIP-4844）

-   处理 blob 类型交易
    
-   使用特殊的淘汰机制（基于对数策略）
    

* * *

### 5\. DevP2P（网络层）

作用：

-   节点之间通信
    
-   广播交易与区块
    
-   节点发现
    

特点：

-   点对点网络（P2P）
    
-   Gossip 传播机制
    
-   每个节点都会验证数据再传播
    

* * *

### 6\. JSON-RPC API（用户接口）

用于与外部交互：

常见方法：

```
eth_sendTransaction
eth_call
eth_getBalance
```

作用：

-   查询链上状态
    
-   发送交易
    

* * *

### 7\. Engine API（CL 与 EL 的桥梁）

Merge 后新增的核心接口，仅供 CL 调用。

（1）engine\_newPayload

作用：

-   验证并执行区块
    

流程：

1.  CL 接收到 Beacon Block
    
2.  提取 execution payload
    
3.  调用 EL
    

EL 执行：

-   检查 parent hash
    
-   执行交易
    
-   更新状态
    

返回状态：

-   VALID
    
-   INVALID
    
-   SYNCING
    
-   ACCEPTED
    

* * *

（2）engine\_forkchoiceUpdated

作用：

-   更新链头（head）
    
-   触发新区块构建
    

输入：

-   head block
    
-   safe block
    
-   finalized block
    
-   可选 payload attributes
    

返回：

-   状态
    
-   payloadId（用于获取构建中的区块）

作用：

（1）防止重放攻击

-   每个交易 nonce 唯一
    
-   重复交易会被拒绝
    

（2）决定合约地址

合约地址计算：

```
contract_address = keccak256(sender, nonce)
```

（3）替换交易（Replace Tx）

-   相同 nonce
    
-   更高 gas fee
    
-   可覆盖旧交易
    

* * *

### 2\. gasPrice（Tₚ）

定义：

```
gasPrice = 每单位 gas 的价格（单位：Wei）
```

关系：

```
1 ETH = 10^18 Wei
```

作用：

-   决定交易优先级
    
-   越高越容易被打包
    

* * *

### 3\. gasLimit（Tg）

定义：

```
gasLimit = 允许消耗的最大 gas
```

特点：

-   执行耗尽则失败（out-of-gas）
    
-   未用完会退回
    

* * *

### 4\. to（Tt）

定义：

```
接收地址（20 bytes）
```

决定交易类型：

| to 值 | 类型 | 说明 |
| --- | --- | --- |
| 空 | 合约创建 | 创建新合约 |
| EOA 地址 | 转账 | ETH 转移 |
| 合约地址 | 调用 | 执行合约 |

* * *

### 5\. value（Tv）

定义：

```
转账金额（单位：Wei）
```

特点：

-   合约创建时 → 初始余额
    
-   调用合约时 → msg.value
    

* * *

### 6\. data / init（Td）

定义：

```
输入数据（byte array）
```

区别：

-   普通调用 → calldata
    
-   合约创建 → init code
    

* * *

### 7\. Signature（v, r, s）

定义：

-   ECDSA 签名
    
-   用于恢复 sender

# Execution Layer 数据结构

* * *

## 一、总体概念

Execution Layer 客户端需要存储两类数据：

1.  当前状态（State）
    
2.  历史区块数据（Blocks / Transactions / Receipts）
    

核心数据结构：

```text
Merkle Patricia Trie（MPT）
```

* * *

## 二、基础组件

* * *

### 1\. RLP（Recursive Length Prefix）

作用：

-   序列化数据（编码）
    
-   用于：
    
    -   交易
        
    -   区块
        
    -   Trie 节点
        

特点：

-   简单
    
-   无 schema
    
-   支持嵌套结构
    

* * *

### 2\. Merkle Tree

定义：

一种基于哈希的二叉树结构，用于数据完整性验证。

* * *

结构特点

-   叶子节点：数据
    
-   非叶子节点：子节点哈希
    
-   根节点：Merkle Root
    

* * *

构建方式

```text
自底向上（bottom-up）
```

* * *

关键性质

1.  防篡改（tamper-proof）
    

-   任意数据变化 → root hash 改变
    

2.  Avalanche Effect
    

-   微小变化 → hash 完全不同
    

* * *

特殊规则

-   若叶子节点数量为奇数：
    
    -   最后一个节点复制一次
        

* * *

作用

-   验证交易是否属于区块（Merkle Proof）
    

* * *

### 3\. Patricia Trie（压缩前缀树）

定义：

一种优化的 Trie（Radix Tree），用于高效存储 key-value。

* * *

特点

1.  前缀压缩
    

-   公共前缀合并
    

2.  边存字符串（不是单字符）
    
3.  单子节点合并（节省空间）
    

* * *

优势

-   更节省存储
    
-   查询效率高
    

* * *

## 三、Merkle Patricia Trie（MPT）

Ethereum 核心数据结构：

```text
Merkle Tree + Patricia Trie
```

* * *

### 1\. 结合优势

| 来源 | 提供能力 |
| --- | --- |
| Merkle Tree | 可验证性（hash） |
| Patricia Trie | 高效存储 |

* * *

### 2\. 节点类型

* * *

（1）Branch Node

结构：

```text
17 个元素：
- 16 个子节点（0~f）
- 1 个 value
```

* * *

（2）Extension Node

作用：

-   路径压缩
    
-   避免单链分支
    

* * *

（3）Leaf Node

存储：

```text
key-value
```

* * *

### 3\. Key 表示

使用：

```text
Nibble（4-bit）
```

特点：

-   每个 hex digit = 1 nibble
    
-   每层最多 16 分支
    

* * *

### 4\. Hash 机制

每个节点：

```text
hash = keccak256(RLP(node))
```

* * *

### 5\. 核心性质

-   任意节点变化 → root 改变
    
-   root 是整个数据结构的承诺（commitment）
    

* * *

## 四、Ethereum 中的四种 Trie

* * *

### 1\. Transaction Trie

* * *

作用

存储：

```text
某个区块的所有交易
```

* * *

结构

```text
key = RLP(tx_index)
value = RLP(transaction)
```

* * *

特点

-   每个 block 一个
    
-   不可变（immutable）
    

* * *

* * *

### 2\. Receipt Trie

* * *

作用

存储：

```text
交易执行结果（receipt）
```

* * *

结构

```text
key = transaction index
value = receipt
```

* * *

功能

-   验证交易执行结果
    
-   支持日志查询
    

* * *

优势

-   不需要执行交易即可验证结果
    
-   加速同步（snap sync）
    

* * *

特殊用途

-   light client 使用 Merkle proof 验证
    

* * *

* * *

### 3\. World State Trie（核心）

* * *

定义

```text
address → account
```

* * *

key

```text
keccak256(address)
```

* * *

value（账户结构）

```text
[nonce, balance, storageRoot, codeHash]
```

* * *

特点

-   全局唯一状态
    
-   每个 block 更新一次
    
-   root 存在 block header
    

* * *

核心作用

```text
state_root = 系统状态承诺
```

* * *

节点分类

| 类型 | 描述 |
| --- | --- |
| EOA | 外部账户 |
| Contract | 合约账户 |

* * *

存储策略

-   Full Node：当前状态 + 部分历史
    
-   Archive Node：所有历史状态
    

* * *

* * *

### 4\. Storage Trie

* * *

定义

```text
storage_slot → value
```

* * *

特点

-   每个合约一个 Storage Trie
    
-   通过 storageRoot 引用
    

* * *

key

```text
keccak256(slot)
```

* * *

value

```text
RLP(value)
```

* * *

操作来源

-   SSTORE（写）
    
-   SLOAD（读）
    

* * *

特点

-   EOA 没有 storage trie
    
-   完全隔离（每个合约独立）
    

* * *

一、执行层的核心本质

执行层（Execution Layer）的唯一目标：

执行状态转换函数 STF

σₜ₊₁ ≡ Π(σₜ, B)

含义：

输入：当前状态 σₜ + 区块 B  
输出：新状态 σₜ₊₁

本质：

区块链 = 状态机  
区块 = 状态输入  
EL = 执行引擎

——————————————————————

二、状态与状态根

状态 σ：

包含所有账户（balance / nonce / storage / code）

但：

区块链不存状态，只存状态的“承诺”

过程：

1）过滤空账户  
Ls(σ) = 非空账户集合

2）构建 Trie

3）得到 root hash

stateRoot = TRIE(Ls(σ))

关键理解：

stateRoot 是“状态的哈希摘要”，不是状态本身

——————————————————————

三、STF 执行流程

执行一个区块的完整流程：

1）读取父区块  
2）验证 Header  
3）验证 blob gas  
4）检查 ommers 为空  
5）执行区块（apply\_body）  
6）生成新状态  
7）校验 stateRoot 等字段  
8）加入链  
9）失败则 Invalid Block

核心原则：

先验证 → 再执行（节省算力）

2）时间规则

3）链结构

-   parentHash 正确
    
-   block number = parent + 1
    

4）PoS 特征

-   difficulty = 0
    
-   nonce = 0
    

5）EIP-1559

-   baseFee 必须正确
    

6）EIP-4844

-   blobGasUsed 合法
    
-   excessBlobGas 正确

f = min(maxPriorityFee, maxFee - baseFee)

给 validator 的小费

十、Gas 费用结算

预扣：

type0/1：gasLimit × gasPrice  
type2：gasLimit × maxFee

执行后：

实际支付：

gasUsed × p

退款：

reserved − actual

十一、费用流向

关键变量：

τ = gasLimit / 2  
ρ = 2  
ξ = 8

规则：

gasUsed > τ → baseFee 上升  
gasUsed < τ → baseFee 下降

最大变化：

本质：

负反馈控制系统

目标：

稳定 gas 使用率

——————————————————————

十三、长期行为

持续高需求：

-   baseFee 指数增长
    
-   gasLimit 缓慢增长
    

最终：

达到新平衡

reserved\_blob = totalBlobGas × maxFeePerBlobGas

blobFee = totalBlobGas × blobGasPrice

reserved − blobFee

# **Account Model vs UTXO Model**

账户模型和UTXO模型

**UTXO Model**

状态表示：`State = Set of UTXOs`

-   无状态验证
    
-   高隐私性
    
-   低可编程性
    

每一笔交易都是独立的，比喻是实物现金，钱包里的不是一个“余额数字”而是一把面额各异的硬币。每次交易，必须把整枚硬币拿出来，花掉，找零，剩下的继续放回钱包

一笔Bitcoin交易的完整生命周期

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741052661-image.png)

UTXO只能被整体花费。Alice有两枚UTXO(0.8 + 0.5 BTC)，她想给Bob转1.0BTC，就必须把两枚都作为input塞进交易，然后显式创建output给自己。

锁定脚本才是真正的“所有权”，UTXO上记录的不是Alice的余额，而是一段脚本`OP_DUP OP_HASH160 <Alice的pubkey哈希> OP_EQUALVERIFY OP_CHECKSIG`。交易时，Alice提供scriptSig(签名+公钥)，脚本执行通过才算解锁成功。节点验证时完全无状态——只需要拿到这一笔交易和其引用的UTXO，不需要知道Alice的历史

守恒是强制的，input总额>=output总额，Bitcoin协议不存在“凭空增加余额”的操作，每个聪的来源都可追溯的某个创币交易

**Account Model**

状态表示： `State = Mapping(address → account)`

账户结构

```
 account = {
   nonce,
   balance,
   storageRoot,
   codeHash
 }
```

Ethereum的做法更像银行系统，有一张全居的大表，每个地址对应一行记录。转账就是直接修改两行数字

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741074443-image.png)

_优势_

-   状态转移更直接 `balance[A] += x` `balance[B] -= x`
    
-   支持全局状态访问和跨合约交互
    
-   更适合DeFi 、AMM 、DEX
    

_缺点_

-   重放保护强依赖于nonce
    
-   状态增长无法pruning状态膨胀
    

Account模型最关键的地方在于全局可读写的storage。当合约A调用合约B，B可以直接读写A的storage（A授权的话），这让DeFi的可组合性成为可能：一笔交易里可以闪电贷、换币贷、质押、还款，全在一个原子操作内完成

但是所有操作都必须按顺序执行，因为前一步的storage变更会影响后一步的读取，这是Ethereum无法像Bitcion那样轻松并行化的原因

**验证模型与状态膨胀的根本区别**

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741096721-image.png)

UTXO验证“无状态”：因为一笔比特币交易里包含了它自己的完整证明：每个input都指向某个具体的`txid:vout`，节点只需要确认那个UTXO存在于当前的UTXO Set，且签名合法，就可以接受这笔交易。不需要知道Alice以前发过多少个交易、当前账户里还有什么。这是Stateless Client和SPV验证的理论依据

Account模型的只增不减：Ethereum的世界的一棵默克尔树（MPT） ，stateRoot是这棵树的哈希，写入区块头。要重新计算一个正确的stateRoot，必须保留完整的树。某个合约弃用了、某个地址再没有活动——对应的trie叶节点依然必须保留，因为历史区块的stateRoot依然引用它。可以通过`Selfdestruct`清空合约存储（已被EIP-6780）限制，但是这在实践中几乎不可发生。Verkle Tree的引入，将把witness体积从几十kB压缩到几kB。

# **序列化：RLP vs SSZ**

区块链的一切最终都要哈希，哈希结果要写进区块头，验证时必须能重现完全相同的字节序列。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741112259-image.png)

为什么Ethereum要从RLP迁移到SSZ

1.  请客户端需要Merkle proof证明状态的一部分（如账户余额），RLP做不到
    
2.  信标链状态字段多达数十个，每个slot更新若干字段，O(N)代价不可接受
    
3.  SSZ的typed schema可以生成代码绑定，减少客户端实现错误
    

RLP的递归前缀编码很优雅，但是无法完全支持Merkle proof——因为整个结构被压成以一个线性字节流，无法从中抽出一个子字段并提供证明

SSZ的核心创新是把序列化和Merkleization分离设计但协同操作：序列化的结果是固定化布局的字节流，Merkleization把每个字段哈希成叶子节点再建树。更新一个字段时，只需要重哈希从该叶节点到根的路径，路径长度是 O(log N)，而不是重新序列化整棵BeasonState

# **共识与最终性**

Ethereum PoS 的共识是两套机制叠加的产物，理解它们要分开看再看合并。

**Casper FFG 解决的问题是最终性（finality）**——即确认某个 checkpoint 永不回滚。**LMD-GHOST 解决的问题是活性（liveness）**——即在有分叉时能持续选出一条链继续出块。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741137988-image.png)

LMD：每个验证者只算其最新一条attestation，防止旧票双重影响

GHOST：每步选票数最多的子树，而非最长链（抗分叉攻击）

`Gasper = Casper FFG + LMD-GHOST`

两者单独都不够，合在一起才是完整的 Gasper

LMD-GHOST 负责活性：网络分叉时继续选出新块，链不停止

Casper FFG负责安全：每 ~12.8 分钟 finalize 一个 checkpoint，给已确认的历史提供经济担保

Casper FFG：检查点投票 + 两阶段确认

两阶段提交：CP 先获 ≥2/3 验证者投票 → Justified；下一个 CP 也被 Justified 后，前者才变为 Finalized安全保证：回滚 Finalized 块 → 需 ≥1/3 stake 被 slash（数十亿美元），经济上不可行

# **网络层：DHT + GossipSub**

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741152944-image.png)

discv5 和 GossipSub 解决的是完全不同层次的问题

discv5 ——节点刚启动时，它不知道网络里有谁，通过 Kademlia 路由表（按节点 ID XOR 距离排列）找到 ENR 记录（包含 IP、UDP 端口、公钥），然后建立 TCP 连接。它本身不传任何区块数据。

GossipSub 是在已建立的 TCP 连接上运行的 pub/sub 协议。每个节点订阅若干 topic（`beacon_block`, `beacon_attestation_0` 到 `beacon_attestation_63` 等），每个 topic 维护一个 D=8 大小的 mesh（全量消息 peer）和若干 gossip peer（只收元数据摘要）。收到新区块时，广播给所有 mesh peer，他们再广播给自己的 mesh peer，一般 2-3 跳就能覆盖大部分网络。

# EL 和 CL

# 一、基本概念

在计算机网络中，EL 和 CL 通常指网络结构中的两个重要层次：

-   EL：Edge Layer（边缘层）
    
-   CL：Core Layer（核心层）
    

这两个概念主要用于描述网络的结构划分，尤其是在互联网或大型局域网架构中。

从整体上看，一个网络可以理解为由“边缘部分”和“核心部分”组成：

-   边缘层（EL）：靠近用户，负责接入网络
    
-   核心层（CL）：位于网络中心，负责数据的高效传输
    

* * *

## 二、边缘层（EL）

### 1\. 定义

边缘层是网络中最靠近用户设备的一层，是用户与网络之间的接口部分。

也可以理解为：用户“接触网络”的地方。

* * *

### 2\. 包含的设备

边缘层通常包括以下设备：

-   个人电脑
    
-   手机
    
-   平板设备
    
-   家用路由器
    
-   接入交换机
    
-   物联网设备（如智能家居设备）
    

* * *

### 3\. 主要功能

边缘层的核心作用是“接入”和“产生数据”，具体包括：

1.  连接用户设备到网络
    
2.  产生数据（例如浏览网页、发送请求）
    
3.  接收来自网络的数据（例如网页内容、视频流）
    

* * *

### 4\. 工作过程举例

当你打开浏览器访问一个网站时：

-   你的电脑发出请求
    
-   请求从你的设备进入网络
    

在这个过程中，你的电脑就属于边缘层设备。

* * *

### 5\. 特点

边缘层具有以下特点：

-   靠近用户
    
-   数量非常多
    
-   设备类型多样
    
-   功能相对简单（以接入为主）
    

* * *

## 三、核心层（CL）

### 1\. 定义

核心层是网络的中心部分，主要负责在不同网络之间高速传输数据。

可以理解为网络中的“中枢系统”。

* * *

### 2\. 包含的设备

核心层通常包括：

-   核心路由器
    
-   骨干网设备
    
-   高性能交换机
    

这些设备通常部署在运营商或大型数据中心中。

* * *

### 3\. 主要功能

核心层的核心任务是“高效传输数据”，具体包括：

1.  数据转发（将数据从源头送到目的地）
    
2.  路由选择（决定数据走哪条路径）
    
3.  高速交换数据包
    
4.  维持网络的稳定性和可靠性
    

* * *

### 4\. 工作过程举例

当你访问一个网站时：

-   数据从你的设备发出（边缘层）
    
-   进入互联网
    
-   经过多个核心网络节点进行转发
    
-   最终到达目标服务器
    

这些中间负责传输的数据网络部分，就是核心层。

* * *

### 5\. 特点

核心层具有以下特点：

-   传输速度要求极高
    
-   设备性能强
    
-   需要高可靠性（不能轻易宕机）
    
-   结构相对集中
    

* * *

## 四、EL 与 CL 的对比

| 对比项 | EL（边缘层） | CL（核心层） |
| --- | --- | --- |
| 所在位置 | 网络边缘 | 网络中心 |
| 面向对象 | 用户设备 | 网络设备 |
| 主要作用 | 接入网络、产生数据 | 转发数据、传输数据 |
| 设备类型 | 电脑、手机、接入设备 | 路由器、核心交换机 |
| 数量 | 多 | 少 |
| 性能要求 | 一般 | 很高 |
| 功能复杂度 | 较低 | 较高 |

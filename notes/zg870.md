## 一、总体架构

Execution Layer Client（执行层客户端，如 Geth）可以抽象为分层结构：

```
Consensus Layer（CL）
        ↓ Engine API
Execution Engine（执行引擎）
        ↓
EVM（执行环境）
        ↓
State（状态数据库）
        ↓
Transaction Pool（交易池）
        ↓
DevP2P（网络层）
```

同时包含两个对外接口：

-   JSON-RPC：面向用户（钱包、DApp）
    
-   Engine API：面向共识层（CL）
    

* * *

## 二、核心本质

执行层的核心功能可以归结为：

**状态转移函数（State Transition Function, STF）**

```
(state, transactions) → new_state
```

即：给定当前状态和交易集合，计算新的状态。

* * *

## 三、核心模块

### 1\. EVM（Ethereum Virtual Machine）

本质：虚拟化执行引擎（类似 CPU）

作用：

-   执行智能合约字节码
    
-   保证所有节点执行结果一致（确定性）
    

特点：

-   与底层硬件无关
    
-   输入相同，输出必然相同
    

* * *

### 2\. State（全局状态）

Ethereum 采用账户模型（Account-based），与 Bitcoin 的 UTXO 模型不同。

State 包含：

-   账户地址
    
-   余额（balance）
    
-   nonce
    
-   合约代码（code）
    
-   存储（storage）
    
-   Merkle-Patricia Trie 结构
    

核心表示：

```
state_root = hash(global_state)
```

* * *

### 3\. Transactions（交易）

交易是状态变化的唯一驱动：

```
state → state'
```

执行流程：

```
Transaction → EVM → 修改 State
```

* * *

### 4\. Transaction Pool（交易池）

作用：

-   存储待打包交易（mempool）
    
-   按优先级排序
    

类型：

Legacy Pool

-   按 gas price 排序
    
-   使用优先队列（heap）
    

Blob Pool（EIP-4844）

-   处理 blob 类型交易
    
-   使用特殊的淘汰机制（基于对数策略）
    

* * *

### 5\. DevP2P（网络层）

作用：

-   节点之间通信
    
-   广播交易与区块
    
-   节点发现
    

特点：

-   点对点网络（P2P）
    
-   Gossip 传播机制
    
-   每个节点都会验证数据再传播
    

* * *

### 6\. JSON-RPC API（用户接口）

用于与外部交互：

常见方法：

```
eth_sendTransaction
eth_call
eth_getBalance
```

作用：

-   查询链上状态
    
-   发送交易
    

* * *

### 7\. Engine API（CL 与 EL 的桥梁）

Merge 后新增的核心接口，仅供 CL 调用。

（1）engine\_newPayload

作用：

-   验证并执行区块
    

流程：

1.  CL 接收到 Beacon Block
    
2.  提取 execution payload
    
3.  调用 EL
    

EL 执行：

-   检查 parent hash
    
-   执行交易
    
-   更新状态
    

返回状态：

-   VALID
    
-   INVALID
    
-   SYNCING
    
-   ACCEPTED
    

* * *

（2）engine\_forkchoiceUpdated

作用：

-   更新链头（head）
    
-   触发新区块构建
    

输入：

-   head block
    
-   safe block
    
-   finalized block
    
-   可选 payload attributes
    

返回：

-   状态
    
-   payloadId（用于获取构建中的区块）
    

* * *

## 四、状态转移函数（STF）

核心逻辑：

```go
func stf(parent, block, state) {
    VerifyHeaders()

    for tx in block:
        state = EVM(tx, state)

    return state
}
```

* * *

### 执行步骤

1\. 区块头验证

包括：

-   gas limit 合法性
    
-   block number 连续性
    
-   base fee 正确性
    
-   timestamp 合法性
    

* * *

2\. 执行交易

逐笔执行：

```
EVM.run(tx)
```

修改：

-   账户余额
    
-   存储
    
-   日志
    

* * *

3\. 更新状态

生成新的 state root：

```
new_state_root
```

写入区块头

* * *

### 关键规则

-   任意一笔交易失败 → 整个区块无效
    

* * *

## 五、同步机制（Sync）

### 1\. Full Sync

流程：

```
Genesis → 执行所有区块 → 当前状态
```

特点：

-   完全信任最小化
    
-   计算量极大
    
-   同步时间长
    

* * *

### 2\. Snap Sync

流程：

1.  选择 pivot block
    
2.  下载状态数据（账户 + storage）
    
3.  验证 Merkle proof
    
4.  执行后续区块
    

* * *

Healing Phase

作用：

-   修复不完整状态
    
-   补齐缺失 trie 节点
    
-   确保状态一致性
    

* * *

### 对比

| 方式 | 安全性 | 速度 | 资源消耗 |
| --- | --- | --- | --- |
| Full Sync | 高 | 慢 | 高 |
| Snap Sync | 中 | 快 | 中 |

* * *

## 六、Merge 后架构变化

### 之前

```
Execution Layer = 执行 + 共识
```

### 现在

```
Consensus Layer → 负责共识
Execution Layer → 负责执行
```

* * *

### 结果

Execution Layer 的职责：

-   执行交易
    
-   验证区块
    
-   维护状态
    

不再负责：

-   区块排序
    
-   fork choice
    
-   共识决策
    

* * *

## 九、总结

Execution Layer Client 可以抽象为三大核心组件：

1.  EVM（执行计算）
    
2.  State（状态存储）
    
3.  Engine API（与共识层通信）
    

其本质是：

**一个确定性的状态转移机器（Deterministic State Transition Machine）**

只负责：

-   接收输入（交易、区块）
    
-   执行计算
    
-   输出新状态
    

不参与：

-   共识决策
    
-   链选择

# **合约字节码**

以太坊只有两种类型的字节码

## **创建字节码**

生成运行时字节码的代码，包括智能合约的构造函数逻辑和构造函数参数。创建字节码相当于创建合约的交易输入数据，前提是该交易的唯一目的是创建合约。

用于检索此代码的solidity代码是`type(ContractName).creationCode`

## **运行时字节码**

存储链上描述智能合约的代码。这段代码不包括合约的构造函数逻辑或构造函数参数，它们与实际创建合约的代码无关。

合约的运行时字节码可以通过使用汇编块并调用`extcodecopy(a)`在链上检索。运行时字节码的哈希值从`extcodehash(a)`返回

这段代码以使用`type(ContractName).runtimeCode`在链上检索

## **字节码**

创建字节码和运行时字节码的统称

## **部署时字节码**

指的是合约运行时字节码

## **初始化字节码**

与创建字节码相同。它是创建存储链上的字节码的代码

每个字节可以是一个操作符、一个完整的操作数或操作数的一部分

### **内存**

类似于计算机的内存，是随机存取内存（RAM），可以在特定位置被读取和存储，而不仅仅是顺序的存储

### **堆栈**

字面意思上是一个按照先进后出的原选择工作的数据堆。这意味着添加堆栈中的数据位于顶部，而移除数据也是从堆栈顶部移除的。

### **调用数据**

调用数据是存储调用或交易有效信息的地方，调用负载通常是要执行函数的签名，然后是函数参数的ABI编码。

不能写入，属于只读位置

### **存储**

是一个永久存储空间，每个合约账户都有自己的存储。不能直接访问另一个账户的存储，其中数据以键值形式存在。

## 一、交易的本质

交易（Transaction）是：

**由外部账户（EOA）发起、经过签名的指令，用于驱动状态转移**

传播方式：

```
用户 → JSON-RPC → Execution Client → DevP2P 网络
```

* * *

## 二、交易结构（核心字段）

一个交易包含以下字段：

* * *

### 1\. nonce（Tₙ）

定义：

```
nonce = 发送者已发送交易数量
```

作用：

（1）防止重放攻击

-   每个交易 nonce 唯一
    
-   重复交易会被拒绝
    

（2）决定合约地址

合约地址计算：

```
contract_address = keccak256(sender, nonce)
```

（3）替换交易（Replace Tx）

-   相同 nonce
    
-   更高 gas fee
    
-   可覆盖旧交易
    

* * *

### 2\. gasPrice（Tₚ）

定义：

```
gasPrice = 每单位 gas 的价格（单位：Wei）
```

关系：

```
1 ETH = 10^18 Wei
```

作用：

-   决定交易优先级
    
-   越高越容易被打包
    

* * *

### 3\. gasLimit（Tg）

定义：

```
gasLimit = 允许消耗的最大 gas
```

特点：

-   执行耗尽则失败（out-of-gas）
    
-   未用完会退回
    

* * *

### 4\. to（Tt）

定义：

```
接收地址（20 bytes）
```

决定交易类型：

| to 值 | 类型 | 说明 |
| --- | --- | --- |
| 空 | 合约创建 | 创建新合约 |
| EOA 地址 | 转账 | ETH 转移 |
| 合约地址 | 调用 | 执行合约 |

* * *

### 5\. value（Tv）

定义：

```
转账金额（单位：Wei）
```

特点：

-   合约创建时 → 初始余额
    
-   调用合约时 → msg.value
    

* * *

### 6\. data / init（Td）

定义：

```
输入数据（byte array）
```

区别：

-   普通调用 → calldata
    
-   合约创建 → init code
    

* * *

### 7\. Signature（v, r, s）

定义：

-   ECDSA 签名
    
-   用于恢复 sender
    

* * *

## 三、交易执行的三种模式

* * *

### 1\. 普通转账

```
to = EOA
data = 空
```

结果：

-   balance 变化
    

* * *

### 2\. 合约调用

```
to = 合约地址
data = 函数调用数据
```

结果：

-   执行 EVM
    
-   修改 storage
    

* * *

### 3\. 合约创建

```
to = 空
data = init code
```

结果：

-   创建新合约账户
    
-   存储 runtime bytecode
    

* * *

## 四、合约创建机制（重点）

* * *

### 1\. 两段代码结构

```
init code + runtime code
```

* * *

### 2\. 执行流程

```
执行 init code
    ↓
返回 runtime code
    ↓
存入合约账户
```

* * *

### 3\. 示例拆解

Runtime 代码

```
6006600702600055
```

逻辑：

```
6 * 7 = 42
存入 storage[0]
```

* * *

Init Code

```
6008600c60003960086000f3 + runtime
```

作用：

1.  CODECOPY（复制 runtime 到 memory）
    
2.  RETURN（返回 runtime）
    

* * *

## 五、交易签名流程

步骤：

* * *

### Step 1：RLP 编码

```
payload → RLP
```

* * *

### Step 2：Keccak256 哈希

```
hash = keccak256(RLP)
```

* * *

### Step 3：ECDSA 签名

```
(v, r, s) = sign(hash, privateKey)
```

* * *

## 一、总体概念

Execution Layer 客户端需要存储两类数据：

1.  当前状态（State）
    
2.  历史区块数据（Blocks / Transactions / Receipts）
    

核心数据结构：


* * *

## 二、基础组件

* * *

### 1\. RLP（Recursive Length Prefix）

作用：

-   序列化数据（编码）
    
-   用于：
    
    -   交易
        
    -   区块
        
    -   Trie 节点
        

特点：

-   简单
    
-   无 schema
    
-   支持嵌套结构
    

* * *

### 2\. Merkle Tree

定义：

一种基于哈希的二叉树结构，用于数据完整性验证。

* * *

结构特点

-   叶子节点：数据
    
-   非叶子节点：子节点哈希
    
-   根节点：Merkle Root
    

* * *

构建方式

```text
自底向上（bottom-up）
```

* * *

关键性质

1.  防篡改（tamper-proof）
    

-   任意数据变化 → root hash 改变
    

2.  Avalanche Effect
    

-   微小变化 → hash 完全不同
    

* * *

特殊规则

-   若叶子节点数量为奇数：
    
    -   最后一个节点复制一次
        

* * *

作用

-   验证交易是否属于区块（Merkle Proof）
    

* * *

### 3\. Patricia Trie（压缩前缀树）

定义：

一种优化的 Trie（Radix Tree），用于高效存储 key-value。

* * *

特点

1.  前缀压缩
    

-   公共前缀合并
    

2.  边存字符串（不是单字符）
    
3.  单子节点合并（节省空间）
    

* * *

优势

-   更节省存储
    
-   查询效率高
    

* * *

## 三、Merkle Patricia Trie（MPT）


* * *

### 1\. 结合优势

| --- | --- |

一、执行层的核心本质

执行层（Execution Layer）的唯一目标：

执行状态转换函数 STF

σₜ₊₁ ≡ Π(σₜ, B)

含义：

输入：当前状态 σₜ + 区块 B  
输出：新状态 σₜ₊₁

本质：

区块链 = 状态机  
区块 = 状态输入  
EL = 执行引擎

——————————————————————

二、状态与状态根

状态 σ：

包含所有账户（balance / nonce / storage / code）

但：

区块链不存状态，只存状态的“承诺”

过程：

1）过滤空账户  
Ls(σ) = 非空账户集合

2）构建 Trie

3）得到 root hash

stateRoot = TRIE(Ls(σ))

关键理解：

stateRoot 是“状态的哈希摘要”，不是状态本身

——————————————————————

三、STF 执行流程

执行一个区块的完整流程：

1）读取父区块  
2）验证 Header  
3）验证 blob gas  
4）检查 ommers 为空  
5）执行区块（apply\_body）  
6）生成新状态  
7）校验 stateRoot 等字段  
8）加入链  
9）失败则 Invalid Block

核心原则：

先验证 → 再执行（节省算力）

——————————————————————

四、区块头验证核心约束

1）Gas 规则

-   gasUsed ≤ gasLimit
    
-   gasLimit 变化受限（±1/1024）
    
-   gasLimit > 5000
    

2）时间规则

-   timestamp 必须递增
    

3）链结构

-   parentHash 正确
    
-   block number = parent + 1
    

4）PoS 特征

-   difficulty = 0
    
-   nonce = 0
    

5）EIP-1559

-   baseFee 必须正确
    

6）EIP-4844

-   blobGasUsed 合法
    
-   excessBlobGas 正确
    

——————————————————————

五、EVM 执行环境 I

执行交易时的上下文：

1）调用

I\_caller = msg.sender  
I\_origin = tx.origin

2）区块

I\_number  
I\_time  
I\_coinbase  
I\_prevRandao  
I\_blockHashes

关系：

gasPrice = baseFee + priorityFee

4）状态

I\_state  
I\_chainId

本质：

I 决定“执行语义”

# **Account Model vs UTXO Model**

账户模型和UTXO模型

**UTXO Model**

状态表示：`State = Set of UTXOs`

-   无状态验证
    
-   高隐私性
    
-   低可编程性
    

每一笔交易都是独立的，比喻是实物现金，钱包里的不是一个“余额数字”而是一把面额各异的硬币。每次交易，必须把整枚硬币拿出来，花掉，找零，剩下的继续放回钱包

一笔Bitcoin交易的完整生命周期

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741052661-image.png)

UTXO只能被整体花费。Alice有两枚UTXO(0.8 + 0.5 BTC)，她想给Bob转1.0BTC，就必须把两枚都作为input塞进交易，然后显式创建output给自己。

锁定脚本才是真正的“所有权”，UTXO上记录的不是Alice的余额，而是一段脚本`OP_DUP OP_HASH160 <Alice的pubkey哈希> OP_EQUALVERIFY OP_CHECKSIG`。交易时，Alice提供scriptSig(签名+公钥)，脚本执行通过才算解锁成功。节点验证时完全无状态——只需要拿到这一笔交易和其引用的UTXO，不需要知道Alice的历史

守恒是强制的，input总额>=output总额，Bitcoin协议不存在“凭空增加余额”的操作，每个聪的来源都可追溯的某个创币交易

**Account Model**

状态表示： `State = Mapping(address → account)`

账户结构

```
 account = {
   nonce,
   balance,
   storageRoot,
   codeHash
 }
```

Ethereum的做法更像银行系统，有一张全居的大表，每个地址对应一行记录。转账就是直接修改两行数字

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741074443-image.png)

_优势_

-   状态转移更直接 `balance[A] += x` `balance[B] -= x`
    
-   支持全局状态访问和跨合约交互
    
-   更适合DeFi 、AMM 、DEX
    

_缺点_

-   重放保护强依赖于nonce
    
-   状态增长无法pruning状态膨胀
    

Account模型最关键的地方在于全局可读写的storage。当合约A调用合约B，B可以直接读写A的storage（A授权的话），这让DeFi的可组合性成为可能：一笔交易里可以闪电贷、换币贷、质押、还款，全在一个原子操作内完成

但是所有操作都必须按顺序执行，因为前一步的storage变更会影响后一步的读取，这是Ethereum无法像Bitcion那样轻松并行化的原因

**验证模型与状态膨胀的根本区别**

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741096721-image.png)

UTXO验证“无状态”：因为一笔比特币交易里包含了它自己的完整证明：每个input都指向某个具体的`txid:vout`，节点只需要确认那个UTXO存在于当前的UTXO Set，且签名合法，就可以接受这笔交易。不需要知道Alice以前发过多少个交易、当前账户里还有什么。这是Stateless Client和SPV验证的理论依据

Account模型的只增不减：Ethereum的世界的一棵默克尔树（MPT） ，stateRoot是这棵树的哈希，写入区块头。要重新计算一个正确的stateRoot，必须保留完整的树。某个合约弃用了、某个地址再没有活动——对应的trie叶节点依然必须保留，因为历史区块的stateRoot依然引用它。可以通过`Selfdestruct`清空合约存储（已被EIP-6780）限制，但是这在实践中几乎不可发生。Verkle Tree的引入，将把witness体积从几十kB压缩到几kB。

# **序列化：RLP vs SSZ**

区块链的一切最终都要哈希，哈希结果要写进区块头，验证时必须能重现完全相同的字节序列。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741112259-image.png)

为什么Ethereum要从RLP迁移到SSZ

1.  请客户端需要Merkle proof证明状态的一部分（如账户余额），RLP做不到
    
2.  信标链状态字段多达数十个，每个slot更新若干字段，O(N)代价不可接受
    
3.  SSZ的typed schema可以生成代码绑定，减少客户端实现错误
    

RLP的递归前缀编码很优雅，但是无法完全支持Merkle proof——因为整个结构被压成以一个线性字节流，无法从中抽出一个子字段并提供证明

SSZ的核心创新是把序列化和Merkleization分离设计但协同操作：序列化的结果是固定化布局的字节流，Merkleization把每个字段哈希成叶子节点再建树。更新一个字段时，只需要重哈希从该叶节点到根的路径，路径长度是 O(log N)，而不是重新序列化整棵BeasonState

# **共识与最终性**

Ethereum PoS 的共识是两套机制叠加的产物，理解它们要分开看再看合并。

**Casper FFG 解决的问题是最终性（finality）**——即确认某个 checkpoint 永不回滚。**LMD-GHOST 解决的问题是活性（liveness）**——即在有分叉时能持续选出一条链继续出块。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741137988-image.png)

LMD：每个验证者只算其最新一条attestation，防止旧票双重影响

GHOST：每步选票数最多的子树，而非最长链（抗分叉攻击）

`Gasper = Casper FFG + LMD-GHOST`

两者单独都不够，合在一起才是完整的 Gasper

LMD-GHOST 负责活性：网络分叉时继续选出新块，链不停止

Casper FFG负责安全：每 ~12.8 分钟 finalize 一个 checkpoint，给已确认的历史提供经济担保

Casper FFG：检查点投票 + 两阶段确认

两阶段提交：CP 先获 ≥2/3 验证者投票 → Justified；下一个 CP 也被 Justified 后，前者才变为 Finalized安全保证：回滚 Finalized 块 → 需 ≥1/3 stake 被 slash（数十亿美元），经济上不可行

# **网络层：DHT + GossipSub**

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741152944-image.png)

discv5 和 GossipSub 解决的是完全不同层次的问题

discv5 ——节点刚启动时，它不知道网络里有谁，通过 Kademlia 路由表（按节点 ID XOR 距离排列）找到 ENR 记录（包含 IP、UDP 端口、公钥），然后建立 TCP 连接。它本身不传任何区块数据。

GossipSub 是在已建立的 TCP 连接上运行的 pub/sub 协议。每个节点订阅若干 topic（`beacon_block`, `beacon_attestation_0` 到 `beacon_attestation_63` 等），每个 topic 维护一个 D=8 大小的 mesh（全量消息 peer）和若干 gossip peer（只收元数据摘要）。收到新区块时，广播给所有 mesh peer，他们再广播给自己的 mesh peer，一般 2-3 跳就能覆盖大部分网络。

# **协议架构概述**

当前的协议架构由两个主要部分组成

执行层（EL）：处理实际的事务和用户交互，是全局计算机执行其程序的地方

共识层（CL）：提供权益证明共识机制，确保所有节点遵循相同的提示并驱动执行层的规范，确保所有节点遵循相同的提示并驱动执行层的规范链

两者通过标准化接口通信，实现了架构解耦

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-08-1775654017594-image.png)

-   Beacon Node（信标节点）：共识层核心，负责Pos共识，信标链数据同步，区块验证、验证者管理、最终性确认
    
-   Execution Engine（执行引擎）：执行层引擎，负责EVM执行、交易处理、账户状态维护、历史数据同步
    
-   Engine API：两层之间唯一的通信桥梁，实现共识层对执行层的驱动
    
-   p2p网络层：两个模块各自独立的P2P网络，分别同步共识层与执行层数据
    
-   Beason API's / JSON-RPC API：面向外部的对外接口，分别服务于验证者和普通用户/DApp
    

**关键交互逻辑**

NaN.  数据流向：
      
      NaN.  信标节点通过`Engine API`向执行引擎下发 待执行的区块任务
            
      NaN.  执行引擎完成交易执行、状态更新后，将状态根返回给信标节点
            
      NaN.  信标节点验证状态根后，将区块纳入信标链，完成最终确认
            
NaN.  对外接口
      
      NaN.  `Beacon APIs`：供外部服务查询信标链数据（如验证者余额、epoch状态）
            
      NaN.  `User APIs`:供用户、Dapp、钱包等调用执行层接口（如发送交易、查询余额）
            
NaN.  架构优势
      
      NaN.  解耦升级：两层可独立迭代，互不影响核心逻辑
            
      NaN.  多客户端兼容：执行层与共识层可自由组合不同客户端
            
      NaN.  安全性的提升：共识层与执行层职责分离，降低单点故障风险

核心负责哪个区块被打包，完全基于Pos机制

-   Blobs：对应Dene升级引入的数据块，存储大容量数据
    
-   p2p：共识层的网络通信层，负责验证者之间的共识消息广播与同步

_交互对象_

Validators（验证者）：通过Beacon API‘s与共识层交互，验证者质押ETH后参与区块提议、证明是Pos网络的核心参与者

**执行层**

核心负责执行交易、计算状态，兼容原以太坊主链逻辑

_核心组件_

_交互对象_

User/web3：通过JSON-RPC API与执行层交互，用户发送交易、查询余额、调用智能合约等。

# **协议设计理念**

以太坊的设计并不是随意的，而是围绕几个核心哲学展开：

-   Simplicity（简单性）
    
    Ethereum在设计初期遵循最小可行协议
    
    -   协议层应尽可能语义简洁
        
    -   降低实现复杂度，减少客户端分歧和共识攻击面
        
    -   设计目标：使协议specification在理论上可被单个开发者完整实现
        
    -   随着协议升级（如EIP-1559、The Merge），简洁转化为模块化隔离复杂性和形式化规范
        
-   Universality(通用计算规范)
    
    -   以太坊不提供特定的应用逻辑，仅提供状态(state)、状态转移函数、EVM执行环境
        
    -   形象化表达`σₜ₊₁ = Υ(σₜ, T)`
        
        -   σ：全局状态
            
        -   T：交易集合
            
        -   Υ：状态转移函数
            

Ethereum 本质上是一个通用计算平台

-   Modularity(模块化架构)
    
    -   具体实现共识层和执行层解耦
        
    -   数据结构与执行逻辑分离
        
    -   设计原则：High cohesion, low coupling(高内聚，低耦合)，支持局部升级和组件替换
        
-   Non-discrimination(协议中立性)
    
    -   Ethereum遵循可信中立
        
    -   原则是协议不对交易内容做语义判断，不限制合约类型和使用场景
        
    -   约束方式：仅通过 gas pricing mechanism(天然气定价机制)内化外部性
        
    -   也就是不进行内容层面的审查(content-level censorship) ,只进行资源级定价(resource-level pricing)
        
-   Agility(演化能力)
    
    -   EIP ：对EVM semantics(EVM语义) 和 Address scheme(地址方案)的修改极其谨慎
        
    -   目标是在安全性、可拓展性、向后兼容之间权衡
        

# **核心设计原则**

-   Complexity Management(复杂性管理)
    
    -   复杂性不是代码行数，而是交互层
        
        -   **Encapsulated complexity（封装复杂性）**
            
        -   **Systemic complexity（系统耦合复杂性）**
            
-   Sandwich Model (分层复杂性分布)
    
    -   结构
        
        -   User Interface Layer （简单）
            
        -   Middleware / Tooling （复杂）
            
        -   Protocol Core （简单）
            
    -   复杂性分布策略：将复杂性推向编译器、编码层和存储模型
        
-   Complexity Placement Priority
    
    复杂性优先放置顺序
    
    ```
     Layer 2 > Client Implementation > Protocol Spec
    ```
    
    -   L1必须保持极简
        
    -   拓展优先通过 Rollup和Off-chain computation
        
-   Freedom(无许可行)
    
    Ethereum遵循Permissionless innovation
    
    -   不限制use-case，通过gas实现incentive compatibility
        
-   Generalization(最小原语设计)
    
    -   提供 low-level opcode
        
    -   避免 high-level abstraction
        
-   No Built-in Features (反特性设计)
    
    -   协议层：minimal
        
    -   应用层：expressive

# EL 和 CL

# 一、基本概念

在计算机网络中，EL 和 CL 通常指网络结构中的两个重要层次：

-   EL：Edge Layer（边缘层）
    
-   CL：Core Layer（核心层）
    

这两个概念主要用于描述网络的结构划分，尤其是在互联网或大型局域网架构中。

从整体上看，一个网络可以理解为由“边缘部分”和“核心部分”组成：

-   边缘层（EL）：靠近用户，负责接入网络
    
-   核心层（CL）：位于网络中心，负责数据的高效传输
    

* * *

## 二、边缘层（EL）

### 1\. 定义

边缘层是网络中最靠近用户设备的一层，是用户与网络之间的接口部分。

也可以理解为：用户“接触网络”的地方。

* * *

### 2\. 包含的设备

边缘层通常包括以下设备：

-   个人电脑
    
-   手机
    
-   平板设备
    
-   家用路由器
    
-   接入交换机
    
-   物联网设备（如智能家居设备）
    

* * *

### 3\. 主要功能

边缘层的核心作用是“接入”和“产生数据”，具体包括：

1.  连接用户设备到网络
    
2.  产生数据（例如浏览网页、发送请求）
    
3.  接收来自网络的数据（例如网页内容、视频流）
    

* * *

### 4\. 工作过程举例

当你打开浏览器访问一个网站时：

-   你的电脑发出请求
    
-   请求从你的设备进入网络
    

在这个过程中，你的电脑就属于边缘层设备。

* * *

### 5\. 特点

边缘层具有以下特点：

-   靠近用户
    
-   数量非常多
    
-   设备类型多样
    
-   功能相对简单（以接入为主）
    

* * *

## 三、核心层（CL）

### 1\. 定义

核心层是网络的中心部分，主要负责在不同网络之间高速传输数据。

可以理解为网络中的“中枢系统”。

* * *

### 2\. 包含的设备

核心层通常包括：

-   核心路由器
    
-   骨干网设备
    
-   高性能交换机
    

这些设备通常部署在运营商或大型数据中心中。

* * *

### 3\. 主要功能

核心层的核心任务是“高效传输数据”，具体包括：

1.  数据转发（将数据从源头送到目的地）
    
2.  路由选择（决定数据走哪条路径）
    
3.  高速交换数据包
    
4.  维持网络的稳定性和可靠性
    

* * *

### 4\. 工作过程举例

当你访问一个网站时：

-   数据从你的设备发出（边缘层）
    
-   进入互联网
    
-   经过多个核心网络节点进行转发
    
-   最终到达目标服务器
    

这些中间负责传输的数据网络部分，就是核心层。

* * *

### 5\. 特点

核心层具有以下特点：

-   传输速度要求极高
    
-   设备性能强
    
-   需要高可靠性（不能轻易宕机）
    
-   结构相对集中
    

* * *

## 四、EL 与 CL 的对比

| 对比项 | EL（边缘层） | CL（核心层） |
| --- | --- | --- |
| 所在位置 | 网络边缘 | 网络中心 |
| 面向对象 | 用户设备 | 网络设备 |
| 主要作用 | 接入网络、产生数据 | 转发数据、传输数据 |
| 设备类型 | 电脑、手机、接入设备 | 路由器、核心交换机 |
| 数量 | 多 | 少 |
| 性能要求 | 一般 | 很高 |
| 功能复杂度 | 较低 | 较高 |

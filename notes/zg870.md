# **Account Model vs UTXO Model**

账户模型和UTXO模型

**UTXO Model**

状态表示：`State = Set of UTXOs`

-   无状态验证
    
-   高隐私性
    
-   低可编程性
    

每一笔交易都是独立的，比喻是实物现金，钱包里的不是一个“余额数字”而是一把面额各异的硬币。每次交易，必须把整枚硬币拿出来，花掉，找零，剩下的继续放回钱包

一笔Bitcoin交易的完整生命周期

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741052661-image.png)

UTXO只能被整体花费。Alice有两枚UTXO(0.8 + 0.5 BTC)，她想给Bob转1.0BTC，就必须把两枚都作为input塞进交易，然后显式创建output给自己。

锁定脚本才是真正的“所有权”，UTXO上记录的不是Alice的余额，而是一段脚本`OP_DUP OP_HASH160 <Alice的pubkey哈希> OP_EQUALVERIFY OP_CHECKSIG`。交易时，Alice提供scriptSig(签名+公钥)，脚本执行通过才算解锁成功。节点验证时完全无状态——只需要拿到这一笔交易和其引用的UTXO，不需要知道Alice的历史

守恒是强制的，input总额>=output总额，Bitcoin协议不存在“凭空增加余额”的操作，每个聪的来源都可追溯的某个创币交易

**Account Model**

状态表示： `State = Mapping(address → account)`

账户结构

```
 account = {
   nonce,
   balance,
   storageRoot,
   codeHash
 }
```

Ethereum的做法更像银行系统，有一张全居的大表，每个地址对应一行记录。转账就是直接修改两行数字

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741074443-image.png)

_优势_

-   状态转移更直接 `balance[A] += x` `balance[B] -= x`
    
-   支持全局状态访问和跨合约交互
    
-   更适合DeFi 、AMM 、DEX
    

_缺点_

-   重放保护强依赖于nonce
    
-   状态增长无法pruning状态膨胀
    

Account模型最关键的地方在于全局可读写的storage。当合约A调用合约B，B可以直接读写A的storage（A授权的话），这让DeFi的可组合性成为可能：一笔交易里可以闪电贷、换币贷、质押、还款，全在一个原子操作内完成

但是所有操作都必须按顺序执行，因为前一步的storage变更会影响后一步的读取，这是Ethereum无法像Bitcion那样轻松并行化的原因

**验证模型与状态膨胀的根本区别**

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741096721-image.png)

UTXO验证“无状态”：因为一笔比特币交易里包含了它自己的完整证明：每个input都指向某个具体的`txid:vout`，节点只需要确认那个UTXO存在于当前的UTXO Set，且签名合法，就可以接受这笔交易。不需要知道Alice以前发过多少个交易、当前账户里还有什么。这是Stateless Client和SPV验证的理论依据

Account模型的只增不减：Ethereum的世界的一棵默克尔树（MPT） ，stateRoot是这棵树的哈希，写入区块头。要重新计算一个正确的stateRoot，必须保留完整的树。某个合约弃用了、某个地址再没有活动——对应的trie叶节点依然必须保留，因为历史区块的stateRoot依然引用它。可以通过`Selfdestruct`清空合约存储（已被EIP-6780）限制，但是这在实践中几乎不可发生。Verkle Tree的引入，将把witness体积从几十kB压缩到几kB。

# **序列化：RLP vs SSZ**

区块链的一切最终都要哈希，哈希结果要写进区块头，验证时必须能重现完全相同的字节序列。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741112259-image.png)

为什么Ethereum要从RLP迁移到SSZ

1.  请客户端需要Merkle proof证明状态的一部分（如账户余额），RLP做不到
    
2.  信标链状态字段多达数十个，每个slot更新若干字段，O(N)代价不可接受
    
3.  SSZ的typed schema可以生成代码绑定，减少客户端实现错误
    

RLP的递归前缀编码很优雅，但是无法完全支持Merkle proof——因为整个结构被压成以一个线性字节流，无法从中抽出一个子字段并提供证明

SSZ的核心创新是把序列化和Merkleization分离设计但协同操作：序列化的结果是固定化布局的字节流，Merkleization把每个字段哈希成叶子节点再建树。更新一个字段时，只需要重哈希从该叶节点到根的路径，路径长度是 O(log N)，而不是重新序列化整棵BeasonState

# **共识与最终性**

Ethereum PoS 的共识是两套机制叠加的产物，理解它们要分开看再看合并。

**Casper FFG 解决的问题是最终性（finality）**——即确认某个 checkpoint 永不回滚。**LMD-GHOST 解决的问题是活性（liveness）**——即在有分叉时能持续选出一条链继续出块。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741137988-image.png)

LMD：每个验证者只算其最新一条attestation，防止旧票双重影响

GHOST：每步选票数最多的子树，而非最长链（抗分叉攻击）

`Gasper = Casper FFG + LMD-GHOST`

两者单独都不够，合在一起才是完整的 Gasper

LMD-GHOST 负责活性：网络分叉时继续选出新块，链不停止

Casper FFG负责安全：每 ~12.8 分钟 finalize 一个 checkpoint，给已确认的历史提供经济担保

Casper FFG：检查点投票 + 两阶段确认

两阶段提交：CP 先获 ≥2/3 验证者投票 → Justified；下一个 CP 也被 Justified 后，前者才变为 Finalized安全保证：回滚 Finalized 块 → 需 ≥1/3 stake 被 slash（数十亿美元），经济上不可行

# **网络层：DHT + GossipSub**

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-09-1775741152944-image.png)

discv5 和 GossipSub 解决的是完全不同层次的问题

discv5 ——节点刚启动时，它不知道网络里有谁，通过 Kademlia 路由表（按节点 ID XOR 距离排列）找到 ENR 记录（包含 IP、UDP 端口、公钥），然后建立 TCP 连接。它本身不传任何区块数据。

GossipSub 是在已建立的 TCP 连接上运行的 pub/sub 协议。每个节点订阅若干 topic（`beacon_block`, `beacon_attestation_0` 到 `beacon_attestation_63` 等），每个 topic 维护一个 D=8 大小的 mesh（全量消息 peer）和若干 gossip peer（只收元数据摘要）。收到新区块时，广播给所有 mesh peer，他们再广播给自己的 mesh peer，一般 2-3 跳就能覆盖大部分网络。

# **协议架构概述**

当前的协议架构由两个主要部分组成

执行层（EL）：处理实际的事务和用户交互，是全局计算机执行其程序的地方

共识层（CL）：提供权益证明共识机制，确保所有节点遵循相同的提示并驱动执行层的规范，确保所有节点遵循相同的提示并驱动执行层的规范链

两者通过标准化接口通信，实现了架构解耦

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-08-1775654017594-image.png)

-   Beacon Node（信标节点）：共识层核心，负责Pos共识，信标链数据同步，区块验证、验证者管理、最终性确认
    
-   Execution Engine（执行引擎）：执行层引擎，负责EVM执行、交易处理、账户状态维护、历史数据同步
    
-   Engine API：两层之间唯一的通信桥梁，实现共识层对执行层的驱动
    
-   p2p网络层：两个模块各自独立的P2P网络，分别同步共识层与执行层数据
    
-   Beason API's / JSON-RPC API：面向外部的对外接口，分别服务于验证者和普通用户/DApp
    

**关键交互逻辑**

NaN.  数据流向：
      
      NaN.  信标节点通过`Engine API`向执行引擎下发 待执行的区块任务
            
      NaN.  执行引擎完成交易执行、状态更新后，将状态根返回给信标节点
            
      NaN.  信标节点验证状态根后，将区块纳入信标链，完成最终确认
            
NaN.  对外接口
      
      NaN.  `Beacon APIs`：供外部服务查询信标链数据（如验证者余额、epoch状态）
            
      NaN.  `User APIs`:供用户、Dapp、钱包等调用执行层接口（如发送交易、查询余额）
            
NaN.  架构优势
      
      NaN.  解耦升级：两层可独立迭代，互不影响核心逻辑
            
      NaN.  多客户端兼容：执行层与共识层可自由组合不同客户端
            
      NaN.  安全性的提升：共识层与执行层职责分离，降低单点故障风险

# **协议设计理念**

以太坊的设计并不是随意的，而是围绕几个核心哲学展开：

-   Simplicity（简单性）
    
    Ethereum在设计初期遵循最小可行协议
    
    -   协议层应尽可能语义简洁
        
    -   降低实现复杂度，减少客户端分歧和共识攻击面
        
    -   设计目标：使协议specification在理论上可被单个开发者完整实现
        
    -   随着协议升级（如EIP-1559、The Merge），简洁转化为模块化隔离复杂性和形式化规范
        
-   Universality(通用计算规范)

# EL 和 CL

# 一、基本概念

在计算机网络中，EL 和 CL 通常指网络结构中的两个重要层次：

-   EL：Edge Layer（边缘层）
    
-   CL：Core Layer（核心层）
    

这两个概念主要用于描述网络的结构划分，尤其是在互联网或大型局域网架构中。

从整体上看，一个网络可以理解为由“边缘部分”和“核心部分”组成：

-   边缘层（EL）：靠近用户，负责接入网络
    
-   核心层（CL）：位于网络中心，负责数据的高效传输
    

* * *

## 二、边缘层（EL）

### 1\. 定义

边缘层是网络中最靠近用户设备的一层，是用户与网络之间的接口部分。

也可以理解为：用户“接触网络”的地方。

* * *

### 2\. 包含的设备

边缘层通常包括以下设备：

-   个人电脑
    
-   手机
    
-   平板设备
    
-   家用路由器
    
-   接入交换机
    
-   物联网设备（如智能家居设备）
    

* * *

### 3\. 主要功能

边缘层的核心作用是“接入”和“产生数据”，具体包括：

1.  连接用户设备到网络
    
2.  产生数据（例如浏览网页、发送请求）
    
3.  接收来自网络的数据（例如网页内容、视频流）
    

* * *

### 4\. 工作过程举例

当你打开浏览器访问一个网站时：

-   你的电脑发出请求
    
-   请求从你的设备进入网络
    

在这个过程中，你的电脑就属于边缘层设备。

* * *

### 5\. 特点

边缘层具有以下特点：

-   靠近用户
    
-   数量非常多
    
-   设备类型多样
    
-   功能相对简单（以接入为主）
    

* * *

## 三、核心层（CL）

### 1\. 定义

核心层是网络的中心部分，主要负责在不同网络之间高速传输数据。

可以理解为网络中的“中枢系统”。

* * *

### 2\. 包含的设备

核心层通常包括：

-   核心路由器
    
-   骨干网设备
    
-   高性能交换机
    

这些设备通常部署在运营商或大型数据中心中。

* * *

### 3\. 主要功能

核心层的核心任务是“高效传输数据”，具体包括：

1.  数据转发（将数据从源头送到目的地）
    
2.  路由选择（决定数据走哪条路径）
    
3.  高速交换数据包
    
4.  维持网络的稳定性和可靠性
    

* * *

### 4\. 工作过程举例

当你访问一个网站时：

-   数据从你的设备发出（边缘层）
    
-   进入互联网
    
-   经过多个核心网络节点进行转发
    
-   最终到达目标服务器
    

这些中间负责传输的数据网络部分，就是核心层。

* * *

### 5\. 特点

核心层具有以下特点：

-   传输速度要求极高
    
-   设备性能强
    
-   需要高可靠性（不能轻易宕机）
    
-   结构相对集中
    

* * *

## 四、EL 与 CL 的对比

| 对比项 | EL（边缘层） | CL（核心层） |
| --- | --- | --- |
| 所在位置 | 网络边缘 | 网络中心 |
| 面向对象 | 用户设备 | 网络设备 |
| 主要作用 | 接入网络、产生数据 | 转发数据、传输数据 |
| 设备类型 | 电脑、手机、接入设备 | 路由器、核心交换机 |
| 数量 | 多 | 少 |
| 性能要求 | 一般 | 很高 |
| 功能复杂度 | 较低 | 较高 |

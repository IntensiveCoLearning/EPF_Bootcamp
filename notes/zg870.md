# **协议架构概述**

当前的协议架构由两个主要部分组成

执行层（EL）：处理实际的事务和用户交互，是全局计算机执行其程序的地方

共识层（CL）：提供权益证明共识机制，确保所有节点遵循相同的提示并驱动执行层的规范，确保所有节点遵循相同的提示并驱动执行层的规范链

两者通过标准化接口通信，实现了架构解耦

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-08-1775654017594-image.png)

-   Beacon Node（信标节点）：共识层核心，负责Pos共识，信标链数据同步，区块验证、验证者管理、最终性确认
    
-   Execution Engine（执行引擎）：执行层引擎，负责EVM执行、交易处理、账户状态维护、历史数据同步
    
-   Engine API：两层之间唯一的通信桥梁，实现共识层对执行层的驱动
    
-   p2p网络层：两个模块各自独立的P2P网络，分别同步共识层与执行层数据
    
-   Beason API's / JSON-RPC API：面向外部的对外接口，分别服务于验证者和普通用户/DApp
    

**关键交互逻辑**

NaN.  数据流向：
      
      NaN.  信标节点通过`Engine API`向执行引擎下发 待执行的区块任务
            
      NaN.  执行引擎完成交易执行、状态更新后，将状态根返回给信标节点
            
      NaN.  信标节点验证状态根后，将区块纳入信标链，完成最终确认
            
NaN.  对外接口
      
      NaN.  `Beacon APIs`：供外部服务查询信标链数据（如验证者余额、epoch状态）
            
      NaN.  `User APIs`:供用户、Dapp、钱包等调用执行层接口（如发送交易、查询余额）
            
NaN.  架构优势
      
      NaN.  解耦升级：两层可独立迭代，互不影响核心逻辑
            
      NaN.  多客户端兼容：执行层与共识层可自由组合不同客户端
            
      NaN.  安全性的提升：共识层与执行层职责分离，降低单点故障风险
            

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/zg870/images/2026-04-08-1775654036087-image.png)

**共识层**

核心负责哪个区块被打包，完全基于Pos机制

_核心组件_

-   Blobs：对应Dene升级引入的数据块，存储大容量数据
    
-   p2p：共识层的网络通信层，负责验证者之间的共识消息广播与同步
    
-   RANDD：随机信标，负责生成随机数，用于验证者抽签、区块提议等
    
-   FORK CHOICE：最新的消息驱动自动分叉选择算法，决定链的主分叉
    
-   Finality：经典的权益最终证明机制，确保区块确认后不可逆转
    

_交互对象_

Validators（验证者）：通过Beacon API‘s与共识层交互，验证者质押ETH后参与区块提议、证明是Pos网络的核心参与者

**执行层**

核心负责执行交易、计算状态，兼容原以太坊主链逻辑

_核心组件_

-   p2p：执行层的网络通信层，同步交易、区块、账户状态等数据
    
-   Txs（mempool）：交易池，暂存待打包的用户交易，等待执行引擎打包
    
-   EVM：以太坊虚拟机，负责执行交易字节码、更新账户状态
    
-   state data：状态数据，存储账户余额、合约代码、存储槽等核心信息
    

_交互对象_

User/web3：通过JSON-RPC API与执行层交互，用户发送交易、查询余额、调用智能合约等。

# **协议设计理念**

以太坊的设计并不是随意的，而是围绕几个核心哲学展开：

-   Simplicity（简单性）
    
    Ethereum在设计初期遵循最小可行协议
    
    -   协议层应尽可能语义简洁
        
    -   降低实现复杂度，减少客户端分歧和共识攻击面
        
    -   设计目标：使协议specification在理论上可被单个开发者完整实现
        
    -   随着协议升级（如EIP-1559、The Merge），简洁转化为模块化隔离复杂性和形式化规范
        
-   Universality(通用计算规范)
    
    -   以太坊不提供特定的应用逻辑，仅提供状态(state)、状态转移函数、EVM执行环境
        
    -   形象化表达`σₜ₊₁ = Υ(σₜ, T)`
        
        -   σ：全局状态
            
        -   T：交易集合
            
        -   Υ：状态转移函数
            

Ethereum 本质上是一个通用计算平台

-   Modularity(模块化架构)
    
    -   具体实现共识层和执行层解耦
        
    -   数据结构与执行逻辑分离
        
    -   设计原则：High cohesion, low coupling(高内聚，低耦合)，支持局部升级和组件替换
        
-   Non-discrimination(协议中立性)
    
    -   Ethereum遵循可信中立
        
    -   原则是协议不对交易内容做语义判断，不限制合约类型和使用场景
        
    -   约束方式：仅通过 gas pricing mechanism(天然气定价机制)内化外部性
        
    -   也就是不进行内容层面的审查(content-level censorship) ,只进行资源级定价(resource-level pricing)
        
-   Agility(演化能力)
    
    -   EIP ：对EVM semantics(EVM语义) 和 Address scheme(地址方案)的修改极其谨慎
        
    -   目标是在安全性、可拓展性、向后兼容之间权衡
        

# **核心设计原则**

-   Complexity Management(复杂性管理)
    
    -   复杂性不是代码行数，而是交互层
        
        -   **Encapsulated complexity（封装复杂性）**
            
        -   **Systemic complexity（系统耦合复杂性）**
            
-   Sandwich Model (分层复杂性分布)
    
    -   结构
        
        -   User Interface Layer （简单）
            
        -   Middleware / Tooling （复杂）
            
        -   Protocol Core （简单）
            
    -   复杂性分布策略：将复杂性推向编译器、编码层和存储模型
        
-   Complexity Placement Priority
    
    复杂性优先放置顺序
    
    ```
     Layer 2 > Client Implementation > Protocol Spec
    ```
    
    -   L1必须保持极简
        
    -   拓展优先通过 Rollup和Off-chain computation
        
-   Freedom(无许可行)
    
    Ethereum遵循Permissionless innovation
    
    -   不限制use-case，通过gas实现incentive compatibility
        
-   Generalization(最小原语设计)
    
    -   提供 low-level opcode
        
    -   避免 high-level abstraction
        
-   No Built-in Features (反特性设计)
    
    -   协议层：minimal
        
    -   应用层：expressive

# EL 和 CL

# 一、基本概念

在计算机网络中，EL 和 CL 通常指网络结构中的两个重要层次：

-   EL：Edge Layer（边缘层）
    
-   CL：Core Layer（核心层）
    

这两个概念主要用于描述网络的结构划分，尤其是在互联网或大型局域网架构中。

从整体上看，一个网络可以理解为由“边缘部分”和“核心部分”组成：

-   边缘层（EL）：靠近用户，负责接入网络
    
-   核心层（CL）：位于网络中心，负责数据的高效传输
    

* * *

## 二、边缘层（EL）

### 1\. 定义

边缘层是网络中最靠近用户设备的一层，是用户与网络之间的接口部分。

也可以理解为：用户“接触网络”的地方。

* * *

### 2\. 包含的设备

边缘层通常包括以下设备：

-   个人电脑
    
-   手机
    
-   平板设备
    
-   家用路由器
    
-   接入交换机
    
-   物联网设备（如智能家居设备）
    

* * *

### 3\. 主要功能

边缘层的核心作用是“接入”和“产生数据”，具体包括：

1.  连接用户设备到网络
    
2.  产生数据（例如浏览网页、发送请求）
    
3.  接收来自网络的数据（例如网页内容、视频流）
    

* * *

### 4\. 工作过程举例

当你打开浏览器访问一个网站时：

-   你的电脑发出请求
    
-   请求从你的设备进入网络
    

在这个过程中，你的电脑就属于边缘层设备。

* * *

### 5\. 特点

边缘层具有以下特点：

-   靠近用户
    
-   数量非常多
    
-   设备类型多样
    
-   功能相对简单（以接入为主）
    

* * *

## 三、核心层（CL）

### 1\. 定义

核心层是网络的中心部分，主要负责在不同网络之间高速传输数据。

可以理解为网络中的“中枢系统”。

* * *

### 2\. 包含的设备

核心层通常包括：

-   核心路由器
    
-   骨干网设备
    
-   高性能交换机
    

这些设备通常部署在运营商或大型数据中心中。

* * *

### 3\. 主要功能

核心层的核心任务是“高效传输数据”，具体包括：

1.  数据转发（将数据从源头送到目的地）
    
2.  路由选择（决定数据走哪条路径）
    
3.  高速交换数据包
    
4.  维持网络的稳定性和可靠性
    

* * *

### 4\. 工作过程举例

当你访问一个网站时：

-   数据从你的设备发出（边缘层）
    
-   进入互联网
    
-   经过多个核心网络节点进行转发
    
-   最终到达目标服务器
    

这些中间负责传输的数据网络部分，就是核心层。

* * *

### 5\. 特点

核心层具有以下特点：

-   传输速度要求极高
    
-   设备性能强
    
-   需要高可靠性（不能轻易宕机）
    
-   结构相对集中
    

* * *

## 四、EL 与 CL 的对比

| 对比项 | EL（边缘层） | CL（核心层） |
| --- | --- | --- |
| 所在位置 | 网络边缘 | 网络中心 |
| 面向对象 | 用户设备 | 网络设备 |
| 主要作用 | 接入网络、产生数据 | 转发数据、传输数据 |
| 设备类型 | 电脑、手机、接入设备 | 路由器、核心交换机 |
| 数量 | 多 | 少 |
| 性能要求 | 一般 | 很高 |
| 功能复杂度 | 较低 | 较高 |

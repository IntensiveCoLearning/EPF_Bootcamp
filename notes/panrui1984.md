## 共识层的工作流 (Data Flow)

1.  **同步与监听**：BN 从 P2P 网络接收到新的区块。
    
2.  **执行验证**：BN 将区块内的交易负载（Execution Payload）通过 Engine API 发给 EL 进行验证。
    
3.  **职责查询**：VC 问 BN：“下一个 Slot 该我干活吗？”
    
4.  **数据请求**：如果是 VC 的轮次，VC 向 BN 请求待签名的区块模板或见证数据。
    
5.  **签名与广播**：VC 使用私钥对数据签名，并发回给 BN。BN 随后将签名后的成品广播到整个 P2P 网络。

EVM 在运行时管理着几种不同类型的数据存储区域，其生命周期和成本（Gas）各不相同：

-   **Stack（栈）**：
    
    -   用于存放操作数和中间计算结果。
        
    -   最大深度为 1024 层。每个元素是 256 位（32 字节）。
        
    -   成本最廉价，但只能直接访问栈顶附近的元素。
        
-   **Memory（内存）**：
    
    -   字节数组，用于临时存储数据（如传递给函数调用的参数）。
        
    -   **生命周期**：仅在单次交易执行期间存在，执行完即销毁。
        
    -   **扩展成本**：随内存使用量的增加而呈二次方增长（Memory Expansion Fee）。
        
-   **Storage（存储）**：
    
    -   持久化的键值对空间（32字节 Key -> 32字节 Value）。
        
    -   **生命周期**：永久存储在区块链状态中。
        
    -   **成本**：极其昂贵（读 SLOAD 和写 SSTORE 是 Gas 消耗大户）。
        
-   **Calldata**：
    
    -   只读的输入区域，包含交易发送的参数。
        
    -   比 Memory 便宜，是 L2 存储数据（Blob 普及前）的主要场所。

# 以太坊共识层 (CL) 概要笔记

## 1\. 共识层的核心角色

如果说执行层（EL）是“身体”，负责处理具体的交易逻辑；那么共识层（CL）就是“大脑”，负责决定哪些区块应该被接纳进主链。

-   **主要职责**：管理验证者（Validators）、协调区块的提议与见证（Attestation）、处理奖励与惩罚、以及确保链的**最终确认性（Finality）**。
    
-   **核心组件**：信标链（The Beacon Chain）。
    

## 2\. 时间体系：Slots 与 Epochs

共识层将时间严格划分为离散的单位，这是 PoS 协调成千上万验证者的基础：

-   **Slot（时隙）**：每 **12 秒** 一个 Slot。每个 Slot 都有机会产生一个新区块。
    
-   **Epoch（时段）**：由 **32 个 Slots** 组成（约 6.4 分钟）。
    
-   **重要性**：验证者的分配、奖励的结算和最终性的达成都是以 Epoch 为单位进行的。
    

## 3\. 验证者（Validators）与质押

-   **准入门槛**：用户需要质押 **32 ETH** 才能激活一个验证者。
    
-   **职责**：
    
    -   **Proposing（提议）**：被选中的验证者负责创建一个新区块。
        
    -   **Attesting（见证/投票）**：大多数验证者在每个 Epoch 都要对他们认为正确的区块头进行投票。
        
    -   **Sync Committees（同步委员会）**：一小组随机选出的验证者，负责支持轻客户端快速同步。
        
-   **激励机制**：
    
    -   **奖励**：正确提议和见证区块。
        
    -   **惩罚（Penalties）**：因离线导致未履行职责。
        
    -   **罚没（Slashing）**：针对恶意行为（如双重投票、包围投票）的严重惩罚，会导致质押金额被强制扣除并逐出网络。
        

## 4\. 共识协议：Gasper

以太坊的共识算法被称为 **Gasper**，它是两个协议的结合体：

1.  **LMD GHOST（分叉选择规则）**：
    
    -   在发生分叉时，根据验证者的投票权重（见证）来决定哪条链是“主链”。
        
    -   原则是“选择累积权重最高的链”。
        
2.  **Casper FFG（最终性机制）**：
    
    -   负责将区块标记为“已确认”。
        
    -   **Justification（理由达成）**：当 2/3 的验证者对某 checkpoint 达成一致。
        
    -   **Finalization（最终确定）**：当连续两个 checkpoint 达成 Justification 时，之前的区块即进入不可逆转的状态。
        

## 5\. 客户端架构

共识层客户端通常分为两个逻辑部分（有时在同一软件中）：

-   **信标节点 (Beacon Node, BN)**：
    
    -   维护信标链状态。
        
    -   通过 P2P 协议与其他节点通信（使用 libp2p）。
        
    -   通过 Engine API 与执行层通信。
        
-   **验证者客户端 (Validator Client, VC)**：
    
    -   持有私钥并进行签名。
        
    -   一个 VC 可以管理多个验证者（即多组 32 ETH）。
        

## 6\. 层间通信：Engine API

这是 CL 与 EL 协同工作的纽带。

-   **指令流**：CL 告诉 EL 什么时候该打包交易生成区块，或者让 EL 验证从网络收到的区块。
    
-   **执行流**：EL 执行完交易后，将结果（如收据、状态根）反馈给 CL，CL 再将其广播到共识网络

执行层是区块链的“大脑”，负责处理交易、管理状态以及运行智能合约。执行层规范定义了以太坊网络的**确定性行为**。

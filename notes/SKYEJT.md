**以太坊预编译合约 笔记**

**一、 什么是预编译合约？**

-   **定义**：预编译合约是以太坊网络中的一组特殊账户，内部包含了**原生内置的函数**（通常是极其复杂的密码学计算），并且具有协议明确规定的固定 Gas 成本。
    
-   **特殊地址**：它们的地址被硬编码在以太坊的最前面，目前范围是 **0x01 到 0x0a**。
    
-   **“伪装”的智能合约**：预编译合约不运行在以太坊虚拟机（EVM）内，而是直接由底层客户端软件（如 Geth, Reth）通过高效的本地代码实现。因此，如果你在 EVM 中去查它们的字节码大小（`EXTCODESIZE`），结果是 0。但在被调用时，它们的表现和普通的智能合约毫无二致。
    
-   **Gas 优惠**：为了节省费用，在 EIP-2929 的设定中，预编译合约默认被包含在交易的“已访问地址（accessed\_addresses）”列表（即“热地址”）中。
    

**二、 为什么要设计预编译？（为何不用 Opcodes？）**

在底层设计中，预编译合约和操作码（Opcodes）的目的都是执行计算，但以太坊选择了预编译来处理复杂算法，主要基于以下考虑：

1.  **操作码空间极其有限**：EVM 的操作码只有 1 个字节，最多只能容纳 256 个指令（`0x00` 到 `0xFF`），好钢必须用在刀刃上，不能随便给特定的密码学算法分配专属操作码。
    
2.  **极致的执行效率**：复杂的密码学运算如果用 Solidity 写成普通的 EVM 字节码，执行起来会极其缓慢且消耗天价 Gas。预编译合约在 EVM 外部用底层语言原生执行，效率极高。
    
3.  **安全与防范 DoS 攻击**：早期 Vitalik 曾设想过“原生合约（让矿工给执行快的合约降低 Gas 定价）”，但这种机制容易被攻击者利用植入后门并对网络发起 DoS 攻击。最终协议选择了退而求其次：由官方直接在协议层面硬编码少量极其常用的密码学操作**3**。
    

**三、 预编译合约的运作与调用机制**

-   **无缝调用**：对开发者而言，调用预编译合约就像调用其他外部智能合约一样，使用的是标准的 `CALL` 系列操作码（如 `STATICCALL` 等）。
    
-   **绝对可预测的 Gas 计费**：预编译合约的 Gas 消耗与输入数据直接挂钩（固定输入 = 固定 Gas）。系统通过严格的硬件基准测试来定价，防止恶意耗尽节点资源的 DoS 攻击。
    
-   **底层实现的双刃剑**：由于各家客户端调用的是高度优化的底层代码库，一旦这些库出现 Bug，可能会直接导致整个以太坊协议层崩溃。因此，新增任何预编译合约都需要极其严苛的测试。
    
-   **无嵌套调用**：为了堵住潜在的安全漏洞，预编译合约在设计上被禁止进行内部的“嵌套调用”。
    

**四、 现有的核心预编译合约 (举例)**

目前以太坊共有 10 个预编译合约，代表性的包括：

-   **0x01 (ECRECOVER)**：用于恢复椭圆曲线公钥（极常用的签名验证）。
    
-   **0x02 (SHA2-256)**：SHA2 哈希算法。
    
-   **0x0a (KZG POINT EVALUATION)**：坎昆升级（EIP-4844）刚引入的最新预编译，用于验证大区块数据 Blob 的 KZG 证明。
    

**五、 未来的演进：新增与淘汰**

-   **新增面临的阻力**：社区目前对增加新的预编译合约非常抗拒，因为每次新增都会带来庞大的测试表面和长期的维护负担，甚至会导致客户端代码臃肿。
    
-   **新增的解决策略**：未来的新密码学支持（如 EIP-2537 的 BLS12-381 或 RIP-7212 的 secp256r1）将倾向于**先在 Layer 2 (L2) 上进行原型设计和测试**，证明稳定并获得广泛采用后，再考虑合并到主网。学术界还在研究“渐进式预编译”，即先在 `CREATE2` 确定的地址上部署高效合约，未来再平滑升级为底层原生预编译。
    
-   **清理门户（移除机制）**：目前社区正在讨论**淘汰部分过时、利用率极低或拖慢节点效率的预编译合约**，如 `IDENTITY`（0x04，已被 `MCOPY` 操作码取代）、`RIPEMD-160` (0x03) 和 `BLAKE2` (0x09)。它们不会被直接删去功能，而是可能被“降级”重写为普通的智能合约代码（虽然这会让调用它们的 Gas 变贵，但能减轻底层协议的负担）。
    

**以太坊区块构建 笔记**

**一、 宏观分工：谁负责指挥？谁负责干活？**

-   **共识层 (CL) 的角色**：它是一个“指挥官”。在每个出块时间点（Slot），共识层会通过伪随机过程选出一个验证者（Validator）作为区块提议者。然后，CL 会通过 Engine API（具体是 `fork choice updated` 接口）给执行层下达构建区块的指令。
    
-   **执行层 (EL) 的角色**：它是一个“打工人”。收到指令后，它会从自己的交易池（Mempool）中捞取交易，组装并执行计算，最终造出一个包含实际数据的区块交还给 CL。
    
-   _注：验证者不一定非要自己构建区块，也可以从外部的专业“构建者（Builders）”那里获取利润最高的区块（这就是 PBS 提议者/构建者分离机制）。_
    

**二、 区块构建的“流水线”步骤 (Payload Building Routine)**

当 EL 收到构建指令后，会进行以下极其严密的流水线操作：

1.  **准备工作台（获取上下文）**：收集构建区块所需的“环境参数”，包括时间戳、区块号、父区块信息、基础费（Base fee）以及从共识层传来的提款记录（Withdrawals）等。
    
2.  **拉取交易**：从交易池中取出待处理的交易。为了让执行客户端实现利润最大化，这些交易通常是**按价值/收益从高到低排序的**。
    
3.  **放入 EVM 逐笔执行**：
    
    -   将环境参数、当前交易和最新状态输入到 EVM 中进行执行。
        
    -   **成功处理**：如果交易执行成功，将其加入区块的交易列表，并累加消耗的 Gas。
        
    -   **失败处理（关键点）**：如果在执行某笔交易时报错（说明这是一笔坏交易），**系统不会立刻终止整个区块的构建**，而是直接跳过这笔交易，继续从池子里拿下一笔交易来试。
        
4.  **触达上限**：系统会不断重复第 2、3 步，直到交易池空了，或者**消耗的 Gas 达到了当前区块的 Gas 上限**（例如主网目前的上限约为 3000 万 Gas）。
    
5.  **最终结账与盖章**：处理完所有交易后，系统会进行最后结算。通过对列表进行“默克尔化”，计算出交易根（Transactions root）、回执根（Receipts root）和提款根（Withdrawals root），并把它们填入区块头。
    

**三、 进阶解析：Geth 客户端源码级的小秘密 (Code walk-through)**

以太坊主流客户端 Geth 在底层实现这套流水线时，还有几个非常精妙的设计：

-   **“保底”的空区块设计**：Geth 收到构建指令后，首先会**立刻创建一个“空区块”**。这是为了确保节点无论如何都有东西可以提交，绝对不会错过出块时间（Miss the slot）。随后，它会在后台开启一个异步线程（Go routine），并发地往这个空区块里尽力塞入真实的交易。
    
-   **Blob 交易的特殊关照**：在从内存池抓取交易并打包时，不仅要看普通交易，还要处理 EIP-4844 引入的 Blob 数据，系统会严格检查每区块允许容纳的 Blob 数量上限。
    
-   **最终交付**：当区块在后台塞满并执行完毕后，共识层（CL）会调用 `getPayload` 接口，把这个装满交易的执行负载拿走，塞进信标链区块里，最后广播给全网。

**以太坊 JSON-RPC笔记**

**一、 什么是 JSON-RPC？**

-   **定义**：它是一种基于 OpenRPC 并使用 JSON 编码的远程过程调用协议。
    
-   **核心作用**：它允许在远程服务器上调用函数并返回结果。无论是普通用户/钱包与以太坊网络的交互，还是以太坊内部**共识层 (CL) 与执行层 (EL) 之间的通信（通过 Engine API）**，都依赖它。
    
-   **请求结构**：所有 JSON-RPC 请求都必须包含四个标准字段：`id`（唯一标识符）、`jsonrpc`（协议版本）、`method`（要调用的方法名）以及 `params`（传递的参数数组）。
    

**二、 API 命名空间 (Namespaces)**

JSON-RPC 的方法是通过不同的前缀（命名空间）来分类的，各个节点客户端必须实现规范要求的基础方法。以下是几个最核心的命名空间：

**1\. Eth 命名空间 (eth\_)**

-   **定位**：最核心、最常用的 API，允许用户与以太坊直接交互。
    
-   **常见方法**：获取区块号 (`eth_blockNumber`)、查询余额 (`eth_getBalance`)、获取 Gas 价格 (`eth_gasPrice`)、执行不上链的模拟调用 (`eth_call`) 等。
    

**2\. Debug 命名空间 (debug\_)**

-   **定位**：提供检查以太坊底层状态和原始数据的能力（如 `debug_getRawBlock`）。
    
-   **限制**：由于这些操作极其消耗节点的计算资源，公共 RPC 提供商通常会限制或直接禁用这个命名空间。
    

**3\. Engine API (engine\_)**

-   **定位**：**专供共识层和执行层进行内部通信**的专属通道，不对外部普通用户开放。
    
-   **安全机制**：它运行在一个独立的端点上，并通过 **JWT (JSON Web Token)** 进行身份验证，防止恶意外部干扰。
    
-   **方法特征**：它的方法（如 `engine_forkchoiceUpdatedV1`、`engine_getPayloadV1`）通常带有版本号（如 V1），以便向后兼容未来的网络升级。
    

**4\. 其他重要命名空间**

-   **admin\_**：用于配置节点，属于**高敏感**权限接口。
    
-   **txpool\_**、**net\_**、**web3\_**：分别用于检查交易池、获取网络信息以及 web3 客户端工具函数。
    

**三、 数据编码规范 (Encoding)**

-   **统一格式**：JSON-RPC 方法的参数统一采用**十六进制 (Hex) 编码**，并且必须带有 `0x` 前缀。
    
-   **细节规则**：数字 0 表示为 `0x0`，65 表示为 `0x41`。对于数量类型，前缀后通常不允许有多余的前导零（例如不能写成诸如带无效前导零的格式）。
    

**四、 传输协议无关性 (Transport Agnostic)**

JSON-RPC 并不绑定于某一种特定的传输网络，它可以在以下三种主要传输协议上运行：

1.  **HTTP**：最基础的单向“请求-响应”模型，服务器返回响应后立刻关闭连接。
    
2.  **WSS (WebSockets)**：支持**双向持续通信**。连接会一直保持打开，非常适合需要**订阅事件**（如监听新区块或新交易）的场景。
    
3.  **IPC (进程间通信)**：用于同一台物理机器上的进程相互通信。它的速度最快，但不具备远程通信能力。
    

**五、 开发者工具 (Tooling)**

开发者有多种方式来调用 JSON-RPC：

-   **底层调用**：可以直接使用终端的 `curl` 命令，或者用 `axios` 在请求体 (Body) 中包裹参数发送 HTTP POST 请求。
    
-   **Web3 库（最推荐）**：由于手写 JSON-RPC 请求非常繁琐，主流开发中最常见的是使用封装好的 Web3 库，比如 Python 的 `web3py`，或者 JS/TS 的 `web3.js` 和 `ethers.js`。它们在底层替你完成了 JSON-RPC 的封装，提供了极其友好的高级函数接口。
    

**以太坊 EOF (EVM Object Format) 核心笔记**

**一、 什么是 EOF？**

-   **目前的痛点**：当前的 EVM 字节码只是一长串毫无结构的“指令流水账 (unstructured sequence of instructions)”。
    
-   **EOF 的定义**：EOF 引入了\*\*“容器 (container)”**的概念，它是一种**可扩展、带版本号\*\*的结构化 EVM 字节码格式。
    
-   **核心机制**：它在智能合约**部署时 (deploy time)** 进行一次性的彻底验证，这使得 EVM 在未来解析和分析代码时变得更加容易和高效。
    

**二、 EOF 的四大核心优势 (Purpose and Benefits)**

引入 EOF 主要是为了解决传统 EVM 字节码格式的诸多限制：

1.  **更完善的验证 (Better Validation)**：允许在合约真正执行之前，对其进行更彻底的静态分析和代码验证。
    
2.  **提升执行效率 (Improved Execution Efficiency)**：通过将代码划分成清晰的区块（sections），极大地方便了 EVM 执行引擎的底层优化。
    
3.  **增强安全性 (Enhanced Security)**：通过强制执行更严格的代码结构规则，封堵了许多潜在的攻击面（漏洞）。
    
4.  **面向未来 (Future-Proofing)**：带版本号的容器设计，为以太坊未来持续添加新功能和改进 EVM 奠定了坚实的基础。
    

**三、 新旧对比：传统 EVM vs. EOF (核心考点)**

| 特性维度 | 传统 EVM (Legacy EVM) | 新版 EOF |
| --- | --- | --- |
| 容器结构 | 非结构化的一团字节码 | 拥有明确定义的独立区块 (sections) |
| 代码验证 | 非常有限，且往往在运行时报错 | 全面彻底，且在执行前预先验证 |
| 跳转机制 | 动态跳转 (使用 JUMP / JUMPI) | 引入了静态跳转 (Static jumps) |
| 类型系统 | 无任何类型概念 | 引入了函数签名和类型系统 |
| 数据处理 | 数据与执行代码混杂在一起 | 拥有完全独立的数据段 (Separate data section) |

**四、 相关的核心改进提案 (EIPs)**

EOF 的升级非常庞大，因此它被拆分成了多个子提案，并由一个**元提案 (Meta EIP) EIP-7692 统一打包追踪**。

其中最关键的两个子提案包括：

-   **EIP-3540**：它是 EOF 的基石，引入了 **EOF 第 1 版 (Version 1)**，正式确立了结构化字节码的初始框架。
    
-   **EIP-7620**：改变了合约的部署方式。它**移除了使用传统 CREATE 或 CREATE2 指令来创建合约的途径**，取而代之的是引入了一对全新的指令：**EOFCREATE 和 RETURNCODE**，专门用于处理 EOF 容器合约的创建。

**RLP (递归长度前缀) 笔记**

**一、 什么是 RLP (Recursive Length Prefix)?**

-   **定义**：RLP 是以太坊执行层（EL）中用于编码和解析数据的核心序列化协议。
    
-   **作用**：将各种数据结构或对象转换为标准的“字节流（Byte Stream）”，以便在分布式网络节点间可靠地传输、存储，并确保所有客户端软件能以完全相同的方式解析数据（涵盖了从交易数据到整个区块链状态的序列化）。
    

**二、 RLP 的设计哲学（为什么不直接用 JSON 或 Protobuf？）**

-   **极度极简**：RLP 不尝试定义任何具体的数据类型（比如不区分布尔值、浮点数或整数），它的唯一目的就是存储“嵌套的字节数组”。至于这些字节具体代表什么意思，完全交给上层协议自己去解释。
    
-   **绝对的确定性**：它保证了字节级别的完美一致性（byte-perfect consistency）。在区块链这种要求所有节点计算结果必须绝对一致的系统中，这种确定性是必不可少的。
    

**三、 RLP 的核心编码规则 (Encoding Rules)**

RLP 根据数据的类型（单个字节、字符串或列表）和大小，应用了极其严谨的前缀规则：

-   **单字节 (值在 0x00 到 0x7F 之间)**：直接照原样编码，不加任何前缀。
    
-   **短字符串 (长度 1 - 55 字节)**：**前缀 = 0x80 + 字符串的长度**，前缀后紧跟字符串本身。例如，长度为3的字符串 "dog"，前缀就是 `0x83` (`0x80` + 3)。
    
-   **长字符串 (长度 > 55 字节)**：**前缀 = 0xb7 + 记录长度所需要的字节数**。然后跟上具体的长度值，最后跟上字符串的字节。
    
-   **短列表 (内部元素总长度 1 - 55 字节)**：**前缀 = 0xc0 + 列表内所有编码项的总长度**。然后紧跟列表内的编码数据。
    
-   **长列表 (内部元素总长度 > 55 字节)**：**前缀 = 0xf7 + 记录总长度所需要的字节数**。然后跟上具体的长度值，最后跟上列表内的编码数据**8**。
    
-   **特殊空值**：空字符串、Null 和 False 统一编码为单字节 `0x80`；空列表 `[]` 编码为 `0xc0`。
    

**四、 RLP 的解码规则 (Decoding Rules)**

解码完全依赖于读取到的**第一个字节（即前缀）**，它就像一把尺子，直接告诉系统接下来是什么数据：

-   **\[0x00 - 0x7F\]**：数据本身，单字节。
    
-   **\[0x80 - 0xB7\]**：这是一个短字符串。字符串长度 = 第一字节减去 `0x80`。
    
-   **\[0xB8 - 0xBF\]**：这是一个长字符串。读取后续几个字节来获取长度（获取几个字节取决于第一字节减去 `0xB7`）。
    
-   **\[0xC0 - 0xF7\]**：这是一个短列表。列表总长度 = 第一字节减去 `0xC0`，然后递归地去解码里面的元素。
    
-   **\[0xF8 - 0xFF\]**：这是一个长列表。读取后续几个字节来获取总长度（取决于第一字节减去 `0xF7`），然后递归解码。
    

**五、 补充进阶：RLP 的局限与未来 (RLP vs SSZ)**

-   虽然 RLP 简单且确定，但它对固定长度数据类型的处理效率低下，并且由于缺乏快速索引（复杂度为 _O_(_N_)），**它不支持默克尔化 (Merkleization)**，这使得轻客户端的证明变得极其困难。
    
-   为了解决这个问题并实现无状态化目标，以太坊 2.0（共识层/信标链）引入了更现代的序列化格式——**SSZ (Simple Serialize)**，以弥补 RLP 的不足。
    

**DevP2P 笔记**

**一、 DevP2P 的“双栈”架构** 以太坊执行层的网络通信为了兼顾效率与可靠性，被划分为两个并行运行的独立协议栈：

-   **节点发现协议栈 (Discovery)**：负责在茫茫网络中寻找其他对等节点（Peers），它运行在 **UDP** 协议上，因为寻找节点不需要建立长连接，UDP 速度更快。
    
-   **信息传输协议栈 (Transport)**：负责节点间实际的交易、区块数据收发，它运行在 **TCP** 协议上，提供面向连接、不丢包、按顺序到达的可靠传输。
    

**二、 节点发现机制 (Discv 协议 - 基于 UDP)** 节点寻找彼此的过程基于类似于 Kademlia 的分布式哈希表（DHT）算法，整个网络由源码中硬编码的“引导节点 (bootnodes)”作为起步入口。

-   **路由表与 K-Buckets**：每个节点会维护一个路由表，里面包含多个“k-buckets”，每个桶最多记录 16 个距离相近的节点。列表按最后活跃时间排序，如果某个节点超过 12 小时无响应，就会被踢出路由表。
    
-   **基础查找流程**：新节点向引导节点发送 `PING`，对方回复 `PONG` 完成验证；接着发送 `FIND-NEIGHBOURS` (在 Discv4 中叫 FindNode) 请求获取其他邻居节点的列表，然后循环此过程。
    
-   **协议升级：Discv4 vs Discv5**
    
    -   **Discv4**：早期的发现协议，节点身份由 secp256k1 密钥对标识，但信息是明文传输的，安全性较低。
        
    -   **Discv5（当前主流）**：进行了巨大的安全升级。它全程使用 **AES-GCM 加密**通信，引入了 `WHOAREYOU` 质询响应机制来防止节点身份被伪造（欺骗）**10more\_horiz**。此外，它还支持“主题广播 (Topic Advertisement)”，允许节点在网络中快速寻找提供特定服务（如轻客户端数据）的其他节点。
        

**三、 节点名片：ENR (以太坊节点记录)**

-   **定义**：ENR (Ethereum Node Record) 是一种标准的节点连接信息数据包（EIP-778），里面封装了该节点的 IPv4/IPv6 地址、TCP/UDP 端口号、公钥以及记录版本号。
    
-   **展现形式**：它通常被 RLP 序列化后编码成一段以 `enr:` 开头的 base64 长字符串。除了 ENR 这种现代格式，以太坊节点也能识别早期的 `multiaddr` 和 `enode://` 格式的节点 URL。
    

**四、 RLPx 传输协议 (Transport - 基于 TCP)** 当节点互相发现后，它们就会使用 RLPx 协议建立安全的 TCP 专属连接。

-   **极致的安全与“前向保密 (Forward Secrecy)”**：在握手阶段，节点不会直接用自己长期的身份私钥进行加密交互，而是**每次连接都动态生成一对临时的 (ephemeral) 密钥**，并通过 ECDH 算法算出共享密钥**more\_horiz**。一旦会话结束，临时密钥立刻销毁。这种设计保证了“前向保密”——即使未来某个节点的长期私钥被黑客盗取，黑客也绝对无法解密该节点过去的通讯记录。
    
-   **消息封装与防篡改**：所有通信内容都被 AES-128-CTR 算法加密，并且每个数据帧都带有 MAC（消息认证码）来确保数据在传输中没有被篡改**20more\_horiz**。
    
-   **多路复用 (Multiplexing)**：一条 RLPx 连接支持“多路复用”，这意味着两个节点可以在同一根网线上，同时并行跑多个不同的业务协议。
    

**五、 应用层子协议 (Application-Level Subprotocols)** 在搭建好上述极其安全的 RLPx 传输通道后，以太坊在这个通道上跑了不同的“子业务”：

-   **eth (以太坊线上传输协议)**：最核心的协议，用于在全网广播和交换区块链基础数据（比如同步区块头、区块体、收发新交易）。
    
-   **snap (快照同步协议)**：专门用于节点的高效状态同步，允许节点分块极速下载以太坊的状态树数据。
    
-   **les (轻以太坊子协议)**：专门为轻客户端设计的协议，让轻客户端不需要下载完整状态就能向全节点请求特定的数据。
    
-   **portal (门户网络)**：一种新型的去中心化检索网络，同样旨在支持轻量级客户端按需获取状态、区块和交易数据。

**Transaction笔记**

**一、 什么是交易？** 交易是由**外部账户**发出的、经过密码学签名的指令，并通过 JSON-RPC 广播到整个以太坊网络中。

**二、 交易的核心字段 (Fields)** 一笔标准的交易包含以下关键信息：

1.  **nonce (交易计数器)**：发送方已发送交易的数量。它有三个极其重要的作用：
    
    -   **防重放攻击**：如果有人截获你的交易并反复广播，EVM 会因为 nonce 重复而直接拒绝。
        
    -   **决定合约地址**：在创建合约时，新地址是由发送者地址和 nonce 共同计算得出的。
        
    -   **替换交易（加速或取消）**：可以使用相同的 nonce 发送一笔 Gas 更高的新交易，来覆盖掉之前卡住的旧交易。
        
2.  **gasPrice (Gas 价格)**：每单位 Gas 你愿意支付的金额（单位是 Wei）。出价越高，越容易被优先打包进区块。
    
3.  **gasLimit (Gas 上限)**：这笔交易最多允许消耗的 Gas 数量。如果执行时把这些 Gas 烧光了，交易就会被强行终止。
    
4.  **to (接收地址)**：20 字节的目标地址。**这个字段直接决定了交易的类型（见下一条）**。
    
5.  **value (金额)**：要转账的以太坊数量（单位是 Wei）。如果是创建合约，这就作为新合约的初始余额。
    
6.  **data / init (附加数据)**：不限长度的字节数组。如果用来创建合约，它就是初始化代码 (init)；如果用来调用合约，它就是具体的指令数据 (data)。
    
7.  **Signature (签名)**：包含 v、r、s 三个值的 ECDSA 密码学签名，用于证明是你本人发出的。
    

**三、 交易的三种主要模式** 交易具体用来干什么，完全取决于 `to` 字段里填了什么：

-   **创建合约 (Contract creation)**：当 `to` 字段为**空 (Empty)** 时触发。此时交易携带的 `init` 代码会被执行一次，执行后返回的才是真正永久保存在链上的“运行时字节码 (runtime bytecode)”。
    
-   **普通转账 (Value transfer)**：当 `to` 字段是一个**普通的外部账户地址**时触发。
    
-   **执行合约 (Contract execution)**：当 `to` 字段是一个**智能合约地址**时，交易就会去触发该合约内部的代码逻辑。
    

**四、 交易回执 (Receipts)** 不管交易是成功还是报错，执行完后都会生成一张“回执单”，它包含了 5 个关键信息：

1.  **交易类型 (Transaction Type)**：区分是传统交易还是新型交易。
    
2.  **状态 (Status)**：**1** 代表执行成功，**0** 代表执行失败。
    
3.  **消耗的 Gas (Gas Used)**：记录真实花掉的 Gas 数量。
    
4.  **日志 (Logs)**：智能合约在执行过程中发出的事件数据。
    
5.  **布隆过滤器 (Logs Bloom)**：一个 256 字节的过滤器，专门用来帮助外部 DApp 或钱包极速检索这笔交易里有没有他们关心的日志或地址。
    

**五、 进阶：类型化交易 (Typed Transactions - EIP-2718)** 为了能在未来方便地增加新的交易功能（比如 EIP-1559 费用机制或 EIP-4844 的 Blob 数据），以太坊引入了“类型化信封”的设计：

-   **信封格式**：`交易类型 (1字节) + 交易负载数据`。
    
-   **如何向后兼容？**：如果一笔交易数据的第一个字节在 `[0x00, 0x7f]` 之间，系统就知道这是一笔**新型的类型化交易**；如果第一个字节 `>= 0xc0`，系统就知道这是一笔按老规矩 (RLP) 编码的**传统遗留交易**。回执（Receipts）也同样采用了这套信封结构。
    

**以太坊数据结构**

以太坊需要保存庞大的当前状态和历史区块数据。为了实现既能高效存储，又能安全验证，以太坊采用了特定的树状数据结构。

**一、 基础概念：两种树的结合**

要理解以太坊的数据结构，首先要了解它背后的两个基础模型：

1.  **Merkle Tree（默克尔树）—— 主攻“防篡改验证”**
    
    -   它是一种自下而上构建的二叉树（必须有偶数个叶子节点）。
        
    -   最底层的叶子节点保存数据，上面的节点保存下方节点的哈希值，直到顶部生成唯一的 **Merkle Root（根哈希）**。
        
    -   这个根哈希存放在区块头中。由于哈希的“雪崩效应”，哪怕最底层的数据被篡改了一点点，最终的根哈希也会完全不同，因此极度安全。
        
2.  **Patricia Tree（帕特里夏树/基数树）—— 主攻“高效存储”**
    
    -   不同于 Merkle 树为了验证而生，它是为了高效存储和检索键值对。
        
    -   它最大的特点是**前缀压缩**：如果多个数据有相同的路径前缀，它会把它们合并到一条边上，并且真实数据只存放在最底层的叶子节点中，极其节省空间。
        

**二、 以太坊的终极武器：MPT (Merkle Patricia Trie)**

以太坊将上述两者的优点结合，创造了 **MPT（默克尔帕特里夏树）**，这也是以太坊执行层状态存储的核心结构。 MPT 包含三种关键的节点类型：

-   **分支节点 (Branch Nodes)**：主要用来岔路导航，包含 16 个分支和 1 个值域（共17个元素）。
    
-   **扩展节点 (Extension Nodes)**：起压缩作用。当一条路径只有单行道（一个子节点）时，系统会用它把路径压缩起来，避免浪费空间。
    
-   **叶子节点 (Leaf Nodes)**：旅途的终点，存放最终的键值对数据。
    

**三、 以太坊的四大核心状态树 (The 4 Ethereum Tries)**

以太坊在运行中，实际上维护着四种不同用途的修改版 MPT：

**1\. 交易树 (Transaction Trie)**

-   **作用**：记录某个具体区块内打包的所有交易。
    
-   **特点**：每个区块都有自己独立的一棵交易树。一旦区块敲定，这棵树就**永远无法更改**。
    
-   **存储方式**：通过“交易在区块中的索引（位置）”作为键，来查找具体的交易内容。
    

**2\. 回执树 (Receipt Trie)**

-   **作用**：记录交易执行后的结果和“凭证”（如交易状态、消耗的 Gas、产生的日志和布隆过滤器）。
    
-   **特点**：同样是区块级别的树，键也是交易索引。它的存在使得轻节点不需要重新运算整笔交易，也能通过根哈希快速验证交易的执行结果。
    

**3\. 世界状态树 (World State Trie)**

-   **作用**：以太坊的“总账本”，代表了以太坊当前所有账户的最新状态。
    
-   **特点**：这是一棵**持续生长的活树 (living structure)**，随着区块的推进不断更新。
    
-   **存储内容**：将账户的地址哈希作为键，对应存放账户的 4 个核心信息：**Nonce（计数器）、Balance（余额）、Code Hash（合约代码哈希）以及 Storage Root（该账户专属存储树的根）**。
    

**4\. 存储树 (Storage Trie)**

-   **作用**：专门用来保存智能合约内部的持久化数据（变量）。
    
-   **特点**：**每个智能合约账户都拥有一棵自己专属的存储树**，它的入口（根哈希）就记录在上述“世界状态树”的账户信息里。如果是普通人的钱包地址（外部账户），这棵树就是空的。
    

**四、 属于未来的数据结构：Verkle Trees (维克尔树)**

-   **面临的危机**：随着时间推移，以太坊目前的 MPT 状态树变得越来越庞大（目前约 1-2TB），普通电脑根本存不下，这阻碍了网络的扩展。
    
-   **解决方案**：以太坊路线图计划用 **Verkle 树** 来彻底替换 MPT。
    
-   **核心优势**：它结合了“向量承诺”技术，树的结构变得“更宽更矮”（每个节点可容纳 256 个子分支）。这使得网络在传输状态证明（witness）时，**数据体积能大幅缩小**（比如 1000 个叶子节点的数据证明能从 4MB 锐减到 150kB），这是未来实现“无状态客户端 (stateless clients)”的基石。

**理清楚EVM**

**1\. 到底什么是 EVM？（全球共享的“虚拟电脑大脑”）**

以太坊网络是由全球成千上万台不同品牌、不同系统的电脑组成的（比如有的是苹果 Mac，有的是 Windows，底层的芯片硬件各不相同）。如果直接让大家各自去运行智能合约，得出的结果很可能五花八门。

为了解决这个问题，以太坊创造了 EVM，它本质上是一个**虚拟出来的 CPU（中央处理器）**。它的核心作用就是提供一个绝对标准的环境，确保所有的智能合约代码，无论在世界上哪台机器上运行，算出来的结果都是百分之百一致的（这就叫确定性）。

**2\. EVM 的核心任务是什么？（账本更新器）**

以太坊本质上是一个巨大的“状态机”，你可以把它理解为一个“全球超级大账本”。EVM 的唯一工作就是处理用户发起的交易，然后根据智能合约的规则，去修改这个大账本上的数据（比如把张三账户里的钱扣掉，加到李四账户里）。

**3\. EVM 是怎么干活的？（临时加工厂）**

当一笔交易进来，需要运行智能合约时，EVM 就像是一个极其严谨的加工厂，它会立刻在内部为你搭建一个专属的“临时工作台”（官方称为“机器状态”）。

这个工作台上只有四件核心工具：

-   **Gas 计数表（燃料表）**：EVM 里没有免费的午餐，执行任何一个微小的动作都要扣除 Gas 燃料费。这个表实时记录你还剩多少燃料可以挥霍。
    
-   **程序计数器（监工的手指）**：就像一个死板的监工，一行一行指着代码，告诉机器“现在正在执行第几步”。
    
-   **堆栈（算盘/叠盘子）**：这是 EVM 最核心的计算区。你可以把它想象成“叠盘子”，每次最多只能放 1024 个盘子。EVM 只能对最上面的盘子进行操作，比如要做加法，它就会把最顶部的两个数字拿出来，加在一起，然后把写着结果的新盘子再放回顶端。
    
-   **内存（临时黑板）**：一块用来记录复杂数据的临时黑板**8**。等这笔交易处理完，黑板马上就会被擦得一干二净。
    

**4\. 运行的大结局（成功还是撤销？）**

机器在工作台上一步步往下算，最后只会迎来两种结局：

-   **完美收工（正常停机）**：遇到了“停止（STOP）”或“返回（RETURN）”指令，说明代码顺利跑完了。此时，EVM 会把你没用完的 Gas 退给你，并将修改后的最终数据盖章，永久记入以太坊的历史账本中。
    
-   **时光倒流（异常回滚）**：如果在中途发现钱（Gas）扣光了，或者代码遇到了错误指令（REVERT），EVM 会立刻拉响警报并强行终止。此时最残酷的是：**它会没收你已经花掉的 Gas 费，并且把刚才计算过程中对账本做的所有修改“一键撤销”，恢复到执行前的模样，仿佛这笔交易从未发生过**。
    

**总结一下：** EVM 就是一台拥有绝对权威的“盲盒计算器”。你给它输入指令并交足手续费，它就在内部用“叠盘子”的方式一板一眼地计算，最后严谨地吐出新的账本结果。绝对公平、给钱才干活、出错就撤销。

# 2026-04-10
<!-- DAILY_CHECKIN_2026-04-10_START -->









今天开始真正从EL层面开始入门

作为以太坊执行层（Execution Layer）的核心工作，状态转移函数其实只需要回答两个最根本的问题**1**：

1.  这个新打包好的区块，到底能不能合法地添加到区块链的末尾？
    
2.  如果把它加进去了，大家账户里的钱和数据（即“状态”）会发生什么变化？
    

为了让你更好理解，官方资料里用了一个非常经典的数学公式来概括这个过程：_σt_+1​≡Π(_σt_​,_B_)。 别被公式吓到，我们把它翻译成大白话就是：**新状态 = 运作规则(旧状态, 新区块)**。

-   _σt_​**（旧状态）**：代表在处理新交易**之前**，那个“超级大账本”当前的模样（比如张三有5块钱，李四有3块钱）。
    
-   _B_**（新区块）**：代表一个装满最新交易的“包裹”（比如里面有一笔记录：张三给李四转了2块钱）。
    
-   Π**（状态转移函数）**：代表以太坊的“运作规则”或处理引擎。它负责打开这个包裹，按照既定规则处理里面的每一笔交易。
    
-   _σt_+1​**（新状态）**：代表处理完包裹里所有交易**之后**，“超级大账本”更新后的最新模样（变成张三剩3块钱，李四变成5块钱）。
    

需要注意的是，以太坊的“状态”并不是像一个文本文档那样保存在某个具体的文件夹里，而是通过运行这些交易后**动态计算推导**出来的。

**那么，这个“运作规则”具体是如何一步步干活的呢？** 根据执行层的代码规范，处理一个新区块（执行状态转移函数）就像是工厂流水线上的质检加加工过程，主要包含以下几个关键步骤：

1.  **核对历史记录**：首先找出区块链上的上一个区块（父区块），看看账本目前处于什么进度。
    
2.  **严格验明正身（Header Validation）**：检查新区块的“区块头”（Header）。比如要核对它附加的数据（如 blob gas）对不对，并将它的参数与父区块进行对比验证，确保它没有伪造或违反基本规则。
    
3.  **清空多余数据**：检查并确保区块中名为 ommers（以前叫叔区块）的字段是空的。
    
4.  **逐一执行交易（Block Execution）**：这是最核心的一步。系统会把区块里打包的交易一笔一笔地执行。执行完后，系统会得出几个结果：这批交易总共消耗了多少 Gas（手续费）、所有交易的最终日志（Logs Bloom）、所有数据的“根目录”（Trie Roots），以及最重要的——**执行完所有交易后的“新状态（State）”**。
    
5.  **最终复核**：系统会把自己刚刚算出来的“新状态”的根哈希值等参数，跟新区块自带的摘要信息进行对比。如果两者完美吻合，说明计算无误。
    
6.  **盖章入库**：如果上述所有检查和计算都顺利通过，这个新区块就会被正式挂到区块链上。
    
7.  **如果发现错误**：在任何一个环节（比如某笔交易不合法，或者算出来的结果对不上），系统就会直接报错抛出“无效区块（Invalid Block）”，并拒绝更新账本。

学习的内容主要围绕**以太坊协议中的共识层（Consensus Layer, CL）架构**及其核心机制展开。以下是核心知识点的详细总结：

**1\. 核心共识机制：Gasper 协议**

以太坊的权益证明（PoS）共识被称为 **Gasper**，它巧妙地结合了两个独立的子协议，以在工程上实现平衡：

-   **LMD GHOST**：侧重于提供**活跃度（Liveness）**。它通过统计验证者的投票数选出主链，确保系统不断产生新区块、保持链向前运行。
    
-   **Casper FFG**：侧重于提供**安全性（Safety）**。它作为检查点机制，通过定期对区块进行最终确认（Finality），防止区块链发生大规模的历史回滚。
    

**2\. 安全性与活跃度的权衡**

-   **CAP 定理限制**：在不可靠的网络环境中，分布式系统无法同时完美保证安全性（不发生坏事）和活跃度（好事总会发生）。
    
-   **以太坊的取舍**：在面临严重的网络故障或分区时，以太坊**优先保证活跃度**。这意味着，即使 Casper FFG 暂时无法完成区块的最终确定，LMD GHOST 也会让网络继续运行、处理用户交易，即使这会带来潜在的链重组安全风险。
    

**3\. 分叉选择规则与链重组（Reorgs）**

-   **分叉选择规则**：由于网络延迟或消息乱序，节点会产生分歧。分叉选择规则通过为候选区块评分，帮助节点选出最有可能成为最终规范链的“头区块（Head block）”。
    
-   **区块链重组（Reorgs）**：当节点接收到新信息重新评估后，如果发现最佳头区块在另一条分支上，就会执行重组——回滚之前的区块，并切换到新头区块所在的分支。由于网络延迟，一两个区块的短暂重组很常见；而长重组则极为罕见，通常是由网络攻击或软件漏洞引发。
    

**4\. 节点与客户端架构**

-   **双层架构**：每个以太坊节点都同时运行着\*\*执行层（EL）**客户端和**共识层（CL）\*\*客户端。两者并行运行，并通过本地的 **Engine-API** 进行通信协同。
    
-   **共识层关键组件**：
    
    -   **信标节点（Beacon Node）**：负责维持共识链、处理区块、以及与其他对等节点通信。
        
    -   **验证者（Validator）**：需要质押 32 ETH，负责在系统中提议和验证区块（代替了传统 PoW 中的矿工）。
        

**5\. 状态转换机制**

-   区块链依赖纯粹的“状态转换函数”来更新节点视图（通过输入旧状态和新区块，计算出新状态）。
    
-   不同于 PoW 机制（由区块驱动），以太坊信标链的状态转换是\*\*由时隙驱动（slot-driven）\*\*的。它定义了每个时隙（per-slot）、每个区块（per-block）和每个纪元（per-epoch）的状态更新规则。任何导致数据溢出或未处理异常的转换都会被视为无效。

**RLP（递归长度前缀序列化）**

**RLP (Recursive Length Prefix)** 是以太坊执行层中用于编码和解析数据的核心序列化协议。它的主要作用是将数据结构或对象（如交易数据和整个区块链状态）转换为字节流，以确保不同语言编写的客户端软件都能以相同的标准格式可靠、高效地读取和传输数据。

-   **设计理念与特点**：RLP 的设计极其极简，它的唯一目的是存储嵌套的字节数组结构，而不试图定义具体的布尔值、浮点数或整数等数据类型，这些数据的具体含义交由以太坊协议本身来解释。这种设计能够保证跨客户端的字节级别完美一致性，这对于区块链操作所需的确定性至关重要。
    
-   **核心编码规则**：
    
    -   **单字节**：如果输入是单字节且值在 `0x00` 到 `0x7F` 之间，则直接进行编码且保持不变。
        
    -   **短字符串（1-55字节）**：前缀为 `0x80` 加上字符串长度，后跟字符串本身。
        
    -   **长字符串（>55字节）**：前缀为 `0xb7` 加上表示长度的字节数，后跟大端格式的字符串长度，最后是字符串内容的字节。
        
    -   **短列表（总负载1-55字节）**：前缀为 `0xc0` 加上列表编码后所有项的总长度，后跟被依次编码的列表项。
        
    -   **长列表（总负载>55字节）**：前缀为 `0xf7` 加上表示长度的字节数，后跟大端格式的负载长度，最后是列表项的内容。
        
    -   **特殊值**：空字符串、Null 和 False 统一编码为单字节 `0x80`；空列表编码为 `0xc0`。
        
-   **解码过程**：解码器通过检查数据的第一个字节（前缀）落在哪个范围（如 `0x00-0x7F`、`0x80-0xB7`、`0xC0-0xF7` 等）来确定后续数据的类型（单字节、字符串还是列表）以及长度，并通过递归的方式对列表内的元素进行解析。
    

**EOF（EVM 对象格式）**

**EOF (EVM Object Format)** 是一种针对以太坊虚拟机（EVM）字节码的可扩展且带有版本号的容器格式。目前的 EVM 字节码是一串无结构的指令序列，而 EOF 引入了“容器”的概念，为字节码带来结构化设计，使其更易于被 EVM 解析和分析，并在部署时会进行一次性的静态验证。

-   **主要目的与核心优势**：
    
    -   **更好的验证**：在智能合约执行前，可以进行更彻底的静态分析和全面验证。
        
    -   **提高执行效率**：通过划分定义明确的代码段（sections），促进 EVM 执行过程的优化。
        
    -   **增强安全性**：强制执行更严格的代码结构规则，从而减少攻击向量。
        
    -   **面向未来**：通过带版本号的容器，为未来 EVM 的进一步升级和添加新功能打下基础。
        
-   **与传统（Legacy）EVM 字节码的区别**：
    
    -   **结构**：传统方式是无结构的字节码，EOF 则是拥有明确定义的区段。
        
    -   **验证**：传统方式仅有有限的运行时验证，EOF 提供全面的预执行验证。
        
    -   **跳转机制**：传统方式使用动态跳转（`JUMP` / `JUMPI`），EOF 则包含了静态跳转。
        
    -   **类型系统**：传统方式无类型概念，EOF 支持函数签名和类型定义。
        
    -   **数据处理**：传统方式将数据与代码混合存放，EOF 引入了独立的数据段将代码与数据隔离。

# Ethereum 执行层网络 —— 一次 DevP2P 的完整理解

当我第一次学习 Ethereum 的时候，我以为节点之间的通信是一件“理所当然”的事情。我以为只要启动一个节点，它就会自动连接到网络、自动同步区块、自动传播交易。但随着我不断深入，我逐渐意识到，这种理解是完全不够的。一个执行层节点并不是天然就属于网络的一部分，它必须通过一整套过程，才能真正成为网络中的成员。

于是，我开始试着用“我”的视角去理解：如果我就是一个刚刚启动的执行层节点，我到底经历了什么，才真正进入 Ethereum 网络？

当我启动节点的那一刻，我其实只是一个孤立的存在。我加载了自己的配置，读取了本地的节点私钥，并生成了对应的公钥。这个公钥就是我的 Node ID，它是我在整个 P2P 网络中的身份标识。从这一刻开始，我“有了身份”，但我仍然是孤独的。我不知道网络里有哪些其他节点，也没有任何连接。

这时候，我意识到，我的第一步并不是同步区块，也不是执行交易，而是一个更基础的问题：我要先找到别人。

于是，我进入了 Discovery 阶段。这是执行层网络的第一层，也是最容易被忽略的一层。这个阶段的目标非常简单，但也非常关键：让我从“谁都不认识”，变成“认识一批节点”。

我不会对全网发出广播，因为那样效率太低，也不现实。相反，我会先去联系一批预先配置好的节点，也就是 Bootnodes。这些节点就像我进入一个陌生环境时，最先能找到的联系人。它们不是中心服务器，但它们能帮我打开第一扇门。

当我联系上一个 bootnode，我不会立刻和它交换区块或交易。我会先做一件很简单的事情：我发一个 Ping，它回一个 Pong。这一步看似简单，但意义重大。它让我确认对方是在线的，是可达的，是一个真实存在的节点。就像我先敲门，确认屋里真的有人。

确认之后，我不会停在这里。我会继续问：“除了你之外，还有谁值得我认识？”于是我发送 FindNode 请求，对方会返回一批它认为“更接近目标”的节点。这时候，我开始逐渐意识到，节点发现不是随机的，而是有结构的。我并不是随便认识人，而是在不断向“更有价值的节点”靠近。

这种查找方式背后，其实就是 Kademlia 的思想。它让我不用扫描整个网络，而是一步一步、沿着更接近目标的方向递归推进。就像问路一样，我不是问全城的人，而是问那些看起来更接近目的地的人。

随着这个过程不断进行，我认识的节点越来越多。我不仅知道它们的存在，还会获取它们的 ENR，也就是它们的“节点名片”。这张名片里包含了它们的 IP、端口、公钥等信息，让我真正知道如何连接它们。

这些信息不会被我随手丢弃。我会把它们整理进自己的路由表中，并按距离分组存放在 k-buckets 里。渐渐地，我不再依赖 bootnode，而是拥有了属于自己的“联系人网络”。这时候，我已经不再是一个刚进入网络的新节点，而是一个开始理解网络结构的节点。

但即便如此，我仍然没有开始真正的区块或交易同步。因为到目前为止，我只是“认识了别人”，还没有和任何人建立正式、安全的通信。

于是，我进入了下一阶段：RLPx。

如果说 Discovery 是“找人”，那么 RLPx 就是“真正建立关系”。我会从已经认识的节点中挑选一些，尝试和它们建立连接。这一次，我不再使用 UDP，而是使用 TCP，因为我需要的是一个可靠的、持续的通信通道。

在建立连接之前，我会进行一次握手。这一步非常关键。我和对方需要确认彼此的身份，并协商出一套用于这次通信的加密方式。我不会直接使用长期私钥来加密所有通信，而是会生成一对临时密钥。这些临时密钥只用于当前会话，并在会话结束后被丢弃。

这样做的意义在于“前向保密”。即使未来我的长期私钥泄露，过去的通信内容也不会被轻易解密。这让我意识到，节点身份和通信安全是两件不同的事情，必须分别处理。

在握手过程中，我和对方通过 ECDH 等机制计算出一个共享秘密，然后再通过一系列方法派生出真正用于通信的会话密钥。这些密钥会被用于加密消息内容，并通过 MAC 来确保消息没有被篡改。

当握手完成之后，我们之间就建立了一条加密的通信通道。但这还不够。我们还需要确认一件事：接下来我们要用什么协议进行通信。

于是，我发送 Hello 消息，告诉对方我支持哪些 capability，比如 `eth`、`snap` 等。对方也会做同样的事情。只有双方都支持的协议，才会真正被使用。

到这里，我和对方之间才真正建立起一条可用的连接。这条连接不仅是加密的，还支持消息分帧和多路复用。这意味着我可以在同一条连接上，同时运行多个上层协议，而不需要为每个协议单独建立连接。

接下来，真正的核心终于开始了。

我会在这条 RLPx 连接之上运行 `eth` 协议。这个协议负责让我和其他节点交换区块头、区块体、交易和回执。我也可能运行 `snap` 协议，用于高效地同步状态数据。

直到这一刻，我才真正开始参与 Ethereum 执行层网络的“业务”。我可以接收交易、传播交易、同步区块、更新状态。我不再是一个孤立的程序，而是网络中的一个活跃节点。

回头再看整个过程，我才真正理解了 DevP2P 的意义。它不是一个简单的网络协议，而是一整套让节点从“孤立存在”变成“网络成员”的完整路径。

而这条路径，也让我终于把之前学过的所有内容串在了一起。

我通过 JSON-RPC 和节点交互，那是应用层和节点之间的通信；  
节点通过 DevP2P 加入网络，那是节点之间的连接过程；  
节点通过 RLPx 和 `eth` 协议交换数据，那是执行层之间的通信；  
最终，这些数据被执行、被记录，形成状态树和区块结构。

当我把这一切连起来之后，我才真正理解：Ethereum 不只是一个“可以写合约的系统”，它更是一个由无数节点通过复杂但有序的协议协同运行起来的网络。

## 一、 灵魂蓝图：以太坊的设计哲学 (Design Rationale)

以太坊的一切架构决策都源于其核心哲学：**“三明治复杂性模型”**。

-   **协议极简 (Simplicity)：** 底层共识协议必须保持精简。宁愿让开发者在应用层多写代码，也绝不轻易往底层增加复杂逻辑。
    
-   **通用性 (Generality)：** 以太坊不为任何特定应用（如转账、交易所）优化，它只提供一个通用的 **EVM**。在底层眼中，所有的操作都是平等的字节码。
    
-   **非歧视性与中立：** 协议不预设立场，不干预应用逻辑，这种“冷酷”的中立是去中心化信任的基石。
    

* * *

## 二、 性能加速：预编译合约 (Precompiled Contracts)

为了在保持“通用性”的同时解决性能瓶颈，以太坊在 EVM 之外开辟了\*\*“硬件加速通道”\*\*。

-   **本质：** 它们不是用 Solidity 写的，而是直接用 Go/Rust 等原生语言集成在客户端内核中。
    
-   **物理地址：** 占据着 $0x01$ 到 $0x0A$ 的低位地址。
    
-   **作用：** 专门处理极其繁重的数学逻辑（如椭圆曲线加法、哈希运算）。
    
-   **工程逻辑：** 它们是 **“以计算换 Gas”**。通过将高频、昂贵的计算逻辑固化，使得 ZK-Rollups 等先进技术在 L1 上的验证成本变得可以接受。
    

* * *

## 三、 系统骨架：模块化架构 (EL & CL)

自“合并（The Merge）”后，以太坊节点进化为\*\*“双头架构”\*\*，两部分通过 **Engine API** 协同工作。

1.  **执行层 (Execution Layer, EL)：** 负责交易执行、状态管理和账户余额，使用 **MPT (Merkle Patricia Trie)** 结构。它是以太坊的“大脑”。
    
2.  **共识层 (Consensus Layer, CL)：** 负责 PoS 共识、验证者调度和最终确定性检查，使用 **SSZ** 序列化格式。它是以太坊的“脊椎”。
    

* * *

## 四、 共识脉搏：时间、随机性与确定性

以太坊如何确保成千上万个节点在混乱的互联网中保持步调一致？

-   **TTD (终端总难度)：** 在合并期间，以太坊放弃了易被操纵的“区块高度”，选择了**累积计算量 (Total Difficulty)** 作为切换点。这反映了以太坊对物理安全（能量消耗）的尊重。
    
    $$TD\_{n} = TD\_{n-1} + D\_{n}$$
    
-   **RANDAO + VDF (公平随机数)：** RANDAO 通过“众筹种子”产生随机性，而 **VDF (可验证延迟函数)** 则通过强制性的数学计算延迟，封锁了最后揭示者的作弊空间。
    
-   **Slots (12s) 与 Epochs (6.4m)：** 以太坊像节拍器一样运行。Slot 是出块机会，Epoch 是结算周期。通过 **LMD GHOST** 规则，系统能在出现分叉时，根据**质押权重**迅速锁定主链。
    

* * *

## 五、 流水线：区块构建循环 (Block Building Loop)

在执行层内部，区块的产生是一个严密的迭代过程：

1.  **环境初始化：** 接收来自共识层的时间戳、Base Fee 等上下文。
    
2.  **贪婪选择：** 从交易池中按价值（Gas Price）排序拉取交易。
    
3.  **状态转换 (STF)：** 将交易丢进 EVM 运行，更新状态数据库（State DB）。如果交易出错，直接跳过并尝试下一笔，直至 **Gas Limit (3,000万)** 耗尽。
    
4.  **默克尔化：** 计算交易根、收据根和提现根，为区块贴上“防伪封条”。
    

* * *

## 六、 存储革命：从 MPT 到 Verkle Tree

这是以太坊目前最关键的技术跨越。

### 1\. MPT (现行结构) —— “深而重的森林”

-   **结构：** 16 叉树，层级极深。
    
-   **痛点：** 验证一个数据需要提供大量“兄弟节点”的哈希。对于 1000 个数据，见证数据 (Witness) 高达 **4MB**，导致普通设备无法验证。
    

### 2\. Verkle Tree (未来结构) —— “广而轻的平原”

-   **结构：** **256 叉树**。高度大幅降低，31 字节 Stem（树干）+ 1 字节 Suffix（后缀）实现了数据的物理聚集。
    
-   **核心黑科技：向量承诺 (Vector Commitments) & KZG 证明。**
    
    -   它不再使用哈希链，而是使用多项式数学。
        
    -   **聚合证明：** 它可以把整条路径的证明压缩成一个微小的数学点。
        
-   **价值：** 见证数据从 4MB 骤减至 **150KB**。

# 以太坊执行层网络通信学习总结

## —— 从 `eth.md` 理解 EL↔EL、EL↔CL 与现代 Ethereum 节点架构

今天我们学习的核心，是 **Ethereum 执行层节点之间的网络通信协议**，也就是 `devp2p` 仓库里的 `caps/eth.md`。这份文档定义的是 **ETH Wire Protocol**，它运行在 **RLPx** 之上，用来让执行层节点彼此交换区块头、区块体、交易池交易、回执等执行层数据。文档本身直接说明，`eth` 协议是用于 peers 之间交换 Ethereum blockchain information 的协议，并且当前基线版本写的是 `eth/69`。

很多人在第一次接触这份文档时，最容易误解的一点是：以为它讲的是“智能合约怎么执行”或者“DApp 怎么和链交互”。其实不是。今天我们已经把这层关系理清了：**前端、钱包、后端程序** 通常是通过 **JSON-RPC** 和节点交互；**共识层 CL 和执行层 EL** 主要通过 **Engine API** 交互；而 `eth.md` 讲的是 **执行层节点和执行层节点之间** 如何互相同步和传播执行层数据。换句话说，它不属于合约开发接口，而属于节点网络层接口。ethereum.org 对节点的说明也印证了这一点：执行层客户端负责监听网络中广播的新交易、在 EVM 中执行它们，并保存最新状态与数据库；共识层客户端负责 PoS 共识，使网络就经过验证的数据达成一致。

* * *

## 一、为什么今天的 Ethereum 不能只有 EL↔CL，还必须有 EL↔EL

这是今天最重要的理解突破之一。你提出的问题非常关键：**“既然 EL 已经和 CL 对接了，为什么 EL 还要和别的 EL 通信？”** 这个问题一旦想通，整个 `eth.md` 的定位就会变得非常清楚。

答案是：**EL↔CL 和 EL↔EL 解决的是两类完全不同的问题。**  
EL↔CL 解决的是“共识层和执行层之间如何协作”；EL↔EL 解决的是“执行层数据如何在整个执行层网络中传播和同步”。Engine API 的 Paris 规范显示，CL 和 EL 的交互核心围绕 `engine_newPayloadV1`、`engine_forkchoiceUpdatedV1`、`engine_getPayloadV1` 等方法展开，其中 `ForkchoiceState` 里有 `headBlockHash`、`safeBlockHash`、`finalizedBlockHash`，而 `ExecutionPayload` 则包含执行层区块内容，例如 `transactions`。这说明 CL 主要是在告诉 EL：当前哪条链头是该接受的，你该围绕哪个 head 更新 forkchoice，或者基于哪个 head 构建新的 payload。

但这还远远不够。因为就算 CL 告诉了 EL “哪个 head 是正统链头”，EL 仍然要自己解决很多现实问题：**交易从哪里来，区块体从哪里下载，回执从哪里补，历史数据从哪里拿，本地交易池如何传播给更多 peer。** 这些都不是 CL 替它做的，而是执行层网络自己完成的。`eth.md` 明确把一个 session 中的高层任务分成两类：**chain synchronization** 和 **transaction exchange**。这就意味着，EL 节点之间的横向通信不是可选装饰，而是执行层能形成一个网络、能让数据流动起来的基础。

所以今天最值得记住的一句话是：

**CL 决定“该信哪条链”，EL 网络负责“把这条链需要的执行层数据搬过来并传播开”。**  
没有 EL↔CL，执行层不知道该跟随哪条 canonical chain；没有 EL↔EL，执行层就没有足够的数据来源和传播网络，整个系统也跑不起来。

* * *

## 二、`eth.md` 的本质：它讲的是执行层节点之间的“数据交换规则书”

从整体上看，`eth.md` 不是在讲“链为什么存在”，也不是在讲“EVM 指令如何执行”，而是在讲一个更基础的问题：**两个执行层节点一旦连上了，应该如何互相说话。**

这份文档规定了哪些消息可以发、先发什么、后发什么、每种消息结构长什么样、响应应该怎么返回。它相当于执行层节点之间的一本“通信规则书”。文档在 Basic Operation 里明确写到：连接建立后，双方必须先交换 `Status` 消息；只有在收到对方 `Status` 之后，这个 Ethereum session 才算激活，在此之前不应该发送其他 `eth` 消息。文档还规定了消息尺寸限制，底层 RLPx 单条消息上限为 16.7 MiB，而 `eth` 协议实践中常见上限更低，典型实现会将单消息限制在约 10 MiB 左右。

这一部分你可以用一句非常朴素的话来理解：  
`eth` **协议不是“随便发消息”，而是“先握手，再按规则同步数据”。**  
这和日常应用层 API 的感觉很不一样，它更接近底层网络协议的思路。

* * *

## 三、Status 握手：不是简单“打招呼”，而是能力与状态通告

今天我重点看了 `Status` 这个消息，因为它是整个 `eth` session 的起点。根据 `eth.md`，`Status` 的格式是：

`[vsn, networkid, genesis, forkid, earliest, latest, latestHash]`

这里面包含了协议版本、网络 ID、创世块哈希、fork 标识，以及当前节点能提供的完整区块历史范围。也就是说，`Status` 并不只是“你好，我在线”，而是“我运行的是哪个协议版本、我在哪条网络、我处于哪个 fork 规则、我现在到底能服务哪一段历史”。

这里还有两个今天特别容易混淆、但我们已经厘清的知识点。

第一，`networkid` **不一定等于** `chainId`。`networkid` 更偏向 P2P 网络层的识别；`chainId` 则更多用于交易签名与重放保护。文档明确提醒，这两者不应想当然地视为同一个值。

第二，`Status` 在 `eth/69` 之后比以前更重要。EIP-7642 明确指出，`eth/69` 做了三件关键事：  
一是让节点在握手里声明自己服务的历史区块范围；  
二是移除了 Merge 后已不再使用的 total difficulty 握手信息；  
三是简化了 receipts 传输格式。

因此，今天你再看 `Status`，不能只把它当成旧时代的“版本对齐”，而应理解为：**现代 Ethereum 执行层节点用它来告知 peer 自己的历史服务能力和当前兼容上下文。**

* * *

## 四、为什么 PoS 之后，`eth.md` 不能再按 PoW 思路去读

这是今天学到的另一个非常关键的点。`eth.md` 本身明确写到：因为 Ethereum 的共识发生在 execution chain 之外，这个协议没有内建机制来决定 canonical chain head，它假定每个节点会通过其他方式，比如和一个 consensus node 通信，来知道 canonical chain。

这意味着，**Merge 之后，**`eth` **协议不再单独承担“哪条链算正统”这件事。**  
在 PoW 时代，执行层网络自己就更接近“既传播数据，也决定哪条链更重”；但在 PoS 时代，canonical head、safe head、finalized head 的判断，已经主要由共识层提供。Engine API 里的 `ForkchoiceState` 正是这种变化的体现，因为它直接让 CL 向 EL 传达 head、safe、finalized 三种哈希。

文档还特别提醒，PoW 转 PoS 之后，**block propagation 已不再由** `eth` **协议负责**；文中保留的 `NewBlock`、`NewBlockHashes` 相关描述，主要只适用于 PoW 和部分 PoA 网络，并会在未来版本中逐步淡出。也就是说，今天阅读 `eth.md` 时，要把它看成一份“带有历史包袱但主线已经现代化”的协议文档：有些内容仍在规范里，但它们对当前主网的核心理解权重要降低。

* * *

## 五、链同步主线：先拿 headers，再拿 bodies，再补 receipts

在今天的学习中，我们把 `eth.md` 里最经典的一条主线理清了：**chain synchronization**。

文档给出的典型流程是这样的：节点先通过 `GetBlockHeaders` 请求区块头，对方返回 `BlockHeaders`；在拿到 header 链之后，再用 `GetBlockBodies` 请求区块体，由对方返回 `BlockBodies`；在某些同步场景下，还要额外用 `GetReceipts` 拉取 receipts，由 `Receipts` 响应。`GetBlockHeaders` 的请求参数中还包含 `startblock`、`limit`、`skip`、`reverse`，这意味着节点不只是“从头到尾一块一块拉”，还可以先取骨架，再并行补全缺失部分。

今天我们特别强调了：  
**为什么一定是先拿 header，再拿 body？**  
因为 header 更轻，它决定了“区块骨架”和各个根承诺；一旦 header 链确定下来，节点就知道每个 block hash 是什么，于是区块体可以向多个 peer 并行请求。之后 receipts 又可以用来校验 `receipts-root`，从而让同步过程不必每次都立刻完整重放所有交易。

这一条流程线，实际上是理解执行层节点“为什么能同步起来”的关键。如果没有 EL↔EL 网络，这条链同步路径本身就不存在，因为 headers、bodies、receipts 都是从其他执行层 peer 拉来的。

* * *

## 六、State sync 为什么不再属于 `eth` 主协议

今天我还学到一个很容易被忽略，但实际上非常体现现代架构变化的点：**state tree 下载已经不再是** `eth` **协议的工作。**

`eth.md` 明确说明，`eth/63` 到 `eth/66` 还可以通过 `eth` 协议同步 state tree，但从 `eth/67` 开始，这项工作已经被交给了辅助协议 `snap`。官方 `snap.md` 也说明，`snap` 运行在 RLPx 之上，用于 peers 之间交换 Ethereum 状态快照。

这意味着今天你理解执行层同步时，不能再想成“一个 `eth` 协议包办一切”。更准确地说：

-   `eth` 主要负责 **headers、bodies、receipts、交易池传播**；
    
-   `snap` 负责 **state 下载**；
    
-   `Engine API` 负责 **EL 和 CL 之间的共识-执行接口**。
    

这恰恰体现了现代 Ethereum 架构正在不断模块化、分层化：不同协议负责不同职责，而不是所有东西挤在一个协议里。

* * *

## 七、交易传播：为什么交易池一定要靠 EL 网络自己扩散

交易交换是今天学习里非常重要的一部分，因为它和你以后理解 mempool、机器人、脚本并发冲突都直接相关。

ethereum.org 说明，执行层客户端会监听网络中广播的新交易并执行它们。`eth.md` 进一步规定，节点之间必须交换 pending transactions，并把它们保存在本地的 transaction pool 中。新 peer 建立连接时，会先用 `NewPooledTransactionHashes` 公告自己本地池中的交易哈希；如果对方发现有自己没有的交易，就会用 `GetPooledTransactions` 拉取完整交易；此外还有 `Transactions` 消息，可直接发送完整交易。

这说明交易传播的核心逻辑是：

**先广播目录，再按需拉正文。**

这么做的好处是节省带宽。不是每个节点都必须把每一笔完整交易无差别地发给所有 peer，而是先告诉别人“我这边有这些交易的哈希、类型、大小”，真正需要的人再来要正文。文档还特别说，节点应该避免把已经知道的交易重复回发给同一个 peer。

这一部分也直接回答了你那个问题：  
**为什么不能只有 EL↔CL？**  
因为交易池传播本来就是 EL 网络内部的职责。CL 负责共识，不负责充当执行层交易广播总线。

* * *

## 八、交易有效，不等于最终执行成功

今天我们还补上了一个非常重要、以后你写 Web3 脚本时一定会反复遇到的概念：**mempool valid 不等于 EVM execute success。**

`eth.md` 在“Transaction Encoding and Validity”一节明确指出，协议所说的交易有效性，关注的是这笔交易是否适合临时放入本地池并与其他 peer 交换，而不是它最终一定能在 EVM 中成功执行。文档要求节点检查交易类型是否已知、签名是否有效、gas 是否满足 intrinsic gas、余额是否足够支付 value 和 gas 成本，还要求交易 nonce 至少不小于发送者当前账户 nonce。

这意味着，一笔交易即使已经进入 mempool，也仍然可能在真正打包执行时失败。失败原因可能是状态变化、滑点变化、余额被其他交易先消耗、或者同 nonce 交易竞争替换。今天虽然没有把这些交易池实现细节展开成一门课，但你已经抓住了最底层的逻辑：  
**网络层接受一笔交易，并不等于链上最终一定成功执行这笔交易。**  
这是理解 Web3 交易行为、尤其是 DeFi 交互和自动化机器人时非常关键的一个思维转折。

* * *

## 九、今天关于 nonce 的真正上下文：交易 nonce 和区块 nonce 不一样

今天专门到了 nonce，这个问题很重要，因为 `eth.md` 里其实出现了两个完全不同的 nonce。

第一种是 **transaction nonce**。它出现在 legacy transaction 的字段里，表示发送者账户的交易序号。文档在交易有效性检查里要求节点确认该 nonce 至少不小于当前账户 nonce，也允许实现自行决定本地池支持多大的 future transaction gap。这个 nonce 主要和账户级交易顺序、重放保护、交易池排队有关。

第二种是 **block nonce**。它出现在 block header 结构中，是历史 PoW 区块头字段的一部分，和 seal 校验有关。文档在 header 编码与 header validity 部分都保留了这个字段，还提到可检查的 seal 包括 `mix-digest` 和 `block-nonce`。

所以今天一个非常重要的收获是：  
**你以后看到“nonce”这个词，绝不能下意识只想到“账户交易序号”。**  
在 Ethereum 协议文档里，它可能是 tx nonce，也可能是 block nonce，必须看上下文。今天在 `eth.md` 里，这两个都出现了。

* * *

## 十、区块结构：为什么一个区块不是“只有交易列表”那么简单

今天我们还系统看了区块结构。`eth.md` 给出的 header 字段远比初学者想象得复杂：有 `parent-hash`、`state-root`、`txs-root`、`receipts-root`、`gas-limit`、`gas-used`、`basefee-per-gas`、`withdrawals-root`、`blob-gas-used`、`excess-blob-gas`、`parent-beacon-root`、`requests-hash` 等。它还定义了 block body 为 `[transactions, ommers, withdrawals]`。

今天最值得记住的，是对“header、body、block”这三者关系的理解：

-   **header** 是摘要和承诺，里面有各种 root 和关键元数据；
    
-   **body** 是正文，真正放交易、ommers、withdrawals 等；
    
-   **完整区块** 则是 header 加上相应正文内容。
    

也正因为 header 里有 `txs-root`、`receipts-root`、`state-root` 等承诺字段，所以同步流程才能先拿 header，再用 body 和 receipts 去校验这些根是否一致。

* * *

## 十一、为什么 header 字段会越来越多：因为 Ethereum 协议还在持续演进

今天我特别提到，`eth.md` 不是一份“死掉的旧规范”，而是一份随硬分叉不断更新的活文档。文档在 header validity 规则里明确写出了不同 fork 之后哪些字段必须存在、之前则必须不存在。比如：

London 之后必须有 `basefee-per-gas`；  
Shanghai 之后必须有 `withdrawals-root`；  
Cancun 之后必须有 `blob-gas-used`、`excess-blob-gas`、`parent-beacon-root`；  
Prague 之后必须有 `requests-hash`。

这让你今天对 Ethereum 的认识更进一步了：  
**Ethereum 不是一个固定不变的“老协议”，而是在持续迭代的系统。**  
每次升级不仅会影响合约开发层，也会改变执行层 header、区块编码、节点网络协议等底层结构。`eth.md` 正是这些演进在执行层 P2P 规范上的反映。

* * *

## 十二、eth/69 的现代重点：历史范围通告与 receipts 简化

今天我们花了不少时间理解 `eth/69` 为什么重要。EIP-7642 给出的核心动机很明确：随着 Ethereum 进入 **history expiry** 时代，并不是所有节点都会永久保存并提供全部历史。因此，协议需要一种机制让节点告诉 peer：**我到底还能服务哪一段历史。** 这就是为什么 `Status` 里新增了 `earliest`、`latest`、`latestHash`，并新增了 `BlockRangeUpdate` 消息。

`BlockRangeUpdate` 的作用，就是在节点历史服务范围变化时，通知 peer 新的 `[earliest, latest, latestHash]`。文档还建议，这种更新没必要每个新区块都发一次，大约每两分钟发一次就可以。

EIP-7642 还指出一个非常现实的优化点：旧版 receipts 传输里带着 Bloom 字段，但客户端实际上通常不会持久化这个字段，因为它可以重新计算。继续在网络上传这个字段会浪费巨量带宽，EIP-7642 估计仅此一项在同步中就会造成大约 **530GB** 的额外传输。因此 `eth/69` 简化了 receipts 编码，移除了 Bloom，并简化了 typed receipts 的表示。

这一部分的本质，是让你看到现代 Ethereum 协议设计正在越来越现实化：  
**不是默认每个节点都全量保留、全量传输、全量服务，而是承认资源边界，并优化真正有价值的数据交换。**

* * *

## 十三、eth/70 草案：为什么 receipts 还要继续分页化

今天我还顺带接触到了 `eth/70` 的方向。`eth.md` 的 changelog 里已经写到了 `eth/70` 的相关更新，而 EIP-7975 则说明了它的目标：支持 **partial block receipt lists**，也就是对超大区块的 receipts 列表进行分页获取。原因是随着主网 block gas limit 提高，单个区块的 receipts 列表未来可能超过客户端常用的约 10 MiB 消息大小限制。需要注意的是，EIP-7975 目前仍是 **Draft**。

这一点虽然还没有完全成为稳定基线，但它很好地说明了协议演进方向：  
**Ethereum 执行层网络协议正在从“大包整传”逐步走向“更细粒度、更可分页、更适应大规模链上数据”的模式。**

* * *

## 十四、今天真正建立起来的整体架构图

如果把今天的学习内容压缩成一张“脑内结构图”，它大概是这样的：

用户、前端、钱包、后端  
通过 **JSON-RPC** 和执行层节点交互；  
执行层节点和共识层节点  
通过 **Engine API** 协作；  
执行层节点彼此之间  
通过 **ETH Wire Protocol (**`eth`**)** 交换区块头、区块体、回执和交易池数据；  
执行层节点如需状态快照  
通过 **SNAP** 协议同步状态；  
共识层节点彼此之间  
则运行各自的共识层网络协议。

今天最大的进步，不只是学会了几个新名词，而是开始真正把这些层次分开看了。以前容易把“链”“节点”“客户端”“RPC”“共识”“执行”“同步”都揉成一团；今天开始，这些概念已经被拆成了不同角色、不同通道、不同职责。

# 我对以太坊协议层的一次系统理解：从 EL、CL 到区块确认与 Engine API

最近我系统学习了以太坊协议层，最大的变化是：我不再只把以太坊看成“一个能跑智能合约的链”，而开始把它理解成一个**分层协作的系统**。

在最开始接触以太坊时，我更关注的是合约怎么写、交易怎么执行、状态怎么变化。但随着我继续往下学，我发现如果只停留在“智能合约执行”这一层，其实很难真正理解以太坊节点到底是怎么工作的。因为今天的以太坊，已经不是一个单体程序就能解释清楚的系统了。The Merge 之后，一个完整的以太坊节点不再只是一个客户端，而是至少由\*\*执行客户端（EL）**和**共识客户端（CL）\*\*两部分组成；如果这个节点还要参与出块和投票，还需要运行验证者客户端。也就是说，现在的“节点”本质上是多个角色协同工作的结果。

在我的理解里，EL 和 CL 的分工非常明确。EL 负责执行交易、运行 EVM、维护账户余额和合约状态，也负责对外提供常见的 JSON-RPC 接口；CL 负责 PoS 共识，接收和传播区块与 attestation，根据共识规则决定当前应该跟随哪条链。简单说，EL 更像“执行者”，CL 更像“裁判和协调者”。以前我总容易把“执行”和“共识”混在一起，现在我会明确区分：**执行层负责状态转移，共识层负责链选择。**

我后来还搞明白了一件很关键的事：EL 和 CL 虽然属于同一个节点，但它们**并不运行在同一套 P2P 网络上**。执行层有自己的网络，主要传播交易；共识层有自己的网络，主要传播区块和 attestation。两者之间不是直接混在一起工作的，而是通过专门的接口协作，这套接口就是 **Engine API**。所以现在我理解的以太坊，不是一条单线网络，而是“执行网络 + 共识网络 + 本地协作接口”组成的系统。

继续往下学之后，我开始真正理解 PoS 下“共识”到底是什么意思。现在的以太坊不再依赖矿工，而是依赖验证者。每个 slot 会随机选出一个区块提议者来提议新区块，同时会有一组验证者对区块进行 attestation。attestation 本质上就是投票：验证者会对自己当前看到的链视图进行签名并广播。共识客户端会不断收集这些投票，然后运行 fork choice，选择当前权重最大的那条链作为链头。也就是说，CL 不是随便决定哪条链是对的，而是根据验证者的投票结果动态选出当前应该跟随的 head。

这一步让我终于明白了一个以前很困惑的问题：**为什么 head 会不稳定。** 一开始我会觉得，既然最后总是要统一，那为什么不能一开始就所有人同时确认？后来我才真正意识到，问题不在于“大家不想统一”，而在于分布式网络里，所有节点不可能在同一瞬间看到完全相同的信息。有的节点先收到了新区块，有的节点还没收到；有的节点先看到了更多 attestation，有的节点稍慢一点。于是，在某个很短的时间窗口里，不同节点可能会基于各自手头的信息，得出不同的“当前最新链头”判断。这不是系统出错，而是分布式系统里非常正常的网络传播延迟现象。

我现在会这样理解 `head`：它表示**当前视图下最新、权重最高的链头**，但它不是最终确认。它之所以会变化，是因为 fork choice 会随着更多 attestation 的到来而动态调整。也就是说，`head` 本质上强调的是“最新”，不是“最终”。这一点特别重要，因为它解释了为什么区块链不能只提供一种确认状态。

接下来我学习到的另一个重点，就是以太坊为什么要区分多种确认层级。现在我会把它们理解成不同强度的确认状态：`head` 代表最新链头，速度最快，但也最容易变化；`safe` 代表更稳妥的安全头；`justified` 是共识层里 checkpoint 已经获得强支持的状态；`finalized` 则是最终确认状态，基本可以认为不会再被回滚。以太坊之所以要分出这些层级，不是因为设计重复了，而是因为不同层级对应着不同程度的确定性。

我后来也想通了：**为什么不能只保留一个最高等级的确认，比如 finalized。** 原因很简单，因为越强的确认通常意味着越慢。如果所有业务都必须等到 finalized 才能继续，那很多需要实时反馈的场景都会变得非常迟钝。比如钱包余额展示、区块浏览器追踪最新区块，这些都更需要 `latest/head`；而跨链桥、大额结算、清算这类更重视安全的场景，就更适合看 `safe` 甚至 `finalized`。所以在我看来，这不是协议“设计得太复杂”，而是它在认真处理一个现实问题：**不同业务对速度和安全的要求并不一样。**

如果用最通俗的话来说，我现在会把这几个确认层级理解成这样：`head` 是“刚刚上链、最新看到的结果”，`safe` 是“已经比较稳了”，`justified` 是“已经获得很强的集体背书”，而 `finalized` 是“最终确认，基本不会再回头”。这样理解以后，我再看这些术语，就不会觉得它们只是一些抽象名词，而是能把它们放到真实的链上确认流程里去理解。

在理解确认层级之后，我又进一步接触到了 EL 和 CL 之间最关键的接口，也就是 **Engine API**。这里我最重要的收获，是把它和普通 JSON-RPC 彻底区分开。普通 JSON-RPC 主要是给钱包、前端、脚本和开发框架使用的，而 Engine API 是给执行层和共识层“内部协作”使用的。它不是给普通 dApp 前端直接调的，而是 CL 和 EL 在同一个节点内部交换链头信息、payload 和兼容性信息的方式。这个区分对我特别重要，因为以前我总觉得“反正都是 RPC”，现在我知道：**外部应用和 EL 说话，CL 和 EL 也在说话，但这是两种完全不同的接口语境。**

在 Engine API 里，我目前最关注的几个方法是 `engine_exchangeCapabilities`、`engine_forkchoiceUpdated`、`engine_getPayload` 和 `engine_newPayload`。一开始这些名字看起来都很像，我也很容易混，但现在我已经能把它们分开理解了。

`engine_exchangeCapabilities` 并不是传区块，也不是执行共识，它只是让 EL 和 CL 先互相告诉对方：**我当前支持哪些 Engine API 方法版本。** 它解决的是接口兼容性问题，而不是链上状态问题。换句话说，它像是一种“先报家底”的过程：在双方正式协作之前，先互相确认“我会哪些方法、你会哪些方法、我们接下来能用哪些版本的接口配合”。我现在会把它理解成一种很轻量、但很有工程意义的能力协商机制。

为什么这一步重要？因为 Engine API 本身不是永远不变的。随着协议演进、硬分叉推进，方法版本可能会升级，旧方法可能会废弃，新方法也可能会加入。如果没有 `engine_exchangeCapabilities` 这样的机制，那么 EL 和 CL 即使各自都正常运行，也可能因为接口版本不一致而合作失败。所以这个方法虽然不直接处理区块和状态，但它其实是在为后续真正的协作打基础。

真正把“共识层当前认哪条链”告诉执行层的核心方法，是 `engine_forkchoiceUpdated`。我现在会把它理解成：CL 先通过验证者投票和 fork choice 算法选出当前应该跟随的链头，然后把这个结果通过 `headBlockHash`、`safeBlockHash` 和 `finalizedBlockHash` 传给 EL。也就是说，EL 并不会自己决定现在该跟哪条链，而是接受 CL 已经计算好的链视图。

更进一步，如果这个调用里还附带了 `payloadAttributes`，那它不仅是在更新链头视图，还相当于在对 EL 说：**“请基于这个 head，开始帮我构建一个新的 execution payload。”** 这一步非常关键，因为它把“共识层选链”和“执行层构建区块执行内容”衔接了起来。也正是从这里开始，我真正理解了 forkchoice 和 block building 并不是两件独立的事，而是同一条工作流里的前后两步。

这时候，`engine_getPayload` 和 `engine_newPayload` 的区别就清楚了。`engine_getPayload` 是把 EL 本地已经构建好的 execution payload 取出来给 CL；而 `engine_newPayload` 是把一个 execution payload 交给 EL，请它按执行规则去验证这个 payload 是否有效。前者更像“把你刚造好的块拿给我”，后者更像“这里有一个块，请你检查它是不是合法的”。

我之前最容易卡住的一点是：EL 又构建、EL 又验证，那不是永远都说自己对吗？后来我才真正想明白，不是“某一个 EL 说自己对就算对”，而是**每个节点的 EL 都会独立重放和验证 payload**。提议者本地 EL 可以先构建候选 payload，但网络里的其他节点收到之后，也会把这个 payload 交给各自的 EL 重新执行。所以最终能不能被全网接受，并不是某一个 EL 说了算，而是全网节点一起独立检查之后的结果。也正因为这样，我才真正理解了为什么执行层的“验证”不是自说自话，而是整个网络一致性的一部分。

把这些内容全部串起来之后，我现在会这样理解以太坊的这套架构：**以太坊节点是 EL 和 CL 的组合；EL 负责执行交易和维护状态，CL 负责共识和链头选择；两者跑在不同的 P2P 网络上，同时通过 Engine API 交换链头信息、payload 和能力信息。**`engine_exchangeCapabilities` **负责先协商接口能力，**`engine_forkchoiceUpdated` **负责把当前链头视图传给 EL，**`engine_getPayload` **负责取回 EL 构建好的 execution payload，**`engine_newPayload` **则负责让 EL 对 payload 进行验证。**

这样一来，我再去看 STF、forkchoice、payload、head、safe、justified、finalized 这些概念时，就不会把它们当成散乱的术语，而会知道它们其实是在共同描述一个完整闭环：**共识层先决定当前链视图，执行层在这个视图下执行和构建，网络中的其他节点再独立验证，最后区块从最新逐步走向最终确认。**

如果让我用一句话概括我这部分学习最大的收获，我会这样说：

**以太坊不是“一个程序执行交易”那么简单，而是一个分层协作的系统。CL 通过 PoS 和验证者投票决定当前链视图，EL 在这个视图下执行和验证状态变化，两者通过 Engine API 协作；而区块从最新到最终确认，也不是一步完成，而是经历从 head 到 safe，再到 justified 和 finalized 的逐步增强过程。**

当我真正把这些东西连起来之后，再回头看交易确认、合约执行、节点架构和 RPC 接口，很多原本零散的概念就终于串成了一条完整的线。

**RLP = Ethereum 的数据序列化协议**

👉 作用：

-   把结构化数据 → 转成 byte\[\]
    
-   用于链上传输 / 存储
    

👉 支持的数据类型只有两种：

```
1. byte[]
2. list（递归结构）
```

❗ 不支持：

-   int
    
-   string
    
-   bool
    

👉 所有类型必须先转 bytes

### 1️⃣ 无类型系统（Type-less）

```
RLP 不关心业务类型，只关心字节结构
```

例如：

```
uint256 → 先转 big-endian bytes → 再编码
```

* * *

### 2️⃣ 递归结构（Recursive）

```
["a", ["b", "c"], "d"]
```

👉 可以无限嵌套

## 1️⃣ 字节数组（string / bytes）

* * *

### ✅ 单字节（< 0x80）

```
编码 = 本身
```

例：

```
0x61 → 0x61
```

* * *

### ✅ 长度 0–55 字节

```
prefix = 0x80 + len(data)
result = prefix + data
```

例：

```
"dog" → len=3
prefix = 0x83

RLP = 0x83 + "dog"
```

* * *

### ✅ 长度 > 55 字节

```
prefix = 0xb7 + len(len(data))
RLP = prefix + len(data) + data
```

👉 多一层长度编码

* * *

## 2️⃣ 列表（list）

* * *

### 编码步骤：

```
1. 每个元素先 RLP encode
2. 拼接 payload
3. 对 payload 再加 prefix
```

* * *

### ✅ payload ≤ 55

```
prefix = 0xc0 + len(payload)
```

* * *

### ✅ payload > 55

```
prefix = 0xf7 + len(len(payload))
```

## 1️⃣ 交易编码（最重要）

```
tx → RLP → keccak256 → 签名
```

* * *

## 2️⃣ 合约地址生成

```
address = keccak256( RLP([sender, nonce]) )
```

* * *

## 3️⃣ 区块数据

-   block header
    
-   receipt
    

* * *

## 4️⃣ 状态树（MPT）

-   account
    
-   storage
    

* * *

## 5️⃣ P2P 网络通信

-   节点间消息传输

**Execution Layer = 状态机 + 可验证数据结构**

-   本质：一个 **Key-Value 状态数据库**
    
-   目标：
    

-   可验证（Verifiable）
    
-   不可篡改（Immutable）
    
-   可追溯（Traceable）
    

* * *

核心数据结构

```
Block
 ├── Header
 │    ├── stateRoot
 │    ├── transactionsRoot
 │    ├── receiptsRoot
 │
 ├── Transactions
 ├── Receipts
```

核心：**3棵Trie**

-   State Trie（状态）
    
-   Transaction Trie（交易）
    
-   Receipt Trie（收据）
    

World State（世界状态）

定义

```
当前链上所有账户的集合
```

结构

```
address → account
```

Account结构

```
{
  nonce,
  balance,
  storageRoot,
  codeHash
}
```

MPT（Merkle Patricia Trie）🔥重点

组成

-   Trie（前缀树）
    
-   Patricia（路径压缩）
    
-   Merkle（哈希树）
    

特点

| 特性 | 说明 |
| 可验证 | hash root |
| 防篡改 | Merkle |
| 高效查找 | Trie |
| 节省空间 | Patricia |

* * *

核心作用

👉 用一个 `root hash` 表示整个状态

三棵核心 Trie

1️⃣ State Trie

```
address → account
```

👉 表示当前世界状态

2️⃣ Transaction Trie

```
txIndex → transaction
```

👉 存储区块中的交易

3️⃣ Receipt Trie

```
txIndex → receipt
```

👉 存储执行结果

Storage Trie（合约存储）

每个合约都有自己的 storage：

```
storageRoot → storage trie
```

Solidity映射

```
mapping(address => uint256) balances;
```

底层：

```
keccak(key + slot) → value
```

Block Header（三个关键字段）

```
stateRoot
transactionsRoot
receiptsRoot
```

含义

| 字段 | 含义 |
| stateRoot | 全局状态 |
| txRoot | 所有交易 |
| receiptRoot | 执行结果 |

核心设计思想🔥

1️⃣ Everything is Tree

-   状态 = Trie
    
-   交易 = Trie
    
-   收据 = Trie
    

2️⃣ Key-Value 本质

```
address → account
storageKey → value
```

3️⃣ 可验证性（Verifiable）

-   Merkle Proof
    
-   轻节点验证
    

4️⃣ 状态更新机制

```
old state → new state
```

👉 每个区块都会生成新的 stateRoot

执行流程（必须理解）

```
用户发送交易
   ↓
执行 EVM
   ↓
修改 storage / balance
   ↓
更新 State Trie
   ↓
生成新的 stateRoot
   ↓
写入 Block Header
```

Besu = 一个“以太坊执行层节点软件”

运行 EVM + 执行交易 + 维护状态 + 对外提供 RPC

Besu = 一个完整的 Execution Layer 实现（客户端）

Besu 是一个 **Ethereum Client（以太坊客户端）**

它包含：

-   执行环境（EVM）
    
-   状态存储
    
-   P2P 网络
    
-   API 接口
    

Execution Environment 执行环境

核心就是：

-   EVM（以太坊虚拟机）
    

作用：

-   执行智能合约
    
-   运行 opcode
    
-   计算结果
    

写的solidity都是在这里执行

Storage 状态存储

存链上数据：

-   账户余额
    
-   合约 storage
    
-   nonce
    

P2P Network 节点网络

Besu 不是单机程序，它会：

-   连接其他节点
    
-   同步区块
    
-   广播交易
    

所有节点：

都维护同一份状态（去中心化）

API 接口层

对外提供：

-   JSON-RPC
    
-   WebSocket
    

支持：

-   eth\_call
    
-   eth\_sendTransaction
    
-   debug

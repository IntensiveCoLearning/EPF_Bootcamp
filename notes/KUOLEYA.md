首先需要介绍的是默克尔树他是一种高效的数据结构，用于验证数据完整性他的叶子节点存数据，非叶子节点存子节点哈希，他的结构为从底部的交易开始逐渐往上延伸成父节点，每个节点逐层向上最后组成顶部的根哈希，而这个根哈希被用来验证整个默克尔树而被存在区块头中，因为使用**Keccak-256**算法所以在默克尔树种微小的变化都会引来巨大的数值误差,所以默克尔树在证明其不可篡改方面非常高效,（需要注意的是因为默克尔树是二叉树结构所以交易节点必须为偶数个，如果交易不满足偶数个交易，则最后一个交易复制自身组成偶数对）

然后是**帕特里夏树**结构直接使用教程里的更清晰

向以太坊的交易树，世界状态树，存储树，收据树等都是这个结构只不过他们存储的值不同

未来以太坊要发展的数据结构是 Verkle 树。它和 MPT 的区别在于：MPT 验证一个节点需要提供路径上所有兄弟节点（见证数据约 4MB），而 Verkle 树使用向量承诺技术，将见证数据压缩到约 150kB，更适合无状态客户端

RLP（递归长度前缀）：本质上来说是一种翻译规则它将负责的数据结构翻译成字节流，方便在网络种进行传输保证不同编程语言编写的客户端都能正确解析他的运行规则是先判断是不是带有0x80标志的特殊值如果是返回结果0x80，如果非空且没有携带0x80的标志就直接输出该字节

如果字节长度>55字节则加上0xB7+长度所占的字节数的数据头在加一个总长度再拼上原来的数据

如果说是一个列表则先把列表的每一项按上面提到的字节编码规则编码然后判断是否小于等于55字节如果是0xC0+总长度的数据头然后拼上原来的数据如果不是则加上0xF7+长度所占的字节数的数据头在加一个总长度再拼上原来的数据

今天学习Transaction anatomy首先交易是外部账户发布的一条有加密签名的指令一般包含有如下的信息nonce，gasprice，gaslimit，to，value，data，Signature，其中比较重要的是

Nonce：防止重放攻击，构建新合约以及，替换交易（简单来说就是假如我发送了一笔Nonce为4的交易但是gas费过低没有矿工接取，我可以在发送一笔更高gas费的交易去替换这个Nonce为4的交易，这样做可以防止后面的nonce5，6，7不会因为nonce始终没有验证而卡住）

To：如果该字段为空则创建合约，如果该字段为外部账户则发起转账，如果该字段为合约账户则调用合约

在现实中交易完成会有收据而在区块链中也是如此在交易完成过后EVM的STF输出的产物无论交易成功还是失败都会返回收据，收据一般包换5个字段

**Transaction Type：交易类型，用于区别传统交易和类型化交易；**

**Status：交易的状态，0表示失败1表示成功**

**Logs：日志**

**Logs Bloom：可以理解为日志搜索框通过快速筛选高效地检查某个地址或事件签名是否包含在日志中**

**收据**最终会提交到区块的**收据 Trie（Receipt Trie）**中

上面提到了**类型化交易他是EIP-2718引入的新概念他旨在新的格式它通过定义了一个Transaction Type用来区分这是什么类型的交易数据**他的格式为

**Typed Transaction = Transaction Type + Transaction Payload**

其中：

-   **Transaction Type**：一个单字节标识符，用于指定交易类型。
    
-   **Transaction Payload**：一个不透明的字节数组，由相应交易类型的规范定义。
    

需要注意的是，**传统交易（legacy transaction）** 的格式为：  
RLP(\[nonce, gasPrice, ..., s\])

收据现在也采用相同的信封模式：

**Typed Receipt = Receipt Type + Receipt Payload**

其中：

-   **Transaction Type**：同上
    
-   **Receipt Payload**：根据关联的 **Transaction Type** 定义进行解释。
    

传统收据（legacy receipt）的格式为：  
RLP(\[status, gasUsed, bloom, logs\])

**如何识别交易或收据的类型**

可以高效地识别交易或收据的类型：

-   如果第一个字节 ∈ \[0x00, 0x7f\]，则是**类型化交易**或**类型化收据**。
    
-   如果第一个字节 ≥ 0xc0，则是**传统交易**或**传统收据**，这是由 RLP 列表编码规则决定的。
    

**类型化收据的第一个字节必须与其关联交易的 Transaction Type 相同。**

今天学习EVM；首先借用资料中的例子EVM是一种根据不同的输入表示出不同状态的状态机，假如我们把EVM看成一太自动售货机那么他会有三种状态，**空闲**（Idle）**等待用户选择**（Selection），**分发产品**（Dispensing）三种输入模式**投币（ InsertCoin），选取商品（**SelectDrink**），取走商品（**CollectDrink**）**

其中系统在某一时刻的状态可以表示为（S）:

_S ∈ {Idle,Selection,Dispensing}；_

他在这个状态的输入表示为(I):

_I ∈ {InsertCoin,SelectDrink,CollectDrink}_

那么它的运行逻辑可以表示为

Υ(S , I) ⟹ S1

其中Y表示状态转移函数S为当前状态，I为输入，S1为下一状态

由此可以通过以上的参数表示出他的运行逻辑；

Y（Idle,InsertCoin） => Selection;

输入空闲阶段和投币那么下个状态表示为待用户选择；

Y（Selection, SelectDrink） => CollectDrink;

输入待用户选择阶段接收和选取商品那么下个状态表示为取走商品

需要注意的是如果输入空闲状态和选取商品那么状态机不会有任何变化，说明当前状态下输入无效：

Y（Idle, SelectDrink） => Idle

由此可见它是一个环形结构用来处理无效操作或者不满足条件输入。

如果EVM是状态机那么以太坊整体可以看成一个基于交易的状态机它通过接受交易作为输入转移到新的状态，而在这一时刻的状态被称为世界状态（输入为用户发起的交易；状态为所有账户的余额，存储，代码等信息；世界状态表示当前时刻以太坊的完整账本状态）

这句话在资料中被NFT市场举例：假如A持有一个NFT的状态为S3，A把NFT卖给我时状态会转移到S4，类似的将NFT卖回给A那么状态会转移到S5。

实际情况是多比交易被打包成一个区块然后区块按顺序连接起来这就成了区块链

以上的信息可以结合起来，用户像EVM发起交易（这里不是向交易对象发起交易是因为无论是用户自身，还是交易对象在EVM这里都只是一串20 字节的地址所以，本质上来说你只是提供了一个地址并让EVM找到这个地址并进行交易），然后EVM根据用户的指令执行，最后完成交易，而其中EVM就决定了以太坊如何根据当前交易更新状态。

**在某一时刻的以太坊状态全部的状态被叫做以太坊世界状态，而世界状态其实就是一个巨大的映射表（address => User\_State）**

在以太坊中账户分为两类合约账户和外部账户两者的对比如下：

| 类型 | 是否有私钥 | EVM 代码 | 用途 |
| 外部账户 | 有 | 空 | 普通用户，通过私钥控制 |
| 合约账户 | 无 | 非空 | 智能合约，代码控制行为 |

**关键点**：合约账户**没有私钥**，它的行为完全由代码决定。

软件在被转译成机器的语言的时候不同的CPU呈现出来也是不同的所以实现上有两种方式一种是实现目标的虚拟机：他的工作原理是源代码被编译成字节码，即代表指令的字节序列。每个字节码对应虚拟机要执行的特定操作

还有一种是基于平台的虚拟机，EVM就属于这种虚拟机：将字节码翻译成本地代码以供执行。

如果虚拟机作为一种抽象层EVM就是他的实现如果把EVM比作CPU那么它的那么他能一次性处理32字节的数据假设EVM被证明在操作账户实例的存储、代码和余额那么他的运行流程如图

他表示EVM 从当前世界状态读取数据，并在收到带有参数（Input）的交易（Transaction）触发后，利用其环境（Environment）中的栈、内存和Gas来执行合约代码（Code），最终产生输出（Output）并计算出新的世界状态（Next world state），然后将存储（Storage）和余额（Balance）的变更永久更新到世界状态中。

**注意**：EVM 是以太坊状态机的**状态转换函数**。它根据输入（交易）和当前状态，决定以太坊如何转换到新的（世界）状态。它被实现为虚拟机，以便能在任何平台上运行，独立于底层硬件。这里是资料中提到其实和上面的NFT市场和自动贩卖机结合的例子一样

EVM在执行期间有四个主要的数据存储位置：**栈（Stack）内存（Memory）存储（Storage）调用数据（Calldata）**

**栈是一个简单的数据结构其中大多数的操作码都是在栈上操作，它负责保存于读写内存和存储的值，它还有两个操作：PUSH 和 POP。PUSH 向栈顶添加一个项，POP 移除栈顶，EVM 栈的最大大小为 1024 个项，每个项 32 字节，并在每次合约执行后重置。只有栈顶的 16 个项是可访问的。如果栈溢出，合约执行将失败。**

**这 16 个项的限制是因为 DUP 和 SWAP 操作码：**

-   **DUPn：复制第 n 个栈项到栈顶**
    
-   **SWAPn：交换栈顶与第 n+1 个项**
    

**并且栈遵循的是先入后出的原则最后添加的元素往往最先被移除如果对空栈执行POP会触发栈溢出错误，EVM 使用栈的主要目的是存储计算中的中间值以及为操作码提供参数。**

EVM在每执行完一条指令的时候EVM就会通过**程序计数器（Program Counter）跳转到下一个要执行的指令操作码的位置其中JUMP 操作码直接设置了程序计数器，实现动态控制流，通过允许灵活的程序执行路径，使 EVM 具备图灵完备性。**

**高级语言如 Solidity 利用 JUMP 来实现 if 条件、for 循环和内部函数调用。**

而在EVM中无限循环是很致命的它会导致Dos攻击所以EVM的gas机制就是用来应对此类威胁，gas机制设计就是用来限制硬件，交易需要使用gas才能使用EVM如果gas耗尽EVM就会终止他们防止资源占用。

Memory（内存）：简单来说EVM内存就像一个临时的草稿本，每次执行合约都会发一本新的，用完就扔。内存通过允许**按索引访问任意大小的数据**来补充栈。栈中的值可以按需存储到内存或从内存加载，这里有几个比较重要的操作码

| MLOAD | 读取内存 | 从指定地址读32字节，推入栈顶 |
| MSTORE | 写入内存 | 从栈取(地址, 32字节值)，写入内存 |
| MSTORE8 | 写入1字节 | 同上，但只写1字节到内存 |
| MSIZE | 查询内存大小 | 返回当前已使用的内存字节数（推入栈顶） |

**调用数据（Calldata）：就是调用指令和交易传递给EVM**

**存储（Storage）：于内存不同存储于以太坊账户关联并作为世界状态的一部分在交易中持久化所以他被看成于智能合约关联的数据库，而存储只能通过关联账户访问，外部账户没有代码因此无法访问自己的存储**

**接着昨天的交易执行分为两个阶段在第一个阶段首先会验证交易签名是否有效如果有效则扣除基本前面提到的至_少_消耗 21,000 Gas，交易附带的数据非零字节每个16gas零字节每个4gas然后增加发送方的nonce然后从发送方的余额中扣除预付款当然也有例外**

**σ0\[Sender\]\_balance ≡ σ\[Sender\]\_balance - upfrontCost  
σ0\[Sender\]\_nonce ≡ σ\[Sender\]\_nonce + 1**

**在这个检查点，系统只改了发送方的钱和计数器，别的什么都没做。哪怕后续交易执行失败了（比如合约抛异常），这两处修改也不会回滚，因为预付款已经扣了，nonce 已经加了，这个也就不能改变了**

**然后进入第二阶段系统会讲交易用一种就叫做Message的格式封装然后交给EVM进行执行**

**如果 To 地址是空的 → 创建新合约。**

**如果 To 地址有内容 → 调用某个合约（或转账）。**

**EVM收到过后会按如下的流程执行：**

**识别要执行的指令→扣除当前指令消耗的GAS→执行指令→程序计数器指向下一条指令，一直到有STOP RETURN SELFDESTRUCT这三种指令(前两者都是执行完成的意思但是RETURN有返回值，STOP没有，SELFDESTRUCT代表自毁冰雪指定合约发起转账)以上是正常的停止方式，也会有异常停止方式包括Gas不够、栈溢出、无效跳转、REVERT等**

**而后交易会处于一个即将上链的准备阶段在这个阶段EVM会把执行合约产生的改动存入一个叫σ\_p的临时结构中 外部查询（比如 eth\_call）还看不到这些改动，因为总账本 σ 还没变，在这个阶段4σ\_p 里的内容还是易失的。如果接下来发生以下情况，这张“草稿纸”会被直接扔掉：**

-   **交易执行失败（比如最后执行了 REVERT）**
    
-   **区块被重组**
    
-   **节点在同步过程中发现分叉**
    

**合约执行完成EVM会把合约执行产生的结果进行执行，把没用完的gas退还，SELFDESTRUCT 的合约把余额转给指定地址，LOG 数据进入区块的日志树**

# 2026-04-09
<!-- DAILY_CHECKIN_2026-04-09_START -->






接着昨天的以太坊通过这些额外的检查为后续从Pow能平稳过渡到Pos提供了坚实的基础，而1599EIP-1559 引入了基础费（Base Fee）+ 优先费（Priority Fee/Tip）的双层机制：也为 PoS 时代的验证者激励机制提供了一个更公平、更可控的框架

这些标准构成了以太坊经济模型的一部分尤其受EIP-1559影响诞生了EIP-4844该提案带来了一种新型交易——blob交易（简单来说4844其实就是Layer 2网络的1559而两者区别在于**EIP-4844 的数据是临时的、执行层不可读的、不挤占普通交易资源的**）

初步的块头验证，可以让STF在不进行执行和计算的前提下向CL报告无效区块，以避免恶意交易挤占计算资源（需要注意的是执行阶段的验证要么成功要么失败不会中途停止，所以进行初步的筛查是很有必要的）

而通过块头验证过后就会进行正式的执行阶段，

1.        **初始化** blobGasUsed **为 0：**在 EIP-4844 之后，区块有了两种 Gas：传统的 gasUsed（普通交易）和 blobGasUsed（Blob 交易）。将它们分开统计，是独立定价和计费的基础

2.        **设置** gasAvailable = H\_gasLimit：给区块设置一个总预算，防止单个交易或整个区块的执行消耗无限的计算资源。

3.        **初始化额外的执行组件**

这些结构用于记录执行结果。收据树证明交易执行结果，提现树记录质押者的 ETH 提取，日志元组则为外部应用（如 DApp）提供查询事件（Event）的能力

**4.       访问信标区块根合约（EIP-4788）：**功能相当于预言机,不过他是个智能合约这个合约就像一个公共公告板。以太坊协议本身每天（每个区块）都会自动把“共识层的快照”贴到这个公告板上

**5.       构建一条系统交易消息：**这条交易是由EL 构造一个特殊交易，由“系统地址”调用信标根合约与普通用户发起的交易不同，这是协议自身发起的内部交易。只有系统地址有权限向信标根合约写入数据而这样做能让智能合约能够以无需信任、无需预言机、无需第三方的方式，直接读取共识层（信标链）的真实状态**。**

6.        **设置 VM 环境并处理消息调用：**虚拟机会执行上述系统交易。将当前区块的“父信标区块根”正式存入合约的存储中。这样一来，未来的任何交易，只要提供正确的时间戳，就能从该合约中查询到那时的共识层状态。

**7.**       删除空账户：字面意思，删除可能产生的余额为 0、代码为空的“空账户”**。**

**8.       处理区块内的交易：这是区块执行的核心，即处理所有用户发起的普通交易**

**具体可分为一下几个步骤：**

**第一步：解码并添加到交易树：**

**输入原始的、压缩的、字节形式的交易数据（RLP编码）。同时，计算这个交易对象的哈希。** **将这些哈希按顺序组织起来，构建一棵交易树。这棵树的根哈希会被填入区块头，起到“目录”作用。任何人只要拥有完整的区块数据，就可以通过这个根哈希验证某笔交易是否真的在这个区块里。**

**第二步：执行交易（细分5个小步骤）**

**这是最核心的部分。对于每一笔交易，节点都会执行以下5步：**

**1\. 恢复发送者地址**

**交易是带着签名的（v, r, s）。节点用这个签名反向计算出发起人的公钥，再推算出他的以太坊地址（from）。确定谁在付Gas费。验证交易没有被篡改（签名有效）。防止伪造交易（没有私钥的人无法盗用别人的地址）。**

**2\. 验证固有有效性检查项：**

-   **to 地址格式是否合法（如果是合约部署，则为空）。**
    
-   **value 值是否为有效数字。**
    
-   **gas 是否小于区块的Gas上限。**
    
-   **发送者的nonce是否与账户的当前nonce一致（防止重放攻击）。**
    

**3\. 计算有效Gas价格**

**根据交易中设定的 maxPriorityFeePerGas（愿意给验证者的小费）和 maxFeePerGas（愿意支付的最高总费用），以及当前区块的 baseFeePerGas（基础费，由网络拥堵程度决定），计算出这笔交易实际要支付的：**

**如果 maxFeePerGas < baseFeePerGas，则交易无效，因为它连基础费都付不起。**

**4\. 初始化执行环境**

-   **创建一个新的EVM实例。**
    
-   **加载当前的世界状态（账户余额、合约代码、存储等）。**
    
-   **设置交易的 gas 上限。**
    
-   **准备 msg.sender（交易发起者）、msg.value（转账金额）等上下文变量。**
    
-   **扣除交易最大可能消耗的Gas（gas \* maxFeePerGas），防止执行过程中Gas不够用。**
    

**5\. 在虚拟机内执行**

-   **EVM 逐条读取交易的指令（例如 PUSH、ADD、SSTORE、CALL 等）。**
    
-   **每执行一条指令，都进行：**
    
    -   **状态验证：比如检查账户余额是否足够。**
        
    -   **Gas扣费：每条指令都有固定的Gas成本，从之前扣留的Gas中逐次扣除。**
        
-   **结果分支：**
    
    -   **成功：所有指令执行完毕，Gas有剩余。状态变更被永久保存到世界状态中。剩余的Gas退还给 from 地址。**
        
    -   **失败：某条指令执行时Gas耗尽，或者触发了 REVERT 指令。所有已经执行的状态变更全部回滚，世界状态回到这笔交易开始前的样子。但是，已经消耗的Gas（用于计算的那部分）不会退还，这些Gas会被验证者收走（作为给他们的补偿，因为他们付出了计算资源）。**

# 2026-04-08
<!-- DAILY_CHECKIN_2026-04-08_START -->







今天学习的是区块头的验证这部分，他的验证有一系列复杂的标准，这里我们直接把他总结成几句话：

1.        一个区块它实际消耗的gasUser不能超过一个区块能容纳的gasLimit上限以确保符合区块容量范围，换句话说就是一个区块消耗的gas费不能超过这个区块的上限

2.        新区块的Gas上限只能在父区块Gas上限附近微调，不能一下子改得太多。

3.        一个交易能达到的最低gas费为5000，以保证基本的交易能够进行

4.        一个区块的时间戳必须大于他的父区块的时间戳确保时间排序正确

5.        新诞生的区块链必须有能够进行追溯的父区块，祖区块以保证这个新区块所处的链是最长合法链，而新诞生的区块链extraData的字段要 ≤ 32 字节，防止区块链过于臃肿

6.        单位Gas的基础费 根据 EIP-1559 中设定的规则计算，反映网络当前对交易处理的需求，这里提到了两个概念一个“基础费”另一个是“EIP-1559”

交易费每笔交易必须支付的“底价”，由网络根据拥堵程度自动计算。网络越忙，基础费越高；越空闲，基础费越低

EIP-1559 ：通俗的来说这是一个基础费+小费的费用模型

其中的1，2，3，6一起规定了 Gas 费如何计算、如何燃烧（baseFee 被销毁）、如何奖励矿工/验证者。

前面提到了EIP-1559它的作用其实就是引入了一个经济模型旨在提升网络效率与稳定性的机制而他的核心要领可以总结为以下几点

l  **设定Gas目标以降低波动性**：意思是以太坊不会让每个区块都满负荷运转，来降低交易环境的波动性

l  **降低网络延迟**：尽可能的优化交易时间，以减少用户的等待时间

l  \*\*稳定区块奖励发行：\*\*通过稳定可预测的方式给予验证者奖励，让用户能规划好自己的行为

l  **可预测的基本费用调整**：EIP-1559引入了可预测的基本费用调整机制，这一功能对钱包尤为有利。这种可预测性有助于提前准确估算交易成本，简化交易创建流程

l  **基础费用烧毁与优先费用**：根据该模式，矿工有权保留优先费用作为激励，而基础费用则被烧毁，实际上将其从流通中移除。这种做法作为对以太坊通胀的对策，通过逐步减少整体供应，促进更健康的经济环境

这句话可以拆开来看在EIP-1559这个模式下假如我要在以太坊上发布交易那么我支付的gas费则会被分为基础费和优先费（也称小费）基础费会作为我上传交易所需的手续费而被销毁，而优先费则会奖励给验证者，这样做既能防止过多的ETH在链上流通也能激励验证者，这个机制巧妙地将网络的使用价值（通过销毁）反哺给了 ETH 代币本身，使以太坊的经济模型更加自洽和可持续

[Architecture部分：](https://epf.wiki/#/wiki/protocol/architecture)介绍了协议是由两个部分——执行层（EL）和共识层（CL）两部分组成

EL负责实际处理用户端发出的交易请求 CL负责监管和验证节点是否遵守规范，

并且EL和CL各自连接不同的P2P网络来获取“原材料”（交易和区块），并通过API进行两个层级之间的数据传输

[Design rationale：](https://epf.wiki/#/wiki/protocol/design-rationale)讲述了以太坊自诞生之初的设计理念：**简洁性**，**普遍性，模块化，非歧视性，敏捷性**：

而协议的设计理念需要遵守如下的原则

-   **满足需求的前提下，协议应尽可能简单。** 当复杂性不可避免时，遵循"夹心模型"——让底层（核心共识）和顶层（用户接口）保持简单，将复杂性推入协议的"中间层"。
    
-   **宁可选择内部复杂但有清晰外部接口的子系统，也不选择各部分纠缠不清的系统。**
    
-   **用户使用协议的目的不应受到限制（网络中立性原则）。**
    
-   **协议功能和操作码应代表尽可能底层的概念，以便能以任意方式组合。**
    
-   **拒绝将常见的高层次用例直接纳入协议。** 如果用户真的需要，完全可以在合约层面通过底层功能模拟实现。

# 2026-04-09
<!-- DAILY_CHECKIN_2026-04-09_START -->
接着昨天的以太坊通过这些额外的检查为后续从Pow能平稳过渡到Pos提供了坚实的基础，而1599EIP-1559 引入了基础费（Base Fee）+ 优先费（Priority Fee/Tip）的双层机制：也为 PoS 时代的验证者激励机制提供了一个更公平、更可控的框架

这些标准构成了以太坊经济模型的一部分尤其受EIP-1559影响诞生了EIP-4844该提案带来了一种新型交易——blob交易（简单来说4844其实就是Layer 2网络的1559而两者区别在于**EIP-4844 的数据是临时的、执行层不可读的、不挤占普通交易资源的**）

初步的块头验证，可以让STF在不进行执行和计算的前提下向CL报告无效区块，以避免恶意交易挤占计算资源（需要注意的是执行阶段的验证要么成功要么失败不会中途停止，所以进行初步的筛查是很有必要的）

而通过块头验证过后就会进行正式的执行阶段，

1.        **初始化** blobGasUsed **为 0：**在 EIP-4844 之后，区块有了两种 Gas：传统的 gasUsed（普通交易）和 blobGasUsed（Blob 交易）。将它们分开统计，是独立定价和计费的基础

2.        **设置** gasAvailable = H\_gasLimit：给区块设置一个总预算，防止单个交易或整个区块的执行消耗无限的计算资源。

3.        **初始化额外的执行组件**

这些结构用于记录执行结果。收据树证明交易执行结果，提现树记录质押者的 ETH 提取，日志元组则为外部应用（如 DApp）提供查询事件（Event）的能力

**4.       访问信标区块根合约（EIP-4788）：**功能相当于预言机,不过他是个智能合约这个合约就像一个公共公告板。以太坊协议本身每天（每个区块）都会自动把“共识层的快照”贴到这个公告板上

**5.       构建一条系统交易消息：**这条交易是由EL 构造一个特殊交易，由“系统地址”调用信标根合约与普通用户发起的交易不同，这是协议自身发起的内部交易。只有系统地址有权限向信标根合约写入数据而这样做能让智能合约能够以无需信任、无需预言机、无需第三方的方式，直接读取共识层（信标链）的真实状态**。**

6.        **设置 VM 环境并处理消息调用：**虚拟机会执行上述系统交易。将当前区块的“父信标区块根”正式存入合约的存储中。这样一来，未来的任何交易，只要提供正确的时间戳，就能从该合约中查询到那时的共识层状态。

**7.**       删除空账户：字面意思，删除可能产生的余额为 0、代码为空的“空账户”**。**

**8.       处理区块内的交易：这是区块执行的核心，即处理所有用户发起的普通交易**

**具体可分为一下几个步骤：**

**第一步：解码并添加到交易树：**

**输入原始的、压缩的、字节形式的交易数据（RLP编码）。同时，计算这个交易对象的哈希。** **将这些哈希按顺序组织起来，构建一棵交易树。这棵树的根哈希会被填入区块头，起到“目录”作用。任何人只要拥有完整的区块数据，就可以通过这个根哈希验证某笔交易是否真的在这个区块里。**

**第二步：执行交易（细分5个小步骤）**

**这是最核心的部分。对于每一笔交易，节点都会执行以下5步：**

**1\. 恢复发送者地址**

**交易是带着签名的（v, r, s）。节点用这个签名反向计算出发起人的公钥，再推算出他的以太坊地址（from）。确定谁在付Gas费。验证交易没有被篡改（签名有效）。防止伪造交易（没有私钥的人无法盗用别人的地址）。**

**2\. 验证固有有效性检查项：**

-   **to 地址格式是否合法（如果是合约部署，则为空）。**
    
-   **value 值是否为有效数字。**
    
-   **gas 是否小于区块的Gas上限。**
    
-   **发送者的nonce是否与账户的当前nonce一致（防止重放攻击）。**
    

**3\. 计算有效Gas价格**

**根据交易中设定的 maxPriorityFeePerGas（愿意给验证者的小费）和 maxFeePerGas（愿意支付的最高总费用），以及当前区块的 baseFeePerGas（基础费，由网络拥堵程度决定），计算出这笔交易实际要支付的：**

**如果 maxFeePerGas < baseFeePerGas，则交易无效，因为它连基础费都付不起。**

**4\. 初始化执行环境**

-   **创建一个新的EVM实例。**
    
-   **加载当前的世界状态（账户余额、合约代码、存储等）。**
    
-   **设置交易的 gas 上限。**
    
-   **准备 msg.sender（交易发起者）、msg.value（转账金额）等上下文变量。**
    
-   **扣除交易最大可能消耗的Gas（gas \* maxFeePerGas），防止执行过程中Gas不够用。**
    

**5\. 在虚拟机内执行**

-   **EVM 逐条读取交易的指令（例如 PUSH、ADD、SSTORE、CALL 等）。**
    
-   **每执行一条指令，都进行：**
    
    -   **状态验证：比如检查账户余额是否足够。**
        
    -   **Gas扣费：每条指令都有固定的Gas成本，从之前扣留的Gas中逐次扣除。**
        
-   **结果分支：**
    
    -   **成功：所有指令执行完毕，Gas有剩余。状态变更被永久保存到世界状态中。剩余的Gas退还给 from 地址。**
        
    -   **失败：某条指令执行时Gas耗尽，或者触发了 REVERT 指令。所有已经执行的状态变更全部回滚，世界状态回到这笔交易开始前的样子。但是，已经消耗的Gas（用于计算的那部分）不会退还，这些Gas会被验证者收走（作为给他们的补偿，因为他们付出了计算资源）。**

# 2026-04-08
<!-- DAILY_CHECKIN_2026-04-08_START -->

今天学习的是区块头的验证这部分，他的验证有一系列复杂的标准，这里我们直接把他总结成几句话：

1.        一个区块它实际消耗的gasUser不能超过一个区块能容纳的gasLimit上限以确保符合区块容量范围，换句话说就是一个区块消耗的gas费不能超过这个区块的上限

2.        新区块的Gas上限只能在父区块Gas上限附近微调，不能一下子改得太多。

3.        一个交易能达到的最低gas费为5000，以保证基本的交易能够进行

4.        一个区块的时间戳必须大于他的父区块的时间戳确保时间排序正确

5.        新诞生的区块链必须有能够进行追溯的父区块，祖区块以保证这个新区块所处的链是最长合法链，而新诞生的区块链extraData的字段要 ≤ 32 字节，防止区块链过于臃肿

6.        单位Gas的基础费 根据 EIP-1559 中设定的规则计算，反映网络当前对交易处理的需求，这里提到了两个概念一个“基础费”另一个是“EIP-1559”

交易费每笔交易必须支付的“底价”，由网络根据拥堵程度自动计算。网络越忙，基础费越高；越空闲，基础费越低

EIP-1559 ：通俗的来说这是一个基础费+小费的费用模型

其中的1，2，3，6一起规定了 Gas 费如何计算、如何燃烧（baseFee 被销毁）、如何奖励矿工/验证者。

前面提到了EIP-1559它的作用其实就是引入了一个经济模型旨在提升网络效率与稳定性的机制而他的核心要领可以总结为以下几点

l  **设定Gas目标以降低波动性**：意思是以太坊不会让每个区块都满负荷运转，来降低交易环境的波动性

l  **降低网络延迟**：尽可能的优化交易时间，以减少用户的等待时间

l  \*\*稳定区块奖励发行：\*\*通过稳定可预测的方式给予验证者奖励，让用户能规划好自己的行为

l  **可预测的基本费用调整**：EIP-1559引入了可预测的基本费用调整机制，这一功能对钱包尤为有利。这种可预测性有助于提前准确估算交易成本，简化交易创建流程

l  **基础费用烧毁与优先费用**：根据该模式，矿工有权保留优先费用作为激励，而基础费用则被烧毁，实际上将其从流通中移除。这种做法作为对以太坊通胀的对策，通过逐步减少整体供应，促进更健康的经济环境

这句话可以拆开来看在EIP-1559这个模式下假如我要在以太坊上发布交易那么我支付的gas费则会被分为基础费和优先费（也称小费）基础费会作为我上传交易所需的手续费而被销毁，而优先费则会奖励给验证者，这样做既能防止过多的ETH在链上流通也能激励验证者，这个机制巧妙地将网络的使用价值（通过销毁）反哺给了 ETH 代币本身，使以太坊的经济模型更加自洽和可持续

今天主要学习的是Execution Layer (EL) 基础部分的[EL Specs](https://epf.wiki/#/wiki/EL/el-specs)在开篇就已经提到过了**黄皮书（巴黎版 v705168a – 2024-03-04）**这个版本已经过时了所以下面主要看的是状态转移函数（STF）解决的两个问题

-   **这个区块是否可以附加到区块链的末端？**
    
-   **状态会因此如何变化**
    

而这个函数的实现被总结为一个公式

**σt+1​≡Π(σt​,B)**

**其中的σt+1是新产生的区块**

**Π代表STF这个函数**

**σt表示为当前区块**

**B表示为待处理的区块**

**其中明确强调了σ（状态） ≠ Python 规范中的 State 类。**

**系统状态不是静态存储的，而是通过状态转换函数动态推导的。**

**这凸显了区块链状态转换的数学模型与软件规范中实际实现细节之间的概念分离**

**而后给出了一张流程图这张图片能总结出几个公式而其中最重要的有**

**第6个_HstateRoot_​≡_TRIE_(_Ls_​(Π(_σ_,_B_)))**

**他表示的是当前区块头里的状态根 = 执行完当前区块所有交易后的新状态，再压缩成哈希。**

**这是验证区块是否有效的核心公式。节点执行完区块后，如果算出的根与区块头里的不一致，直接拒绝该区块。**

**第3_个Ls​(σ)≡{p(a):σ\[a\]=∅}_：世界状态折叠函数**

**把"地址 → 账户"的映射转换成可以插入默克尔树（Trie）的键值对列表。**

**没有这个转换，庞大的世界状态就无法被压缩成一个32字节的根哈希。它是连接"原始数据"和"密码学验证"的桥梁。**

**第一个_TRIE_(_LI_∗​(_σ_\[_a_\]_s_​))≡_σ_\[_a_\]_s_​**

**每个账户的存储空间（键值对数据库）被压缩成一个存储根哈希。**

**这保证了账户内部的数据（比如你的ERC-20代币余额、NFT编号）一旦被篡改，根哈希就会变化，立刻被全网发现。**

**这些公式共同组成了以太坊的“数据一致性检测系统”。”**

今天主要学习了第一周课程的The Protocol部分

其中Prehistory部分讲述了以太坊的发展历程它代表的是根植于**早期互联网的开放精神**、**Unix的简洁哲学**、**自由开源软件运动**、**密码朋克的隐私抗争**，以及**比特币的去中心化货币实验**

[Architecture部分：](https://epf.wiki/#/wiki/protocol/architecture)介绍了协议是由两个部分——执行层（EL）和共识层（CL）两部分组成

EL负责实际处理用户端发出的交易请求 CL负责监管和验证节点是否遵守规范，

并且EL和CL各自连接不同的P2P网络来获取“原材料”（交易和区块），并通过API进行两个层级之间的数据传输

[Design rationale：](https://epf.wiki/#/wiki/protocol/design-rationale)讲述了以太坊自诞生之初的设计理念：**简洁性**，**普遍性，模块化，非歧视性，敏捷性**：

而协议的设计理念需要遵守如下的原则

-   **满足需求的前提下，协议应尽可能简单。** 当复杂性不可避免时，遵循"夹心模型"——让底层（核心共识）和顶层（用户接口）保持简单，将复杂性推入协议的"中间层"。
    
-   **宁可选择内部复杂但有清晰外部接口的子系统，也不选择各部分纠缠不清的系统。**
    
-   **用户使用协议的目的不应受到限制（网络中立性原则）。**
    
-   **协议功能和操作码应代表尽可能底层的概念，以便能以任意方式组合。**
    
-   **拒绝将常见的高层次用例直接纳入协议。** 如果用户真的需要，完全可以在合约层面通过底层功能模拟实现。
    

**在最后的**[**Evolution**](https://epf.wiki/#/wiki/protocol/history)**部分讲述了以太坊从发布之初的几个重要节点以及其引入的一些特色协议**

## ERC-721 与 ERC-1155 深度研读笔记

### 一、 哲学起源：独立个体 vs. 工业集群

**ERC-721：独特的原子** ERC-721 的核心设计目标是“非同质化”。每一个代币都拥有一个全网唯一的数字身份。它的设计逻辑是\*\*“面向资产”\*\*的，将每一个 Token 视为一个独立的、不可分割的实体。

**ERC-1155：高效的容器** ERC-1155 由 Enjin 团队提出，其核心目标是“多代币管理”。它不再纠结于单一资产的独特性，而是引入了\*\*“面向类别”\*\*的设计。在一个合约内，它可以同时容纳无数种同质化代币（FT）和非同质化代币（NFT）。

* * *

### 二、 存储架构：映射逻辑的降维打击

**映射结构的差异**

-   **ERC-721** 内部使用 `mapping(uint256 => address)`，即“ID 对应 所有人”。如果你想知道某人有多少币，必须遍历或查看另一个专门的计数映射。
    
-   **ERC-1155** 使用 `mapping(uint256 => mapping(address => uint256))`，即“ID 对应 地址 对应 余额”。这让它在处理“半同质化”资产时具备天然优势，比如游戏里的 100 瓶红药水，只需要修改一个余额数值，而不是生成 100 个 ID。
    

* * *

### 三、 性能革命：Gas 消耗的帕累托改进

**批量操作（Batch Operations）** 这是两者在技术表现上的最大分水岭。

-   **ERC-721 的线性成本：** 如果你需要转移 10 个不同的 NFT，你必须发起 10 笔独立的交易，或者在循环中调用 10 次转移函数。每一笔操作都要支付固定的基础 Gas 费。
    
-   **ERC-1155 的常数级优化：** 通过 `safeBatchTransferFrom` 函数，你可以在一笔交易中打包发送几十种不同 ID 和数量的代币。由于减少了交易确认的次数和状态修改的冗余，批量操作的 Gas 成本较前者可降低 **60% 以上**。
    

* * *

### 四、 接口与元数据：灵活度与标准化的博弈

**元数据管理**

-   **ERC-721：** 每个 Token 通常需要单独设置 `tokenURI`。虽然灵活性极高，但对于拥有数万个文件的项目方来说，维护链上字符串是一笔巨大的开销。
    
-   **ERC-1155：** 引入了 `ID 替换机制`。合约只需存储一个带有 `{id}` 占位符的通用 URI。当外部查询时，程序会自动将十六进制 ID 填入，实现了一套逻辑管理无限资产的目标。
    

**接收安全性** 两者都要求接收方（如果是合约）必须实现特定的“接收钩子”函数，以防止代币锁死在不支持 NFT 的合约中。但 ERC-1155 的钩子会额外传递“转账数量”和“额外数据”等参数，这使得它在处理复杂的链上自动化交易（如自动化兑换、合成）时更加智能。

* * *

### 五、 商业适用性：如何进行选型？

**选择 ERC-721 的场景：**

-   **高净值艺术品：** 每一件作品都有独特的历史和叙事。
    
-   **数字身份（DID）：** 如 ENS 域名，每个名字都是唯一的。
    
-   **实体资产证明：** 房产证、奢侈品证书等一一对应的资产。
    

**选择 ERC-1155 的场景：**

-   **大型游戏生态：** 既有独特的皮肤（NFT），又有成千上万的消耗品材料（FT）。
    
-   **社交票务系统：** 比如发放 500 张完全一样的会议门票，ERC-1155 的铸造和分发效率极高。
    
-   **跨平台互操作：** 需要在一个合约里通过低成本 Gas 实现多种复杂资产流转。
    

* * *

### 六、 总结语

ERC-721 确保了\*\*“稀缺性的绝对性”**，它是 NFT 精神的图腾。 ERC-1155 实现了**“操作的极致效率”\*\*，它是大规模 Web3 应用的基石。

这两者不应看作是“新旧更替”，而应看作是“特种兵”与“工业化集团军”在不同战场上的专业化分工。在实际开发中，开发者应根据业务中资产的\*\*“重复性”**和**“流转频率”\*\*来决定最终的标准选型。

# **ERC-6551（代币绑定账户）深度研读笔记**

## **一、概述：什么是 ERC-6551？**

ERC-6551 是一个以太坊代币标准提案，于 2023 年 5 月 7 日在以太坊主网上正式发布。它的核心创新是：**为每一个 ERC-721 NFT 分配一个独立的智能合约账户**，这个账户被称为“代币绑定账户”（Token Bound Account，简称 TBA）。

用最直观的方式来理解：传统的 NFT 只是一张“图片”或“证书”，被你放在钱包里；而引入 ERC-6551 之后，这张“小图片”本身变成了一个“钱包”。你的钱包控制着这个 NFT，而这个 NFT 又可以拥有其他资产——这就形成了一种“钱包套钱包”的层级关系。

该标准由 Future Primitive 团队联合 Manifold 及 Token Bound 社区共同开发，旨在突破 ERC-721 代币“无法作为链上资产所有人”或“无法像传统以太坊账户那样执行操作”的核心限制。

## **二、核心技术原理**

**系统架构的两部分**

第一部分是注册表（Registry），一个无需许可、不可变、无所有者的单例合约，负责为 NFT 部署和计算 TBA 地址。

第二部分是账户接口（Account Interface），定义了 TBA 必须实现的标准功能，包括 token()、state()、isValidSigner() 等。

**CREATE2 机制与确定性地址**

TBA 的地址是通过 CREATE2 操作码确定性地派生出来的。这意味着，在账户实际部署之前，就可以计算出它的地址。

地址的派生基于以下参数的组合：

-   实现合约地址（implementation）
    
-   盐值（salt）
    
-   链 ID（chainId）
    
-   NFT 合约地址（tokenContract）
    
-   NFT 的 ID（tokenId）
    

这种设计的优势在于：dApp 可以在 TBA 尚未部署时就引用其地址，提前为其充值或设置授权。

**最小代理模式**

注册表使用 ERC-1167 最小代理来部署每个 TBA。这是一种 Gas 高效的部署方式——每个账户都指向同一个实现合约，但拥有独立的存储空间。部署后的字节码结构会将上述参数（salt、chainId、tokenContract、tokenId）作为不可变数据附加到代理末尾。

**所有权与控制权逻辑**

TBA 的控制权逻辑非常精巧：

-   账户的 owner() 函数会动态读取当前 NFT 的持有者地址
    
-   当 NFT 被转移时，TBA 的控制权自动跟随到新的持有者，无需任何密钥或签名转移操作
    
-   当 TBA 执行操作时，会验证操作发起者是否与当前 NFT 持有者地址一致
    

这意味着：**卖 NFT 就等于卖这个 NFT 下面的整个“资产包”**，所有绑定在 TBA 中的资产会随 NFT 一起转移。

## **三、核心接口规范**

**注册表接口（IERC6551Registry）**

注册表需要实现以下核心功能：

-   一个 AccountCreated 事件，记录账户创建信息
    
-   createAccount 函数，用于创建 TBA 账户
    
-   account 函数，用于根据参数查询 TBA 地址
    

**账户接口（IERC6551Account）**

账户需要实现以下核心功能：

-   receive 函数，使账户可以接收 ETH
    
-   token 函数，返回该账户绑定的 NFT 信息（链 ID、合约地址、代币 ID）
    
-   state 函数，返回账户的当前状态
    
-   isValidSigner 函数，验证给定签名者是否为有效签名者
    

账户还必须实现 ERC-165（接口检测）和 ERC-1271（签名验证）。

## **四、应用场景深度分析**

**场景一：链游——角色即账户**

这是最直观、也最被看好的应用方向。

在传统链游中，你的游戏角色 NFT 和装备 NFT 是分离的，都存放在你的个人钱包里。想卖掉角色时，装备得一个一个转移，非常繁琐。

有了 ERC-6551 之后，每个游戏角色 NFT 都可以拥有自己的 TBA。你在游戏中获得的装备、药水（ERC-20 代币）、成就徽章等，都可以直接存入这个角色的 TBA 中。当你决定“退坑”时，只需要把角色 NFT 卖掉，所有配套资产会随之一并转移——相当于卖了一个完整的“游戏账号”。这极大地提升了资产的流动性和交易效率。

更进一步，角色 NFT 还可以记录多维度的数据——比如像 CS:GO 的皮肤那样记录“杀敌数”，这些数据随着角色 NFT 的转移而转移。

**场景二：SocialFi 与去中心化身份**

ERC-6551 为链上身份带来了全新的范式。

此前，Vitalik Buterin 提出的 SBT（灵魂绑定代币）思路是“让钱包绑定资产来证明身份”；而 ERC-6551 的思路是“让 NFT 成为钱包”——你持有的某个 NFT 就是你的链上身份。

这种设计的优势在于：

-   可移植的声誉系统：一个 NFT 可以跨应用携带其社交历史、信誉评分和参与记录。你不再需要在每个新平台上重新建立信任。
    
-   POAP 与参与证明：项目方可以直接查看某个 NFT 的 TBA 中积累了多少 POAP，以此判断用户是否为“优质用户”，淘汰白名单 PUA 的不良风气。
    
-   低成本的信任机制：TBA 能够迅速验证链上行为记录，大大降低 Web3 社交中的信任成本。
    

SocialFi 的核心诉求——无平台依赖、数据可移植、高可组合性——恰好与 TBA 的特性高度契合。

**场景三：NFT 金融化与可编程资产**

积分交易：Pichi Finance 已经实现了这样的场景——用户创建一个“Michi Wallet NFT”，向关联的 ERC-6551 钱包存入可产生积分的代币。积分积累后，用户可以取出产生积分的代币但保留积分在钱包中，然后将整个 NFT 卖出。本质上是在交易“积分”本身，这为 TGE（代币生成事件）前后的积分定价提供了全新的解决方案。

空投革新：传统的空投需要项目方进行“快照”——在某个时间点记录所有持有者的地址。有了 ERC-6551，项目方可以直接向 NFT 的 TBA 地址进行空投，快照可以被淘汰了。

资产管理：投资组合 NFT 可以将多种资产（ERC-20 代币、其他 NFT）打包进一个 TBA，作为一个整体进行交易或作为抵押品。

**场景四：可组合的 NFT 与数字时尚**

一个 PFP 项目可以通过 ERC-6551 实现更复杂的可组合性。比如，你可以为自己的“无聊猿”购买一件虚拟潮牌服饰，这件服饰直接进入该无聊猿的 TBA。你展示的不仅仅是“我拥有这只猿”，更是“这只猿拥有这些衣服”——NFT 成为了它自己的数字衣橱。

这种模式为虚拟时尚品牌（如 RTFKT）与 PFP 项目的合作提供了技术基础。

## **五、安全模型与风险**

**授权残留问题**

这是最核心的安全问题。如果 TBA 中存在未清理的授权（比如对某个合约的 ERC-20 无限额授权），当 NFT 被转移时，这些授权会一并转交给买家。这意味着：前一个持有者授权的恶意合约，仍然可以动用新持有者 TBA 中的资产。

应对策略有三点：第一，在转移 NFT 或挂单出售前主动清理所有授权；第二，优先使用带支出上限的授权而非无限额授权；第三，市场方应在 UI 中显式展示 TBA 的授权状态并发出警告。

**所有权循环风险**

理论上可能出现 NFT 被转移到其自身 TBA 中的情况。这会导致资产永久无法访问——因为要访问这个 TBA 需要持有该 NFT，但 NFT 本身在 TBA 里，形成死循环。应用程序需要采取措施限制此类转移。

**欺诈风险**

用户可能会出售一个声称持有特定资产的 NFT，但在交易完成后将资产从 TBA 中提取出去。去中心化市场需要建立机制来防范此类欺诈行为，比如通过时间锁或托管机制。

**重放攻击风险**

如果 TBA 支持链下签名（isValidSignature），需要防范跨链和跨合约的重放攻击。推荐使用域分隔的 EIP-712 结构化签名。

## **六、局限性讨论**

**Gas 成本问题**

TBA 是智能合约账户，因此部署和执行交易的 Gas 费用显著高于普通的 EOA（外部拥有账户）。在网络拥堵时，这可能会成为大规模应用的障碍。

**生态支持尚不完善**

目前并非所有钱包、市场和平台都原生支持 ERC-6551。用户在实际使用中可能遇到工具不兼容的问题。不过，随着标准逐步成熟，主流开发者平台（如 Alchemy、thirdweb）已开始提供相关指南和模板。

**与传统 NFT 的兼容性问题**

并非所有 NFT 项目都能平滑迁移到 ERC-6551。不依赖 ownerOf 方法的项目（如 CryptoPunks 的早期实现）与该标准存在兼容性问题。

## **七、横向对比：ERC-6551 vs. ERC-7656**

在研读过程中，有必要了解一个相关的演进提案。ERC-7656 是 ERC-6551 的泛化版本，由同一生态中的开发者提出。它的核心思想是：不局限于“账户”，而是允许 NFT 关联任何类型的服务合约。

ERC-6551 关联的对象是智能合约账户，注册表采用单例模式全局唯一，灵活性相对限定为账户逻辑。而 ERC-7656 关联的对象是任意类型的服务合约，注册表允许多实例且项目可定制，灵活性可扩展至借贷、碎片化、身份等多种场景。ERC-7656 兼容 ERC-6551，可以理解为对 ERC-6551 思想的“去约束化”扩展。但目前 ERC-6551 仍是生态采纳的主流标准。

## **八、未来展望**

ERC-6551 的长期价值在于：它将 NFT 从“静态的所有权凭证”转变为“动态的、可交互的链上实体”。

展望未来，可能出现以下演进方向：

第一，与账户抽象（EIP-4337）的协同。TBA 可以天然兼容账户抽象，实现 Gas 代付、会话密钥等高级功能，进一步提升用户体验。

第二，多链场景。同一个 NFT 在不同链上拥有独立的 TBA，需要 UI 明确标注链上下文以避免混淆。

第三，更复杂的治理机制。持有特定 NFT 的 TBA 可以参与 DAO 投票，实现“资产投票权”的新范式。

第四，创作者经济。艺术家可以将多个作品打包进一个 TBA，作为一个整体发行和交易。

## **九、总结**

ERC-6551 的核心贡献在于重新定义了 NFT 与资产所有权之间的关系。它让 NFT 从一个被持有的对象，变成了一个可以主动持有其他对象的“主体”。这种范式转换——虽然目前受限于 Gas 成本和生态成熟度——为链游、社交、金融等多个赛道打开了此前无法想象的设计空间。

理解 ERC-6551 的关键，不是记住它的技术细节，而是把握住这个核心意象：**你的小图片，现在是一个钱包了。**

账户抽象（EIP-4337）深度研读笔记

一、 背景：为什么要“抽象”账户？

在以太坊传统的架构中，账户分为两类：

EOA（外部账户）： 如 MetaMask，由私钥直接控制。

CA（合约账户）： 如智能合约，代码控制。

痛点： EOA 是唯一的交易发起者。这意味着用户必须保存助记词、必须持有 ETH 作为 Gas 费、且操作逻辑是硬编码的（无法自定义）。 EIP-4337 的目标： 让账户变成“智能合约”，同时允许它像 EOA 一样直接发起交易，从而实现账户的完全可编程化。

二、 核心组件：EIP-4337 是如何运行的？

EIP-4337 巧妙地在不改动以太坊底层共识的前提下，通过引入一套“链外中继”机制实现了 AA。

UserOperation (用户操作对象)：

它不是传统的交易（Transaction），而是一个包含用户意图的结构体。

里面记录了：谁发起的、做什么、Gas 限制、签名数据等。

Bundler (打包器)：

一个特殊的节点角色。它负责从专门的“UserOp 内存池”里捞取这些操作。

它把多个 UserOperation 打包成一个真正的以太坊交易，并提交给链上。

获利方式： Bundler 支付 Gas 费给矿工，同时从用户那里收取补偿（通常会有差价收益）。

EntryPoint (入口点合约)：

全链唯一的官方标准合约。

它负责两件事：验证（验证签名对不对、账户里有没有钱）和执行（把钱转出去或调用目标合约）。

这种设计避免了每个钱包都要写复杂的验证逻辑，提高了安全性。

Paymaster (代付合约)：

核心亮点： 它是 Gas 策略的调节器。

它可以替用户交 Gas（项目方引流），或者允许用户用 USDT/USDC 支付 Gas，由 Paymaster 背后将其换算成 ETH 给矿工。

三、 交易执行生命周期

一个 EIP-4337 交易要经历以下五个阶段：

Step 1: 发起 —— 用户在智能钱包点击“发送”，生成 UserOperation。

Step 2: 传播 —— 传入专用的 UserOp P2P 内存池。

Step 3: 打包 —— Bundler 捡起它，模拟执行无误后，打包提交给 EntryPoint。

Step 4: 验证 —— EntryPoint 调用用户钱包合约的 validateUserOp 函数。

Step 5: 执行 —— 验证通过后，EntryPoint 真正去执行目标业务逻辑。

四、 EIP-4337 带来的“杀手级”体验升级

无助记词化（社交恢复）： 通过代码逻辑，可以设定“3 个监护人中有 2 个同意即可重置私钥”，不再担心弄丢那 12 个单词。

Gas 自由： 新用户进场不需要先去交易所买 ETH，项目方可以设置 Paymaster 直接为新用户补贴 Gas。

多签与限额： 可以像银行卡一样设置“单笔最高 500U”，或者“转账给陌生地址需要双重验证”。

原子化批处理： 在 Uniswap 交易时，原本需要“授权”+“交易”两次签名。AA 可以将其封装成一个动作，一键完成。

五、 进阶思考：目前的局限性

成本问题： 由于涉及多次智能合约调用，在 Layer 1（主网）上的 Gas 费会比普通转账贵 2-3 倍。因此 L2（如 Arbitrum, Optimism, Base）才是 AA 的真正战场。

兼容性： 并不是所有现有的 DApp 都能无缝识别智能合约发起的调用（例如某些项目会检查 tx.origin），这需要生态系统的整体适配。

## EIP-1559 协议深度研读笔记

EIP-1559 是以太坊历史上最重要的革新之一（于 2021 年伦敦升级中引入）。它从根本上改变了以太坊的**手续费结构**和**经济模型**。

* * *

### 一、 核心痛点：为什么要改？

在 EIP-1559 之前，以太坊使用的是 **“第一价格密封拍卖”** 机制。

-   **体验差：** 用户像在黑市拍卖，不知道给多少钱能成交。给少了交易卡住，给多了又成了“冤大头”。
    
-   **效率低：** 手续费波动剧烈，钱包预测费用的准确度很低。
    
-   **通胀隐患：** 所有的手续费（Gas Fee）全部给到矿工，代币没有回收机制。
    

* * *

### 二、 EIP-1559 的三大支柱

1\. 费用拆分（Base Fee + Priority Fee）

原本统一的 Gas Fee 被拆分为两部分：

-   **基础费（Base Fee）：**
    
    -   由系统自动根据网络拥堵情况计算。
        
    -   **必须支付：** 它是交易进入区块的门槛。
        
    -   **核心特性：直接燃烧（Burned）**。这部分钱不再给矿工，而是直接从代币总量中永久销毁。
        
-   **小费（Priority Fee）：**
    
    -   由用户自愿支付给验证者（原矿工）。
        
    -   作用是让验证者在网络极度拥堵时优先打包你的交易。
        

2\. 动态区块大小（Elastic Block Size）

EIP-1559 引入了“目标容量”概念：

-   **目标：** 每个区块的目标装载量是 $50\\%$（即 15M Gas）。
    
-   **动态调整：**
    
    -   如果上一个区块超过 $50\\%$，基础费就会自动上调。
        
    -   如果上一个区块低于 $50\\%$，基础费就会自动下调。
        
    -   区块最大可以达到 $100\\%$（30M Gas），这为流量峰值提供了缓冲，不再像以前那样导致网络直接瘫痪。
        

3\. 燃烧机制（The Burn）

这是该协议最具话题性的部分。每当一笔交易发生，其 Base Fee 都会被销毁。

-   **通缩压力：** 如果网络交易非常活跃，燃烧掉的 ETH 数量可能会超过新产出的 ETH 数量。
    
-   **超声波货币（Ultrasound Money）：** 这让 ETH 有了从“无限通胀”转向“净通缩”的可能。
    

* * *

### 三、 给小白的形象比喻：打车模型

-   **旧模型（拍卖）：** 路上只有 10 辆出租车，大家在路边喊价，谁出的钱多谁上。你可能出 100 块，其实 50 块就够了，但你不知道。
    
-   **新模型（EIP-1559）：** 1. **起步价（Base Fee）：** 官方定价，当前全城统一。路越堵起步价越高。但这笔钱不给出租车司机，而是被政府（系统）回收销毁。
    
    2\. **加急费（Tip）：** 你想让司机快点接你，额外塞给司机的辛苦钱。
    
    3\. **弹性空间：** 如果人多，每辆车可以从坐 4 个人变成挤 8 个人（动态区块），保证大家不至于完全等不到车，只是起步价会飞速上涨。

## 现代以太坊：从架构解耦到 L2 扩容的技术全景

自“合并”（The Merge）完成以来，以太坊已经从一个单一的区块链演变为一个高度模块化的分布式系统。理解这一演进，是进入 Web3 技术核心领域的必经之路。

### 一、 结构之变：执行层与共识层的“分家”

在合并之前，以太坊节点是一个耦合的整体。现在，一个完整的以太坊节点由两个独立的软件（客户端）组成，它们各司其职，通过 **Engine API** 协同工作：

-   **执行层 (Execution Layer, EL)：** 相当于以太坊的“大脑”。它运行 EVM（以太坊虚拟机），负责处理智能合约、计算账户余额和更新系统状态。代表客户端是 **Geth** 和 **Nethermind**。
    
-   **共识层 (Consensus Layer, CL)：** 相当于以太坊的“法律”。它负责 PoS（权益证明）逻辑，管理验证者节点，决定哪个区块应该被接纳进主链。代表客户端是 **Prysm** 和 **Lighthouse**。
    

**为什么要分家？** 这种架构解耦不仅提高了安全性，还为后续的模块化升级（如数据分片）提供了前提，允许开发者针对不同的层级进行专门的性能优化。

* * *

### 二、 性能之跃：The Surge 与 EIP-4844 的“外挂空间”

在 Vitalik 的路线图中，**The Surge** 是最关乎用户体验的阶段，其核心目标是让以太坊达到每秒 10 万次以上的交易处理能力。

为了实现这一点，以太坊引入了 **EIP-4844（Proto-Danksharding）**。这是通过一种名为 **Blob** 的新数据结构实现的：

-   **什么是 Blob？** 它是附加在区块上的“数据气泡”。
    
-   **为什么更便宜？** 以前 L2 必须把数据存在 `calldata` 里（昂贵且永久保存），而 Blob 空间是临时的（约 18 天后自动删除）。
    
-   **解耦费用：** Blob 有自己独立的“Gas 价格”。这意味着即使主网有人在抢购 NFT 导致 Gas 暴涨，L2 的扩容数据也不会受到直接干扰，从而大幅降低了用户的转账手续费。
    

* * *

### 三、 旅程之终：一笔交易的“入链”全程

当我们通过节点（如 Geth）发起一笔交易时，它会经历以下完整的生命周期：

1.  **进入 Mempool：** 交易被签名发送后，首先进入节点的执行层内存池（Mempool）。
    
2.  **构建 Payload：** 当该节点被选为提议者时，EL 客户端会从池中打包交易，执行它们并生成一个 **Execution Payload**（执行载荷）。
    
3.  **共识封装：** CL 客户端接过载荷，将其包裹在信标区块（Beacon Block）中，并向全网广播。
    
4.  **证明与投票（Attestation）：** 遍布全球的验证者收到区块，验证交易的合法性并进行投票。
    
5.  **最终确认（Finality）：** 随着新区块不断产生，当该交易所在的区块经过了足够的周期（Epoch）验证后，它便进入了“不可逆”状态。
    

* * *

### 四、 总结：技术中心化运营视角

作为一名深耕区块链的技术人员，理解这些不仅是为了写代码，更是为了理解**系统的边界**。

从 **Geth** 的交易处理逻辑，到 **EIP-4844** 对成本结构的改变，以太坊正在从一个“昂贵的超级计算机”转型为“廉价且安全的结算层”。在找实习的过程中，这种对底层架构（EL/CL 分离）和前沿提案（Blob）的清晰认知，将是你区别于普通开发者、体现“技术中心化运营”专业度的核心竞争优势。

## **从交易发起到最终确认的完整生命周期**

### **一、什么是节点客户端？**

以太坊节点是运行客户端软件的计算机。合并（The Merge）之后，运行一个全节点需要两个客户端协同工作。

**执行层客户端（EL）** 负责处理交易执行、维护账户状态、运行EVM，代表项目有Geth、Nethermind、Erigon。

**共识层客户端（CL）** 负责运行PoS、管理验证者、决定哪条链是规范的，代表项目有Lighthouse、Prysm、Teku。

两者通过Engine API通信——CL是“老板”，决定接受什么区块；EL是“工人”，执行交易并构建区块。

### **二、交易的完整生命周期**

**阶段1：交易创建与签名**

用户创建交易（包含to、value、gas、nonce等字段），用私钥签名后通过RPC提交到节点。

**阶段2：进入Mempool（交易池）**

交易进入节点的Mempool（内存池）——等待被打包的“候车区”。节点执行四项验证：检查余额是否足够、验证签名是否正确、确认nonce是否等于该地址下一笔交易的序号（防重放攻击）、检查Gas价格是否足够高。验证通过后，交易被广播给全网其他节点。

**阶段3：区块构建（Engine API驱动）**

EL等待CL的指令。CL发送请求告诉EL当前head区块是什么，以及是否需要构建新区块。如果需要构建，EL会在12秒内反复重建payload，目的是捕获后到达的“高小费”交易，最大化验证者的MEV（最大可提取价值）。

**阶段4：交易排序与MEV**

交易在区块中的顺序决定价值——超过50%的区块价值集中在前12.5%的交易中。

MEV供应链分为三层：Searcher（搜索者）扫描mempool发现套利机会，打包成bundle；Block Builder（区块构建者）聚合mempool交易、私密订单流和bundle，构建最优区块；Validator/Proposer（验证者/提议者）在每12秒的slot中，选择最高出价的区块并广播到网络。

举例说明三明治攻击：攻击者在用户交易前买入、后卖出，从差价中获利。

**阶段5：区块传播与确认**

第一次确认时，区块被挖出或提议，交易被包含。后续确认中，新区块不断堆叠在它上面。以太坊的最终确定性需要约2到3个epoch（大约13分钟），之后交易不可逆。

**阶段6：交易状态**

交易可能处于以下几种状态：Pending表示在mempool中等待打包；Mined或Success表示已上链成功；Failed表示执行失败但消耗了Gas；Dropped表示因Gas过低或nonce错误被移除；Dropped and Replaced表示被同nonce但更高Gas的交易替换。

### **三、Gas机制详解**

Gas Limit是交易最多消耗的计算单位。Base Fee是基础费，根据区块拥堵动态调整，会被燃烧。Priority Fee是小费，给验证者以加速打包。Effective Gas Price等于Base Fee加Priority Fee。

典型Gas消耗：普通ETH转账需要21,000 gas；ERC-20转账大约45,000 gas；合约部署则视复杂度而定。

### 一、 节点客户端基础

节点客户端是实现以太坊协议规范的软件，负责点对点通信、交易验证及状态维护。

1.  **执行层 (Execution Layer, EL)**：负责处理智能合约逻辑和交易执行。
    
    -   **Geth (Go-Ethereum)**：最主流的 Go 语言实现，生态最全，资料最丰富。
        
    -   **Nethermind**：基于 .NET 构建，以高性能和对企业级用户友好著称，是寻找实习中常被提及的 Geth 替代方案。
        
    -   **Erigon**：侧重于优化数据存储，显著降低磁盘占用。
        
2.  **共识层 (Consensus Layer, CL)**：负责实现权益证明 (PoS) 共识。
    
    -   **Lighthouse / Prysm**：常见的信标链客户端实现。
        
3.  **节点类型**：
    
    -   **全节点 (Full Node)**：存储所有链上数据，参与交易校验。
        
    -   **存档节点 (Archive Node)**：保留每一时刻的历史状态，常用于数据爬取和深层分析。
        

* * *

### 二、 交易从发起至确认的全生命周期

了解交易如何从你的钱包流动到区块链。

1\. 发起与签名 (Sign)

-   **私钥签名**：用户在本地（如 Metamask）使用私钥对交易（包含 Nonce、Gas Price、目标地址、数据等）进行签名。签名证明了交易权，且无法被篡改。
    

2\. 广播与进入交易池 (Mempool)

-   **广播**：签名后的交易发送至节点的 RPC 接口。
    
-   **Mempool (内存池)**：每个节点维护一个 Mempool。你的交易被传播给其他节点，等待被打包。此时交易状态为 "Pending"。
    

3\. 验证与打包 (Validation & Inclusion)

-   **验证**：执行层客户端（如 Geth）检查交易的余额是否充足、Nonce 是否连续、签名是否有效。
    
-   **打包**：区块构建者（Builder）从 Mempool 筛选交易（通常按 Gas 价格排序）并尝试构建区块。
    

4\. 共识与执行 (Consensus & Execution)

-   **提议区块**：被选中的验证者（Validator）向全网发布新区块。
    
-   **执行状态**：节点收到区块后，执行层客户端重新运行区块内的所有交易，更新状态树（State Trie）。
    

5\. 确认与最终性 (Confirmation & Finality)

-   **确认**：区块被收录在链上。
    
-   **最终性**：在 PoS 下，通常需要经过约 2 个 Epoch（约 12.8 分钟），交易才被视为不可篡改。

## 🌊 The Surge（大浪潮）：让交易快如闪电，便宜如水

**核心目标：** 解决以太坊的“贵”和“慢”，目标是实现每秒 10 万次以上的交易处理量 ($>100,000$ TPS)。

### 1\. 什么是 Blob (EIP-4844)？

以前，Layer 2 (L2) 扩容方案（如 Arbitrum, Optimism）需要把交易数据存在以太坊的 `calldata` 里，这就像在繁华市中心的写字楼里租仓库，价格昂贵且空间有限。

-   **Blob 的形象比喻**：Blob 就像是以太坊专门为 L2 挂载的“临时大外挂货柜”。它不在市中心写字楼里占地方，而是挂在建筑外侧。
    
-   **为什么便宜？** Blob 数据在以太坊上是**临时存储**的（大约 18 天后会被删除）。因为它不永久占用宝贵的以太坊账本空间，所以费用大幅下降。
    
-   **EIP-4844（坎昆升级）**：这就是引入 Blob 的技术协议。它让 L2 的交易手续费降低了 10 倍甚至更多。
    

### 2\. 对普通人的意义

-   **低费用**：你在 L2 上交互 DeFi 或转账，手续费从几美金降到了几美分。
    
-   **以 Rollup 为中心**：以太坊不再亲自处理所有小额交易，而是变成一个“结算层”和“数据安全层”，具体的活儿交给 L2 去干。
    

* * *

## 👹 The Scourge（天灾/肃清）：确保去中心化，不让大户独大

**核心目标：** 解决以太坊在价值分配和中心化方面的隐患，主要是针对 **MEV（最大可提取价值）**。

### 1\. 为什么要“肃清”？

现在的以太坊节点在打包交易时，由于可以看到交易顺序，大户或专业团队可以通过抢跑、夹击（Sandwich Attack）等手段赚取额外利润。这导致了两个问题：

-   **用户受损**：普通用户的交易滑点变大，被“撸羊毛”。
    
-   **节点中心化**：只有拥有强大计算能力的专业团队才能赚到这些钱，普通验证者难以生存。
    

### 2\. 核心举措

-   **PBS（区块提议者与构建者分离）**：强制将“决定哪些交易入场”和“真正打包交易”这两个权力分开，防止单个节点权力过大。
    
-   **抗审查性**：确保即使某些节点想封锁特定地址的交易，其他节点也能通过机制让这些交易顺利上链。
    

* * *

## 💡 开发者笔记总结

**The Surge** 侧重于**性能与廉价**，通过 Blob 技术让 L2 变得超级便宜，从而承载数亿级别的用户 。 **The Scourge** 侧重于**公平与民主**，旨在解决 MEV 导致的中心化问题，保护普通交易者不被系统性收割 。

# 以太坊合并（The Merge）后的**模块化架构**

## \## 1. 架构总览：双客户端模型

在合并之前，你只需要运行一个客户端（如 Geth）就能搞定一切。现在，一个完整的以太坊节点必须同时运行两个软件：

1.  **执行层 (Execution Layer, EL):** 负责“干活”，处理具体的交易逻辑。
    
2.  **共识层 (Consensus Layer, CL):** 负责“决策”，确保全网达成一致。
    

* * *

## \## 2. 执行层 (Execution Layer) —— 处理“发生了什么”

执行层保留了以太坊自 2015 年诞生以来的所有代码逻辑。

-   **前身：** 即原有的 Eth1 链。
    
-   **主流客户端：** Geth (最常用), Nethermind, Besu, Erigon。
    
-   **核心职责：**
    
    -   **EVM 执行：** 运行智能合约代码，处理账户转账。
        
    -   **状态维护：** 维护所有账户余额、合约代码和存储的“状态树”。
        
    -   **交易池 (Mempool) 管理：** 接收用户发出的交易，并进行初步校验。
        
    -   **生成执行载荷 (Execution Payload)：** 把交易打包成一个“活儿”，交给共识层去确认。
        

* * *

## \## 3. 共识层 (Consensus Layer) —— 处理“顺序是什么”

共识层是合并时引入的“新大脑”，它不再关心交易内容，只关心谁有权出块。

-   **前身：** 信标链 (Beacon Chain)。
    
-   **主流客户端：** Prysm, Lighthouse, Teku, Nimbus, Lodestar。
    
-   **核心职责：**
    
    -   **权益证明 (PoS)：** 管理验证者（Validators）的质押、奖励和惩罚（Slashing）。
        
    -   **出块逻辑：** 随机选出提议者（Proposer）和委员会（Committees）进行投票。
        
    -   **分叉选择规则 (LMD-GHOST)：** 决定哪条链才是“主线”。
        
    -   **区块传播：** 在 P2P 网络中广播共识数据。
        

* * *

## \## 4. 关键连接：Engine API

既然分开了，它们怎么沟通？答案是 **Engine API**。这是一组基于 JSON-RPC 的指令集，让 CL 能够指挥 EL。

-   **典型流程：**
    
    1.  **CL** 告诉 **EL**：“嘿，轮到我出块了，给我一个执行载荷（一堆交易）。”
        
    2.  **EL** 打包好交易，发回给 **CL**。
        
    3.  **CL** 把这个载荷包在自己的共识数据里，广播给全网。
        
    4.  其他节点的 **CL** 收到后，把载荷拆出来发给自己的 **EL**：“你跑一下看看这些交易合法吗？”
        
    5.  **EL** 返回“VALID”，**CL** 确认，区块上链。
        

## **打卡笔记灵感：**

The Merge 后的双客户端模型。EL 像体力劳动者，负责跑 EVM；CL 像指挥官，负责 PoS 逻辑。这种模块化设计不仅让以太坊摆脱了挖矿，更为未来的分片（Sharding）打下了基础。Engine API 则是两者协同的生命线。下步准备去看看这些客户端的具体配置差异。

### 通用 zkVM 的定义、构造与优化

一、 核心概念：什么是通用 zkVM (Universal zkVM)？

通用 zkVM 是一个能够执行任意指令集（如 RISC-V、WASM）并为执行结果生成零知识证明的系统。

-   **三大安全属性**：
    
    -   **完整性 (Completeness)**：正确的计算必能生成证明。
        
    -   **可靠性 (Soundness)**：错误的计算绝无法伪造证明。
        
    -   **零知识性 (Zero-Knowledge)**：不泄露私密输入。
        
-   **工程意义**：它让开发者可以用 Rust/C++ 等传统语言编写逻辑，而无需手动构建复杂的 ZK 电路。
    

二、 技术架构：从代码到证明的转化

1.  **算术化 (Arithmetization)**：将程序执行轨迹 (Trace) 转化为多项式约束。论文中提到了 **Plonkish** 和 **R1CS** 的平衡选择。
    
2.  **指令集映射 (ISA Mapping)**：如何将 CPU 指令（如加减法、跳转）翻译成 ZK 友好的等式。
    
3.  **查找协议 (Lookup Protocols)**：利用 **LogUp** 等高效查找论证处理复杂的位运算和内存操作，避免电路膨胀。
    

三、 性能瓶颈与前沿优化（结合 2026/668）

证明生成速度是限制 zkVM 大规模应用的最大障碍，最新的研究方向包括：

-   **递归与分段证明 (Recursion & Sharding)**：
    
    -   **分段**：将长程序切分为多个 Chunks，并行生成证明。
        
    -   **递归聚合**：将成千上万个子证明压缩成一个最终证明，极大地降低了链上验证成本。
        
-   **内存一致性优化**：通过高效的排列检查（Permutation Check）确保内存读写在成千上万个 Cycle 中保持一致且无开销。
    
-   **预编译（Precompiles）**：针对常用的加密算法（如 Keccak、SHA256），直接编写专用电路以跳过昂贵的虚拟机模拟过程。
    

四、 对区块链实习/就业的启发

作为区块链专业的学生，在共学中可以重点关注以下应用方向：

1.  **Layer 2 扩容**：zkVM 是实现通用型 zkRollup 的基石，支持开发者将复杂的 dApp 逻辑直接迁移到 L2。
    
2.  **跨链通信**：利用 zkVM 在目标链上验证源链的虚拟机状态转换，实现去中心化的轻客户端桥。
    
3.  **隐私保护**：在不暴露用户交易细节的前提下，证明用户符合某种信用评级或资格。

## 笔记：基于 secp256k1/r1 的 ECDSA 批量验证优化

### 1\. 核心问题

-   **背景**：ECDSA（椭圆曲线数字签名算法）广泛用于区块链（如比特币、以太坊），但其原生结构不支持高效的**批量验证（Batch Verification）**。
    
-   **痛点**：在高吞吐量场景下，逐个验证签名会成为系统的计算瓶颈。
    

### 2\. 改进方案：两个变体

为了实现高效批量验证，通过在签名中嵌入\*\*额外比特（Additional Bits）\*\*构造了两个变体：

-   **ECDSA\_rec**：侧重于恢复（Recovery）特性的改进。
    
-   **ECDSA\_ast**：性能表现通常优于 `ECDSA_rec`。
    

### 3\. 关键技术对比

研究对比了两种随机化批量验证技术：

-   **HSS-rand**：本次研究证明其在大多数情况下效率最高。
    
-   **NMVR-rand**：传统的随机化验证技术，效率略逊于 HSS-rand。
    

### 4\. 性能数据总结 (关键量化指标)

| 曲线类型 | 变体版本 | 批量验证提升 (vs 单独验证) | HSS-rand vs NMVR-rand |
| secp256k1 (比特币/以太坊常用) | ECDSA_ast | +30.9% | +10.6% |
|   | ECDSA_rec | +17.2% | +9.8% |
| secp256r1 (标准 TLS/国标常用) | ECDSA_ast | +53.9% | +27.6% |
|   | ECDSA_rec | +40.5% | +22.9% |

## CLWE 问题的搜索到决策归约

### 1\. 核心概念定义

-   **CLWE (Continuous LWE)**：连续型 LWE 问题，由 Bruna 等人在 2021 年提出，与传统离散空间的 LWE 不同，它处理的是连续概率分布。
    
-   **搜索版本 (Search)**：目标是根据给定的样本直接计算出秘密向量 $s$。
    
-   **决策版本 (Decision)**：目标是区分给定的样本是来自 CLWE 分布还是来自纯随机均匀分布。
    

### 2\. 现有研究局限（背景）

-   **2021 年 (STOC 2021)**：初步提出 CLWE 概念。
    
-   **2022 年 (Gupte 等人)**：实现了从 CLWE 到标准 LWE 的经典归约。
    
-   **瓶颈**：现有的反向归约（CLWE $\\leftarrow$ LWE）仅针对“离散 CLWE”有效。对于**一般性 CLWE**，此前缺乏从“搜索”到“决策”的直接归约证明。
    

### 3\. 本文核心贡献

本文填补了上述空白，提出了一种针对一般 CLWE 的搜索到决策归约方法。

-   **核心算法**：设计了一个相对简单的算法。
    
-   **实现路径**：利用\*\*决策预言机（Decision Oracle）\*\*作为子程序。
    
-   **达成目标**：通过决策预言机的反馈，以较小的误差逼近秘密向量，从而证明了：**只要能解决 CLWE 的决策问题，就足以解决其搜索版本。**
    

### 4\. 技术意义

-   **安全性等价性**：证明了 CLWE 的决策版本与搜索版本在计算难度上是等价的（或至少搜索不比决策难）。
    
-   **理论完备性**：为基于 CLWE 的密码学构造提供了更坚实的底座，使其安全性证明不再局限于离散子集。

### **以太坊协议层核心架构 (EPF Wiki)**

**1\. EPF 核心定位**

-   **非应用层开发**：EPF 专注于以太坊协议本身（Core Protocol），而非智能合约或 DApp 开发。
    
-   **目标**：培养核心贡献者（Core Contributors），深入研究执行层（EL）、共识层（CL）及协议路线图。
    

**2\. 执行层 (Execution Layer, EL)**

执行层负责处理交易、智能合约执行和状态管理。

-   **核心组件**：
    
    -   **EVM (以太坊虚拟机)**：执行字节码的核心引擎。
        
    -   **状态管理**：使用 **Merkle Patricia Tree (MPT)** 组织账户平衡、合约代码和存储。
        
    -   **客户端实现**：常见的有 Geth (Go), Besu (Java), Nethermind (.NET), Reth (Rust)。
        
-   **关键协议**：包含 P2P 网络（DevP2P）和交易池管理。
    

**3\. 共识层 (Consensus Layer, CL)**

共识层负责通过权益证明（PoS）确保全网对数据达成一致。

-   **Beacon Chain (信标链)**：PoS 的指挥中心。
    
-   **核心机制**：
    
    -   **Gasper**：结合了 Casper FFG（最终性）和 LMD GHOST（分叉选择规则）。
        
    -   **验证者角色**：质押 32 ETH 参与出块与见证（Attestation）。
        
-   **客户端实现**：Prysm (Go), Lighthouse (Rust), Teku (Java), Lodestar (TS)。
    

**4\. 引擎 API (Engine API)**

-   这是连接 EL 和 CL 的桥梁。在合并（The Merge）后，两个客户端通过此 API 通信，确保执行结果被共识层认可。
    

**5\. 2026 路线图进阶**

-   **Cryptography (密码学)**：重点关注 KZG 承诺、椭圆曲线配对（Pairings），这些是支撑 EIP-4844 和 Danksharding 的数学基础。
    
-   **zkEVM & Lean Consensus**：研究如何通过零知识证明简化执行校验，以及共识机制的轻量化。

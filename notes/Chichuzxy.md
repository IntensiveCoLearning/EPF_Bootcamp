# 以太坊第一周学习笔记：协议基础与执行层 (EL)

## 一、The Protocol 全区（以太坊整体协议）

### 1\. Prehistory（史前历史）

-   **核心背景**：比特币只能做简单转账，无法运行复杂的自定义程序（智能合约）
    
-   **关键节点**：2013 年 Vitalik 发布以太坊白皮书，提出 "世界计算机" 概念；2015 年 7 月以太坊主网正式上线
    
-   **核心目标**：打造一个去中心化的通用计算平台，任何人都能在上面部署和运行智能合约
    

### 2\. Architecture（整体架构）

-   **合并后核心结构**：执行层 (EL) + 共识层 (CL) 完全分离
    
    -   执行层：管交易执行、EVM 运行、账户状态管理
        
    -   共识层：管 PoS 共识、区块验证、链的最终性和安全
        
-   **其他基础层**：网络层（节点通信）、数据层（区块和状态存储）、应用层（DApp、钱包）
    
-   **优势**：模块化设计，两层可以独立升级，互不影响
    

### 3\. Design rationale（设计理念）

-   **账户模型 vsUTXO**：选择账户模型（而非比特币的 UTXO），方便跟踪账户余额和执行复杂合约逻辑
    
-   **Gas 机制**：防止网络资源被滥用，同时给验证者提供激励；用户为计算和存储资源付费
    
-   **去中心化优先**：牺牲部分性能，保证网络抗审查、无单点故障
    
-   **渐进式升级**：通过 EIP（以太坊改进提案）逐步优化，避免硬分叉分裂
    

### 4\. Evolution（演进历史）

表格

| 升级阶段 | 时间 | 核心内容 |
| --- | --- | --- |
| 前沿 (Frontier) | 2015 | 主网正式上线，仅支持基础功能 |
| 家园 (Homestead) | 2016 | 第一个稳定版本，修复安全漏洞 |
| 伦敦 (London) | 2021 | EIP-1559，改变 Gas 定价机制，开始燃烧 ETH |
| 合并 (Merge) | 2022 | 从 PoW 转为 PoS，EL+CL 分离，能耗降低 99% |
| 上海 (Shanghai) | 2023 | 允许质押的 ETH 解锁提现 |
| 坎昆 (Cancun) | 2024 | EIP-4844，引入 Blob 数据，大幅降低 L2 手续费 |

## 二、执行层 (EL) 基础部分

### 1\. EL Specs（执行层核心规范）

-   所有 EL 客户端（Geth、Nethermind 等）必须遵守的 "宪法"
    
-   明确定义了交易执行规则、状态计算方式、区块验证标准
    
-   保证不同客户端运行结果完全一致，维护链的统一性
    

### 2\. Client architecture（客户端架构）

EL 客户端核心模块：

-   P2P 网络模块：发现其他节点，同步区块和交易
    
-   交易池：存放等待打包的未确认交易
    
-   EVM 执行引擎：执行交易和智能合约的核心
    
-   状态数据库：存储所有账户的余额、合约代码和数据
    
-   JSON-RPC 接口：对外提供 API 服务，供钱包和 DApp 调用
    

### 3\. EVM（以太坊虚拟机）

-   以太坊的 "心脏"，一个沙盒式的执行环境
    
-   所有智能合约和交易都在 EVM 中运行，保证所有节点执行结果一致
    
-   基于栈的虚拟机，有自己的指令集（如加减乘除、合约调用等）
    
-   合约代码需要编译成 EVM 字节码才能运行
    

### 4\. Transaction anatomy（交易结构）

**核心字段**：

-   `from`：发送方地址（必须是外部拥有账户 EOA）
    
-   `to`：接收方地址（EOA 或合约地址）
    
-   `value`：转账的 ETH 数量
    
-   `gasLimit`：最多愿意花费的 Gas 上限
    
-   `maxFeePerGas/maxPriorityFee`：Gas 价格（EIP-1559 后）
    
-   `nonce`：发送方的交易序号，防止重放攻击
    
-   `data`：合约调用数据或部署合约的代码
    
-   `signature`：发送方的数字签名，证明交易合法性
    

**生命周期**：用户签名→广播到网络→进入交易池→被打包进区块→EVM 执行→状态更新→交易确认

### 5\. Data Structures（核心数据结构）

-   **账户**：分为 EOA（外部拥有账户，有私钥）和合约账户（无私钥，由代码控制），都包含 nonce、balance、codeHash、storageRoot
    
-   **区块**：由区块头（父哈希、状态根、交易根等）、交易列表、收据列表组成
    
-   **Merkle Patricia Trie**：以太坊特有的树状数据结构，用于存储账户状态、交易和收据，根哈希在区块头中，保证数据不可篡改
    

### 6\. RLP Serialization（RLP 编码）

-   以太坊自定义的序列化格式，用于所有需要传输和存储的数据
    
-   特点：简单、高效、紧凑，只编码字节数组和列表
    
-   作用：把区块、交易等复杂对象转换成字节流，方便在网络上传输和存储
    

### 7\. DevP2P（P2P 网络协议）

-   以太坊节点之间通信的底层协议
    
-   主要功能：节点发现（找到网络中的其他节点）、加密通信、区块同步、交易广播
    
-   基于 Kademlia DHT 算法实现节点发现，保证网络的去中心化
    

### 8\. JSON-RPC（节点对外接口）

-   以太坊节点对外提供的标准 API 接口
    
-   基于 HTTP 或 WebSocket，使用 JSON 格式通信
    
-   常用方法：`eth_getBalance`（查余额）、`eth_sendTransaction`（发交易）、`eth_call`（调用合约）、`eth_getBlockByNumber`（查区块）
    
-   钱包（如 MetaMask）和 DApp 都是通过 JSON-RPC 与以太坊节点交互
    

## 三、执行层 (EL) 进阶部分

### 1\. EOF（EVM 对象格式）

-   EVM 的重大升级，解决传统 EVM 字节码的痛点
    
-   把合约代码分成明确的段（代码段、数据段、类型段等），不再是混合的字节流
    
-   优势：更容易验证、执行效率更高、支持更复杂的优化，为未来 EVM 升级打下基础
    
-   已在坎昆升级中部分启用
    

### 2\. Precompiled Contracts（预编译合约）

-   内置在 EVM 中的特殊合约，由客户端原生代码实现
    
-   用于执行计算密集型操作，如 SHA256 哈希、ECDSA 签名验证、零知识证明配对运算
    
-   优势：比用普通 EVM 指令实现 Gas 成本低得多，大幅提升网络性能
    
-   地址范围：0x01 ~ 0x09（后续升级会增加更多）
    

### 3\. Block Building（区块构建）

-   **PoS 合并后流程**（MEV-Boost 模式）：
    
    1.  用户发送交易，进入各个节点的交易池
        
    2.  Builder（区块构建者）从交易池中选择交易，按最大 MEV（最大可提取价值）顺序打包
        
    3.  Builder 把区块模板发给 Relay（中继）
        
    4.  Relay 把模板发给当前 slot 的 Proposer（区块提议者）
        
    5.  Proposer 签名并广播区块，其他节点验证
        
-   优势：最大化 MEV 收益，同时保证链的安全和去中心化

# **以太坊共识层（CL）核心知识体系 · 学习笔记**

## **一、共识层概述：执行层与共识层的"双引擎"架构**

2022年9月的"The Merge"是以太坊历史上最关键的转折点。在这次升级中，原有的PoW链（执行层）与全新的PoS链——**信标链（Beacon Chain，即共识层）** 合并，形成了现在的双层架构。

### **1.1 分层结构**

| 层级 | 核心职责 | 主流客户端 |
| --- | --- | --- |
| 共识层 (CL)：以太坊的"心脏" | 区块广播、共识逻辑、验证者管理、分叉选择、奖惩分发 | Prysm、Lighthouse、Teku、Nimbus |
| 执行层 (EL)：以太坊的"大脑" | 交易执行、智能合约运行、状态管理、EVM操作 | Geth、Nethermind、Erigon、Besu |

两层之间通过 **Engine API** 实现通信：共识层负责"方向盘"（什么时候该出块），执行层负责"发动机"（打包并执行交易）。

### **1.2 合并前后对比**

-   **合并前**：PoW单一架构，Geth既处理交易又负责挖矿出块；
    
-   **合并后**：一个完整节点必须同时运行EL和CL两个客户端，职责清晰分离，模块化程度大幅提升。
    

这个双层设计带来的好处是：谁的事情谁管，共识的事情交给共识层，交易执行的事情交给执行层，互不干扰，各自优化。

## **二、CL Specs：共识层规范体系**

共识层的所有行为都由一套**可执行的规范**来定义。这套规范存放在 `ethereum/consensus-specs` 仓库中，是用 Markdown 写的 Python 代码，可以直接通过 `pyspec` 工具转化为可执行程序，用来生成客户端测试向量。

### **2.1 规范的核心特点**

-   **可执行**：不是死文档，而是能跑起来的代码；
    
-   **以分叉为单位组织**：每个硬分叉（Phase 0 → Altair → Bellatrix → Capella → Deneb → Electra → Fulu...）都继承并扩展上一版本的规范；
    
-   **配置分层**：通过 `preset` 和 `config` 系统支持不同网络的配置（主网、测试网等）。
    

### **2.2 规范包含的内容**

| 模块 | 内容 |
| --- | --- |
| 类型与容器 | BeaconState、BeaconBlock 等核心数据结构的类型定义 |
| 状态转换函数 | process_block()、process_epoch() 等核心逻辑 |
| 分叉选择算法 | LMD-GHOST 实现 |
| 网络协议 | P2P 通信规范 |
| 验证者生命周期 | 存款、激活、退出、提款全流程 |

想深入了解，可以直接看 `eth2book.info` 的《Upgrading Ethereum》带注释版规范，它把枯燥的规范代码逐行拆解，非常适合学习。

## **三、SSZ + Merkleization：数据怎么"打包"**

SSZ（Simple Serialize）是共识层的标准序列化格式，替代了执行层的 RLP。它的核心功能有两个：**序列化**（把结构体变成字节流）和 **Merkle化**（把数据变成默克尔树，方便做密码学证明）。

### **3.1 SSZ 类型系统**

SSZ 的类型分为两大类：

**基本类型**（固定大小）：

-   `uintN`：N 位无符号整数（uint8/16/32/64/128/256）
    
-   `boolean`：True 或 False
    
-   `byte`：8 位不透明数据（和 uint8 序列化相同，但语义不同）
    

**复合类型**：

-   固定大小：`Container`（结构体）、`Vector[T, N]`（定长数组）、`Bitvector[N]`
    
-   可变大小：`List[T, N]`（变长列表）、`Bitlist[N]`、`Union[...]`
    

判断标准很简单：只要包含可变大小成分的类型，整体就是可变大小的。

### **3.2 序列化核心原理**

-   **固定大小类型**：直接把每个元素按小端序编码，拼起来就行；
    
-   **可变大小类型**：在元素位置放一个**偏移量**（offset），指向数据堆里实际内容的起始位置，实际数据统一塞在序列化结果末尾的"堆区"。
    

举个例子：一个结构体 `{number1: 37, number2: 55, vector: [1,2,3,4], number3: 22}`，序列化后会变成：

text

```
[37, 0, 0, 0,    ← number1
 55, 0, 0, 0,    ← number2
 16, 0, 0, 0,    ← offset（指向vector在字节16的位置）
 22, 0, 0, 0,    ← number3
 1, 2, 3, 4]     ← vector 实际数据
```

### **3.3 Merkleization 是什么**

Merkleization 就是把 SSZ 序列化后的数据转化成一棵默克尔树，方便生成**密码学证明**（比如"某个验证者的余额确实在 BeaconState 里的某个位置"）。每个数据块（chunk）大小是 **32 字节**，数据被切分成 chunk 后逐层哈希，最终得到 HashTreeRoot。

**为什么这很重要？** 轻客户端依赖 SSZ 的默克尔证明来验证链上数据而不需要同步全部区块，这是以太坊去信任验证的基础设施。

## **四、Beacon API：CL 节点的对外接口**

Beacon API 是一组 RESTful HTTP 接口，让我们可以查询共识层节点的各种信息。所有端点都在 `/eth/v1/` 命名空间下。

### **4.1 常用端点分类**

| 类别 | 端点示例 | 用途 |
| --- | --- | --- |
| Beacon 状态 | /eth/v1/beacon/states/{state_id}/root | 查询状态根 |
|   | /eth/v1/beacon/states/{state_id}/pending_deposits | 查询待处理存款 |
| Beacon 区块 | /eth/v1/beacon/blocks/{block_id}/root | 获取区块根 |
| 节点信息 | /eth/v1/node/syncing | 查询同步状态 |
|   | /eth/v1/node/identity | 获取节点身份 |
| 配置 | /eth/v1/config/spec | 获取链规范参数 |
|   | /eth/v1/config/fork_schedule | 获取分叉时间表 |
| 验证者 | /eth/v1/validator/duties/... | 获取验证者职责 |

### **4.2 关键参数说明**

-   **state\_id**：可以是 `head`（最新头状态）、`genesis`（创世）、`finalized`（已最终确认）、`justified`（已证明）、某个 slot 号、或 stateRoot 哈希值；
    
-   **block\_id**：类似地，可以是 `head`、`genesis`、`finalized` 或具体 slot/root；
    
-   API 返回中的 `execution_optimistic` 字段表示数据是否引用了尚未被 EL 验证的执行载荷，⚠️ 需注意此信息稍后可能被作废。
    

### **4.3 与执行层 API 的关系**

-   执行层用 **JSON-RPC**（如 `eth_getBalance`），由 EL 客户端提供；
    
-   共识层用 **REST API**（Beacon API），由 CL 客户端提供；
    
-   日常交互（查余额、发交易）用的是 JSON-RPC；查询验证者状态、链共识信息则用 Beacon API。
    

## **五、CL Networking：共识层网络架构**

共识层有自己独立的 P2P 网络，与执行层的 P2P 网络是分开的。执行层传播交易，共识层传播区块和证明。

### **5.1 网络堆栈层次**

共识层网络建立在 **libp2p** 框架之上，分为三层协议：

| 域 | 协议 | 作用 |
| --- | --- | --- |
| Gossip 域 | GossipSub | 一对多广播：传播区块、证明（attestation）、同步委员会消息 |
| Discovery 域 | Discv5 | 节点发现：找到网络中的其他对等节点 |
| Req/Resp 域 | 请求-响应协议 | 一对一定向通信：请求特定区块、状态等 |

### **5.2 GossipSub 工作机制**

GossipSub 是共识层消息广播的核心协议，可理解为"流言式广播"：

-   每个节点随机维护一批对等节点（mesh），收到消息后随机转发给 mesh 中的几个节点；
    
-   不同类型的消息走不同的 **topic**（主题）：
    
    -   **证明（attestation）主题**：每秒数千条小消息（几百字节），需要高吞吐和低延迟；
        
    -   **区块主题**：每个 epoch 只有少量但体积大（可达 MB 级），需要可靠传播；
        
-   不同主题面临不同挑战，社区正在研究**分主题参数优化**：对小消息降低 gossip 因子减少带宽浪费，对大消息适当缩小 mesh 加快传播速度。
    

### **5.3 传输与加密**

-   传输层必须支持 **TCP**，可选支持 **QUIC (UDP)**；
    
-   加密使用 **libp2p-noise** 安全信道握手，身份采用 `secp256k1` 密钥；
    
-   多路复用支持 **mplex** 或 **yamux**，让一个连接上同时跑多个数据流。
    

### **5.4 节点发现**

新节点启动时通过 **bootnode**（启动节点）接入网络，使用类 Kademlia 协议的 **Discv5** 发现其它节点。Discv5 还能发送"广告"展示节点支持的子协议，方便对等节点协商通信协议。

## **六、Weak Subjectivity & Syncing：弱主观性与同步**

### **6.1 什么叫"弱主观性"**

区块链的"主观性"指需要依赖社会信息（从他处获取的可信状态）来确认当前正确链的程度。

-   **PoW** 是客观的：最长链就是规范链，任何新节点只看算力就能判断；
    
-   **PoS** 则天然带有主观性：因为在 PoS 里选择"正确的分叉"需要**计数历史投票**，这就给**长程攻击**留下了可乘之机——早期验证者提款后可以构建一条比当前规范链更长的替代链来欺骗新节点。
    

**弱主观性**是这样解决问题的：要求新节点从一个"足够近"的**可信检查点（weak subjectivity checkpoint）** 开始同步。这个检查点就像一个"不可逆转的锚"，区块链在检查点之前的历史被视为绝对正确、不可篡改。

### **6.2 检查点的核心机制**

-   弱主观性检查点是网络中所有节点都认可的**状态根**；
    
-   检查点之间的间隔小于验证者提款周期，这样即使有攻击者尝试分叉，在能提款之前就已经被罚没（slashing）了；
    
-   检查点不等于最终确认（finalized）区块：如果出现两个冲突的 finalized 区块，表明共识出现了严重问题，节点会陷入两难；但如果有区块与弱主观性检查点冲突，节点直接**拒绝**它。
    

### **6.3 轻客户端同步：Sync Committee**

有没有办法让手机或浏览器这样资源受限的设备也能安全地验证以太坊状态，而不用跑完整节点？这就是 **轻客户端（Light Client）** 要做的事。

Altair 升级引入的 **同步委员会（Sync Committee）** 是关键组件：

-   每约 27.3 小时（256 个 epoch）从活跃验证者中随机选出 512 个组成同步委员会；
    
-   这 512 个验证者负责在链上对每个 slot 的区块头签名；
    
-   轻客户端只需追踪这 512 个签名，而不是全部 **~300,000+** 验证者的证明，数据量从"天文数字"骤降至"伸手可及"；
    
-   BeaconState 中同时保有 `current_sync_committee` 和 `next_sync_committee`，给验证者预留一整个周期做好轮换准备。
    

### **6.4 同步模式选择**

| 同步模式 | 说明 | 适用场景 |
| --- | --- | --- |
| Checkpoint Sync | 从一个最近的弱主观性检查点开始同步，只下载检查点之后的数据 | 快速启动新节点，只需信任一个近期状态 |
| Snap Sync / State Sync | 直接下载最新的全局状态，然后才验证后续区块 | 全节点快速追赶 |
| Full Sync | 从创世开始逐块验证 | 安全要求最高的场景（如区块浏览器） |

**最佳实践**：启动新 CL 节点时，从可信来源（如多个独立区块浏览器交叉验证）获取最近的检查点状态，然后用 Checkpoint Sync 快速追到最新状态 —— 这是安全与速度之间的最佳平衡点。

## **七、学习路线建议**

对于初学者，建议按以下顺序逐层深入：

1.  **先搞懂"两层架构为啥这么设计"**（本节第一、二部分），理解 EL 和 CL 的分工；
    
2.  **掌握 SSZ 的类型与序列化**（第三部分），这是理解 CL 所有数据结构的基础；
    
3.  **了解 Beacon API 的常用查询**（第四部分），能独立查询节点状态；
    
4.  **理解共识层网络架构**（第五部分），知道证明和区块在节点间怎么传播；
    
5.  **深入弱主观性与同步**（第六部分），这是安全启动节点的必备知识。
    

每个主题都对应一个独立的子页面（如 Merkleization 子页链接在附件中已给出），可以在掌握概念后根据需要深入阅读。推荐的进阶学习资源：

-   **《Upgrading Ethereum》**（[eth2book.info](https://eth2book.info/)）：带注释版的共识层规范，逐步拆解每种状态转换

-   [**ethereum.org**](https://ethereum.org/) **开发者文档**：各板块的入门级教程
    
-   [**ethresear.ch**](https://ethresear.ch/)：前沿改进提案与社区讨论

# PeerDAS（对等数据可用性采样）学习笔记

## 一、核心概念

-   **全称**：Peer Data Availability Sampling（点对点数据可用性采样）
    
-   **标准**：EIP-7594，以太坊 Fusaka 升级的核心技术
    
-   **本质**：从 "全量下载验证" 转向 "概率性抽样验证" 的革命性数据可用性方案
    
-   **通俗理解**：就像质检员不用检查每一个产品，只需随机抽验少量样品，就能高概率判断整批产品是否合格
    

## 二、解决的核心问题

Dencun 升级（EIP-4844）引入了 Blob 数据类型，专门存储 Layer2 提交的交易数据，大幅降低了 L2 交易费。但**所有全节点必须下载完整的 Blob 数据**，导致：

-   节点硬件要求越来越高，普通人难以参与，去中心化程度下降
    
-   为了不压垮节点，不得不限制每个区块的 Blob 数量（目标 3 个，最大 6 个），成为 L2 扩容的瓶颈
    

## 三、工作原理（通俗版）

1.  **数据扩展**：将每个 Blob 通过**纠删码**技术扩展为原来的 2 倍大小（原始数据 + 冗余数据）
    
2.  **矩阵划分**：把所有扩展后的 Blob 排列成一个大矩阵，然后按**列**切分成 128 个 "数据列"（DataColumnSidecars）
    
3.  **节点分工**：每个节点根据自己的 ID，**确定性地负责保管 4 个数据列**（约占总数据的 1/32）
    
4.  **抽样验证**：每个时隙，节点随机向其他节点请求**另外 4 个随机数据列**进行验证
    
5.  **可用性判断**：只要节点能获取到超过 50% 的列（64 个），就能完整恢复所有 Blob 数据；只要能成功获取 1/8 的列，就能以极高概率确认数据可用
    

## 四、关键技术

-   **纠删码（Reed-Solomon）**：丢失 50% 的数据仍能完整恢复原始数据
    
-   **KZG 多项式承诺**：用于验证数据列的真实性和完整性，无需下载完整数据
    
-   **Gossipsub 子网**：为每个数据列建立专用的 P2P 广播通道，提高数据分发效率
    
-   **确定性分配**：节点根据 ID 固定负责某些列，防止攻击者针对性欺骗
    

## 五、与传统方案对比

表格

| 对比项 | 传统全量验证 | PeerDAS |
| --- | --- | --- |
| 数据下载量 | 100% Blob 数据 | 约 1/8 Blob 数据 |
| 存储需求 | 完整存储所有 Blob | 仅存储 4 个数据列（约 1/32） |
| 节点门槛 | 高（需要大带宽和硬盘） | 低（普通电脑即可运行） |
| Blob 容量上限 | 每区块 6 个 | 可安全提升至 48 个（8 倍），未来潜力更大 |
| 去中心化程度 | 逐渐降低 | 显著提升 |

## 六、核心优势

1.  **大幅降低节点负担**：带宽和存储需求降低约 8 倍，普通人也能运行以太坊节点
    
2.  **指数级提升 L2 容量**：Blob 容量提升 8 倍，直接带来 L2 交易费大幅下降和 TPS 提升
    
3.  **保持同等安全级别**：基于数学和密码学的概率验证，安全性不低于全量验证
    
4.  **灵活扩容**：配合 EIP-7892（BPO），未来可通过轻量级更新调整 Blob 数量，无需硬分叉
    

## 七、潜在挑战

-   **网络复杂性增加**：需要维护 128 个专用子网，对 P2P 网络稳定性要求更高
    
-   **数据恢复依赖节点协作**：如果大量节点离线，可能影响数据恢复
    
-   **激励机制待完善**：需要更明确的激励措施鼓励节点诚实存储和分享数据
    

## 八、应用场景与影响

-   **Layer2 扩容**：为 Optimism、Arbitrum、zkSync 等 L2 提供近乎无限的数据空间，交易费有望再降一个数量级
    
-   **链上游戏与 NFT**：支持更复杂的链上游戏和大规模 NFT 铸造，降低数据存储成本
    
-   **去中心化应用**：使数据密集型 DApp（如社交媒体、内容平台）在以太坊上成为可能
    
-   **以太坊长期发展**：是实现完整版 Danksharding 的关键一步，为以太坊成为全球结算层奠定基础

# 以太坊 PBS 家族核心术语速记笔记

（全是共识层 MEV 相关的升级方案，一句话讲明白）

## 1\. PBS（Proposer-Builder Separation）

-   **全称**：提议者 - 构建者分离
    
-   **通俗解释**：把 "谁来打包交易" 和 "谁来确认区块" 这两个活分开干
    
-   **核心作用**：
    
    -   原来验证者既要打包又要确认，现在专业的人干专业的事
        
    -   普通家庭验证者不用懂 MEV 套利，直接卖打包权赚钱
        
    -   防止大机构垄断区块生产
        
-   **现状**：现在以太坊用的是链外版 PBS（MEV-Boost），依赖第三方中继器
    

## 2\. ePBS（Enshrined PBS）

-   **全称**：协议内嵌 PBS
    
-   **通俗解释**：把 PBS 直接写进以太坊代码里，不用再靠第三方中继器
    
-   **核心作用**：
    
    -   解决现在 MEV-Boost 中继器中心化、单点故障的问题
        
    -   协议直接保障交易安全，不用信任任何人
        
    -   对应 EIP-7732，是以太坊 Glamsterdam 升级的核心
        
-   **关键改进**：把共识验证和执行验证分开，减轻节点 4 秒关键窗口期的压力
    

## 3\. PTC（Payload Timeliness Committee）

-   **全称**：有效载荷及时性委员会
    
-   **通俗解释**：ePBS 里的 "监考老师"，专门盯着构建者按时交作业
    
-   **核心作用**：
    
    -   每个 slot 随机选 512 个验证者当 PTC 成员
        
    -   检查构建者有没有在规定时间内公布完整区块内容
        
    -   没按时交就罚构建者的质押币，保障网络不卡壳
        

## 4\. PEPC（Protocol-Enforced Proposer Commitments）

-   **全称**：协议强制提议者承诺
    
-   **发音**：pepsi（百事可乐），好记
    
-   **通俗解释**：让验证者可以对自己的区块提条件，协议帮你强制执行
    
-   **核心作用**：
    
    -   原来验证者只能选最高报价的区块，现在可以加规则
        
    -   比如 "必须包含某笔交易"、"不能用某构建者的区块"
        
    -   给验证者更多话语权，防止被构建者绑架
        

## 5\. TBHL（Two-Block HeadLock）

-   **全称**：双区块头锁
    
-   **通俗解释**：ePBS 的一种具体实现方案，一个 slot 出两个小区块
    
-   **核心作用**：
    
    -   第一个区块：提议者公布选了哪个构建者的报价
        
    -   第二个区块：构建者公布完整的交易内容
        
    -   用 "头锁" 机制保护构建者，防止提议者偷 MEV
        

## 6\. ET（Execution Ticket）

-   **全称**：执行凭证
    
-   **通俗解释**：ePBS 的另一种竞争方案，直接拍卖 "打包交易的门票"
    
-   **核心作用**：
    
    -   谁买到票谁就能打包区块，验证者只负责确认
        
    -   卖票的钱全部销毁（类似 EIP-1559），让所有 ETH 持有者受益
        
    -   验证者收入更稳定，家庭质押门槛更低
        

* * *

### 一句话关系总结

PBS 是基础理念 → ePBS 是把它写进协议的官方版 → PTC 是 ePBS 里的监管机制 → TBHL 和 ET 是 ePBS 的两种不同技术实现 → PEPC 是给验证者加 buff 的扩展功能

# 以太坊共识层（CL）架构学习笔记

核心定位：以太坊 PoS 共识的核心载体，负责链上共识达成、区块最终确认、节点网络协同，与执行层（EL）并行配合，共同支撑以太坊网络的完整运行。

* * *

## 一、核心基础概念

### 1\. 分叉与重组

-   **分叉**：区块链因网络延迟、中断、恶意行为等，节点对链状态产生不同视图，出现多条并行分支的现象。
    
-   **重组（Reorg）**：节点根据新的区块 / 投票信息，按分叉选择规则切换到最优链，回滚原链区块、采纳新分支区块的过程。
    
    -   短重组（1-2 个块）：多由网络延迟导致，属于正常现象；
        
    -   长重组：通常由链攻击、协议漏洞或实现缺陷导致，属于异常情况。
        

### 2\. 安全（Safety）与活性（Liveness）

-   **安全**：「坏事永远不会发生」，核心是保证所有诚实节点对链状态达成一致，杜绝双花、冲突区块确认，提供**区块最终性**（确认后不可回滚）。
    
-   **活性**：「好事最终会发生」，核心是保证链能持续产出新区块，永远不会陷入停滞僵局。
    

### 3\. CAP 定理与以太坊的取舍

分布式系统无法同时满足一致性、可用性、分区容错性。网络不可靠时，无法设计出永远同时完美满足安全和活性的协议。

-   以太坊的核心选择：**网络异常时优先保障活性**，先保证链不停止出块，网络恢复后再重新达成共识，哪怕牺牲临时的安全性。
    

* * *

## 二、以太坊核心共识协议：Gasper

Gasper 是以太坊 PoS 共识的核心，由两个独立协议互补结合而成，分工明确、能力互补。

1.  **LMD GHOST**
    
    -   核心作用：保障**活性**，提供链的实时性，负责持续出块、选定链头，让链持续前进不卡壳。
        
    -   分叉选择规则：「贪婪最重观察子树」，统计验证者对区块及其子区块的累计投票，投票权重最高的子树顶端，即为链的头块。
        
2.  **Casper FFG**
    
    -   核心作用：保障**安全**，提供链的**最终性**，区块被最终确认后绝对不可回滚，杜绝长分叉和双花风险。
        
    -   分叉选择规则：永远遵循「包含最高正当检查点的链」，且绝不回退已最终确认的区块。
        
3.  组合优势
    
    LMD GHOST 负责 “让链持续往前走”，Casper FFG 负责 “给前进的链拍板定案”；哪怕 Casper FFG 暂时无法完成区块最终确认，LMD GHOST 也能保证链持续出块，完美匹配以太坊优先活性的设计。
    

* * *

## 三、分叉选择规则

### 1\. 核心作用

解决节点因网络问题 / 恶意行为出现的状态分歧，让所有诚实节点最终收敛到同一条唯一的规范链上，是共识协议的核心规则。

### 2\. 基础前提

任何被选中的区块，必须满足：本身及所有祖先区块都符合协议规则、完全有效；无效区块及其子区块会被直接忽略。

### 3\. 主流分叉选择规则对比

表格

| 规则类型 | 适用场景 | 核心逻辑 | 关键说明 |
| --- | --- | --- | --- |
| 最重链规则 | 比特币、以太坊 PoW | 选择累计工作量最大的链的末端作为头块 | 误区纠正：以太坊 PoW从未使用 GHOST 规则，白皮书曾规划但因攻击风险未落地，一直使用最简单稳定的最重链规则 |
| Casper FFG 规则 | 以太坊 PoS | 选择包含最高正当检查点的链，绝不回退最终区块 | 核心保障区块最终性，锚定链的安全底线 |
| LMD GHOST 规则 | 以太坊 PoS | 选择验证者累计投票权重最高的子树顶端作为头块 | 保障链的实时出块和活性，与 Casper FFG 规则配合使用 |

* * *

## 四、共识层（CL）整体架构与核心组件

以太坊网络由 **执行层（EL）+ 共识层（CL）** 两层组成，每个全节点必须同时运行两层客户端软件。

### 1\. 核心组件

（1）信标节点（共识层客户端）

-   定位：CL 的核心协调者，PoS 共识的 “大脑”。
    
-   主流实现：Prysm、Teku、Lighthouse、Nimbus、Lodestar。
    
-   核心职责：
    
    1.  与其他信标节点 P2P 通信，同步区块、投票信息；
        
    2.  通过 Engine-API 与本地执行层客户端交互；
        
    3.  对接本地验证者客户端，下发指令、广播其工作成果。
        

（2）验证者客户端

-   准入门槛：质押 32 ETH，获得参与共识的资格。
    
-   定位：PoS 共识的 “执行者”，替代了 PoW 时代的矿工。
    
-   核心职责：按协议规则提议新区块、对区块进行投票证明；仅和本地信标节点通信，不直接对接 P2P 网络。
    

### 2\. 网络分层

-   主网：以太坊生产环境，处理真实 ETH 交易，承载实际资产价值。
    
-   测试网：供开发者测试协议、合约功能，使用无实际价值的测试 ETH，规避主网风险。
    
-   通用规则：无论主网 / 测试网，都必须包含 EL+CL 两层，缺一不可。
    

* * *

## 五、CL 与 EL 的协同工作流程

两层通过本地 RPC 的**Engine-API**通信，并行运行，共享节点 ENR，各自持有独立密钥（EL 对应 eth1 密钥，CL 对应 eth2 密钥）。

### 场景 1：共识客户端不是区块生产者

1.  通过 P2P 区块八卦协议接收新的信标块；
    
2.  对区块进行预验证，过滤无效块；
    
3.  将区块内的交易执行载荷发送给执行层；
    
4.  执行层执行交易，验证区块状态合法性，返回验证结果给共识层；
    
5.  共识层确认区块有效后，将其加入区块链，生成认证信息并广播到全网。
    

### 场景 2：共识客户端是区块生产者（轮到该节点出块）

1.  收到协议通知，确认当前节点为本轮区块生产者；
    
2.  调用执行层客户端的创建区块方法；
    
3.  执行层访问交易内存池，打包合法交易，执行后生成区块哈希；
    
4.  共识层将交易数据、区块哈希打包进信标块，通过 P2P 网络广播新区块；
    
5.  其他节点验证区块有效后进行投票证明；
    
6.  区块获得足够验证者证明后，被加入链头，逐步完成最终确认。
    

* * *

## 六、状态转换机制

状态转换是区块链的核心：所有节点从相同的创世状态出发，按相同顺序执行相同的区块，最终必须得到完全一致的状态，否则即为共识失败。

### 1\. 核心定义

状态转换函数是**纯函数**（输出仅由输入决定，无额外副作用），核心公式：

`S' ≡ f(S, B)`

-   S：区块执行前的信标链状态
    
-   B：待执行的信标块
    
-   S'：区块执行后的新状态
    

### 2\. 信标链的特殊设计

与 PoW 的**区块驱动**不同，信标链是**槽位驱动**：状态更新由槽位进度决定，哪怕该槽位没有出块，也会执行对应的状态更新。

### 3\. 三类核心状态转换

1.  **每槽位转换**：每个槽位（12 秒）都会执行，无论该槽位是否有区块；
    
2.  **每区块转换**：仅当槽位有有效区块时执行，处理区块内的交易、投票等内容；
    
3.  **每时代转换**：每个时代（32 个槽位，约 6.4 分钟）的起始槽位执行，处理验证者奖惩、质押进出、检查点确认等时代级操作。
    

### 4\. 无效转换判定

出现以下情况的状态转换视为无效：

-   执行中出现断言失败、越界访问等未处理异常；
    
-   出现 uint64 数值溢出 / 下溢；
    
-   区块签名验证失败；
    
-   区块声明的状态根与执行后的实际状态根不匹配。
    

* * *

## 七、CL 网络通信核心技术栈

共识层拥有独立的 P2P 网络，核心技术组件如下：

1.  **libp2p**：P2P 通信核心协议
    
    模块化的点对点通信 “万能工具箱”，支撑 CL 节点的所有网络通信，支持多传输协议、可插拔适配不同网络环境。核心协议栈从下到上分为：
    
    -   发现层：mdns、kademlia (DHT)、rendezvous，负责找到网络中的对等节点；
        
    -   NAT / 中继层：relay、dcutr，解决防火墙 / NAT 阻挡导致的直连失败问题；
        
    -   传输层：TCP（必选）、QUIC、WebSockets 等，负责节点间数据传输；
        
    -   安全层：libp2p-noise（默认）、TLS，负责连接加密和节点身份认证；
        
    -   复用层：yamux（默认优先）、mplex，让单条连接同时承载多个独立通信流；
        
    -   应用层：Gossipsub、Req/Resp、ping，负责区块 / 投票广播、节点心跳等共识逻辑。
        
2.  **discv5**：节点发现协议
    
    运行在 UDP 上，与 libp2p 并行，专门负责节点发现；核心作用是动态交换、更新以太坊节点记录（ENR），让节点实时获取最新的可连接对等节点信息。
    
3.  **ENR（以太坊节点记录）**
    
    以太坊节点的 “身份名片”，结构化存储节点的身份、连接地址、支持的协议等信息；核心组成包括：签名（保证真实性）、序列号（更新时递增，识别最新版本）、键值对（存储连接细节）。
    
4.  **SSZ（简单序列化）**
    
    共识层的核心编码方案，替代了执行层的 RLP 编码；核心优势是序列化结果确定性强，原生支持高效默克尔化，完美适配 PoS 共识的轻节点、状态验证需求。
    
5.  **Snappy**：压缩算法
    
    谷歌开发的高速压缩算法，优先保证压缩 / 解压速度，兼顾合理压缩比，用于降低 P2P 网络传输的数据量。

## 八、关键误区纠正与补充

1.  以太坊 PoW 共识从未使用 GHOST 分叉选择规则，仅在白皮书规划过，因攻击风险最终未落地，一直使用最重链规则；
    
2.  区块最终性仅由 Casper FFG 提供，LMD GHOST 不具备正式的安全保障，仅负责选链和出块；
    
3.  网络分区时，以太坊会优先保证链持续出块（活性），但会暂停区块最终确认，网络恢复后再重新达成共识；
    
4.  验证者仅能通过本地信标节点接入网络，不能直接对接 P2P 网络，所有通信都由信标节点中转。

# 以太坊共识层学习笔记

## 一、核心概念与目标

### 1\. 共识层的本质

-   **解决的问题**：在**不可靠的基础设施**（节点可能掉线、出错、被攻击，网络不稳定）上，构建**可靠的分布式系统**
    
-   **以太坊的目标**：让全球数万个独立节点的账本**完全一致**，快速达成交易顺序和结果的共识
    

### 2\. 拜占庭容错（BFT）

-   系统能容忍**任意故障**（包括节点恶意搞破坏），依然正常运行
    
-   去中心化网络的核心要求：节点之间无法互相信任，必须靠机制保证共识
    

## 二、共识机制的关键澄清

-   ❌ 错误认知：PoW（工作量证明）、PoS（权益证明）是共识协议
    
-   ✅ 正确理解：它们是**Sybil 抵抗机制**（防止攻击者低成本创建大量假节点攻击系统）
    
-   以太坊真正的共识协议：**Gasper** = LMD GHOST（选链头） + Casper FFG（最终性）
    

## 三、区块链基础

-   **区块**：交易的批次打包（优化共识开销），每个区块指向父区块，形成链式结构
    
-   以太坊区块内容演变：
    
    -   合并前：执行链存用户交易，信标链存验证者证明
        
    -   合并后：信标链块同时包含执行载荷（用户交易）
        
    -   EIP-4844（Deneb 升级）后：新增 Blob 数据（临时存储，供 L2 降低手续费）
        

## 四、PoS 过渡（合并）

-   **时间**：2022 年 9 月 15 日（巴黎硬分叉）
    
-   **激活方式**：按**终端总难度（TTD）** 而非区块高度激活（避免恶意分叉）
    
-   **核心变化**：
    
    -   信标链接管共识，矿工被验证者取代
        
    -   能耗降低 99% 以上，安全性和可扩展性提升
        

## 五、信标链（PoS 核心）

### 1\. 验证者

-   **准入门槛**：质押至少 32 ETH（最多 2048 ETH）作为抵押
    
-   **核心职责**：
    
    1.  被随机选中时**提出新区块**
        
    2.  验证他人提出的区块，提交**证明（投票）**
        
    3.  参与共识，最终确定区块链状态
        
-   **运行方式**：由验证者客户端运行，一个客户端可管理多个验证者
    

### 2\. 时间单位

-   **时段（Slot）**：12 秒，每个时段是出块的机会（可能空块）
    
-   **纪元（Epoch）**：32 个时段 = 6.4 分钟，是共识的大周期
    

### 3\. 委员会

-   每个时段，验证者被随机分成**至少 128 人一组**的委员会
    
-   负责验证该时段的区块，投票按质押金额加权
    
-   **随机性来源**：RANDAO（伪随机）+ VDF（可验证延迟函数），防止被操控
    
-   每个纪元重新洗牌分配委员会，保证去中心化
    

### 4\. Blob（EIP-4844）

-   作用：临时存储数据，供 L2 使用，大幅降低 L2 手续费
    
-   存储：每个块 3~6 个 Blob，保留 4096 个纪元（约 18 天）后自动删除
    
-   额外存储需求：52~104GB
    

## 六、检查点与最终性

### 1\. 基本概念

-   **检查点**：每个纪元第一个时段的区块（无则取最近的有效区块）
    
-   **正当化**：检查点获得≥2/3 质押金额的投票
    
-   **最终确定**：当后一个检查点被正当化时，前一个检查点及其之前的所有区块**不可撤销**
    

### 2\. 最终性时间

-   理论：2 个纪元（12.8 分钟）
    
-   实际平均：14~16 分钟（交易通常在纪元中间被打包）
    

### 3\. 两种投票

-   **LMD GHOST 投票**：每个时段的委员会投，选当前链头
    
-   **Casper FFG 投票**：所有验证者每个纪元投，选检查点，负责最终性
    

## 七、激励与惩罚机制

### 1\. 奖励

-   认证奖励：投票与多数一致获得
    
-   提案奖励：提出有效区块获得（约占总奖励 1/8）
    
-   举报人奖励：举报违规验证者获得
    

### 2\. 惩罚

-   轻微惩罚：离线、未及时投票，扣少量 ETH
    
-   **非活动泄露**：最终性延迟超过 4 个纪元时，不活跃验证者会被持续扣 ETH，直到恢复最终性
    
-   **削减（Slashing）**：严重违规，罚没 0.5~32 ETH 并被踢出网络
    
    -   可削减的 4 种行为：双重提案、LMD 双重投票、FFG 环绕投票、FFG 双重投票
        

## 八、验证者生命周期

1.  **激活**：存入 32 ETH 到存款合约，等待队列（每个纪元激活数量有限）
    
2.  **运行**：履行提案和验证职责，获得奖励或被惩罚
    
3.  **退出**：
    
    -   自愿退出：服务满 2048 个纪元（约 9 天）可申请
        
    -   强制退出：余额低于 16 ETH 时被停用
        
4.  **提取**：
    
    -   诚实退出：4 个纪元延迟后，约 27 小时可提取余额
        
    -   被削减：延迟约 36 天才能提取剩余余额

# RLP 递归长度前缀 极简学习笔记

## 一、核心定义

-   以太坊**执行层**专属标准序列化协议，把结构化数据转成统一字节流
    
-   唯一目的：编码 / 解码**嵌套字节数组**，保证不同语言客户端数据完全一致
    
-   应用范围：交易、账户状态、区块头等所有执行层核心数据
    
-   对比：以太坊共识层使用更现代的 SSZ（简易序列化）
    

## 二、编码核心规则（速记）

所有编码都由**数据类型 + 长度**决定，前缀是唯一判断依据

1.  **单字节（0x00~0x7F）**：直接使用原字节
    
    -   示例：0x2a → 0x2a
        
2.  **短字符串（长度 1~55 字节）**：前缀 = 0x80 + 长度，后跟内容字节
    
    -   示例："dog"(3 字节) → 0x83 0x64 0x6f 0x67
        
3.  **长字符串（长度 > 55 字节）**：前缀 = 0xb7 + 长度的字节数，后跟长度（大端）+ 内容
    
    -   示例：56 字节字符串 → 0xb8 0x38 + 56 字节内容
        
4.  **短列表（总编码长度 1~55 字节）**：前缀 = 0xc0 + 总长度，后跟所有项的编码
    
    -   示例：\["cat","dog"\] → 0xc8 0x83 0x63 0x61 0x74 0x83 0x64 0x6f 0x67
        
5.  **长列表（总编码长度 > 55 字节）**：前缀 = 0xf7 + 总长度的字节数，后跟总长度（大端）+ 所有项的编码
    
    -   示例：57 字节总长度 → 0xf8 0x39 + 57 字节编码内容
        
6.  **特殊值编码**：
    
    -   空字符串 /null/false → 0x80
        
    -   空列表 \[\] → 0xc0
        

## 三、解码核心规则（编码逆过程）

看 **第一个字节（前缀）** 直接判断类型和长度，列表需递归解码

表格

| 前缀范围 | 数据类型 | 解码步骤 |
| --- | --- | --- |
| 0x00~0x7F | 单字节 | 直接输出该字节 |
| 0x80 | 空值 | 输出空字符串 /null/false |
| 0x81~0xB7 | 短字符串 | 长度 = 前缀 - 0x80，取后续对应字节 |
| 0xB8~0xBF | 长字符串 | 长度字节数 = 前缀 - 0xB7 → 读取实际长度 → 取后续对应字节 |
| 0xC0~0xF7 | 短列表 | 总长度 = 前缀 - 0xC0 → 递归解码后续字节中的每个项 |
| 0xF8~0xFF | 长列表 | 长度字节数 = 前缀 - 0xF7 → 读取总长度 → 递归解码后续字节中的每个项 |

## 四、RLP 设计本质与优势

1.  **极简主义**：不定义布尔、整数、浮点数等数据类型，只处理嵌套数组结构，数据含义完全由上层协议决定
    
2.  **绝对确定性**：相同输入在任何客户端都生成完全相同的字节流，这是区块链共识的基础
    
3.  **高效紧凑**：对嵌套字节数组的编码比 JSON、XML 等通用格式更省空间
    

## 五、常用工具与核心资源

-   工具：Geth RLP、RLP Dump、Node.js/Python/Rust/Nethermind RLP 库
    
-   核心资源：以太坊黄皮书、官方 RLP 文档

DevP2P 学习笔记

## 一、DevP2P 核心基础（EL 执行层网络协议）

DevP2P 是以太坊执行层（EL）的网络协议，核心依赖两种传输层协议，通俗理解如下：

-   **TCP（传输控制协议）**：面向连接，发送数据前先“握手”建立连接，可靠但稍慢，能保证数据按顺序、无错误传递（比如寄快递，先确认收件人能收，再送，确保包裹完好送达）。
    
-   **UDP（用户数据报协议）**：无连接，直接发送数据不提前确认，速度快但不可靠，无法保证数据顺序和完整性（比如发短信，直接发送，不管对方是否及时收到、是否乱序）。
    

## 二、EL 网络规范（点对点通信规则）

以太坊是点对点网络，每个节点有两个核心组件（执行客户端、共识客户端），各自有专属点对点网络：执行客户端负责“八卦”交易，共识客户端负责“八卦”区块。

EL 网络分两层，对应两种协议：

-   **发现层**：负责找其他节点，用 UDP 协议（只需让其他节点知道自己能通信，不用可靠连接）。
    
-   **传输层**：负责节点间传消息，用 TCP 协议（需要可靠连接，确保消息有序、无错）。
    

## 三、Discv 协议（节点发现协议）

### 核心作用

让新节点在网络中找到其他节点，核心依赖“引导节点”和类似 Kademlia 的算法（简单说就是分布式“通讯录”，节点通过这个“通讯录”找邻居）。

### 节点发现流程（通俗版）

1.  新节点先找“引导节点”（网络中所有节点都认识的“老节点”），发送 PING 消息。
    
2.  引导节点回复 PONG 消息，两者匹配后，新节点就能和引导节点建立连接。
    
3.  新节点向引导节点发送 FIND-NEIGHBOURS 请求，获取其他可连接的“邻居节点”，再和邻居节点重复 PING-PONG 流程，逐步融入网络。
    

### 有线协议（核心消息格式）

就是节点间发送消息的“规范格式”，核心消息有 5 种，无需记复杂结构，重点记功能：

-   PING/PONG：确认节点是否在线、可连接。
    
-   FindNode：请求获取靠近目标节点的邻居列表。
    
-   邻居包：回复 FindNode，提供可连接的节点列表（最多 16 个）。
    
-   ENR 请求/响应：获取/提供节点的连接信息（类似节点的“名片”）。
    

## 四、Discv4 与 Discv5（发现协议的两个版本）

### 核心区别（通俗对比）

| 特点 | Discv4 | Discv5 |
| --- | --- | --- |
| 安全性 | 明文传输（不安全） | AES-GCM 加密（安全） |
| 服务发现 | 功能有限 | 支持基于主题的精准查找 |
| 扩展性 | 一般，适配小型网络 | 优化，适配大型网络 |
| 核心优势 | 简单，早期主流 | 安全、灵活，目前主流（多数客户端已采用） |

补充：目前部分客户端仍在从 Discv4 过渡到 Discv5。

## 五、ENR（以太坊节点记录）

### 核心作用

节点的“电子名片”，存储节点的连接信息（IP、端口）、公钥等，方便其他节点连接。

### 关键细节

-   必填字段：节点ID方案（如“v4”），其他字段（IP、端口等）可选。
    
-   编码格式：RLP 编码（最大300字节），还有可直接查看的文本格式（前缀为 enr:）。
    
-   其他格式：除了 ENR，节点还支持 multiaddr、enode 两种连接格式（类似不同风格的“名片”）。
    

## 六、RLPx 协议（节点传输协议）

### 核心作用

基于 TCP 协议，负责节点间的**安全消息交换**（发现节点后，真正传数据靠它），名字来源于 RLP 序列化格式。

### 核心特点

-   安全连接：通过加密握手建立安全会话，用 AES-128-CTR 加密消息，保证隐私和完整性。
    
-   前向保密：每次会话生成临时密钥，会话结束后丢弃，即使长期密钥泄露，过去的通信也不会被破解。
    
-   消息帧：消息被封装成“帧”，支持多协议同时在一个连接上运行（复用）。
    

### 核心消息与子协议

-   基础消息（必支持）：Hello（宣布支持的功能）、Disconnect（断开连接）、Ping/Pong（检查节点在线）。
    
-   应用级子协议（基于 RLPx 运行，负责具体功能）：
    
    -   eth：交换区块链数据（区块、交易）。
        
    -   snap：同步节点状态。
        
    -   les：支持轻客户端（不用存储完整区块链数据）。
        
    -   门户：轻客户端的去中心化检索网络。
        

## 总结

DevP2P 是 EL 节点的“通信体系”：Discv 协议（UDP）负责“找节点”，RLPx 协议（TCP）负责“传数据”，ENR 是节点的“身份名片”，Discv5 是目前更安全、更高效的发现版本，整体保障以太坊点对点网络的稳定运行。

# 以太坊交易学习笔记

## 一、交易基础

-   **定义**：由外部账户（EOA）发出的**加密签名指令**，通过 JSON-RPC 广播到全网，是触发 EVM 状态转换的唯一方式
    
-   **核心作用**：实现 ETH 转账、部署智能合约、调用智能合约功能
    

## 二、交易核心字段（必背）

表格

| 字段 | 通俗解释 | 关键作用 |
| --- | --- | --- |
| nonce | 发送方的交易序号（从 0 开始严格递增） | 1. 防重放攻击（同 nonce 交易只能执行一次）2. 计算合约地址（发送方地址 + nonce 哈希后 20 字节）3. 替换 / 取消待处理交易（同 nonce + 更高 gasPrice） |
| gasPrice | 每单位 gas 的价格（单位：wei） | 决定交易优先级，价格越高矿工越先打包 |
| gasLimit | 最多愿意为交易支付的 gas 总量 | 防止无限循环消耗 gas，不足时交易回滚且已消耗 gas 不退 |
| to | 接收方 20 字节地址 | 决定交易类型：- 空值 → 部署新合约- EOA 地址 → 普通 ETH 转账- 合约地址 → 调用合约函数 |
| value | 转账金额（单位：wei） | 部署合约时为新合约的初始余额 |
| data/init | 任意长度字节数组 | - 部署合约：init 字节码（仅执行一次，生成 runtime 代码）- 调用合约：函数签名 + 参数编码 |
| v/r/s | ECDSA 签名三组件 | 证明交易由发送方私钥签署，不可伪造 |

## 三、合约创建全流程

### 1\. 核心概念

-   **init 字节码**：部署阶段执行的临时代码，唯一作用是**生成并返回 runtime 字节码**
    
-   **runtime 字节码**：合约的永久业务代码，存储在链上，每次合约调用都会执行
    
-   **合约地址计算**：`keccak256(rlp([发送方地址, nonce]))` 取后 20 字节
    

### 2\. 极简示例（乘法合约）

-   目标：部署一个计算 6×7 并将结果存入存储槽 0 的合约
    
-   **runtime 字节码**（业务逻辑）：`6006600702600055`
    
    -   对应指令：PUSH1 06 → PUSH1 07 → MUL → PUSH1 00 → SSTORE
        
-   **init 字节码**（部署逻辑）：`6008600c60003960086000f3` + runtime 字节码
    
    -   作用：将 runtime 代码复制到内存，然后返回给 EVM 永久存储
        

### 3\. 部署交易结构

json

```
[
  "0x",          // nonce=0（账户第一笔交易）
  "0x77359400",  // gasPrice=2Gwei
  "0x13880",     // gasLimit=80000（合约部署标准值）
  "0x",          // to为空，标识合约创建模式
  "0x05",        // 给新合约预存5wei初始余额
  "0x6008600c60003960086000f36006600702600055"  // 完整init字节码
]
```

## 四、合约执行流程

-   交易`to`字段填写**已部署合约的地址**
    
-   `data`字段填写**函数调用数据**（函数选择器 + 参数编码）
    
-   示例：调用上述乘法合约（无自定义函数，data 为空）
    

json

```
[
  "0x1",          // nonce=1（账户第二笔交易）
  "0x77359400",   // gasPrice=2Gwei
  "0x13880",      // gasLimit=80000
  "0x5fbdb2315678afecb367f032d93f642f64180aa3",  // 合约地址
  "0x",           // 不转账
  "0x"            // 无调用数据，执行合约默认逻辑
]
```

-   执行结果：合约存储槽 0 的值变为`0x2a`（十进制 42）
    

## 五、交易收据（Receipt）

-   每笔交易执行后生成的**执行结果凭证**，永久存储在链上
    
-   核心内容：
    
    1.  交易类型：区分旧版交易和类型化交易
        
    2.  状态：`0x1`表示成功，`0x0`表示失败
        
    3.  Gas Used：交易实际消耗的 gas 量
        
    4.  日志：合约触发的事件（用于前端监听和数据查询）
        
    5.  日志布隆：256 字节的过滤器，快速定位区块内的相关日志
        

## 六、EIP-2718：类型化交易标准

-   **解决问题**：旧版 RLP 编码难以扩展新交易类型，设计脆弱
    
-   **统一格式**：`交易类型(1字节) + 交易载荷`
    
-   **向后兼容规则**：
    
    -   首字节在`0x00-0x7f` → 类型化交易
        
    -   首字节≥`0xc0` → 旧版 RLP 交易
        
-   **意义**：轻松支持 EIP-1559（动态 gas）、EIP-4844（Blob 交易）等新特性
    

## 七、Foundry 本地实操（快速验证）

Foundry 是 Rust 编写的高性能以太坊开发工具，包含：

-   **Anvil**：极速本地测试节点
    
-   **Cast**：命令行工具，用于发送交易、查询链上数据
    

### 实操步骤

1.  启动本地节点：`anvil`（自动生成 10 个测试账户，每个预存 10000ETH）
    
2.  准备交易 payload（如上述部署交易）
    
3.  签名交易：`node sign.js '[payload数组]' [测试账户私钥]`
    
4.  发布交易：`cast publish [签名后的十六进制交易]`
    
5.  验证结果：
    
    -   查看合约代码：`cast code [合约地址]`
        
    -   查看合约余额：`cast balance [合约地址]`
        
    -   查看合约存储：`cast storage [合约地址] 0x`
        

## 八、底层技术速览

-   **RLP 编码**：以太坊标准序列化格式，高效打包任意嵌套的二进制数据
    
-   **Keccak-256**：以太坊使用的加密哈希函数，生成 32 字节唯一指纹
    
-   **ECDSA**：椭圆曲线数字签名算法，确保交易的真实性和不可篡改性

# 以太坊执行层核心数据结构学习笔记

## 总览

-   以太坊执行层用 **Merkle Patricia Tree（MPT，默克尔帕特里夏树）** 存储所有状态和历史数据
    
-   所有数据先经过**RLP 编码**再存入树中
    
-   所有树的根哈希都存在区块头，用于密码学验证和全网共识
    

## 一、基础编码：RLP

-   以太坊专属的**序列化编码方式**，把任意复杂数据（整数、字符串、数组、对象）统一转换成字节数组
    
-   核心作用：统一数据格式，用于计算节点哈希、网络传输和磁盘持久化
    
-   特点：极简高效，只处理字节和列表两种基本类型
    

## 二、基础树结构

### 1\. 默克尔树（Merkle Tree）

-   核心能力：**数据完整性验证**
    
-   结构：二叉树，叶节点存原始数据，非叶节点存两个子节点的哈希值，最顶层是**默克尔根**
    
-   构建方式：自下而上反复哈希成对节点；奇数个叶节点时复制最后一个凑偶数
    
-   关键特性：
    
    -   防篡改：任何一个叶节点数据变化，都会导致根哈希完全不同
        
    -   轻量验证：只需提供从叶节点到根的路径（Merkle 证明），就能验证数据是否存在，无需下载全部数据
        
-   以太坊应用：区块头中的交易根、收据根、状态根都是默克尔根
    

### 2\. 帕特里夏树（Patricia Trie / 基树）

-   核心能力：**高效键值存储与查找**
    
-   结构：前缀压缩的字典树，把公共前缀合并成一条边，避免每个字符都建一个节点
    
-   关键特性：
    
    -   空间效率高：大幅减少节点数量
        
    -   查找速度快：直接按前缀遍历
        
    -   只在叶节点存值，中间节点仅用于导航
        

## 三、核心：默克尔帕特里夏树（MPT）

-   完美结合了默克尔树的**验证能力**和帕特里夏树的**高效存储能力**，是以太坊所有数据存储的基础
    
-   基本单位：**Nibble（半字节 / 4 位）**，对应一个十六进制字符，所以每个节点最多有 16 个分支
    
-   三种节点类型：
    
    1.  **分支节点**：17 元素数组（16 个分支 + 1 个值），用于导航，当一个节点有多个子节点时使用
        
    2.  **扩展节点**：优化节点，当一个分支节点只有一个子节点时，把路径压缩成一个扩展节点
        
    3.  **叶节点**：存储实际的键值对，包含剩余路径和对应的值
        
-   节点哈希：每个节点的哈希是其 RLP 编码内容的 Keccak-256 哈希，哈希值就是该节点的引用地址
    

## 四、以太坊的四大 MPT 树

### 1\. 交易树（Transaction Trie）

-   作用：存储一个区块内的所有交易
    
-   键值对：`RLP(交易索引) → RLP(交易对象)`
    
-   特点：每个区块一个，一旦确认就**永久不可变**
    
-   交易对象包含：nonce、gasLimit、手续费参数、收发地址、转账金额、输入数据、签名等
    

### 2\. 收据树（Receipt Trie）

-   作用：存储一个区块内所有交易的**执行结果**
    
-   键值对：`RLP(交易索引) → RLP(收据对象)`
    
-   特点：每个区块一个，不可变
    
-   收据内容：交易状态、gas 使用量、日志（合约事件）等
    
-   用途：快速查询交易结果、轻客户端验证、快照同步（大幅加快节点同步速度）
    

### 3\. 世界状态树（World State Trie）

-   作用：存储**所有以太坊账户**的当前状态，是以太坊的 "全局数据库"
    
-   键值对：`Keccak256(账户地址) → RLP(账户对象)`
    
-   账户对象包含：
    
    -   nonce：该账户发送的交易数量
        
    -   balance：账户 ETH 余额（单位：wei）
        
    -   codeHash：合约代码的哈希（普通账户为空字符串的哈希）
        
    -   storageRoot：该账户存储树的根哈希
        
-   特点：
    
    -   全局唯一，**随每个区块动态更新**
        
    -   本身不存储在链上，只有根哈希存在区块头
        
    -   完整节点保留当前状态和回滚所需的历史状态；档案节点保留所有历史状态
        

### 4\. 存储树（Storage Trie）

-   作用：存储**智能合约的持久化变量**
    
-   键值对：`Keccak256(存储槽索引) → RLP(存储值)`
    
-   特点：
    
    -   每个合约账户**独立拥有一个存储树**，普通账户没有
        
    -   通过世界状态树中的`storageRoot`引用
        
    -   Solidity 等高级语言会自动把合约变量映射到不同的存储槽
        
-   安全设计：存储槽索引先哈希再作为键，防止攻击者构造恶意键导致 DOS 攻击
    

## 五、未来升级：Verkle 树

-   以太坊路线图的核心升级，将**全面替代 MPT**
    
-   核心目标：实现**无状态客户端**，大幅降低节点运行门槛
    
-   与 MPT 的核心区别：
    
    -   结构：MPT 是 16 元树，Verkle 树是 256 元树，树的深度更浅
        
    -   验证：MPT 用哈希验证，Verkle 树用**向量承诺**验证
        
    -   见证数据：Verkle 树的见证数据比 MPT 小 95% 以上（1000 个叶子从 4MB 降到 150KB）
        
-   优势：无状态客户端无需存储完整状态，只需区块头和见证数据就能验证交易
    
-   挑战：从 MPT 到 Verkle 树的迁移需要大量计算和数据转换
    

## 学习意义

-   理解以太坊底层数据存储原理，能更深入地掌握合约执行、交易验证、gas 消耗的本质
    
-   对合约存储优化、安全审计、轻客户端开发等高级主题至关重要
    
-   了解 Verkle 树等未来升级，能提前把握以太坊技术发展方向

# 以太坊预编译合约 极简学习笔记

## 一、一句话搞懂

以太坊官方内置的 **"超级工具账户"**（地址`0x01~0x0a`），专门干 EVM 跑起来又慢又费 gas 的复杂加密计算，调用方式和普通合约完全一样。

## 二、核心基础信息

-   不是普通智能合约：代码**不在 EVM 里**，是以太坊节点软件直接写死的原生功能
    
-   特殊标识：用`EXTCODESIZE`查它的代码长度会返回`0`（看起来像空账户），但调用就能正常执行
    
-   Gas 优化：EIP-2929 把预编译地址加入交易初始访问列表，省了首次访问的 gas 费
    

## 三、为什么不用操作码？

两者都能执行计算，但预编译是更优解：

1.  **操作码不够用**：EVM 只有 256 个 1 字节操作码（`0x00~0xFF`），资源极其稀缺
    
2.  **速度天差地别**：预编译在 EVM 外原生运行，避开 EVM 指令开销，复杂加密能快几十上百倍
    
3.  **安全妥协的结果**：曾想过让矿工投票给常用合约降 gas，但怕攻击者用陷阱门搞 DoS 攻击，最终选了协议指定固定预编译的安全方案
    

## 四、现有预编译速查表（主网已上线）

表格

| 地址 | 名称 | 一句话功能 | 上线版本 |
| --- | --- | --- | --- |
| 0x01 | EC 恢复 | 从签名恢复公钥（最常用） | 创世（边境） |
| 0x02 | SHA2-256 | 计算 SHA256 哈希 | 创世 |
| 0x03 | RIPEMD-160 | 计算 RIPEMD160 哈希 | 创世 |
| 0x04 | 身份 | 原封不动返回输入 | 创世 |
| 0x05 | MODEXP | 大数字模指数运算 | 拜占庭 |
| 0x06 | ECADD | 椭圆曲线点加法 | 拜占庭 |
| 0x07 | ECMUL | 椭圆曲线点乘法 | 拜占庭 |
| 0x08 | 配对 | 椭圆曲线配对检查（零知识证明用） | 拜占庭 |
| 0x09 | BLAKE2 | BLAKE2 哈希压缩 | 伊斯坦布尔 |
| 0x0a | KZG 证明验证 | 验证 Blob 数据的 KZG 证明 | 坎昆（EIP-4844） |

## 五、怎么工作 & 怎么用

### 工作原理

-   统一接口：和调用普通合约**完全相同**，用`CALL/STATICCALL`等操作码，开发者不用学新东西
    
-   明码标价：gas 费和输入数据大小 / 复杂度直接挂钩，固定输入对应固定 gas，防止 DoS 攻击
    
-   安全设计：禁止预编译内部嵌套调用，降低漏洞风险
    

### 最简调用示例（SHA256 哈希 "Hello"）

asm

```
// 1. 把输入存入内存
PUSH5 0x48656C6C6F  // "Hello"的编码
PUSH1 0
MSTORE

// 2. 调用0x02地址的SHA256预编译
PUSH1 0x20  // 返回32字节
PUSH1 0x20  // 结果存在内存0x20位置
PUSH1 5     // 输入长度5字节
PUSH1 0x1B  // 输入在内存0x1B位置
PUSH1 2     // 预编译地址
PUSH4 0xFFFFFFFF  // 给足gas
STATICCALL

// 3. 从内存读结果
PUSH1 0x20
MLOAD
```

结果：`0x185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969`

## 六、预编译的未来

### 新增预编译

-   流程：提交 EIP→社区讨论→硬分叉上线
    
-   阻力大：测试成本高、客户端维护重、gas 定价错了会出大问题
    
-   优化思路：先在 Layer2 跑通验证，证明好用再上主网
    
-   热门待上线：
    
    -   RIP-7212：secp256r1 曲线（苹果 / 安卓手机原生签名用）
        
    -   EIP-2537：BLS12-381 曲线（共识和零知识证明用）
        
    -   EIP-7545：Verkle 证明验证（未来以太坊升级用）
        

### 移除预编译

-   候选：过时、没人用的（身份预编译、RIPEMD-160、BLAKE2）
    
-   不会直接删：会改成普通智能合约实现，保证老代码能用，但 gas 费会变高
    

## 七、主流客户端实现位置

-   Geth：`core/vm/contracts.go`
    
-   Besu：`org.hyperledger.besu.evm.precompile` 包
    
-   Nethermind：`Nethermind.EVM.Precompiles` 命名空间
    
-   Reth：`revm-precompile` 库
    

## 八、前沿方向

**渐进式预编译**：先把预编译用普通合约部署在固定的 CREATE2 地址，主网和 L2 都能用；等客户端原生实现上线后自动切换，无缝过渡。

## 核心速记

预编译 = 以太坊官方外挂，地址 01 到 0a，调用和普通合约一样，专门搞定加密计算，快又省 gas，加难删也难。

# Besu 执行客户端学习笔记

# EVM（以太坊虚拟机）极简学习笔记

## 一、核心定位

**EVM 是以太坊 "世界计算机" 的心脏**，本质是**以太坊状态机的状态转换函数**：接收交易作为输入，执行计算，输出新的世界状态并永久上链。所有以太坊节点运行相同的 EVM 规范，保证全网状态一致。

## 二、先懂：以太坊是 "基于交易的状态机"

### 1\. 状态机通俗理解

-   类比自动售货机：有固定状态（闲置→选品→发货），输入（投币 / 选品）触发状态切换
    
-   核心三要素：**状态**（系统当前配置）+ **输入**（触发动作）+ **转移函数**（状态怎么变）
    

### 2\. 以太坊的状态机

-   **世界状态**：所有以太坊账户的集合（地址→账户状态的映射）
    
-   **输入**：用户发起的交易
    
-   **转移函数**：就是 EVM
    
-   实际运行：一批交易打包成一个区块，区块执行后生成新的世界状态，链式串联就是区块链
    

### 3\. 以太坊两种账户

表格

| 账户类型 | 控制方式 | 有无 EVM 代码 | 核心用途 |
| --- | --- | --- | --- |
| 外部账户（EOA） | 私钥控制 | 无 | 用户转账、发起交易 |
| 合约账户 | 代码控制 | 有（智能合约） | 执行自定义逻辑、存储数据 |

## 三、为什么用 "虚拟机" 范式？

解决**跨平台兼容**问题：

1.  高级语言（Solidity）→ 编译成**EVM 字节码**（平台中立）
    
2.  不同硬件 / 操作系统上的以太坊客户端，都能运行同一份字节码
    
3.  优势：一次编写，全网运行；隔离硬件复杂性
    

## 四、EVM 基础特性

-   字长：32 字节（256 位），是 EVM 处理数据的基本单位
    
-   执行模型：基于**栈**的虚拟机（和 JVM 类似）
    
-   拟图灵完备：理论上能执行任何计算，但受 Gas 限制，无法无限运行
    

## 五、EVM 执行核心：四大数据位置（必记）

### 1\. 堆栈（Stack）

-   特点：后进先出（LIFO），**临时存储**，合约执行完自动清空
    
-   限制：最大 1024 项，每项 32 字节；只能直接访问栈顶 16 项
    
-   用途：所有操作码的 "工作台"，存计算中间值、操作码参数
    
-   成本：极低
    

### 2\. 内存（Memory）

-   特点：字节数组，**临时存储**，合约执行完自动清空；初始全 0，动态扩展
    
-   计费：按 "页"（32 字节）扩展计费，越往后扩展越贵
    
-   用途：存大块临时数据（如函数参数、返回值、数组）
    
-   成本：中等
    

### 3\. 存储（Storage）

-   特点：**永久存储**，和合约账户绑定，交易间数据保留；是键值数据库（2^256 个槽位，每个槽 32 字节）
    
-   限制：只能被当前合约的代码访问
    
-   优化：Solidity 会自动把多个小变量打包进同一个 32 字节槽，节省 Gas
    
-   成本：**极高**（是内存的上千倍），清空存储槽可获得 Gas 退款
    

### 4\. 呼叫数据（Calldata）

-   特点：**只读**，是交易 / 合约调用时传入的输入数据
    
-   用途：传递函数参数、调用数据
    
-   成本：极低（零字节更便宜）
    

## 六、EVM 字节码与控制流

### 1\. 字节码是什么

-   是 EVM 能直接执行的机器码，由 **操作码（指令）和操作数（参数）** 组成
    
-   通常用十六进制表示，有对应的人类可读助记符（EVM 汇编）
    
-   常用操作码速记：
    
    -   `PUSH1-PUSH32`：把数据压入栈
        
    -   `ADD/MUL`：加减乘除运算
        
    -   `MLOAD/MSTORE`：读写内存
        
    -   `SLOAD/SSTORE`：读写存储
        
    -   `JUMP/JUMPDEST`：实现跳转（控制流）
        
    -   `RETURN`：停止执行并返回结果
        

### 2\. 程序计数器（PC）

-   跟踪下一条要执行的操作码在字节码中的偏移量
    
-   正常执行时 PC 自动 + 1，遇到`JUMP`指令会跳转到指定位置
    
-   `JUMPDEST`是合法跳转目标的标记，防止恶意跳转
    

## 七、Gas 机制：EVM 的安全卫士

### 1\. 为什么需要 Gas

-   防止无限循环（如`while(true)`）和 DOS 攻击
    
-   让计算资源有成本，抑制滥用
    

### 2\. 核心规则

-   每执行一个操作码都要消耗对应数量的 Gas（越复杂越贵）
    
-   用户发起交易时指定 Gas 上限和 Gas 价格
    
-   交易执行完，剩余 Gas 退还；如果 Gas 耗尽，交易回滚，但已消耗的 Gas 不退还
    
-   这就是 EVM 是 "拟图灵完备" 的原因：计算能力受 Gas 限制
    

## 八、EVM 升级

-   主要通过 **EIP（以太坊改进提案）** 添加新操作码或优化现有逻辑
    
-   原则：不破坏现有合约兼容性
    
-   重要升级方向：EOF（以太坊对象格式），是新一代字节码格式，能提升执行效率和安全性
    

## 九、快速学习资源

-   交互式操作码查询：
    
    [evm.codes](http://evm.codes)
    
    File
    
-   权威文档：以太坊黄皮书、《掌握以太坊》
    
-   实战工具：Ethervm、[evm.storage](http://evm.storage)
    
-   入门视频：Whiteboard Crypto《EVM 动画非技术介绍》

# 以太坊执行层客户端架构学习笔记

## 一、核心概述

执行层客户端是以太坊网络的 “执行大脑”，**不再负责共识**（合并后移至共识层），核心只做**状态转换**和配套服务。

### 核心职责

1.  存储区块链数据和本地状态副本
    
2.  通过 DevP2P 协议与其他节点通信，传播交易和区块
    
3.  维护交易池，筛选合法交易
    
4.  执行交易和智能合约，更新全局状态
    
5.  通过 Engine API 响应共识层的指令（验证区块、构建区块）
    

### 整体架构分层

plaintext

```
共识层(CL) ←Engine API→ 执行层(EL) ←DevP2P→ 其他节点
                      ↑
外部(钱包/DApp) ←JSON-RPC→
```

## 二、核心组件详解

### 1\. EVM（以太坊虚拟机）

-   **本质**：以太坊的 “通用虚拟 CPU”，**保证所有节点执行相同代码得到完全一致的结果**，是共识的基础
    
-   **三明治复杂度模型**：
    
    -   顶层：Solidity 等高级语言（简单易用）
        
    -   中间层：编译器（复杂，把高级语言转成 EVM 字节码）
        
    -   底层：EVM 字节码（简单，直接执行）
        

### 2\. 全局状态与状态机

-   以太坊是**全局状态机**：输入交易→执行→输出新状态
    
-   **状态**：所有账户余额、合约代码、合约存储、网络参数的总和，用 Merkle-Patricia Trie 存储
    
-   与比特币区别：比特币只存 UTXO（未花费交易输出），以太坊存完整全局状态
    

### 3\. 交易

-   唯一能触发状态转换的外部输入
    
-   流程：用户签名→JSON-RPC 发送→执行层验证→进入交易池→被打包进区块→EVM 执行→状态更新
    

### 4\. 网络与 API 接口

表格

| 接口类型 | 作用 | 使用者 |
| --- | --- | --- |
| DevP2P | 节点间点对点通信，传播交易和区块 | 其他执行层客户端 |
| JSON-RPC | 标准化外部接口，支持查询状态、发送交易 | 钱包、DApp、开发者工具 |
| Engine API | 共识层与执行层的专用内部通信接口，JWT 认证 | 仅共识层客户端 |

### 5\. Engine API（重中之重）

共识层通过这两个核心方法驱动执行层：

（1）`engine_newPayload`（验证区块）

-   作用：共识层收到新块后，把执行部分发给执行层验证
    
-   验证流程：检查父块哈希→验证块头合法性→执行所有交易→更新状态
    
-   返回状态：
    
    -   ✅ **有效**：完全验证通过
        
    -   ❌ **无效**：块或其祖先有问题
        
    -   ⏳ **同步中**：执行层还没追上链，缺少数据
        
    -   ⚠️ **已接受**：基础检查通过，完整验证未完成（浅状态客户端）
        

（2）`engine_forkchoiceUpdated`（同步状态 + 构建区块）

-   作用：共识层告诉执行层 “哪条链是最新的正确链”；如果需要出块，同时触发区块构建
    
-   返回状态：同上，出块时会额外返回`payloadId`，供共识层获取构建好的区块
    

前置步骤：能力交换

-   启动时`engine_exchangeCapabilities`：协商双方支持的 Engine API 版本（V1/V2/V3），保证兼容
    

## 三、关键工作流程

### 1\. 节点启动完整流程

1.  共识层与执行层交换 Engine API 能力
    
2.  共识层发送初始分叉选择更新，告诉执行层当前链头
    
3.  执行层开始同步数据（返回 “同步中” 状态）
    
4.  同步完成后返回 “有效” 状态，节点正常运行
    

### 2\. 状态转换函数（STF）

执行层的**核心逻辑**，所有区块验证最终都走这个流程：

go

运行

```
func 状态转换(父块, 当前块, 旧状态) (新状态, 错误) {
    1. 验证块头合法性（gas限制、时间戳、父哈希等）
    2. 按顺序执行区块内的每一笔交易
    3. 所有交易执行成功，返回新状态
    4. 任何一笔交易失败，返回错误，整个块无效
}
```

-   块头验证失败常见情况：gas 限制变动超过 1/1024、时间戳早于父块、基础费用计算错误
    

### 3\. 两种同步模式对比

表格

| 模式 | 原理 | 优点 | 缺点 | 现状 |
| --- | --- | --- | --- | --- |
| 全同步 | 从创世块开始，重放所有历史交易，逐步重建状态 | 绝对安全，验证所有数据 | 极慢（主网几天），耗大量 CPU / 磁盘 | 即将被 EIP-4444 淘汰 |
| 快拍同步 (Snap Sync) | 1. 选最近确定的块作为起点2. 直接下载该块的状态快照3. 验证快照完整性4. 同步该块之后的所有交易 | 快（主网几小时），资源消耗低 | 依赖弱主观性检查点 | 目前主流默认模式 |

## 四、内部核心模块

### 1\. 交易池

临时存储待打包的合法交易，按优先级排序供矿工选择

-   **遗留池**：按 gas 价格排序，用双堆管理，交易过多时淘汰价格最低的
    
-   **Blob 池**：专门处理 EIP-4844 的 Blob 交易，驱逐策略用对数函数优化
    

### 2\. 内部共识引擎（ethone）

-   兼容历史共识机制：PoW（Ethash）、PoA（Clique）
    
-   合并后 PoS 模式下：只处理块头验证、取款处理等剩余工作，核心共识由共识层完成
    

### 3\. 存储后端

执行层用**嵌入式键值数据库**存储区块链和状态数据，主流选择：

（1）LevelDB

-   传统选择，基于 LSM 树架构（先写内存，满了刷磁盘，后台合并）
    
-   缺点：已停止维护，写入放大问题严重，性能跟不上以太坊负载
    

（2）Pebble（重点）

-   **CockroachDB 开发**，Geth 现在的**默认存储后端**，完全兼容 LevelDB
    
-   针对以太坊 “高写入、低延迟” 工作负载优化：
    
    -   支持多个活跃内存表，减少写入卡顿
        
    -   更精细的压缩控制，降低写入放大
        
    -   更强的批处理和快照支持，更好地对齐块执行和并发读取
        
-   项目地址：[https://github.com/cockroachdb/pebble](https://github.com/cockroachdb/pebble)
    

（3）MDBX

-   另一个高性能选择，基于 B + 树，读性能更好，适合查询密集型场景
    

## 五、主流执行客户端

-   Geth（Go-Ethereum）：最广泛使用，生态最完善，默认 Pebble 存储
    
-   Besu：Java 编写，适合企业级场景
    
-   Erigon：高性能，优化了存储和同步速度
    
-   Nethermind：.NET 编写，功能丰富
    
-   Reth：Rust 编写，极致性能，新兴选择

# 以太坊燃气（Gas）核算 学习笔记

## 核心定位

Gas 是以太坊网络中衡量交易 / 合约执行消耗计算资源的单位，用于防止恶意攻击、分配网络资源，所有链上操作均需支付 Gas 费用。本笔记基于**上海升级规范**和以太坊执行层规范（EELS）整理。

## 一、本征气体（Intrinsic Gas）计算

### 1\. 定义

本征气体是交易开始执行所需的**最低 Gas 成本**，涵盖 EVM 初始化、数据传输等固定开销，从交易的`gasLimit`中扣除，是交易有效性的前提条件。

### 2\. 上海升级后公式

交易访问列表

各部分拆解：

表格

| 组成部分 | 计算逻辑 | 说明 |
| --- | --- | --- |
| 初始化代码成本 | 长度长度长度长度否则 | 仅当CALLDATA=初始化代码时计算，按 32 字节对齐 |
| 交易数据成本 | 输入数据零字节非零字节 | 零字节成本远低于非零字节，鼓励优化交易数据 |
| 合约创建成本 | 合约创建否则普通转账 | 合约创建需额外消耗 Gas |
| 基础交易成本 | 交易 | 所有交易的固定基础开销 |
| 访问列表成本 | 访问列表访问列表地址存储密钥访问列表存储 | EIP-2930 引入，优化状态访问的 Gas 成本 |

### 3\. 关键常量

-   GTXDATAZERO​=4，Gtxdatanonzero​=16
    
-   交易（普通转账基础 Gas）
    
-   Gtxcreate​=32000（合约创建额外 Gas）
    

## 二、有效燃气价格与优先费用

### 1\. 交易类型区分

以太坊支持 4 种交易类型，不同类型的 Gas 价格计算逻辑不同：

表格

| 交易类型 | 标识 | 有效燃气价格（p） | 优先费用（f） |
| --- | --- | --- | --- |
| 遗留交易 | 0/1 | p=TgasPrice​ | f=TgasPrice​−HbaseFeePerGas​ |
| EIP-1559 交易 | 2 | p=HbaseFeePerGas​+min(TmaxPriorityFeePerGas​,TmaxFeePerGas​−HbaseFeePerGas​) | f=min(TmaxPriorityFeePerGas​,TmaxFeePerGas​−HbaseFeePerGas​) |
| Blob 交易 | 3 | 同类型 2 + Blob Gas 费用 | 同类型 2 |

### 2\. 核心概念

-   **基础费用（**HbaseFeePerGas​**）**：协议自动计算，随区块拥堵程度动态调整，会被燃烧销毁。
    
-   **优先费用（小费）**：支付给矿工 / 验证者，激励打包交易，越高越容易被优先处理。
    
-   **有效燃气费用**：effectiveGasFee=effectiveGasPrice×TgasLimit​（交易预付款的核心组成）
    

## 三、Blob 燃气（EIP-4844）

针对大型数据（如 Rollup 交易数据）设计的低成本存储方案，Blob 数据不进入 EVM，仅存储哈希。

### 1\. 总 Blob Gas 计算

totalBlobGas=GgasPerBlob​×len(TblobVersionedHashes​)

其中 GgasPerBlob​=217（固定值）

### 2\. Blob Gas 价格动态调整

Blob Gas 价格随网络**超额 Blob Gas**累积量指数增长，公式简化为：

blobGasPrice≈1×e3338477excessBlobGas​

-   当区块 Blob Gas 使用量 ≤ 目标值（393216）时，价格 = 1
    
-   超过目标值后，价格指数上升，抑制过度使用
    

### 3\. Blob 燃气费用

blobGasFee=totalBlobGas×blobGasPrice

-   Blob 费用**单独计算**，需叠加到普通交易的 Gas 费用中
    
-   预付款：maxBlobFee=TmaxFeePerBlobGas​×totalBlobGas
    

## 四、交易执行全流程

以太坊交易执行由状态转移函数 Υ 驱动，分为 5 个核心阶段：

### 阶段 1：状态检查与预付款扣除

1.  **交易内在有效性校验**：检查发送者账户、nonce、余额、gasLimit 是否满足要求（共 10 项，如`intrinsicGas ≤ T_{gasLimit}`、`maxGasFee + T_{value} ≤ 发送者余额`）。
    
2.  **不可逆状态变更**：
    
    -   扣除预付款：v0​=effectiveGasFee+blobGasFee
        
    -   发送者 nonce+1
        

### 阶段 2：交易标准化与子状态初始化

1.  **交易转消息**：所有交易统一转换为`Message`结构，分为两种类型：
    
    -   → 合约创建消息
        
    -   地址 → 合约调用消息
        
2.  **子状态初始化**：初始化自毁集合、日志序列、访问地址列表等，为 EVM 执行做准备。
    

### 阶段 3：EVM 主执行循环

1.  **机器状态（**μ**）**：包含可用 Gas、程序计数器、内存、堆栈等核心参数。
    
2.  **单步执行逻辑**：
    
    -   读取当前操作码 → 计算 Gas 成本 → 执行操作（如栈运算、存储读写）→ 更新程序计数器
        
    -   跳转指令（JUMP/JUMPI）需验证目标地址是否为`JUMPDEST`，否则视为无效。
        
3.  **停机规则**：
    
    -   正常停机：返回输出数据，剩余 Gas 退还
        
    -   异常停机（如 OOG、REVERT）：状态回滚，已消耗 Gas 不退还
        

### 阶段 4-5：临时状态与最终状态确认

-   临时状态：处理 Gas 退款、日志提交等中间操作
    
-   最终状态：生成交易收据，更新全局状态树，完成状态转移
    

## 五、区块整体有效性

区块是交易的集合，其有效性需满足以下条件：

1.  **状态初始化**：从父区块状态根派生当前区块初始状态 σ0​。
    
2.  **顺序执行交易**：通过块转移函数 Π 按顺序执行所有交易，前一笔交易的状态是后一笔的输入。
    
3.  **气体积累验证**：区块内所有交易累计 Gas 使用量 ≤ 区块 Gas 上限。
    
4.  **承诺验证**：交易收据根、最终状态根必须与区块头声明的哈希一致，否则区块无效。

# 以太坊执行层（EL）规范学习笔记

## 一、规范概述

-   **最新规范**：EELS（Ethereum Execution Layer Specification）Python 实现
    
-   **过时参考**：黄皮书巴黎版本（2024-03-04），未包含合并后（PoS）更新
    
-   **核心依据**：EIP 提案 + 客户端实现共识
    

## 二、核心：状态转移函数（STF）

执行层唯一核心职责：执行块级状态转移，回答两个问题

1.  该区块能否追加到区块链末端？
    
2.  追加后区块链状态如何变化？
    

### 数学定义

σt+1​≡Π(σt​,B)

-   σt+1​：执行当前块后的新状态
    
-   Π：块级状态转移函数（执行所有交易）
    
-   σt​：执行前的旧状态
    
-   B：待执行的当前块
    

> 注意：σ 是**动态推导**的概念状态，而非 Python 规范中的 `State` 类实例

### 执行步骤（9 步）

1.  检索父块头
    
2.  验证过剩 Blob Gas 匹配
    
3.  验证当前块头合法性
    
4.  检查 `ommers` 字段为空（PoS 后无叔块）
    
5.  执行块内所有交易，输出：Gas 用量、Trie 根、日志布隆、新状态
    
6.  验证执行结果与块头参数一致（重点：`state_root`）
    
7.  追加块到区块链
    
8.  修剪旧块：仅保留最近 255 个块
    
9.  验证失败则抛出 `Invalid Block` 错误
    

## 三、块头验证核心规则（PoS 后）

表格

| 验证维度 | 关键条件 |
| --- | --- |
| Gas 限制 | 1. HgasUsed​≤HgasLimit​2. 与父块 gasLimit 偏差 ≤ 1/10243. 最小 gasLimit = 5000 |
| 时间与序号 | 1. 时间戳 > 父块时间戳2. 块号 = 父块号 + 1 |
| 哈希与签名 | 1. 父哈希 = KEC (RLP (父块头))2. ommers 哈希 = 空哈希 |
| PoS 特征 | 1. 难度 = 02. nonce = 0x0000000000000000 |
| 经济模型 | 1. 基础费符合 EIP-1559 计算2. withdrawal 哈希非空 |
| Blob 交易 | 1. blobGasUsed ≤ 786432（6 个 blob）2. blobGasUsed 是 131072（217）的整数倍3. excessBlobGas 计算正确 |

## 四、经济模型

### 1\. EIP-1559 基础费机制

-   **核心目标**：降低 gas 价格波动，提高费用可预测性
    
-   **关键参数**：
    
    -   弹性乘数 ρ=2：gas 目标 = gasLimit / 2
        
    -   最大调整分母 ξ=8：单块基础费最大涨幅 12.5%，最大跌幅 10%
        
-   **调整规则**：
    
    -   父块 gasUsed = gas 目标 → 基础费不变
        
    -   父块 gasUsed > 目标 → 基础费上涨
        
    -   父块 gasUsed < 目标 → 基础费下跌
        
-   **费用分配**：基础费**全部燃烧**，优先费（小费）归区块提议者
    

### 2\. EIP-4844 Blob Gas 机制

-   **Blob 参数**：
    
    -   目标 blobGas / 块 = 393216（3 个 blob）
        
    -   最大 blobGas / 块 = 786432（6 个 blob）
        
-   **价格规则**：
    
    -   过剩 blobGas 累积未超过阈值时，价格 = 1
        
    -   持续超目标使用 → 过剩 blobGas 累积 → 价格**指数级上涨**
        
    -   单块 blobGas 低于目标 → 过剩 blobGas 清零，价格重置
        

## 五、块执行流程

1.  **初始化**：`blobGasUsed=0`，`gasAvailable=块gasLimit`，初始化收据 / 提款 Trie 和日志
    
2.  **系统交易**：执行信标块根存储交易（EIP-4788），将父信标块根写入系统合约
    
3.  **交易执行**：按顺序解码并执行所有交易
    
4.  **提款处理**：执行信标链验证的提款（EIP-4895），转账 ETH 到指定地址
    
5.  **状态清理**：删除所有空账户
    

## 六、交易执行与 Gas 会计

### 1\. 交易类型

表格

| 类型 | 名称 | 核心特征 |
| --- | --- | --- |
| 0 | Legacy | 固定 gasPrice |
| 1 | EIP-2930 | 支持访问列表，降低状态访问 gas |
| 2 | EIP-1559 | 动态基础费 + 优先费 |
| 3 | EIP-4844 | 支持 blob 数据，额外支付 blob gas 费 |

### 2\. 内在 Gas 计算（最低执行成本）

g0​=Gtransaction​+Gtxcreate​+Gcalldata​+Gaccesslist​

-   Gtransaction​=21000（所有交易基础费）
    
-   Gtxcreate​=32000（合约创建额外费）
    
-   Gcalldata​：0 字节 4 gas，非 0 字节 16 gas
    
-   Gaccesslist​：每个地址 2400 gas，每个存储键 1900 gas
    

### 3\. 有效 Gas 价格

-   类型 0/1：p=TgasPrice​
    
-   类型 2/3：p=HbaseFeePerGas​+min(TmaxPriorityFeePerGas​,TmaxFeePerGas​−HbaseFeePerGas​)
    

### 4\. 交易有效性检查（10 项）

1.  发送方账户存在且非合约
    
2.  nonce 与发送方当前 nonce 匹配
    
3.  内在 gas ≤ 交易 gasLimit
    
4.  发送方余额 ≥ maxGasFee + 转账 value
    
5.  maxFeePerGas ≥ 块基础费
    
6.  类型 2/3：maxFeePerGas ≥ maxPriorityFeePerGas
    
7.  交易 gasLimit ≤ 块剩余 gas
    
8.  输入数据大小 ≤ 49152 字节
    
9.  nonce < 264−1
    
10.  blob 交易参数合法
     

## 七、EVM 执行核心

### 1\. 机器状态

μ≡(μgasAvailable​,μprogramCounter​,μmemoryContents​,μactiveWordsInMemory​,μstackContents​)

-   栈最大深度：1024 项
    
-   内存按 32 字节（1 字）为单位扩展
    

### 2\. 执行周期

1.  根据 PC 获取当前指令
    
2.  扣除指令 gas 成本（含内存扩展费）
    
3.  执行指令操作（栈操作、内存读写、算术运算等）
    
4.  更新 PC：普通指令 +1，PUSH 指令 +(1 + 数据长度)，JUMP/JUMPI 跳转到目标地址
    
5.  重复直到停机
    

### 3\. 停机条件

-   **正常停机**：STOP、RETURN、REVERT、SELFDESTRUCT
    
-   **异常停机**：Gas 不足、栈溢出 / 下溢、无效指令、非法跳转等
    
-   异常停机时：所有状态回滚，已消耗 gas 不退还
    

## 八、块整体有效性

块是原子性的状态转移单元，必须同时满足：

1.  所有交易顺序执行后的**状态根**与块头 `state_root` 一致
    
2.  所有交易收据的**收据根**与块头 `receipts_root` 一致
    
3.  累计 gas 使用量 ≤ 块 gasLimit
    
4.  所有验证步骤全部通过
    

> 任何一项不满足则整个块无效，不接受部分交易执行结果

# 以太坊协议架构学习笔记

## 一、协议总览

以太坊合并后采用**双层分离架构**，由执行层和共识层组成，两层通过标准 API 通信，各自运行独立客户端和 P2P 网络，共同维护区块链的安全与运行。

## 二、核心两层架构

### 1\. 执行层（EL）

-   **核心职责**：处理用户交易、运行 EVM 智能合约、管理全局账户状态、维护交易内存池
    
-   **对外接口**：JSON-RPC API，供钱包、DApp、交易所等用户侧应用调用
    
-   **内部通信**：通过**Engine API**与共识层交互，接收共识层的出块指令，返回执行结果
    
-   **P2P 网络**：传播交易和已执行的区块数据
    
-   **常见客户端**：Geth、Nethermind、Besu
    

### 2\. 共识层（CL）

-   **核心职责**：运行权益证明（PoS）共识，选举出块者、收集验证者投票、决定规范链、实现区块终结性
    
-   **对外接口**：Beacon API，供验证者客户端调用，提交投票和出块
    
-   **内部通信**：通过 Engine API 向执行层发送出块请求，验证执行结果
    
-   **P2P 网络**：传播验证者投票、出块信息、Blob 数据
    
-   **常见客户端**：Lighthouse、Prysm、Teku
    

## 三、核心设计理念与原则

### 核心理念

1.  **简洁性**：协议规范尽可能简单，让普通开发者能理解和实现，避免少数团队垄断
    
2.  **普遍性**：只提供图灵完备的 EVM 底层，不内置任何特殊功能，任何逻辑都可通过智能合约实现
    
3.  **模块化**：组件独立封装，修改单个组件不影响整体，方便升级和跨项目复用
    
4.  **非歧视性**：不限制任何合法用途，按计算量公平收取交易费，遵循网络中立原则
    
5.  **敏捷性**：通过 EIP 开放提案流程，持续迭代优化协议
    

### 补充原则

-   优先管理和封装复杂性，将不可避免的复杂度推到中间层（编译器、序列化工具等）
    
-   功能设计追求底层化、可组合性，拒绝内置高层用例
    
-   保障用户使用自由，不偏袒或反对任何类型的合约和交易
    

## 四、区块链层关键技术

### 1\. 账户模型（对比比特币 UTXO）

-   以太坊采用**账户余额模型**：直接记录每个地址的余额和交易序号（nonce）
    
-   **优点**：节省存储空间、交易体积更小、代币完全可互换、实现简单、支持 DEX 等复杂逻辑
    
-   **缺点**：需要 nonce 防止重放攻击，废弃账户无法从全局状态中删除
    

### 2\. 状态数据结构

表格

| 技术 | 现状 | 核心特点 | 演进方向 |
| --- | --- | --- | --- |
| Merkle Patricia Trie（MPT） | 当前在用 | 生成唯一状态根，支持密码学证明，增删查效率 O (logn) | 逐步被 Verkle 树替代 |
| Verkle 树 | 研发中 | 证明体积极小，是实现无状态客户端的关键，大幅降低节点存储压力 | 未来将全面替换 MPT |

### 3\. 序列化格式

-   **RLP（老格式）**：简单确定性，保证序列化后字节完全一致，但效率低，不支持默克尔化
    
-   **SSZ（新格式，共识层专用）**：效率更高，原生支持默克尔化，支持快速索引，是轻客户端证明的基础
    

## 五、PoS 共识机制（Gasper）

Gasper = **RANDAO（随机数）** + **LMD-GHOST（分叉选择）** + **Casper FFG（终结性）**

1.  **RANDAO**
    
    -   生成不可预测的随机数
        
    -   作用：随机选择出块者和验证者委员会，防止提前预谋攻击
        
2.  **LMD-GHOST（分叉选择规则）**
    
    -   核心逻辑：不选 "最长链"，选**验证者投票权重最高**的子树
        
    -   算法流程：从创世块开始，逐层向下选择子节点中权重最高的块，平局按区块哈希排序
        
    -   作用：实时决定当前链的头部，处理临时分叉
        
3.  **Casper FFG（终结性机制）**
    
    -   核心定义：被终结的区块绝对不可篡改，篡改需烧毁至少 33% 的质押 ETH
        
    -   流程：验证者对检查点（每 32 个区块一个）进行两轮投票，超过 2/3 同意后，检查点被永久终结
        
    -   作用：提供链的最终性，保证交易不可回滚
        

## 六、P2P 网络架构（混合设计）

结合结构化和非结构化网络的优点，兼顾效率和鲁棒性

### 1\. 节点发现：Discv5 DHT（结构化网络）

-   基于 Kademlia 算法，查找节点的通信开销为 O (logn)
    
-   作用：新节点通过引导节点快速发现其他节点，存储节点的连接信息（ENR 记录）
    
-   注意：**DHT 只用来找节点，不传播区块和交易**
    

### 2\. 数据传播：GossipSub（非结构化八卦网络）

-   核心逻辑：节点将收到的区块 / 交易转发给相邻节点，像八卦一样扩散到全网
    
-   优点：高流量下稳定性强，适合区块链高频数据传播
    

### 3\. 三种 P2P 网络对比

表格

| 网络类型 | 通信开销 | 节点状态开销 | 典型应用 | 核心特点 |
| --- | --- | --- | --- | --- |
| 中心化服务器 | O(1) | O(n) | Napster | 最快但存在单点故障 |
| DHT（结构化） | O(logn) | O(logn) | BitTorrent、IPFS | 去中心化，查找高效 |
| 纯洪泛（非结构化） | O(n) | O(1) | 早期 Gnutella | 简单但效率极低 |

## 七、易错点辨析

1.  ✅ 执行层管 "交易执行"，共识层管 "链的合法性"，两者缺一不可
    
2.  ✅ LMD-GHOST 负责 "选链"，Casper FFG 负责 "敲定链"，分工不同
    
3.  ✅ 以太坊用账户模型，比特币用 UTXO 模型，不要混淆
    
4.  ✅ 区块和交易通过 GossipSub 传播，DHT 仅用于节点发现
    
5.  ✅ 区块终结性是 PoS 的核心优势，PoW 没有真正的终结性
    

## 八、核心学习要点

1.  重点掌握两层架构的分工和通信方式
    
2.  理解 Gasper 共识的三个组成部分及各自作用
    
3.  理清 P2P 混合架构的设计思路和各组件职责
    
4.  关注以太坊技术演进方向：MPT→Verkle 树、RLP→SSZ、全节点→无状态客户端

## 二、共识层（CL）相关知识 & 挑战解答

### 🔍 知识卡整理

1.  **CL 核心职责**：共识层（Consensus Layer）负责**区块提议、证明与最终性**，是以太坊 PoS 共识的核心，保障网络的安全与稳定。
    
2.  **Fork choice 作用**：分叉选择规则（Fork choice）用于在多条链中**选择当前最优链头**，确保全网共识一致。
    
3.  **Finality 机制**：最终性（Finality）通过检查点机制，为链上交易提供**经济上强不可逆保障**，一旦交易被最终确认，几乎不可能被回滚。
    
4.  **Slashing 机制**：罚没（Slashing）用于惩罚恶意验证者的行为（如双签、长程攻击），保障共识安全。
    

### 🎯 Boss 挑战：解释 “链头正确” 和 “经济上最终不可逆” 的层次差异

这两个概念是以太坊 PoS 共识中**两个完全独立、递进的安全层次**，核心差异如下：

1.  **链头正确（Fork choice 层面）**
    
    -   本质：是**实时的、概率性的共识选择**，由 Fork choice 规则选出当前最长 / 最优链头。
        
    -   特点：链头是动态可变的，在极端情况下（如网络分区、51% 攻击），链头可能发生重组（reorg），交易存在被回滚的可能。
        
    -   作用：保障网络的实时可用性，让全网快速达成临时共识。
        
2.  **经济上最终不可逆（Finality 层面）**
    
    -   本质：是**确定性的、经济惩罚保障的最终状态**，由检查点机制和验证者质押资产背书。
        
    -   特点：一旦交易被最终确认，想要回滚交易需要验证者发动攻击，面临**全部质押资产被 Slashing 罚没**的惩罚，攻击成本极高，在经济上几乎不可行。
        
    -   作用：为交易提供绝对的最终性保障，让用户无需担心交易被回滚。
        

**总结**：链头正确是 “暂时的、可变更的共识”，最终不可逆是 “永久的、经济保障的最终状态”，二者是从 “临时一致” 到 “绝对安全” 的两个递进层次。

## 三、执行层（EL）& 共识层（CL）协同知识 & 挑战解答

### 🔍 知识卡整理

1.  **Engine API 核心定位**：Engine API 是以太坊 Merge 之后，**执行层（EL）与共识层（CL）之间的核心接口**，是两层协同的桥梁。
    
2.  **接口交互逻辑**：CL 负责提供 payload（区块数据）请求，EL 负责执行交易、返回执行结果，二者通过 Engine API 完成数据交互。
    
3.  **接口异常影响**：Engine API 接口异常会直接影响区块提议、网络同步，严重时会导致节点离线、网络共识异常。
    
4.  **版本兼容性关键**：EL/CL 客户端升级时，**版本兼容性**是核心，必须保证接口前后兼容，否则会导致节点无法正常运行。
    

### 🎯 Boss 挑战：“EL 与 CL 解耦但协同” 的工程含义

**一句话总结**：执行层（EL）和共识层（CL）在代码、架构、升级上完全解耦，可独立迭代开发，但通过标准化的 Engine API 接口严格协同，共同保障以太坊网络的正常运行。

## 模块一：区块链执行层核心基础

1.  执行层（EL）核心定位：区块链的计算与状态变更核心引擎，核心职责是交易执行与区块链全局状态更新。
    
2.  Nonce 核心作用：一是防止交易重放攻击，避免同一笔交易被恶意重复执行；二是严格约束交易的发送与执行顺序，保证同账户交易按 Nonce 递增顺序执行。
    
3.  Gas 核心机制：为 EVM 的计算、存储资源做量化计价，同时通过 Gas 上限限制，从机制上避免合约代码无限执行耗尽节点资源，是网络安全与资源公平分配的核心设计。
    
4.  交易执行产物：交易执行完成后，会生成对应的事件日志、交易回执（收据），并完成区块链全局状态的变更。
    

## 模块二：EVM 以太坊虚拟机核心规则

1.  EVM 本质：具备确定性的**基于栈的虚拟机（栈机）**，所有运算操作遵循栈 “后进先出” 的执行原则。
    
2.  存储区域核心特性
    
    -   Memory（内存）：临时存储区域，仅在当前合约调用的生命周期内有效，调用结束后数据立即清空。
        
    -   Storage（存储）：持久化状态区域，数据会永久记录在区块链上，是账户与合约状态的核心存储载体。
        
3.  Opcode 操作码规则：不同操作码对应不同的节点资源消耗成本，其 Gas 定价直接决定单笔交易的总 Gas 消耗。
    
4.  Revert 操作规则：执行后会回滚本次交易对区块链状态做出的所有变更，同时向调用方返回错误信息；已消耗的 Gas 不予退还，仅退还未使用的 Gas。
    

## 模块三：交易全生命周期核心流程

1.  交易发起前提：交易必须先经过账户私钥签名，完成签名后才会被广播到区块链网络节点。
    
2.  交易暂存规则：节点接收到广播的合法交易后，会将交易放入交易池（txpool/mempool）中，等待打包。
    
3.  交易打包流程：交易被区块提议者（矿工 / 验证者）从交易池中选中后，会被打包进新生成的区块。
    
4.  交易最终性规则：交易的最终性由区块链共识层完成确认，仅被打包进区块不等于交易完成最终确认，需经过共识多轮验证后，交易才具备不可逆的最终性。
    

## 挑战 1：用 “输入→执行→产出” 三步解释状态转移函数

状态转移函数是区块链的核心底层函数，区块链本质是一个分布式状态机，该函数的作用就是驱动区块链的全局状态从旧状态向新状态完成可信流转，三步拆解如下：

1.  **输入阶段**：状态转移函数的核心输入包含两部分，一是**区块链的前序全局旧状态**，即上一个区块执行完成后，全网已达成共识的最终账户、合约数据状态；二是**合法的已签名交易集合**，即本次需要执行的、经过校验的有效交易，是触发状态变更的核心源。
    
2.  **执行阶段**：以 EVM 为执行载体，由执行层（EL）按规则依次执行输入的交易集合。对每一笔交易，EVM 会解析交易指令，调用对应的 Opcode 操作码，在虚拟机环境中完成交易逻辑（如代币转账、合约方法调用等）；同时通过 Gas 机制约束资源消耗，处理执行异常（如 Revert 状态回滚），最终计算出每笔交易执行后的状态变更结果。
    
3.  **产出阶段**：执行完成后，状态转移函数输出两大核心产物，一是**全新的全局新状态**，即经过所有交易执行、异常回滚处理后，全网需要达成共识的最新账户与合约数据，该状态会作为下一次状态转移的输入旧状态；二是**交易执行配套凭证**，包含交易回执、执行事件日志、Gas 消耗明细等，用于验证交易执行结果与状态变更的合法性。
    

## 挑战 2：解释为什么 “确定性” 是全网共识前提

这里的 “确定性”，特指 EVM 的执行确定性 —— 对于同一个初始状态、同一笔交易 / 交易集合，在任何节点的 EVM 中执行，都必须得到完全一致的最终状态、执行结果与 Gas 消耗。它是全网共识的核心前提，原因如下：

1.  **确定性是分布式节点达成状态一致的基础**。区块链是去中心化的分布式系统，无中心化权威节点，全网成千上万的节点都需要独立验证区块与交易。只有执行过程具备确定性，所有节点对同一笔交易、同一个区块执行后，才能得到完全相同的最终状态。若执行结果不确定，不同节点执行同一区块得到不同状态，节点间就无法达成一致共识，区块链分布式账本会出现分叉、数据不一致，失去账本的唯一性与可信度。
    
2.  **确定性是区块合法性验证的核心前提**。区块链共识流程中，节点收到新的区块后，必须独立重新执行区块内的所有交易，验证执行后的状态根、交易根等数据是否与区块头一致，以此确认区块合法性。若执行过程不具备确定性，节点无法复现区块提议者的执行结果，就无法完成区块合法性校验，共识机制将无法正常运转。
    
3.  **确定性保障了区块链的不可篡改性与可追溯性**。区块链的核心价值是不可篡改的分布式账本，每一个区块的状态都依赖前一个区块的最终状态，整条链的状态流转环环相扣。只有每一次状态转移的执行过程都是确定的，才能保证整条链的状态流转可复现、可追溯、不可篡改。若执行结果不确定，账本的历史状态就无法被统一验证，区块链的不可篡改特性将无从谈起。
    
4.  **确定性是区块链去信任设计的核心支撑**。去中心化网络中节点互不信任，只有执行逻辑具备确定性，节点才能无需信任其他节点，仅通过自身独立执行就能验证数据的正确性。若执行结果不确定，节点必须依赖第三方的执行结果，违背了区块链去中心化、去信任的核心设计原则。
    

## 挑战 3：按顺序写出交易生命周期至少 5 个节点

以下为交易从发起至完成最终确认的完整生命周期，按顺序共 6 个核心节点，覆盖全流程关键环节：

1.  **交易构造与签名**：用户在钱包中构造交易（包含接收地址、转账金额、Gas 参数、Nonce、合约交互数据等核心字段），使用账户私钥对交易进行加密签名，生成合法可广播的交易原始数据，这是交易生命周期的起点，只有完成签名的交易才具备网络合法性。
    
2.  **交易广播与节点接收校验**：签名完成的交易通过钱包节点广播到区块链 P2P 网络，网络中的全节点接收到交易后，先完成基础合法性校验（签名有效性、Nonce 合规性、账户余额是否足够支付 Gas 等），过滤无效交易。
    
3.  **交易进入 Mempool（交易池）**：通过基础校验的合法交易，会被节点放入本地交易池（Mempool/txpool）暂存，等待区块提议者筛选；同时节点会将该交易继续广播给网络中的其他节点，让交易同步到全网交易池。
    
4.  **交易被选中与区块打包**：区块提议者（矿工 / 验证者）按照 Gas 费高低、交易优先级等规则，从交易池中筛选交易，将选中的交易打包进新构造的区块中，完成区块构建后向全网广播该新区块。
    
5.  **区块共识验证与交易上链**：全网节点接收到新区块后，对区块完成合法性校验，同时独立重新执行区块内的所有交易，验证执行结果与区块头数据是否一致；区块通过共识机制验证后，会被添加到区块链主链上，交易正式上链，从 “待打包” 变为 “已打包” 状态。
    
6.  **交易最终确认（最终性达成）**：区块被添加到主链后，随着后续新区块不断链接在该区块之后，经过共识层多轮确认，交易所在的区块被不可逆地固定在主链上，交易完成最终确认，具备不可回滚的最终性，交易生命周期正式完成。

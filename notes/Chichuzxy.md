# 以太坊协议架构学习笔记

## 一、协议总览

以太坊合并后采用**双层分离架构**，由执行层和共识层组成，两层通过标准 API 通信，各自运行独立客户端和 P2P 网络，共同维护区块链的安全与运行。

## 二、核心两层架构

### 1\. 执行层（EL）

-   **核心职责**：处理用户交易、运行 EVM 智能合约、管理全局账户状态、维护交易内存池
    
-   **对外接口**：JSON-RPC API，供钱包、DApp、交易所等用户侧应用调用
    
-   **内部通信**：通过**Engine API**与共识层交互，接收共识层的出块指令，返回执行结果
    
-   **P2P 网络**：传播交易和已执行的区块数据
    
-   **常见客户端**：Geth、Nethermind、Besu
    

### 2\. 共识层（CL）

-   **核心职责**：运行权益证明（PoS）共识，选举出块者、收集验证者投票、决定规范链、实现区块终结性
    
-   **对外接口**：Beacon API，供验证者客户端调用，提交投票和出块
    
-   **内部通信**：通过 Engine API 向执行层发送出块请求，验证执行结果
    
-   **P2P 网络**：传播验证者投票、出块信息、Blob 数据
    
-   **常见客户端**：Lighthouse、Prysm、Teku
    

## 三、核心设计理念与原则

### 核心理念

1.  **简洁性**：协议规范尽可能简单，让普通开发者能理解和实现，避免少数团队垄断
    
2.  **普遍性**：只提供图灵完备的 EVM 底层，不内置任何特殊功能，任何逻辑都可通过智能合约实现
    
3.  **模块化**：组件独立封装，修改单个组件不影响整体，方便升级和跨项目复用
    
4.  **非歧视性**：不限制任何合法用途，按计算量公平收取交易费，遵循网络中立原则
    
5.  **敏捷性**：通过 EIP 开放提案流程，持续迭代优化协议
    

### 补充原则

-   优先管理和封装复杂性，将不可避免的复杂度推到中间层（编译器、序列化工具等）
    
-   功能设计追求底层化、可组合性，拒绝内置高层用例
    
-   保障用户使用自由，不偏袒或反对任何类型的合约和交易
    

## 四、区块链层关键技术

### 1\. 账户模型（对比比特币 UTXO）

-   以太坊采用**账户余额模型**：直接记录每个地址的余额和交易序号（nonce）
    
-   **优点**：节省存储空间、交易体积更小、代币完全可互换、实现简单、支持 DEX 等复杂逻辑
    
-   **缺点**：需要 nonce 防止重放攻击，废弃账户无法从全局状态中删除
    

### 2\. 状态数据结构

表格

| 技术 | 现状 | 核心特点 | 演进方向 |
| --- | --- | --- | --- |
| Merkle Patricia Trie（MPT） | 当前在用 | 生成唯一状态根，支持密码学证明，增删查效率 O (logn) | 逐步被 Verkle 树替代 |
| Verkle 树 | 研发中 | 证明体积极小，是实现无状态客户端的关键，大幅降低节点存储压力 | 未来将全面替换 MPT |

### 3\. 序列化格式

-   **RLP（老格式）**：简单确定性，保证序列化后字节完全一致，但效率低，不支持默克尔化
    
-   **SSZ（新格式，共识层专用）**：效率更高，原生支持默克尔化，支持快速索引，是轻客户端证明的基础
    

## 五、PoS 共识机制（Gasper）

Gasper = **RANDAO（随机数）** + **LMD-GHOST（分叉选择）** + **Casper FFG（终结性）**

1.  **RANDAO**
    
    -   生成不可预测的随机数
        
    -   作用：随机选择出块者和验证者委员会，防止提前预谋攻击
        
2.  **LMD-GHOST（分叉选择规则）**
    
    -   核心逻辑：不选 "最长链"，选**验证者投票权重最高**的子树
        
    -   算法流程：从创世块开始，逐层向下选择子节点中权重最高的块，平局按区块哈希排序
        
    -   作用：实时决定当前链的头部，处理临时分叉
        
3.  **Casper FFG（终结性机制）**
    
    -   核心定义：被终结的区块绝对不可篡改，篡改需烧毁至少 33% 的质押 ETH
        
    -   流程：验证者对检查点（每 32 个区块一个）进行两轮投票，超过 2/3 同意后，检查点被永久终结
        
    -   作用：提供链的最终性，保证交易不可回滚
        

## 六、P2P 网络架构（混合设计）

结合结构化和非结构化网络的优点，兼顾效率和鲁棒性

### 1\. 节点发现：Discv5 DHT（结构化网络）

-   基于 Kademlia 算法，查找节点的通信开销为 O (logn)
    
-   作用：新节点通过引导节点快速发现其他节点，存储节点的连接信息（ENR 记录）
    
-   注意：**DHT 只用来找节点，不传播区块和交易**
    

### 2\. 数据传播：GossipSub（非结构化八卦网络）

-   核心逻辑：节点将收到的区块 / 交易转发给相邻节点，像八卦一样扩散到全网
    
-   优点：高流量下稳定性强，适合区块链高频数据传播
    

### 3\. 三种 P2P 网络对比

表格

| 网络类型 | 通信开销 | 节点状态开销 | 典型应用 | 核心特点 |
| --- | --- | --- | --- | --- |
| 中心化服务器 | O(1) | O(n) | Napster | 最快但存在单点故障 |
| DHT（结构化） | O(logn) | O(logn) | BitTorrent、IPFS | 去中心化，查找高效 |
| 纯洪泛（非结构化） | O(n) | O(1) | 早期 Gnutella | 简单但效率极低 |

## 七、易错点辨析

1.  ✅ 执行层管 "交易执行"，共识层管 "链的合法性"，两者缺一不可
    
2.  ✅ LMD-GHOST 负责 "选链"，Casper FFG 负责 "敲定链"，分工不同
    
3.  ✅ 以太坊用账户模型，比特币用 UTXO 模型，不要混淆
    
4.  ✅ 区块和交易通过 GossipSub 传播，DHT 仅用于节点发现
    
5.  ✅ 区块终结性是 PoS 的核心优势，PoW 没有真正的终结性
    

## 八、核心学习要点

1.  重点掌握两层架构的分工和通信方式
    
2.  理解 Gasper 共识的三个组成部分及各自作用
    
3.  理清 P2P 混合架构的设计思路和各组件职责
    
4.  关注以太坊技术演进方向：MPT→Verkle 树、RLP→SSZ、全节点→无状态客户端

3.  **密码朋克捍卫隐私**：1990 年代美国政府将加密技术列为军品管控，引发 "密码战争"。菲尔・齐默尔曼发布 PGP 加密工具并以出版源代码的方式对抗出口禁令；1992 年埃里克・休斯、蒂莫西・梅等人创立密码朋克邮件列表，提出 "密码朋克写代码"，主张用技术而非法律保障隐私与自由；同期大卫・乔姆、戴伟、尼克・萨博等先后尝试 DigiCash、B-money、BitGold 等早期数字货币，为去中心化货币积累了关键设计经验。
    
4.  **比特币破局与以太坊诞生**：2008 年金融危机后，中本聪发布比特币白皮书，首创基于工作量证明的去中心化区块链与数字货币，解决了双花问题，但功能单一难以支撑复杂应用。维塔利克・布特林发现比特币局限后，联合加文・伍德等人于 2014 年完成以太坊设计，2015 年 7 月 30 日以太坊主网上线，成为首个支持通用智能合约的 "世界计算机"，开启了去中心化应用的新时代。

## 一、EIP 相关知识 & 挑战解答

### 🔍 知识卡整理

1.  **EIP 核心要素**：一份合格的 EIP 必须包含**动机**（为什么要做这个改进）、**规范**（技术实现的详细标准）、**向后兼容性**（不破坏现有网络和用户资产）等核心内容。
    
2.  **流程特点**：EIP 流程强调**公开讨论与迭代**，所有提案都要在社区、开发者、验证者等多方参与下充分讨论、反复修改。
    
3.  **提案落地规则**：**并非所有想法都会进入主网**，只有经过严格审核、测试、社区共识的提案才会最终部署。
    
4.  **可信度保障**：**测试与参考实现**是提升提案可信度的关键，只有代码跑通、测试网验证通过，提案才具备落地基础。
    

### 🎯 Boss 挑战：EIP 草案标题与问题定义

提案标题（示例）

**EIP-XXXX: 以太坊 L2 跨链消息轻量级验证优化**

问题定义

当前以太坊 Layer2 与主网的跨链消息验证依赖完整的区块证明，存在 Gas 成本高、验证延迟长的问题，导致跨链交互体验差、用户成本高。本提案旨在设计一种基于 ZK-SNARK 的轻量级验证方案，在不降低安全性的前提下，将跨链消息验证的 Gas 成本降低 80% 以上，同时缩短验证确认时间，提升 L2 生态的用户体验与可扩展性。

## 二、共识层（CL）相关知识 & 挑战解答

### 🔍 知识卡整理

1.  **CL 核心职责**：共识层（Consensus Layer）负责**区块提议、证明与最终性**，是以太坊 PoS 共识的核心，保障网络的安全与稳定。
    
2.  **Fork choice 作用**：分叉选择规则（Fork choice）用于在多条链中**选择当前最优链头**，确保全网共识一致。
    
3.  **Finality 机制**：最终性（Finality）通过检查点机制，为链上交易提供**经济上强不可逆保障**，一旦交易被最终确认，几乎不可能被回滚。
    
4.  **Slashing 机制**：罚没（Slashing）用于惩罚恶意验证者的行为（如双签、长程攻击），保障共识安全。
    

### 🎯 Boss 挑战：解释 “链头正确” 和 “经济上最终不可逆” 的层次差异

这两个概念是以太坊 PoS 共识中**两个完全独立、递进的安全层次**，核心差异如下：

1.  **链头正确（Fork choice 层面）**
    
    -   本质：是**实时的、概率性的共识选择**，由 Fork choice 规则选出当前最长 / 最优链头。
        
    -   特点：链头是动态可变的，在极端情况下（如网络分区、51% 攻击），链头可能发生重组（reorg），交易存在被回滚的可能。
        
    -   作用：保障网络的实时可用性，让全网快速达成临时共识。
        
2.  **经济上最终不可逆（Finality 层面）**
    
    -   本质：是**确定性的、经济惩罚保障的最终状态**，由检查点机制和验证者质押资产背书。
        
    -   特点：一旦交易被最终确认，想要回滚交易需要验证者发动攻击，面临**全部质押资产被 Slashing 罚没**的惩罚，攻击成本极高，在经济上几乎不可行。
        
    -   作用：为交易提供绝对的最终性保障，让用户无需担心交易被回滚。
        

**总结**：链头正确是 “暂时的、可变更的共识”，最终不可逆是 “永久的、经济保障的最终状态”，二者是从 “临时一致” 到 “绝对安全” 的两个递进层次。

* * *

## 三、执行层（EL）& 共识层（CL）协同知识 & 挑战解答

### 🔍 知识卡整理

1.  **Engine API 核心定位**：Engine API 是以太坊 Merge 之后，**执行层（EL）与共识层（CL）之间的核心接口**，是两层协同的桥梁。
    
2.  **接口交互逻辑**：CL 负责提供 payload（区块数据）请求，EL 负责执行交易、返回执行结果，二者通过 Engine API 完成数据交互。
    
3.  **接口异常影响**：Engine API 接口异常会直接影响区块提议、网络同步，严重时会导致节点离线、网络共识异常。
    
4.  **版本兼容性关键**：EL/CL 客户端升级时，**版本兼容性**是核心，必须保证接口前后兼容，否则会导致节点无法正常运行。
    

### 🎯 Boss 挑战：“EL 与 CL 解耦但协同” 的工程含义

**一句话总结**：执行层（EL）和共识层（CL）在代码、架构、升级上完全解耦，可独立迭代开发，但通过标准化的 Engine API 接口严格协同，共同保障以太坊网络的正常运行。

## 模块一：区块链执行层核心基础

1.  执行层（EL）核心定位：区块链的计算与状态变更核心引擎，核心职责是交易执行与区块链全局状态更新。
    
2.  Nonce 核心作用：一是防止交易重放攻击，避免同一笔交易被恶意重复执行；二是严格约束交易的发送与执行顺序，保证同账户交易按 Nonce 递增顺序执行。
    
3.  Gas 核心机制：为 EVM 的计算、存储资源做量化计价，同时通过 Gas 上限限制，从机制上避免合约代码无限执行耗尽节点资源，是网络安全与资源公平分配的核心设计。
    
4.  交易执行产物：交易执行完成后，会生成对应的事件日志、交易回执（收据），并完成区块链全局状态的变更。
    

## 模块二：EVM 以太坊虚拟机核心规则

1.  EVM 本质：具备确定性的**基于栈的虚拟机（栈机）**，所有运算操作遵循栈 “后进先出” 的执行原则。
    
2.  存储区域核心特性
    
    -   Memory（内存）：临时存储区域，仅在当前合约调用的生命周期内有效，调用结束后数据立即清空。
        
    -   Storage（存储）：持久化状态区域，数据会永久记录在区块链上，是账户与合约状态的核心存储载体。
        
3.  Opcode 操作码规则：不同操作码对应不同的节点资源消耗成本，其 Gas 定价直接决定单笔交易的总 Gas 消耗。
    
4.  Revert 操作规则：执行后会回滚本次交易对区块链状态做出的所有变更，同时向调用方返回错误信息；已消耗的 Gas 不予退还，仅退还未使用的 Gas。
    

## 模块三：交易全生命周期核心流程

1.  交易发起前提：交易必须先经过账户私钥签名，完成签名后才会被广播到区块链网络节点。
    
2.  交易暂存规则：节点接收到广播的合法交易后，会将交易放入交易池（txpool/mempool）中，等待打包。
    
3.  交易打包流程：交易被区块提议者（矿工 / 验证者）从交易池中选中后，会被打包进新生成的区块。
    
4.  交易最终性规则：交易的最终性由区块链共识层完成确认，仅被打包进区块不等于交易完成最终确认，需经过共识多轮验证后，交易才具备不可逆的最终性。
    

## 挑战 1：用 “输入→执行→产出” 三步解释状态转移函数

状态转移函数是区块链的核心底层函数，区块链本质是一个分布式状态机，该函数的作用就是驱动区块链的全局状态从旧状态向新状态完成可信流转，三步拆解如下：

1.  **输入阶段**：状态转移函数的核心输入包含两部分，一是**区块链的前序全局旧状态**，即上一个区块执行完成后，全网已达成共识的最终账户、合约数据状态；二是**合法的已签名交易集合**，即本次需要执行的、经过校验的有效交易，是触发状态变更的核心源。
    
2.  **执行阶段**：以 EVM 为执行载体，由执行层（EL）按规则依次执行输入的交易集合。对每一笔交易，EVM 会解析交易指令，调用对应的 Opcode 操作码，在虚拟机环境中完成交易逻辑（如代币转账、合约方法调用等）；同时通过 Gas 机制约束资源消耗，处理执行异常（如 Revert 状态回滚），最终计算出每笔交易执行后的状态变更结果。
    
3.  **产出阶段**：执行完成后，状态转移函数输出两大核心产物，一是**全新的全局新状态**，即经过所有交易执行、异常回滚处理后，全网需要达成共识的最新账户与合约数据，该状态会作为下一次状态转移的输入旧状态；二是**交易执行配套凭证**，包含交易回执、执行事件日志、Gas 消耗明细等，用于验证交易执行结果与状态变更的合法性。
    

## 挑战 2：解释为什么 “确定性” 是全网共识前提

这里的 “确定性”，特指 EVM 的执行确定性 —— 对于同一个初始状态、同一笔交易 / 交易集合，在任何节点的 EVM 中执行，都必须得到完全一致的最终状态、执行结果与 Gas 消耗。它是全网共识的核心前提，原因如下：

1.  **确定性是分布式节点达成状态一致的基础**。区块链是去中心化的分布式系统，无中心化权威节点，全网成千上万的节点都需要独立验证区块与交易。只有执行过程具备确定性，所有节点对同一笔交易、同一个区块执行后，才能得到完全相同的最终状态。若执行结果不确定，不同节点执行同一区块得到不同状态，节点间就无法达成一致共识，区块链分布式账本会出现分叉、数据不一致，失去账本的唯一性与可信度。
    
2.  **确定性是区块合法性验证的核心前提**。区块链共识流程中，节点收到新的区块后，必须独立重新执行区块内的所有交易，验证执行后的状态根、交易根等数据是否与区块头一致，以此确认区块合法性。若执行过程不具备确定性，节点无法复现区块提议者的执行结果，就无法完成区块合法性校验，共识机制将无法正常运转。
    
3.  **确定性保障了区块链的不可篡改性与可追溯性**。区块链的核心价值是不可篡改的分布式账本，每一个区块的状态都依赖前一个区块的最终状态，整条链的状态流转环环相扣。只有每一次状态转移的执行过程都是确定的，才能保证整条链的状态流转可复现、可追溯、不可篡改。若执行结果不确定，账本的历史状态就无法被统一验证，区块链的不可篡改特性将无从谈起。
    
4.  **确定性是区块链去信任设计的核心支撑**。去中心化网络中节点互不信任，只有执行逻辑具备确定性，节点才能无需信任其他节点，仅通过自身独立执行就能验证数据的正确性。若执行结果不确定，节点必须依赖第三方的执行结果，违背了区块链去中心化、去信任的核心设计原则。
    

## 挑战 3：按顺序写出交易生命周期至少 5 个节点

以下为交易从发起至完成最终确认的完整生命周期，按顺序共 6 个核心节点，覆盖全流程关键环节：

1.  **交易构造与签名**：用户在钱包中构造交易（包含接收地址、转账金额、Gas 参数、Nonce、合约交互数据等核心字段），使用账户私钥对交易进行加密签名，生成合法可广播的交易原始数据，这是交易生命周期的起点，只有完成签名的交易才具备网络合法性。
    
2.  **交易广播与节点接收校验**：签名完成的交易通过钱包节点广播到区块链 P2P 网络，网络中的全节点接收到交易后，先完成基础合法性校验（签名有效性、Nonce 合规性、账户余额是否足够支付 Gas 等），过滤无效交易。
    
3.  **交易进入 Mempool（交易池）**：通过基础校验的合法交易，会被节点放入本地交易池（Mempool/txpool）暂存，等待区块提议者筛选；同时节点会将该交易继续广播给网络中的其他节点，让交易同步到全网交易池。
    
4.  **交易被选中与区块打包**：区块提议者（矿工 / 验证者）按照 Gas 费高低、交易优先级等规则，从交易池中筛选交易，将选中的交易打包进新构造的区块中，完成区块构建后向全网广播该新区块。
    
5.  **区块共识验证与交易上链**：全网节点接收到新区块后，对区块完成合法性校验，同时独立重新执行区块内的所有交易，验证执行结果与区块头数据是否一致；区块通过共识机制验证后，会被添加到区块链主链上，交易正式上链，从 “待打包” 变为 “已打包” 状态。
    
6.  **交易最终确认（最终性达成）**：区块被添加到主链后，随着后续新区块不断链接在该区块之后，经过共识层多轮确认，交易所在的区块被不可逆地固定在主链上，交易完成最终确认，具备不可回滚的最终性，交易生命周期正式完成。

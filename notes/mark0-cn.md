# 协议组成

协议包含两个主要部分——执行层和共识层。执行层（EL）负责处理实际的交易和用户交互，这是全局计算机运行程序的地方。共识层（CL）提供权益证明共识机制——这是一种密码经济学安全机制，可确保所有节点遵循同一最新状态，并推动执行层的主链运行。

![](https://epf.wiki/wiki/protocol/img/clients-overview.png)![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/mark0-cn/images/2026-04-06-1775476081960-image.png)

# 协议设计理念

1.  **简洁性：**理想情况下，任何普通程序员都应能理解并实现整个规范，主要是为了减少个人或精英开发团队对协议的影响。
    
2.  **普适性：**以太坊旨在成为一个让新时代开发者能够构建去中心化且真正无需信任的应用程序的平台，而无需担心底层的复杂性。
    
3.  **模块化：**尽管以太坊远非完美，但与协议存在同步的持续且严谨的研究与工程实践一直在推进。在开发过程中，应能轻松在某一处对协议进行小幅修改，且应用层无需任何额外修改即可继续运行。
    
4.  **无歧视性：**无歧视性是以太坊设计理念核心架构的基础。该协议不会主动限制或阻止特定类别的使用，协议中的所有监管机制均旨在直接规制对协议本身造成的损害，而非针对特定不受欢迎的应用。
    
5.  **灵活性：**以太坊协议的细节并非一成不变。以太坊改进提案流程是一项开放标准，用于向协议提出新的修改建议。
    

# 区块链层级协议

## Accounts over UTXOs

区块链最早的实现形式（包括比特币及其衍生币）将用户余额存储在基于未花费交易输出（UTXOs）的结构中。而以太坊则采用基于账户的模型。

**_UTXO：未花费交易输出（UTXO）是部分数字货币模型中的一个独特要素。它代表了发送方授权的一定数量的加密货币，可供接收方花费。_**

因此，用户在系统中的“余额”是指用户拥有能生成有效签名的私钥的那组硬币的总价值。基于账户的模型更具灵活性，支持更复杂的交易。

以太坊采用基于**账户**的模型，而非未花费交易输出（UTXO）模型。虽然未花费交易输出能提供更高程度的隐私，但也会给以太坊这样的系统带来更大的复杂性。

## 账户的好处

1.  **节省空间**：一个账户拥有5个未花费交易输出（UTXO），那么从UTXO模型切换到账户模型可将空间占用从（20 + 32 + 8）× 5 = 300字节（20字节为地址、32字节为交易ID、8字节为金额）降至20 + 8 + 2 = 30字节（20字节为地址、8字节为金额、2字节为随机数）。实际节省的空间虽远不及这个数值，但仍相当可观——因为账户需要存储在帕特里夏树中，而帕特里夏树本身体积较大。此外，交易规模也可更小（例如，以太坊中交易为100字节，而比特币中为200-250字节），因为每笔交易仅需一次引用和一次签名，且仅生成一个输出。
    
2.  **高度可替代性：**账户具有完全可替代性，因为任何硬币都可以被其他任何硬币替代。
    
3.  **简洁性**：账户模型比未花费交易输出（UTXO）模型更易于实现和理解。未花费交易输出（UTXO）模型需要更复杂的交易验证算法，且其灵活性和功能也不如账户模型
    

账户范式的一个缺点是，为了防止重放攻击，每笔交易都必须包含一个**随机数**，账户会记录已使用的随机数，且仅当交易的随机数是上一个已使用随机数加1时才接受该交易

# Merkle Patricia Trie (MPT)

![image](https://p6-flow-imagex-sign.byteimg.com/labis/image/b2d0cd8c0d35a4efa4708e331a742d4d~tplv-a9rns2rl98-pc_smart_face_crop-v1:512:384.image?lk3s=8e244e95&rcl=20260406201346ACB621EB7364C50C0626&rrcfp=cee388b0&x-expires=2090837634&x-signature=%2B9Nm2%2B9UcZN3TbSeD64piCdfzdA%3D)

它借鉴了 PATRICIA (the Practical Algorithm To Retrieve Information Coded in Alphanumeric) 的部分特性，同时也是为了高效检索构成以太坊状态的各项数据而设计的。

Merkle-Patricia trie具有确定性且可通过密码学验证：生成状态根的唯一方式是从状态的每个独立部分计算得出，而两个相同的状态可通过比较根哈希及生成根哈希的哈希值（默克尔证明）轻松证明其一致性。反之，不存在两个不同状态却拥有相同根哈希的情况，任何试图用不同值修改状态的操作都会生成不同的状态根哈希。从理论上讲，这种结构为插入、查找和删除操作提供了时间复杂度为 O(log(n)) 的“终极方案”。

# **Verkle trees**

![image](https://p11-flow-imagex-sign.byteimg.com/isp-i18n-media/img/58ac836596786a57796832f8d52ef3cb~tplv-a9rns2rl98-pc_smart_face_crop-v1:512:384.image?lk3s=8e244e95&rcl=20260406201435604AA3E177812567F782&rrcfp=cee388b0&x-expires=2090837687&x-signature=muaEx69gNk51tKGiqukCVu4De6k%3D)

MPT 目前被应用于多种需要在网络上传输成员资格证明的场景，包括协议、公钥目录、比特币等加密货币以及安全文件系统。一棵具有_n_ 叶子，并且有_O_(_log_2​_n_)大小的证明。即使_O_(_logn_)复杂度足够安心，然而在发送大量数据的时候会造成大量带宽。分支因子为𝑘的Verkle树可实现𝑂(𝑘𝑛)的构建时间复杂度与𝑂(logₖ𝑛)的成员证明大小。这意味着分支因子𝑘在计算能力与带宽之间形成了一种权衡。

以太坊当前面临的紧迫问题之一是其状态规模。据估算，当前状态大小约为1-2TB（在撰写本文时）。节点本身将状态存储在工作内存中，甚至存储在速度较慢的永久存储中都不切实际，因此，无状态化对于网络的发展变得至关重要。Verkle树及其向量承诺能够生成小得多的证明（**称为见证**）。与默克尔树需要提供每一层所有“兄弟节点”的哈希值不同，证明者只需提供从每个叶子节点到根节点路径上的所有父节点（以及一份称为可选项的额外证明）即可。

# **Recursive Length Prefix (RLP)**

创建一种新的序列化方案的基本原理在于其他方案的概率特性。RLP 通过简洁且确定性的序列化解决了这一问题，并保证了绝对的逐字节一致性。RLP 并未尝试定义布尔值、浮点数、双精度浮点数甚至整数等任何特定数据类型——相反，它仅用于以嵌套数组的形式存储结构。键/值映射也未得到显式支持；支持键/值映射的半官方建议是将此类映射表示为 `[[k1, v1], [k2, v2], ...]`，其中`k1, k2...`按照字符串的标准排序规则进行排序。

随着时间的推移，数据结构对序列化算法完全匿名的理念，对于布尔值和整数等固定长度数据类型而言效率低下。以太坊 2.0 引入了简单序列化（SSZ），它支持可变长度和固定长度数据类型，还具备默克尔化等额外功能。

```
RLP 规则（极简版）
单个字节，值在 [0x00, 0x7F] 之间：
→ 直接原样输出
字符串长度 > 1 或 字节 > 0x7F：
→ 前面加一个长度前缀
列表（数组）：
→ 先把里面所有元素 RLP 编码，再整体加一个列表长度前缀

dog 对应的 ASCII：
d = 0x64
o = 0x6F
g = 0x67
长度 = 3
3 小于 55（0x37），所以前缀是：
0x80 + 3 = 0x83
最终 RLP 编码：
0x83 0x64 0x6F 0x67

嵌套列表例子（以太坊常用）
比如编码：
["dog", "cat"]
先编码 "dog" → 0x83646F67
再编码 "cat" → 0x83636174
拼接起来：0x83646F6783636174
总长度 = 8
列表前缀：0xC0 + 8 = 0xC8
最终 RLP：
0xC8 83 64 6F 67 83 63 61 74
```

# **Simple serialize (SSZ)**

序列化是将数据结构转换为可传输格式并在之后进行重构的过程。SSZ 是以太坊 2.0 信标链中使用的一种序列化格式。它被设计为一种非自描述的序列化方案——相反，它依赖于必须预先知晓的模式。与 RLP 相比，SSZ 具有诸多优势，例如对对象的高效重哈希和快速索引，而这些都是 RLP 所缺乏的，这也导致了_O_(_N_) 复杂度。

根据[Vitalik 的评论](https://ethresear.ch/t/replacing-ssz-with-rlp-zip-and-sha256/5706/12)，SSZ 试图解决的一个主要问题是 RLP 不支持默克尔化，这意味着任何事物都无法通过简洁的轻客户端证明，也就失去了实现无状态性的可能——而无状态性仍是当前以太坊研发的核心目标。

```
SSZ 核心特点（一句话）
固定长度类型：直接按二进制存
可变长度类型：存 偏移量 (offset)，数据放后面
结构严格按 schema（类型定义） 解析，不像 RLP 只靠长度猜
自带 默克尔化支持（每一段都能做哈希树）

2. 最简单例子：uint64
类型：uint64
值：123
SSZ 序列化（小端序）：
0x7B 0x00 0x00 0x00 0x00 0x00 0x00 0x00
就是直接 8 字节小端存储。

3. 固定长度数组 example
类型：vector[uint64, 2]
值：[1, 2]
每个 uint64 占 8 字节，直接拼接：
0x0100000000000000 0200000000000000

4. 可变长度例子（最能体现 SSZ）
类型：List[uint64, 4]（最多 4 个，但长度可变）
值：[10, 20]
SSZ 会分成两部分：
offsets 区：一个偏移量，指向数据开始位置
data 区：真正的数据
最终序列化：
0x04000000          # 偏移量 4（32位小端）
0x0A00000000000000  # 10
0x1400000000000000  # 20
```

# DHT

分布式哈希表（DHT）的主要优势在于，查找操作在网络中仅产生对数级别的通信开销。这使其适用于在点对点（p2p）网络中查找（查询）内容。但随即出现了一个问题：如果以太坊中的大多数节点都只对同一种内容——最新区块——感兴趣，那我们为何需要在以太坊中_查找_内容呢？基于共识槽的链头始终是相同的，而每个共识槽只需广播一个区块。分布式哈希表（DHT）被应用于[BitTorrent](https://www.bittorrent.org/beps/bep_0005.html)和星际文件系统（IPFS）等协议中，这些协议存储着海量内容，用户会通过它_查找_自己感兴趣的内容。而在以太坊的网络架构中，分布式哈希表（DHT）的作用是查找不同的节点，而非区块。

以太坊网络层的发现协议采用了基于卡德梅利亚（Kademlia）的分布式哈希表（discv5）来存储节点记录（ENR）。节点记录包含互联网层的路由信息，用于在节点之间建立连接。加入网络的节点会使用引导节点在分布式哈希表中为自身的节点ID转发查找查询。在此过程中，它们会发现其他节点的节点记录，从而帮助填充自身的路由表。通常，节点还会查找随机的节点ID来遍历网络，即找到所有节点。

以太坊网络中的区块通过点对点协议栈的流言协议进行分发。在通过底层分布式哈希表发现节点后，节点会使用覆盖网络（[流言子协议](https://github.com/libp2p/specs/blob/f25d0c22e5ef045c8c050bc91c297468de35f720/pubsub/gossipsub/gossipsub-v1.0.md)）在整个网络中传播区块。该覆盖网络会创建包含路由信息的专属路由表，以建立连接并传递覆盖网络的特定信息（已订阅的主题、扇出值等）。这种覆盖网络本质上是一个非结构化网络。

DHT 会额外采取步骤加入非结构化网络，而非仅连接节点（[友邻模型](https://en.wikipedia.org/wiki/Friend-to-friend)或 PEX）并直接下载/传播所需内容。为何要通过 DHT 这一额外步骤，再加入非结构化网络呢？从网络启动的角度来看，Kademlia 能提供全局视角，而友邻网络本质上只能提供网络的本地视角。简单来说，DHT 为节点加入网络提供了公开且非本地化的机制（也可以说，其去中心化程度也略高）。这种利用结构化网络（DHT）启动并接入非结构化网络的混合方式，在 BitTorrent 的 [对等点交换（PEX）](https://www.bittorrent.org/beps/bep_0011.html)协议中也能看到。[非结构化网络](https://en.wikipedia.org/wiki/Peer-to-peer#Unstructured_networks)作为覆盖网更受青睐，因为它们在高动态变化的网络中具有鲁棒性。

26.4.18 今天学习了Beacon 节点接口说明  
  
**一、信标链概述**

-   信标链是以太坊 PoS 的核心系统链，负责存储和管理验证者注册表
    
-   成为验证者的方式：向以太坊 PoW 链上的存款合约发送单向 ETH 交易（Capella 升级后可提款）
    
-   验证者激活条件：信标链处理存款收据 + 达到激活余额 + 完成排队
    
-   退出方式：自愿退出 或 因违规被强制罚没
    
-   信标链的主要负载来源：**见证消息**（同时包含分片块可用性投票和信标块的 PoS 投票）
    

* * *

### **二、Beacon API**

-   共识层客户端（信标节点）提供的 REST 接口
    
-   用于读取共识相关信息，验证者客户端也通过该接口与信标节点交互
    

* * *

### **三、核心数据结构（Container）**

**1\. BeaconState**

包含以下主要字段组：

| 分类 | 字段 |
| --- | --- |
| 版本信息 | genesis_time, genesis_validators_root, slot, fork |
| 历史信息 | latest_block_header, block_roots, state_roots, historical_roots |
| Eth1 相关 | eth1_data, eth1_data_votes, eth1_deposit_index |
| 验证者注册 | validators, balances |
| 随机性 | randao_mixes |
| 罚没 | slashings（每 epoch 被罚没的有效余额总和） |
| 参与率 | previous_epoch_participation, current_epoch_participation |
| 最终性 | justification_bits, previous_justified_checkpoint, current_justified_checkpoint, finalized_checkpoint |
| 不活跃性 | inactivity_scores |
| 同步委员会 | current_sync_committee, next_sync_committee |
| 执行层 | latest_execution_payload_header |
| 提款 | next_withdrawal_index, next_withdrawal_validator_index |
| 深度历史（Capella 起） | historical_summaries |
| 待处理项 | pending_deposits, pending_partial_withdrawals, pending_consolidations |

**2\. BeaconBlockBody**

信标区块体包含以下组件：

-   randao\_reveal：RANDAO 随机数揭示
    
-   eth1\_data：Eth1 数据投票
    
-   graffiti：任意数据
    
-   操作列表：
    
    -   proposer\_slashings（提议者罚没）
        
    -   attester\_slashings（见证者罚没）
        
    -   attestations（见证消息）
        
    -   deposits（存款）
        
    -   voluntary\_exits（自愿退出）
        
-   sync\_aggregate：同步聚合
    
-   execution\_payload：执行载荷
    
-   bls\_to\_execution\_changes：BLS 到执行层的凭证变更
    
-   blob\_kzg\_commitments：Blob 的 KZG 承诺
    
-   execution\_requests：执行请求

26.4.17 今天学习了以太坊弱主观性  
  
**一、客观性与主观性**

-   **客观信息**：正确性可被完全验证
    
-   **主观信息**：正确性无法完全验证，需要一定程度的信任
    
-   大多数信息处于客观与主观之间的光谱上
    

**合并前（PoW）**

-   从创世块开始，客户端可以客观验证之后的每个区块
    
-   客户端从对等节点获取最新"最终"链头，通过父哈希回溯到创世块
    
-   篡改历史需要巨大的算力，因此已验证的历史是**客观**的
    
-   创世块本身仍需一定信任（但可视为"弱客观"而非主观，因为大多数客户端自带创世块及其校验和）
    
-   "最终"区块：距离当前链头第7个区块（回滚6个PoW区块被认为困难）
    

**合并后（PoS）**

-   引入验证者成员制和投票机制
    
-   共识层（CL）与执行层（EL）逻辑分离
    
-   历史验证机制大致相同，但最新最终区块从共识层获取
    
-   EL验证区块执行的正确性，CL在此基础上投票并达成共识
    
-   **创世块同步变得"不安全"** —— 因此引入**弱主观性**
    

* * *

### **二、弱主观性下的同步**

**三个主要差异**

**1\. 同步方向相反**

-   问题：已退出的验证者仍可对过去的区块重新投票/见证，创建替代历史
    
-   已同步的节点不受影响（已处理验证者退出）
    
-   正在同步的节点无法知道验证者的未来状态，可能跟随错误链
    
-   解决：信标链的同步方向**反向**（与执行块同步相反）
    

**2\. 信任锚点**

-   链的历史在特定条件下可被改变，同步目标无法客观验证
    
-   同步目标（称为**弱主观性检查点**）需要通过**带外**方式共享（不能信任P2P层的随机连接）
    
-   同步目标是"最终确认"的检查点，因此是**弱主观**的（允许一定程度验证，但存在性无法证明）
    

**3\. 时间限制**

-   节点离线时间过长也会受到同样攻击
    
-   攻击窗口：足够让验证者退出并发起攻击的时间段，称为**弱主观性周期**
    
-   同步目标也有时效性：若回溯或EL同步时间过长，目标会变陈旧
    
-   解决方案：EL同步时，持续导入新的信标块但不检查其执行载荷，称为**乐观导入区块**
    

* * *

### **三、弱主观性同步步骤**

1.  通过**带外方式**获取弱主观性检查点
    
2.  从弱主观性检查点回溯所有区块直到创世块
    
3.  更新执行链的目标头
    
4.  **乐观地**跟随链头，持续更新执行链的目标头
    
5.  EL同步完成后，将CL槽位标记为已验证，节点可视为**完全同步**

26.4.15 今天学习了主流 CL 客户端对比  
  
**一、概述**

-   共识客户端（原称 eth2 客户端）运行以太坊的 PoS 共识算法，使网络能够就信标链的链头达成一致
    
-   共识客户端不参与验证/广播交易，也不执行状态转换——这些由执行客户端负责
    
-   共识客户端本身不发起见证或提议新区块——这些由验证者客户端（共识客户端的可选附加组件）完成

26.4.14 今天学习了CL 客户端内部设计  
  
**  
一、概述**

-   以太坊共识协议是 **forkful** 的：使用分叉选择规则，可能发生重组
    
-   **Gasper** = LMD GHOST（活性）+ Casper FFG（安全性）
    
-   活性：正确的事情总会发生；安全性：错误的事情永远不会发生。实际协议无法同时保证两者
    

* * *

### **二、分叉选择机制**

**目标**：让所有诚实节点最终同意唯一的线性历史链

**分叉选择规则**

-   从候选区块中选出一个头区块，唯一确定一条回到创世块的链
    
-   选中的区块及其所有祖先必须有效
    

**不同共识的分叉选择规则**

-   PoW（以太坊/比特币）：最重链规则（累积工作量最大），实际上未使用GHOST
    
-   Casper FFG（PoS）：跟随包含最高合理检查点的链，永不回滚最终确认的区块
    
-   LMD GHOST（PoS）：贪婪最重观察子树，累计验证者投票
    

**重组**

-   节点收到新区块后重新运行分叉选择规则
    
-   若新区块在另一分支上且得分更高，节点回退原区块，切换到新区块所在分支
    
-   短重组（1-2个区块）常见；长重组应罕见
    

* * *

### **三、安全性与活性**

-   **安全性**：最终确认的检查点不冲突，无双花攻击，所有诚实节点状态一致
    
-   **活性**：链能持续添加新区块，不死锁
    
-   CAP定理：分布式系统无法同时保证一致性、可用性和分区容错性
    

**以太坊的选择**

-   良好网络条件下同时追求安全与活性
    
-   网络分区时**优先保证活性**：两侧节点继续出块但无法最终确认；分区持续可能导致两条独立链
    

* * *

### **四、Gasper：LMD GHOST + Casper FFG**

| 组件 | 作用 | 特性 |
| --- | --- | --- |
| LMD GHOST | 保证活性 | 持续出块，但可能分叉，非形式安全 |
| Casper FFG | 保证安全性 | 周期性最终确认区块，防止长回滚 |

-   组合效果：LMD GHOST 推动链前进，Casper FFG 提供稳定性
    
-   即使 Casper FFG 无法最终确认，链仍可继续增长
    

* * *

### **五、共识层架构组件**

**节点类型**

-   普通节点：运行客户端软件，参与 P2P 网络
    
-   验证者节点：质押 32 ETH，运行验证者客户端，参与共识
    

**共识客户端（CL）**

-   信标节点：协调 PoS 共识，如 Prysm、Teku、Lighthouse、Nimbus
    
-   验证者客户端：允许质押 ETH，提议区块，仅与本地信标节点通信
    

**网络层次**

-   主网（Mainnet）：生产环境，管理真实 ETH
    
-   测试网：用于开发测试，管理测试 ETH
    

**EL 与 CL 的关系**

-   每个节点同时运行 EL 和 CL 软件
    
-   CL 负责共识链（信标链），处理信标区块和见证消息
    
-   CL 与 EL 通过 Engine API 进行本地 RPC 通信
    
-   两者共享 ENR，但各有独立的密钥（eth1 key 和 eth2 key）
    

* * *

### **六、控制流程**

**CL 作为非区块生产者**

1.  通过区块 gossip 协议接收区块
    
2.  预验证区块
    
3.  将区块中的交易作为执行载荷发送给 EL
    
4.  EL 执行交易并验证区块状态
    
5.  EL 返回验证数据给 CL
    
6.  CL 将区块加入链并广播见证消息
    

**CL 作为区块生产者**

1.  收到成为下一区块生产者的通知
    
2.  调用 EL 的创建区块方法
    
3.  EL 访问交易池，打包交易，执行并生成区块哈希
    
4.  CL 将交易和区块哈希加入信标区块
    
5.  CL 通过区块 gossip 协议广播区块
    
6.  其他客户端验证并见证
    
7.  获得足够见证后，区块成为链头，被合理化并最终确认
    

* * *

### **七、状态转换**

**基本原理**

-   纯函数：输出仅取决于输入，无副作用
    
-   所有节点从同一创世状态开始，应用相同区块，得到相同最终状态
    

**信标链状态转换函数**

-   `S' = f(S, B)`：给定前一状态 S 和区块 B，产生新状态 S'
    
-   每槽转换：`S' = f_s(S)`，无论是否有区块
    
-   每区块转换：`S' = f_b(S, B)`
    
-   每 epoch 转换：`S' = f_e(S)`
    

**有效性条件**

-   状态转换不能触发未处理异常（断言失败、越界访问）
    
-   不能导致 uint64 溢出或下溢
    

**代码示意**

text

```
def state_transition(state, signed_block):
    block = signed_block.message
    process_slots(state, block.slot)      # 处理槽（包括空槽）
    assert verify_block_signature(state, signed_block)
    process_block(state, block)           # 处理区块
    assert block.state_root == hash_tree_root(state)
```

26.4.13 今天学习了交易打包与出块流程  
  
**一、概述**

-   区块构建是以太坊区块链的核心功能，决定验证者在提议区块前如何获取区块
    
-   以太坊网络由共识客户端（CL）和执行客户端（EL）共同组成
    
-   验证者被选中提议区块时，会从 CL 获取区块，也可以使用外部构建者构建的区块
    
-   本文聚焦 EL 如何构建区块以及交易执行流程
    

* * *

### **二、Payload 构建流程**

-   CL 通过 Engine API 的 `fork_choice_updated` 端点指令 EL 开始构建区块
    
-   构建出的 payload 的 fee recipient 可能与 payload 属性中建议的不同
    

**交易来源**

-   节点通过 gossip 协议广播交易
    
-   交易经过验证（nonce 正确性、余额充足、签名正确）后存入交易池（mempool）
    

**区块提议者选择**

-   每个 slot 有一个指定的区块提议者，由 CL 通过伪随机过程选出
    
-   被选中的验证者，其 CL 通过 `fork_choice_updated` 方法向 EL 提供构建区块所需的上下文信息
    

* * *

### **三、区块构建简化逻辑**

**输入参数**

-   environment：包含时间戳、区块号、父区块、base fee、提款等信息（来自 CL）
    
-   pool：交易池，假设按交易价值升序排列
    
-   state：状态数据库，用于执行所有交易
    

**输出**

-   构建好的区块
    
-   累积了所有交易的状态数据库
    
-   可能的错误
    

**核心循环逻辑**

1.  初始化 gasUsed = 0，transactions 为空数组
    
2.  当 gasUsed < 30,000,000 且交易池非空时循环执行
    
3.  从交易池中取出一笔交易
    
4.  在 EVM 中执行该交易
    
5.  若执行出错，跳过该交易继续循环（交易无效或交易池数据过时）
    
6.  若无错误，将交易加入 transactions 列表，累加 gasUsed
    
7.  循环结束后，调用 Finalize 生成完整区块（计算交易根、收据根、提款根等）
    

* * *

### **四、Geth 代码流程详解**

**1\. Engine API 调用**

-   验证者被选为区块构建者后，通过 Engine API 调用 `engine_forkchoiceUpdatedV2`，EL 启动区块构建
    

**2\. 创建初始区块**

-   Geth 的 miner 模块负责区块构建和交易执行的核心逻辑
    
-   `buildPayload` 函数首先创建一个空区块，确保节点不会错过 slot
    
-   同时启动一个 goroutine，负责后续填充区块（并发执行）
    

**3\. 获取密封区块**

-   goroutine 等待多个通信操作，首次调用 `getSealingBlock` 时指定 `noTxs: False`（区块不应为空）
    
-   `getSealingBlock` 通过 `getWorkCh` 通道发送请求，该通道在 `mainLoop` 中被监听
    
-   数据经 `getWorkCh` 后发送给 `w.generateWork`
    

**4\. 填充交易**

-   `generateWork` 函数执行实际的交易填充
    
-   `fillTransactions` 从 mempool 获取所有待处理交易（包括 blob 交易）
    
-   交易按费用排序后传入 `commitTransactions`
    

**5\. 提交交易**

-   `commitTransactions` 检查每笔交易是否有足够的剩余 Gas
    
-   同时检查每个区块允许的 blob 数量（EIP-4844 规定）
    
-   调用 `applyTransaction` 执行交易
    

**6\. 执行交易**

26.4.12 今天学习了节点对外接口规范  
  
**一、概述**

-   JSON-RPC 是一种基于 JSON 的远程过程调用协议，属于执行 API 规范的一部分
    
-   允许用户在客户端上调用远程服务器函数并获取返回结果
    
-   共识层（CL）与执行层（EL）之间通过 Engine API 进行交互，同样基于 JSON-RPC
    

* * *

### **二、API 通用结构**

所有 JSON-RPC 请求遵循统一格式：

-   id：请求的唯一标识符
    
-   jsonrpc：JSON-RPC 协议版本（如 "2.0"）
    
-   method：要调用的方法，格式为 `命名空间_方法名`
    
-   params：方法参数数组，无参数时为空数组
    

* * *

### **三、命名空间**

方法由命名空间前缀和方法名组成，用下划线分隔。客户端需实现规范要求的最小方法集，也可提供客户端特有的方法。

**常见命名空间**

-   eth：核心以太坊交互接口，用于读取余额、创建交易等，可能涉及敏感数据
    
-   web3：提供 web3 客户端的实用功能，非敏感
    
-   net：提供节点网络信息，非敏感
    
-   txpool：检查交易池，非敏感
    
-   debug：检查以太坊状态，支持 Geth 风格追踪，非敏感
    
-   trace：检查以太坊状态，支持 Parity 风格追踪，非敏感
    
-   admin：配置节点，敏感
    
-   rpc：提供 RPC 服务器及其模块信息，非敏感
    

敏感命名空间（如 admin、eth）可用于配置节点或访问节点中存储的账户数据。

* * *

### **四、常用方法示例**

**eth 命名空间**

-   eth\_blockNumber：返回最新区块号
    
-   eth\_call：立即执行消息调用，不上链
    
-   eth\_chainId：返回当前链 ID
    
-   eth\_estimateGas：估算交易或调用的 Gas 用量
    
-   eth\_gasPrice：返回当前每单位 Gas 的价格（Wei）
    
-   eth\_getBalance：查询指定地址的余额
    
-   eth\_getBlockByHash / eth\_getBlockByNumber：按哈希或区块号查询区块信息
    
-   eth\_getCode：查询指定地址的代码
    
-   eth\_getLogs：返回匹配过滤条件的日志数组
    
-   eth\_getStorageAt：查询指定存储位置的值
    

**debug 命名空间**

提供对以太坊状态的原始数据访问，适用于区块浏览器或研究场景。部分方法计算量大，公共 RPC 提供方常限制或仅允许安全方法。

-   debug\_getBadBlocks：返回客户端见过的近期坏区块
    
-   debug\_getRawBlock：返回 RLP 编码的区块
    
-   debug\_getRawHeader：返回 RLP 编码的区块头
    
-   debug\_getRawReceipts：返回 EIP-2718 二进制编码的收据数组
    
-   debug\_getRawTransactions：返回 EIP-2718 二进制编码的交易数组
    

* * *

### **五、Engine API**

-   执行层在独立的认证端口上提供 Engine API，而非普通 HTTP JSON-RPC 端口
    
-   该接口仅供共识层访问，不是面向用户的 API
    
-   用于共识层与执行层之间的通信，交换共识、分叉选择和区块验证信息
    
-   通信通过 HTTP 上的 JSON-RPC 进行，使用 JSON Web Token（JWT）认证发送方身份（不提供流量加密）
    
-   恶意外部方无法访问该接口
    

**核心方法**

-   engine\_exchangeTransitionConfigurationV1：在 CL 和 EL 之间交换配置详情
    
-   engine\_forkchoiceUpdatedV1：更新分叉选择状态，可选启动区块构建
    
-   engine\_getPayloadBodiesByHashV1：根据区块哈希返回对应的执行载荷体
    
-   engine\_getPayloadV1：获取 EL 构建的执行载荷
    
-   debug\_newPayloadV1：返回执行载荷验证详情，用于调试
    

带星号的方法有多个版本以支持网络升级。

* * *

### **六、编码规范**

参数采用十六进制编码，带 `0x` 前缀

-   数字 65 表示为 `"0x41"`
    
-   数字 0 表示为 `"0x0"`
    
-   无效写法：`"0x"`（无数字）、`"ff"`（无前缀）
    
-   哈希、账户地址、字节数组同样使用 `0x` 前缀的十六进制编码，如 `"0x400"` 表示十进制 1024，不允许前导零
    

* * *

### **七、传输协议无关性**

JSON-RPC 不依赖特定传输协议，可运行于：

-   HTTP：单向请求-响应模型，响应后关闭连接
    
-   WebSockets（WSS）：双向协议，连接保持开放，支持基于订阅的事件驱动通信
    
-   IPC：同一机器上进程间通信，速度快但不适合远程通信（如本地 JS 控制台）
    

* * *

-   ethers.js：`await provider.getBlockNumber()`

26.4.11 今天学习了 DevP2P  
  
**一、概述**

-   执行层使用两种传输协议：**TCP**（可靠、面向连接）和 **UDP**（快速、无连接）
    
-   网络分为两个并行的栈：**发现栈**（UDP）和 **传输栈**（TCP）
    

* * *

### **二、发现协议（Discv）**

**Discv4**

-   基于Kademlia DHT的去中心化节点发现
    
-   节点标识：secp256k1密钥对，公钥作为Node ID
    
-   节点间距离：通过哈希公钥的XOR计算
    
-   路由表：256个k桶，每桶最多16个节点，按最近/最久访问排序
    
-   12小时无响应则从表中移除
    
-   端点验证：通过Ping/Pong防止放大攻击
    

**Discv5**

-   Discv4的改进版，增加加密通信和服务发现
    
-   使用AES-GCM加密，ECDH建立会话密钥
    
-   支持WHOAREYOU挑战-响应机制防止欺骗
    
-   支持主题广告和服务发现
    
-   多数执行客户端已采用（Geth、Nethermind、Reth），部分仍在迁移（Besu、Erigon等）
    

**核心消息类型**

-   Ping（0x01）和 Pong（0x02）：验证节点可用性
    
-   FindNode（0x03）和 Neighbors（0x04）：查找并返回附近节点
    
-   ENRRequest（0x05）和 ENRResponse（0x06）：请求并返回ENR记录
    

* * *

### **三、以太坊节点记录（ENR）**

-   标准格式（EIP-778），包含节点连接信息
    
-   必填字段：id（如"v4"）
    
-   可选字段：secp256k1公钥、IP地址、TCP端口、UDP端口、IPv6地址等
    
-   编码：RLP列表 `[signature, seq, k, v, ...]`，最大300字节
    
-   文本表示：base64编码，前缀 `enr:`
    

**其他可理解的格式**

-   multiaddr：如 `/ip4/127.0.0.1/tcp/30303/<node-id>`
    
-   enode：URL格式，如 `enode://<node-id>@127.0.0.1:30303?discport=30301`
    

* * *

### **四、RLPx传输协议**

-   TCP基础上的安全传输协议，负责连接建立和消息交换
    

**安全连接建立流程**

1.  发起方生成临时密钥对（secp256k1）
    
2.  发送认证消息（包含临时公钥和nonce）
    
3.  接收方验证并返回确认消息
    
4.  发送首个加密帧（Hello消息，包含端口、ID、客户端信息、协议等）
    

**加密机制**

-   使用椭圆曲线集成加密方案（ECIES）
    
-   椭圆曲线：secp256k1
    
-   密钥派生函数：NIST SP 800-56连接KDF
    
-   消息认证码：HMAC-SHA-256
    
-   加密算法：AES-128-CTR
    

**前向保密**

-   静态共享密钥：基于长期密钥的ECDH，会话间不变，长期密钥泄露则历史通信可解密
    
-   临时密钥：每次握手生成，会话结束后丢弃，即使长期密钥泄露也无法解密历史通信
    

**消息帧结构**

-   header-ciphertext：加密的帧元数据
    
-   header-mac：头部MAC
    
-   frame-ciphertext：加密的消息数据
    
-   frame-mac：消息数据MAC
    

**P2P能力消息（必需）**

-   Hello（0x00）：宣布支持的能力
    
-   Disconnect（0x01）：发起断连
    
-   Ping（0x02）和 Pong（0x03）：存活检查
    

**断连原因代码**

-   0x00：主动请求断开
    
-   0x02：协议违规
    
-   0x03：无用节点
    
-   0x05：已连接
    
-   0x06：协议版本不兼容
    
-   0x09：身份异常
    

* * *

### **五、应用层子协议**

-   **eth（以太坊线协议）**：区块链数据交换、区块传播、交易中继
    
-   **snap（快照协议）**：状态同步，下载状态树片段
    
-   **les（轻以太坊子协议）**：支持轻客户端，从全节点请求数据
    
-   **portal（门户网络）**：为轻客户端提供去中心化的状态、区块、交易检索

-   二叉树结构，叶子节点数为奇数时复制最后一个哈希
    

* * *

### **三、帕特里夏树基础**

-   n叉树，用于高效存储数据而非验证
    
-   边可标记字符序列，空间效率高
    
-   只有一个子节点的节点与父节点合并
    
-   值仅存储在叶子节点

### **四、默克尔帕特里夏树（MPT）**

-   以太坊执行层的主要数据结构，结合了两种树的特性
    
-   从默克尔树继承：加密验证属性，节点包含子节点哈希
    
-   从帕特里夏树继承：基于前缀的高效键值存储和检索
    

**三种节点类型**

-   分支节点：17元素数组（1个节点值 + 16个分支），用于导航分支
    
-   扩展节点：当分支节点只有一个子节点时，压缩路径存储
    
-   叶子节点：存储键值对，键为节点哈希，值为内容
    

**关键概念**

-   nibble（半字节）：一个十六进制数字，是MPT中的基本键单位
    
-   每个节点有哈希值，通过SHA-3计算其内容
    
-   节点哈希同时作为引用该节点的键
    

* * *

### **五、以太坊中的四种MPT**

**1\. 交易树**

-   每个区块有独立的交易树
    
-   键：RLP编码的交易索引，值：RLP编码的交易对象
    
-   区块最终确认后不可更改
    

**2\. 收据树**

-   每个区块有独立的收据树
    
-   键：交易索引，值：对应的收据
    
-   主要作用：提供规范、可验证的交易结果记录
    
-   用于快照同步，无需重新执行历史交易
    

**3\. 世界状态树**

-   核心数据结构，表示以太坊当前状态
    
-   键：keccak-256哈希后的20字节账户地址
    
-   值：RLP编码的账户状态
    
-   不存储在链上，其状态根存储在区块头中
    
-   每个区块处理完所有交易后更新状态根
    

**账户字段**

-   nonce：成功发送的交易数量
    
-   balance：拥有的ETH余额（Wei为单位）
    
-   codeHash：合约代码哈希；EOA为空字符串哈希
    
-   storageRoot：指向账户存储树的根哈希（32字节）
    

**4\. 存储树**

-   每个合约账户独立的MPT
    
-   键：keccak-256哈希后的256位存储槽索引
    
-   值：RLP编码的256位值
    
-   通过SSTORE写入，通过SLOAD读取
    
-   EOA的存储树为空
    
-   使用哈希键防止DoS攻击（避免构造长路径）
    

* * *

### **六、未来：Verkle树**

**设计目标**

-   替代当前MPT，实现更高效的无状态客户端
    
-   关键升级：The Verge
    

**结构特点**

-   类似MPT，但分支因子更大（256子节点/节点）
    
-   键长32字节：31字节茎 + 1字节后缀
    
-   使用向量承诺而非简单哈希

26.4.9 今天学习了交易字段与生命周期  
  
**一、交易概述**

-   交易是由**外部账户**发出的加密签名指令，通过 JSON-RPC 广播到全网
    
-   交易包含以下字段：nonce、gasPrice、gasLimit、to、value、data/init、签名
    

* * *

### **二、交易字段详解**

**nonce**

-   等于发送方已发送的交易数量
    
-   三种作用：防止重放攻击、确定合约账户地址、替换卡住的事务
    

**gasPrice**

-   每单位 Gas 支付的 Wei 数量，用于优先执行（价格越高越容易被矿工打包）
    

**gasLimit**

-   交易执行中可使用的最大 Gas 量，耗尽则停止执行
    

**to（接收方地址）**

-   为空：合约创建模式
    
-   指向外部账户：价值转移模式
    
-   指向合约账户：合约执行模式
    

**value**

-   转账的 Wei 数量；在合约创建模式下成为新合约账户的初始余额
    

**data / init**

-   无限长度的字节数组，指定 EVM 的输入
    
-   合约创建模式下为 init 字节码，否则为输入数据
    

**签名**

-   发送方的 ECDSA 签名
    

* * *

### **三、合约创建**

-   init 字节码由两部分组成：init 字节码 + runtime 字节码
    
-   init 代码仅在账户创建时执行一次，其返回值成为 runtime 字节码
    
-   runtime 字节码存储在合约账户中，每次合约被调用时执行
    
-   部署流程：构造 init 字节码 → 签名 → 提交到节点
    

* * *

### **四、合约代码执行**

-   交易中的 to 指向已部署的合约地址
    
-   value 和 data 可以为空
    
-   执行后合约内部状态发生变化（如存储槽被写入）
    

* * *

### **五、交易收据**

每个交易（成功或失败）都会产生收据，包含五项内容：

-   交易类型（区分传统交易和类型化交易）
    
-   状态（1=成功，0=失败）
    
-   Gas 用量
    
-   日志列表（日志地址、索引主题、事件数据）
    
-   日志布隆过滤器（256字节，用于快速搜索日志）
    

收据被提交到区块的**收据树**

* * *

### **六、类型化交易与收据（EIP-2718）**

-   引入统一的**信封格式**，支持扩展新交易/收据类型，同时向后兼容
    

**交易格式**：`交易类型 + 交易载荷`

-   交易类型：单字节标识符
    
-   交易载荷：各类型规范定义的字节数组
    

**收据格式**：`收据类型 + 收据载荷`

-   收据类型必须与对应交易的交易类型相同
    

**识别规则**

-   首个字节在 0x00-0x7f 之间：类型化交易/收据
    
-   首个字节 ≥ 0xc0：传统交易/收据（RLP列表编码）
    

传统交易格式为：`RLP([nonce, gasPrice, ..., s])`  
传统收据格式为：`RLP([status, gasUsed, bloom, logs])`

26.4.8 今天学习了演变历史和EVM 执行模型基础  
  
**一、Frontier（2015年7月30日）**

-   以太坊首个版本，属于测试性发布，供开发者学习与构建DApp
    
-   Gas上限硬编码为5000，后通过“Frontier解冻分叉”提升至3,141,592
    
-   金丝雀合约：发出0/1信号，若多个合约变为1，客户端停止挖矿并提示升级，防止链上长时间中断
    

* * *

### **二、Homestead（2016年3月14日）**

-   第二个主要版本，以太坊从测试阶段进入成熟稳定平台
    

**EIP-2 包含的改动**

-   合约创建Gas费用从21,000提升至53,000（仅交易创建，CREATE指令不变）
    
-   无效高s值签名，防止交易延展性攻击
    
-   合约创建Gas不足时直接失败，不留下空合约
    
-   修改难度调整算法，稳定出块时间
    

**EIP-7 包含的改动**

-   新增 DELEGATECALL 操作码（0xf4）
    
-   功能类似 CALLCODE，但保留父级的发送者与价值
    
-   无额外Gas补贴，Gas管理更可预测
    

**EIP-8 包含的改动**

-   增强 devp2p 网络的前向兼容性
    
-   忽略版本号、hello/ping中的额外列表元素
    
-   静默丢弃未知包类型
    
-   接受新的加密握手编码
    

* * *

### **三、The Merge（2022年9月15日）**

-   通过 EIP-3675 激活，共识机制从 PoW 切换为 PoS
    
-   信标链自2020年12月1日起独立运行并达成共识，经长期验证后成为以太坊的共识层
    
-   Merge 名称来源于执行层网络与共识层网络的合并
    
-   关键变化：淘汰高能耗PoW挖矿、共识与执行逻辑分层、引入独立p2p网络和共识层
    

### **一、以太坊即状态机**

-   状态机由三部分组成：状态、输入、状态转换函数
    
-   以太坊是**基于交易的状态机**：接收交易作为输入，通过状态转换函数进入新状态
    
-   当前状态称为**世界状态**，本质是20字节地址到账户状态的映射
    
-   EVM就是以太坊状态机的**状态转换函数**
    

**账户类型**

-   外部账户：由私钥控制，EVM代码为空
    
-   合约账户：由非空EVM代码控制，该代码即智能合约
    

* * *

### **二、虚拟机范式**

-   直接为不同硬件平台编译软件存在碎片化问题
    
-   解决方案：源代码编译为**字节码**，由**虚拟机**翻译为原生代码执行
    
-   优势：可移植性 + 硬件抽象
    

**EVM特点**

-   字长：32字节
    
-   作为虚拟机器实现，可在任何平台上运行，不依赖底层硬件
    

* * *

### **三、EVM字节码**

-   字节码中的每个字节要么是**操作码**，要么是**操作数**
    
-   操作码有人可读的助记符，即EVM汇编（最低可读形式）
    
-   只有PUSH\*类操作码带有操作数（操作数长度为X字节）
    

* * *

### **四、EVM数据存储位置**

**1\. 栈**

-   最大1024项，每项32字节，每次合约执行后重置
    
-   只能访问顶部16项（DUP和SWAP操作码的限制）
    
-   多数操作码在栈上操作：消耗栈顶数据，压入结果
    
-   栈下溢会导致执行失败
    

**程序计数器**

-   跟踪下一个要执行的操作码在字节码数组中的偏移位置
    
-   JUMP操作码直接设置程序计数器，实现动态控制流
    
-   JUMPDEST标记跳转的目标位置
    

**2\. 内存**

-   字节数组，理论上无限大（2^256字节）
    
-   所有位置初始为零，存储整个程序相关的临时数据
    
-   动态分配，以“页”（每页32字节）为单位扩展
    
-   扩展时会消耗Gas
    
-   MSTORE写入32字节，MSTORE8写入1字节，MLOAD读取
    

**3\. 调用数据**

-   只读的输入数据，通过消息调用或交易传入
    
-   使用CALLDATALOAD读取32字节到栈，或CALLDATACOPY复制到内存
    

**4\. 存储**

-   词寻址的词数组（2^256个槽位，每槽32字节）
    
-   与账户关联，跨交易持久化（属于世界状态）
    
-   只能被所属账户的代码访问
    
-   SSTORE写入，SLOAD读取
    
-   从非零改为零时获得Gas退款
    
-   存储操作非常昂贵
    

* * *

### **五、Gas机制**

-   作为计算资源的“货币”，防止网络DoS攻击
    
-   Gas成本与硬件限制（存储容量、处理能力）挂钩
    
-   交易用ETH支付Gas，Gas耗尽则执行停止（如无限循环）
    
-   由于Gas限制计算为有限步数，EVM被称为**准图灵完备**
    

* * *

### **六、EVM升级**

-   EVM变更较为谨慎，避免破坏现有合约
    
-   主要变更包括：新增操作码、修改现有操作码（不破坏逻辑）
    
-   示例EIP：1153、4788、5000、5656（新增）、6780（禁用SELFDESTRUCT）
    
-   EOF（EVM Object Format）是未来重大升级：为字节码定义格式，使EVM更易理解和处理

26.4.7 今天学习了设计理念章节  
  
**一、设计哲学（六大核心）**

1.  **简洁性**
    
    -   普通程序员应能理解并实现全部规范。
        
    -   模块化和清晰的规范保障了简洁性。
        
2.  **通用性**
    
    -   以太坊“没有内置功能”，只提供图灵完备的EVM（以太坊虚拟机）。
        
    -   开发者可自由构建任何数学上可定义的智能合约或交易类型。
        
3.  **模块化**
    
    -   封装复杂性，子系统内部复杂但对外接口简单。
        
    -   便于升级、调试和复用（如Dagger、Patricia树、SSZ等模块）。
        
4.  **非歧视性**
    
    -   不主动限制或阻止特定用途。
        
    -   收费机制按计算步骤计费，允许无限循环（只要付费）。
        
5.  **敏捷性**
    
    -   通过EIP（以太坊改进提案）持续优化。
        
    -   核心组件（如EVM、地址系统）可谨慎修改以提升扩展性/安全性。
        
6.  **无内置功能**
    
    -   不把高级用例内建到协议中，留给子协议或合约实现（如锁时功能、代币系统）。
        

* * *

### **二、指导原则**

-   **管理复杂性**：尽可能简化协议，复杂部分放入“中间层”（如编译器、存储结构、网络协议）。
    
-   **封装 vs 系统复杂度**：封装复杂度优于系统级相互缠绕的复杂度。
    
-   **自由度**：类似网络中立性，不限制或偏袒特定用途。
    
-   **通用化**：提供低层原语，允许任意组合（如分离“函数调用”和“事件通知”）。
    

* * *

### **三、账户模型 vs UTXO**

-   **UTXO（比特币等）**：余额由未花费交易输出组成，隐私性高但复杂、不利于去中心化交易所。
    
-   **账户模型（以太坊）**：
    
    -   省空间（约30字节 vs 300字节）。
        
    -   完全可互换（无染色币问题）。
        
    -   更简单灵活，支持复杂逻辑。
        
    -   缺点：需nonce防重放，账户无法删除。
        

* * *

### **四、核心数据结构**

| 结构 | 特点 |
| --- | --- |
| Merkle Patricia Trie (MPT) | 确定性、可加密验证，增删改查 O(log n)，但状态过大（1-2TB）。 |
| Verkle树（研究中） | 证明尺寸更小（O(log_k n)），支持无状态客户端，分支因子可权衡算力与带宽。 |
| RLP编码 | 简单确定性序列化，仅支持嵌套数组，不支持整数/布尔等固定类型。 |
| SSZ序列化 | 支持固定+变长类型，可默克尔化，用于Eth2信标链，支持快速索引与轻客户端证明。 |

* * *

### **五、共识与最终性**

-   **Casper FFG**：PoS下的最终性机制，要求至少33%质押ETH被罚没才能回滚区块。
    
-   **LMD-GHOST**：分叉选择规则，基于最新消息驱动的最重子树。
    
-   **Gasper** = Casper FFG + LMD-GHOST，是Eth2的理想化共识抽象。
    

* * *

### **六、网络层：DHT + 非结构化覆盖网**

-   **discv5（Kademlia DHT）**：存储ENR记录，用于发现节点（而非区块）。
    
-   **区块传播**：通过gossipsub（非结构化网络）广播。
    
-   **混合策略**：
    
    -   DHT提供全局、去中心化的启动视图。
        
    -   非结构化网络在高变动（churn）环境下更鲁棒。
        
    -   简单性仍是最大优势。

26.4.6 今天阅读学习了史前历史，就是关于以太坊的起源与背景：  
以太坊的诞生并非偶然，而是植根于多个技术与社会运动的交汇：

1.  **互联网精神**：起源于ARPANET，打破国界，强调开放与连接。
    
2.  **Unix理念**：模块化、简洁、可组合，强调“做好一件事”。
    
3.  **密码学突破**：从默克尔谜题到Diffie-Hellman、RSA，实现了无需信任的安全通信。
    
4.  **自由与开源运动**：Richard Stallman发起GNU，Linus Torvalds创建Linux，强调“自由”而非“免费”。
    
5.  **密码朋克运动**：主张用代码保护隐私，反对政府监控，代表有PGP、Cypherpunk宣言、加密无政府主义。
    
6.  **数字货币尝试**：DigiCash、E-gold、B-money、BitGold均未成功，但为比特币铺路。
    
7.  **比特币诞生**：2008年中本聪提出去中心化电子现金系统，首次实现无中心化信任的共识机制。
    
8.  **以太坊的崛起**：Vitalik Buterin发现比特币局限，与Gavin Wood等人于2015年推出以太坊，目标是构建“世界计算机”，支持通用去中心化应用。
    

以太坊不仅是技术创新，更是自由、反叛、协作精神的延续。

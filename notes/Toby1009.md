## PBS**（Proposer-Builder Separation）设计理念：公平与去中心化**

今天討論到了 PBS 的部分，蠻有趣的

一般來說，驗證者也要負責去建構和驗證，也就是 Builder 和 Proposer 的工作

那會導致中心化問題，為啥會有中心化問題？

因為 MEV 很卷，很卷的話小 builder 收益就不行，收益不行就影響到生存空間，最終只會留下大的。

因此就想出可以透過把 Builder 和 Proposer 的工作分離，讓 Proposer 不需要去擔心這方面的問題，進而增加去中心化程度

其中還有一些好處，就是「抗審查」，審查者無法透過控制 base fee 來審查或排除特定交易

這是因為原本攻擊者可以透過提高 Base fee 導致對方付不起而排除特定交易，而 PBS 則是可以提前預定下個 builder，因此攻擊者無法透過提高 base fee 來審查。

## EVM

-   evm.go：負責的是：要不要執行、誰跟誰、gas 夠不夠、失敗怎麼回滾。就是「進場前的準備」和「出場後的善後」
    
-   interpreter.go：`evm.Run()` 進去之後的東西。那邊有一個大迴圈，每次讀一個 byte、查 JumpTable 找對應的 handler、扣 gas、執行。那個才是 EVM 的心臟。
    
-   instructions.go：每個 opcode 的具體實作在 `instructions.go`。像 `SSTORE` 要處理 EIP-2200 的 gas 計算、`SLOAD` 在 Berlin 之後有 cold/warm access 的區分、`LOG` 要寫 event，每個都有自己的邏輯。
    

完整的 picture 是：

`evm.go`（調度 / 進出場）→ `interpreter.go`（opcode 迴圈）→ `instructions.go`（單一 opcode 實作）→ `StateDB`（狀態讀寫）

## 為什麼「建構區塊」的指令會塞在 fork choice update 裡？

這是 The Merge 之後 CL/EL 分工架構的設計選擇。流程是這樣的：

1.  **CL（共識層）跑 fork choice rule**，決定哪個區塊是 canonical head
    
2.  CL 透過 Engine API 呼叫 `engine_forkchoiceUpdated`，告訴 EL：「head 現在是這個 block hash」
    
3.  **同一個呼叫裡**，CL 可以帶一個可選的 `payloadAttributes` 參數
    
4.  如果帶了 `payloadAttributes`，就是在說：「順便，下一個 slot 輪到我們提議區塊了，請開始建構 payload」
    
5.  EL 收到後開始組裝交易，之後 CL 再用 `engine_getPayload` 把組好的區塊拿回來
    

所以 `forkchoiceUpdated` 其實是一個**多功能的呼叫**，做了兩件事：

-   **必做**：同步 fork choice 狀態（告訴 EL 誰是 head、safe、finalized）
    
-   **可選**：觸發區塊建構（只有當這個 validator 是下一個 proposer 時才帶 `payloadAttributes`）
    

### 為什麼不拆成兩個獨立的 API？

實務上的原因：EL 要開始建構區塊之前，**必須先知道自己該基於哪個 head 來建構**。這兩件事有強依賴——你不可能在還沒確定 head 的情況下就開始打包交易。把它們合在一個呼叫裡保證了原子性：EL 永遠是基於最新的 fork choice 狀態來建構區塊，不會有 race condition。

### 簡單記法

-   `forkchoiceUpdated` = 「更新 head」+（可選）「準備建構下一個區塊」
    
-   `getPayload` = 「把建好的區塊給我」
    
-   `newPayload` = 「驗證別人提議的區塊」
    

fork choice 本身確實是投票選 head 沒錯，但 Engine API 把它跟區塊建構的觸發綁在一起，是工程上的設計，不是概念上 fork choice 的職責變了。

## Gas Limit 決定區塊容量

每個區塊有一個 gas limit（目前大約 36M gas，正在逐步提高），block proposer 把交易塞進去直到累計 gas 逼近上限。這就是區塊的「容量」。

## 最高收益的組合

理論上 proposer 應該自己從 mempool 挑交易、按 priority fee 排序來最大化收益。但實務上 post-Merge 的以太坊，**大多數 proposer 把排序外包給 builder 了**（PBS / MEV-Boost 架構）：

1.  **多個 builder 競爭出價** — 每個 builder 從 mempool（以及私有 orderflow）組出能最大化 MEV + priority fee 的區塊，然後把 `(出價金額, 區塊 header/hash)` 送到 relay。
    
2.  **Relay 的角色** — relay 介於 builder 和 proposer 之間，負責：
    
    -   驗證區塊有效性
        
    -   確保 builder 的出價真的會付給 proposer
        
    -   保管區塊內容，等 proposer 簽名後才揭露完整區塊（這就是「commitment」機制）
        
3.  **Proposer 只看出價，選最高的** — proposer 拿到多個 relay 回傳的 `bid`，選最高出價那個的 header 簽名，再把簽名送回 relay。
    

**關於「Proposer 看不到區塊內容」**

這正是設計的核心 — 稱為 **commit-reveal scheme**：

-   Proposer 簽的是 **block header**（不含完整交易列表）
    
-   簽名完成後，relay 才把完整的區塊內容釋出給網路
    
-   這樣 proposer 無法「先看內容、偷走 MEV、再自己重組區塊」
    

這個機制叫做 **blind signing**，是 MEV-Boost 設計的關鍵 trade-off — proposer 必須「盲目信任」relay 不會做惡（例如 relay 理論上可以在最後一刻不揭露區塊，導致 proposer 錯過出塊機會）。

**目前的問題**

-   MEV-Boost 是 **out-of-protocol**（協議外）的解決方案，依賴 relay 的誠信
    
-   Ethereum roadmap 上有 **ePBS（enshrined PBS）** 的計畫，希望把這個機制寫進協議本身，減少對中心化 relay 的依賴
    

Relay 的誠信問題不是假交易，而是它在 PBS 流程中扮演了一個**被雙方信任的中間人**角色，這個信任是沒有密碼學保證的。

### Relay 到底在做什麼

在 MEV-Boost 的流程裡，relay 夾在 builder 和 proposer 中間：

1.  Builder 組好區塊，把**完整區塊 + 出價**交給 relay
    
2.  Relay **驗證**這個區塊是合法的、出價是真的
    
3.  Relay 只把**區塊 header + 出價**轉給 proposer（隱藏區塊內容）
    
4.  Proposer 選最高出價，簽名 header
    
5.  Relay 看到簽名後，才把**完整區塊內容**公開出去
    

### 信任問題在哪

Relay 同時要對兩邊守信用，但都沒有鏈上機制強制執行：

**對 Proposer 的信任：**

-   Proposer 信任 relay 說的出價是真的——如果 relay 虛報出價（說 builder 出了 5 ETH 但其實只有 1 ETH），proposer 簽了之後拿到的錢比預期少
    
-   Proposer 信任 relay 會在簽名後真的公開完整區塊——如果 relay 扣住區塊不放，proposer 等於簽了一個空 slot，miss 掉提議機會
    

**對 Builder 的信任：**

-   Builder 信任 relay 不會在 proposer 簽名前洩漏區塊內容——如果 relay 把內容偷偷給 proposer 看，proposer 就可以偷走裡面的 MEV 策略，自己重新排序交易（叫做 proposer stealing）
    
-   Builder 信任 relay 真的把自己的區塊轉交給 proposer，而不是偷偷偏袒別的 builder
    

### 為什麼不驗證就好？

區塊本身是可以驗證的，relay 確實會驗證區塊合法性（交易有效、gas 沒超限等）。問題不在區塊內容的正確性，而在**行為層面**的承諾沒辦法用密碼學約束：

-   「不偷看」——沒辦法證明你沒看過資料
    
-   「不偏袒」——沒辦法證明你公平轉發了所有 builder 的出價
    
-   「不扣住區塊」——沒辦法在鏈上強制 relay 必須在某個時間內公開
    

這些都是**行為承諾**，不是**計算承諾**，所以沒辦法用 ZKP 或簽名來解決。

### 實際造成的問題

-   **中心化**：目前只有少數幾個 relay 在運作（BloXroute、Flashbots、Ultra Sound 等），大部分 proposer 都指向同樣幾個 relay，形成單點故障
    
-   **審查風險**：relay 可以選擇不轉發包含特定交易的區塊（OFAC 合規事件就發生過——某些 relay 拒絕轉發包含 Tornado Cash 交易的區塊）
    
-   **活性風險**：relay 掛了，依賴它的 proposer 就只能退回本地建構區塊，可能錯過 MEV 收益
    

### ePBS 想解決什麼

這就是為什麼 **enshrined PBS（ePBS）** 在路線圖上——把 builder 和 proposer 的分離直接寫進協議裡，用鏈上機制取代 relay 的信任角色。Glamsterdam（2026 上半年）有在討論把 ePBS 納入。這樣 relay 這個角色就不需要存在了，信任假設回到協議本身。

### 節點啟動與驗證者操作

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-08-1775655674938-image.png)

啟動後進入正常的 validator 操作循環，這裡有兩條並行路徑：自己出塊 vs. 收到別人的區塊。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-08-1775655692426-image.png)

幾個值得注意的細節：

`engine_forkchoiceUpdated` 這個 RPC 身兼兩職——平常只傳 `forkchoiceState`（更新 head/safeBlock/finalizedBlock），當 validator 被選為 proposer 時額外帶上 `payloadAttributes`（包含 `timestamp`、`prevRandao`、`suggestedFeeRecipient`），EL 就開始非同步地打包交易、建構新區塊，並回傳一個 `payloadId`。

之後 CL 在需要廣播之前，用 `engine_getPayload(payloadId)` 把建好的 payload 取回來，這個設計讓 EL 有充裕時間做 MEV 優化或 PBS（proposer-builder separation）。

### **Engine API** 是由 **EL 開放給 CL** 使用的 API

這個設計是 **The Merge（合併）** 後才引入的，為了讓原本獨立的執行層與共識層能夠互相溝通。兩者之間透過本地的 authenticated HTTP/IPC 連線通訊，並使用 JWT token 做身份驗證，確保安全性。

-   **Engine API** 是一組 JSON-RPC 介面，由 EL 客戶端（如 Geth、Nethermind、Besu 等）實作並暴露。
    
-   **CL 客戶端**（如 Lighthouse、Prysm、Teku 等）透過這個 API 與 EL 溝通，協調區塊的生產與驗證流程。
    
-   主要用途包括：
    
    -   `engine_forkchoiceUpdatedV*` — CL 通知 EL 目前的 fork choice（最新的 head、finalized、safe block），也可以觸發 EL 開始建構新區塊（payload building）。
        
    -   `engine_getPayloadV*` — CL 向 EL 索取已建構好的 payload（新區塊內容）。
        
    -   `engine_newPayloadV*` — CL 將新區塊傳給 EL 執行並驗證。
        

### 為啥不用 grpc 要用 json rpc

以兼容性、易用性來說 json rpc 大勝

以速度來說的話 grpc 贏

-   JSON-RPC：它是文字協議，基於 HTTP/1.1。所有的瀏覽器、JavaScript 引擎都能原生解析 JSON。開發者只需一個簡單的 fetch 請求就能呼叫 eth\_getBalance
    
-   gRPC：基於 HTTP/2，採用 Protocol Buffers (protobuf) 這種二進位格式。瀏覽器對原生 gRPC 的支援較差，通常需要 grpc-web 外掛程式或代理層（如 Envoy）來轉換，這大大增加了前端開發的複雜度。
    

以調適成本來說，json rpc 也是贏

-   JSON-RPC 是人類可讀的。你可以直接用 curl 指令發送一個請求，並肉眼觀察傳回的十六進位資料。
    
-   gRPC 是二進位流。如果你沒有原始的 .proto 定義文件，你根本無法理解傳輸的資料是什麼。對於一個追求「開源、透明、去中心化」的系統，JSON 這種低門檻的互動方式更符合直覺。
    

替換成本也蠻高的。以太坊在 2014-2015 年設計時，gRPC 還非常年輕（Google 2015 年才開源）。

-   當時 JSON-RPC 已經是成熟的標準。隨著以太坊生態的爆發，所有的錢包、交易所、索引器（如 The Graph）都基於 JSON-RPC 建構了基礎設施。
    
-   替換成本極高：如果要從 JSON-RPC 切換到 gRPC，表示整個 Web3 產業的程式碼庫都要重寫。
    

性能需求？

gRPC 的優點在於高並發、低延遲、雙向串流傳輸（非常適合微服務架構內部通訊）。

但對於區塊鏈用戶來說，瓶頸通常不在“網路傳輸速度”，而是在鏈上確認速度（區塊時間 12 秒）。使用者一分鐘可能只會發幾筆交易，JSON-RPC 的額外開銷（Header 大、文字解析慢）在動輒幾十美金的 Gas 費和秒級的確認時間面前，幾乎可以忽略不計。

不過 gRPC 在後端越來越多

-   共識層 (CL)：許多共識客戶端（如 Prysm）內部以及共識層與驗證者（Validator）之間的通信，其實大量使用了 gRPC。因為這裡需要極高的同步速度和效率。
    
-   Engine API：雖然 Engine API 規範定義是 JSON-RPC，但在實際高效能生產的環境中，有些實作會嘗試使用更有效率的序列化方式。
    

### Payload 驗證流程

![](https://epf.wiki/images/el-architecture/payload-validation-routine.png)![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-08-1775658827582-image.png)

`newPayload` 是 CL 視角的最外層，它對 EL 來說就是一個 bool——成功或失敗。`process_execution_payload` 在 CL 內部做輕量的高層驗證（parentHash、timestamp），不涉及 EVM。`notify_payload` 是那條沒有實作的介面邊界，把 payload 丟給 execution engine。真正重的工作都在 EL 的 `stf` 裡面：header 驗證 → loop 執行每筆 tx → 累積 state。任何一步出錯，整個 block 就 invalid，CL 收到 `false`。

### **VerifyHeaders 驗證的欄位與失敗案例**

驗證分兩層：不需要父區塊的靜態檢查，以及需要對比父區塊的轉換規則檢查。

**靜態檢查（單一 block header 就能判斷）**

`extraData` 長度不能超過 32 bytes。`gasUsed` 不能大於 `gasLimit`。`logsBloom` 必須是合法的 2048-bit bloom filter 格式。`blockHash` 必須和 header 的 RLP 雜湊一致。

**轉換規則檢查（需要對比 parent）**

`blockNumber` 必須剛好是 parent + 1，跳號或倒退都 fail。

`timestamp` 必須嚴格大於 parent 的 timestamp，相同或更早都 fail。

`gasLimit` 漂移限制（EIP-1559 之前就有）：每個 block 的 gasLimit 只能在 parent gasLimit 的 ±1/1024 範圍內調整。所以如果 parent 是 30,000,000，這個 block 最多只能調到 30,029,296 或最少 29,970,704。超出就 fail。這是防止 block producer 突然把 gas limit 調爆的護欄。

`baseFee` 計算（EIP-1559）：baseFee 必須根據 parent 的 `gasUsed` 和 `gasTarget`（gasLimit / 2）用公式算出來，不能亂填：

```
if gasUsed > gasTarget:
    baseFee = parentBaseFee + parentBaseFee * (gasUsed - gasTarget) / gasTarget / 8
else:
    baseFee = parentBaseFee - parentBaseFee * (gasTarget - gasUsed) / gasTarget / 8
```

填錯一個 wei 都會 fail。

`parentHash` 必須和 parent block 的 hash 完全吻合。這個看起來很基本，但在 sync 過程中如果拿到孤兒區塊（orphan block），就會在這裡卡住。

**go-ethereum 對應位置**

實際程式碼在 `consensus/beacon/consensus.go` 的 `VerifyHeader()` 和 `core/block_validator.go` 的 `ValidateBody()`。PoS 之後 difficulty 和 uncle 相關的驗證都被移除了，剩下的主要就是上面這些。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-08-1775659077269-image.png)

幾個重點：

-   **PC（Program Counter）** 從 0 開始，每執行一個 opcode 就往前推進。`PUSH32` 這類帶參數的 opcode，PC 會一次跳 33 bytes（1 opcode + 32 data）。
    
-   **gas 分靜態和動態**：`ADD` 固定 3 gas，`SLOAD` 要看 slot 是否是 warm（100）或 cold（2100），`SSTORE` 最複雜，還要看 slot 的 original / current value 狀態來決定退款。
    
-   **REVERT vs out of gas** 的差別在 receipt：REVERT 的 `status = 0` 但 gas 只扣用掉的部分，剩餘退還；out of gas 的 `status = 0` 且 gas 全吃光。兩者都讓 state 回滾到這筆 tx 執行前，但不讓整個 block invalid——tx 本身是 invalid，block 還算 valid。
    
-   `CALL` **opcode** 會遞迴進入另一個 `evm.Call`，形成 call stack，最深 1024 層。
    

### newPayload 函數做了什麼？

`newPayload` 是 CL 視角最薄的那一層——它本身幾乎不做事，就是把 payload 往下傳，然後把 STF 的結果翻譯成 Engine API 規範定義的三種狀態之一。

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-08-1775659181125-image.png)

`newPayload` 實際做的事可以拆成三層：

-   **進 STF 之前的三道快速檢查**，這些不需要跑 EVM，純查記憶體或 db：blockHash 如果已知就直接回 `VALID`（避免重複執行）；parent 不存在就回 `SYNCING`（不是 `INVALID`，因為我只是還沒同步到，不代表 block 本身有問題）；payload 的 `blockHash` 要和 header 欄位重新算一遍的 hash 對得上，withdrawals root 也要吻合。
    
-   **呼叫 STF**，就是上面那張圖的全部流程。
    
-   **把結果翻譯成 Engine API 的回傳格式**：`{ status, latestValidHash, validationError }`。`latestValidHash` 的用途很關鍵——當回傳 `INVALID` 時，這個值告訴 CL「我最後確認合法的 block 是這個 hash」，CL 用它來決定要從哪裡開始 reorg。如果連 parent 都是 invalid，這個值會是 `null`。
    

`SYNCING` 這個狀態值得特別注意：EL 回 `SYNCING` 不是說「block 有問題」，而是說「我現在資料不夠，沒辦法驗證，你等我一下」。CL 收到之後會繼續監控，等 EL 同步完再重新問。

## 設計原理的名詞筆記

### Verkle tree

以太坊目前使用 MPT 來儲存狀態，但目前以太坊狀態很大，儲存和驗證成本很高

另外就是 MPT 每次都要提供路徑上每個節點的雜湊值，資料量大

這些都導致「無狀態客戶端」的實現，無狀態客戶端就是不用儲存完整狀態，僅需要收到帶有 witness 的區塊即可驗證

![Screenshot 2026-04-07 at 2.43.11 PM.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-07-1775544201754-Screenshot_2026-04-07_at_2.43.11_PM.png)

### 為什麼 Verkle Tree 的 Proof 更小？

在 Merkle Tree 中，要證明「葉節點 X 存在」，你必須提供路徑上**每一層所有兄弟節點的雜湊**，讓驗證者能重新算出根雜湊。

在 Verkle Tree 中，因為使用向量承諾（通常是**KZG 多項式承諾**），每個父節點的承諾本身就能直接「承諾」它底下所有子節點的值，驗證者不需要看兄弟節點，只需要沿路徑走下來的節點＋一個額外的 proof。

### 分支因子的取捨

設分支因子為 **k**：

-   樹高為 **O(log\_k n)**，k 越大樹越矮，路徑越短
    
-   但建構時間為 **O(kn)**，k 越大計算成本越高
    

這是頻寬（proof 大小）與計算力之間的取捨。以太坊目前研究傾向使用 **256** 的分支因子。

### 對以太坊的意義：Stateless Clients

Verkle Tree 是實現**無狀態驗證**的關鍵：

-   礦工/驗證者打包區塊時，附上所有被存取狀態的 **witness（證明）**
    
-   其他節點收到區塊後，**不需要自己儲存完整狀態**，直接用 witness 驗證即可
    
-   Witness 大小因為 Verkle Tree 的特性而大幅縮小（估計可從 MB 級降到數十 KB 級）
    
-   這讓**輕客戶端、手機節點、瀏覽器節點**都能安全地驗證以太坊狀態
    

所以就不需要信任任何第三方，可以自己驗證

### MPT

MPT 跟 MPT Root

**MPT Root** 放在區塊頭裡，區塊頭是所有節點都會同步的，所以 Root 是**人人都有、公開的**。

**MPT 本身（完整的樹）** 則是各個全節點**自己存在本地硬碟**上，網路上沒有一個「公告欄」存著完整的樹，你必須自己跑全節點才有。

這就產生了一個有趣的分工：

-   **Root** 是共識的一部分，所有人都認同「這個區塊的狀態根是 0xabc…」
    
-   **樹的內容** 是各自負責存，全節點存完整的，輕客戶端只存 Root
    

所以當你用 MetaMask 查餘額，它背後其實是去問一個 RPC 全節點「給我帳戶 A 的餘額」，然後**信任**那個全節點的回答——因為 MetaMask 預設不會幫你驗證 MPT Proof。

而如果你跑自己的全節點，你有完整的 MPT，不需要問任何人，直接自己查。

### MPT 的信任問題

從上面可以發現 鏈上真正承諾的是 state root，而不是在每次查詢時直接把完整的 MPT 或對應 proof 都提供給使用者。

一般錢包查詢餘額時，通常只是向 RPC 節點取得結果，例如 eth\_getBalance 的回傳值，而不會預設拿到一份 state proof 來自行對照區塊中的 state root 做驗證。

這代表使用者在日常情境下，往往仍需信任 RPC 節點提供的是正確且最新的狀態資料。若 RPC 回傳錯誤或過時資訊，可能導致使用者對自己的資產、nonce、交易狀態或可執行操作做出錯誤判斷。

例如：錢包告訴你餘額不足，讓你誤以為交易失敗，但其實交易已經發生

因此，從信任最小化的角度來看，最理想的方向仍然是：使用者能夠更容易自行驗證狀態，而不是單純依賴第三方節點的回答。

### 但有一個例外：合約裡用 Proof

有一種情況 Proof 會「上鏈」——就是當一個**智慧合約需要驗證某個狀態是否真實**的時候，呼叫者會把 Proof 當作交易的參數傳入合約，合約在鏈上執行驗證邏輯。

這就是一些跨鏈橋、Oracle、或 L2 → L1 狀態驗證的做法：Proof 作為 calldata 上鏈，但它本身不是「被儲存的狀態」，只是「被執行的輸入參數」，用完就沒了。

### LMD-GHOST 跟 Casper FFG 關係

昨天知道 LMD-GHOST 是 fork choice rule，解決的就是在分叉時選擇誰的情況，也就是許則當前的主鏈頭

那看到 Casper FFG 則是負責最終確定性

這時就想知道，這兩者差在哪？

差別就是 LMD-GHOST 雖然可以選擇當前的主鏈頭，但還是有例外可以被推翻，就是有人重組了最長鏈，那這時就會透過 Casper FFG 去搞

這關鍵問題就在於：「票數最多」!= 不能被推翻

舉個例子：

```
A → B → C → D → E  （目前主鏈，票數較多）
A → B → F → G      （分叉鏈）
```

原本因為 CDE 票數最多當主鏈，但這時攻擊者有 40% 驗證者，他把票灌去 FG

而 LMD-GHOST 只看「最新的票」，所以這 40% 的票瞬間從主鏈移走，加上原本就在分叉鏈的票，主鏈可能就被推翻了。

這叫做**鏈重組（reorg）**，在沒有 FFG 的情況下，理論上任何深度的歷史都可能被重組。

這蠻危險的

假設你在區塊 D 收到一筆付款，賣了東西給買家。

LMD-GHOST 告訴你 D 是主鏈的一部分，看起來很安全。

但如果之後鏈重組把 D 幹掉，這筆付款就消失了，你貨也出了，錢也沒了。

那FFG 解決什麼？

FFG 加了一個**數學保證**：一旦 checkpoint 被 finalized，要推翻它就必須讓超過 1/3 的驗證者被 slash（燒掉所有質押的 ETH）。

這不是「票數不夠」的問題，而是**經濟上不可能**——攻擊成本極高，而且攻擊者的錢會被直接銷毀。

所以：

-   **LMD-GHOST** 說「跟這條鏈」，但這個選擇是暫時的
    
-   **FFG** 說「這個點之前的歷史，永遠不會變了」，這個保證是永久的
    

簡單來說 LMD-GHOST 的「票數最多」是動態的，票可以被收回或重投，所以歷史仍然可以被重組。FFG 的 finality 是靠**經濟懲罰綁定**的，不是靠票數，兩者保護的是不同層次的安全性。

### **Gossip Protocol（八卦協議）**

Gossip Protocol 是分散式系統中一種節點間互相傳播訊息的機制，其靈感來自真實世界的「謠言散播」——每個人隨機告訴幾個人，消息就能迅速傳遍整個群體。

核心思想很簡單：每個節點定期隨機挑幾個鄰居，把自己知道的狀態「低語」給對方，對方再轉傳出去，如此反覆。不需要中央協調者，最終整個叢集的狀態就會收斂一致。

**傳播速度**是對數級的。每輪訊息量翻倍（大約），所以 N 個節點只需要 log₂(N) 輪就能全部收斂。100 個節點約 7 輪，1000 個節點約 10 輪，非常快。

**容錯性極強**。節點掛掉、網路丟包都沒關係——因為每條訊息有多條路徑，不依賴任何單一路徑。這是 Gossip 比 broadcast 更適合分散式環境的核心原因。

**最終一致性（Eventual Consistency）**。Gossip 不保證「現在所有人都知道」，但保證「最終所有人都會知道」。這個 trade-off 是它的設計哲學

**實際應用**很廣泛：Cassandra 和 DynamoDB 用它來同步節點狀態、偵測故障；Ethereum 的 devp2p 用它廣播交易和區塊；Consul 和 Serf 用 SWIM 協議（Gossip 的變種）做服務發現；BitTorrent 的 DHT 也有類似的傳播機制。

### DHT

**DHT（Distributed Hash Table，分散式雜湊表）** 是一種讓大量節點在沒有中央伺服器的情況下，共同維護一個巨大鍵值表的機制。核心問題是：**「誰負責存這筆資料？」**——DHT 的答案是用數學距離決定，不是地理距離。

先從最核心的問題開始：在一個有幾百萬個節點的網路裡，「鍵 K 的資料由誰負責存？」這個問題必須有確定性的答案，而且不需要詢問任何中央伺服器。DHT 的解法是**把節點 ID 和資料 key 都對映到同一個數線（ID 空間）上，用距離近的節點來負責**。

核心機制：

**ID 空間與一致性雜湊（Consistent Hashing）**

每個節點和每個 key 都被雜湊到同一個數線（例如 0–2^160 的環）。資料由「在環上順時針方向第一個遇到的節點」負責——這就是 `responsible(key)` 的定義。這樣做的關鍵優勢是：當新節點加入或舊節點離開，只有「相鄰區間」的 key 需要搬遷，其他節點完全不受影響。

**路由問題：O(N) vs O(log N)**

最原始的 DHT 做法，每個節點只認識「下一個節點」，查詢 key 時沿著環一步步傳遞，最壞需要 N 跳。Chord 協議的解法是讓每個節點維護一個 **finger table**——存放「在環上距離 1, 2, 4, 8… 的節點」，每次路由都能把距離減半，從而達到 O(log N) 跳就找到目標節點。

**實際系統的變體**

BitTorrent DHT（Kademlia）用 XOR 距離而非環上距離，好處是路由更對稱、距離的定義更有彈性。Ethereum 的 discv5 節點發現協議也是基於 Kademlia。IPFS 的內容路由同樣建在 Kademlia DHT 之上——你給一個 CID，它會幫你找到存有這份資料的節點。

### DHT 在 Ethereum 裡不是用來找資料，而是用來找人

![image.png](https://raw.githubusercontent.com/IntensiveCoLearning/EPF_Bootcamp/main/assets/Toby1009/images/2026-04-07-1775552594679-image.png)

**第一層：DHT 在 Ethereum 裡解決的是「找人」問題**。區塊只有一份最新的，不需要 content-addressed 查詢；但節點有幾萬個，新節點加入時必須有辦法在沒有任何朋友的情況下找到它們。discv5 存的是 ENR（路由資訊），不是區塊。

**第二層：為什麼不能直接用 gossip 自舉？** f2f / PEX 只有局部視野——你得先有幾個朋友才能認識更多人，這是個先有雞還是先有蛋的問題。Kademlia 提供的是一個全局可查詢的目錄，任何新節點拿到一個 bootstrap node 就能找到整個網路。

**第三層：為什麼找到朋友之後要換成非結構化 overlay？** DHT 的路由表維護在高 churn（節點頻繁進出）的環境下代價很高——每次有節點加入或離開都需要調整。GossipSub 是非結構化的，節點掉線只是少了幾條 gossip 路徑，整體網路完全不受影響，韌性遠勝於維護嚴格路由關係的結構化網路。

這個「結構化 bootstrap → 非結構化 overlay」的模式值得記住，它在 BitTorrent、IPFS、Ethereum 裡反覆出現，是分散式系統裡的經典架構決策。

### 從設計原理中我們可以學到什麼？

-   管理複雜性：API 介面（頂層）和底層 infra 應該盡量簡單，把不可避免的複雜性推進中間層（middleware、serialization、adapter 層）
    
-   **不要把高層業務邏輯硬編碼進核心系統**，讓核心保持通用，讓上層自己組合。
    
-   可驗證性優先於可信任性：MPT Root 的設計哲學是：**不要要求對方值得信任，要讓對方的行為可被驗證。**與其在服務間建立信任關係，不如設計成每個回應都帶有可驗證的 proof 或 checksum，與其依賴部署流程的正確性，不如讓系統在執行時自我驗證狀態的一致性 - 這是 zero-trust architecture 的底層思想
    
-   Finality vs Liveness 的取捨，Casper FFG + LMD-GHOST 的組合解決的是分散式系統的經典難題：**你不能同時擁有完美的一致性和完美的可用性（CAP theorem）**。
    
-   DHT 的啟示：bootstrapping 問題。任何去中心化系統都有冷啟動問題，以太坊用硬編碼的 bootstrap 節點解決。這個「先用中心化解決冷啟動，再過渡到去中心化」的模式在很多系統都適用，不必追求從第一天就完美去中心化。

### 為啥密碼學的突破是「去信任化」的前提？

在公鑰密碼學出現之前，所有加密系統都依賴「共享同一把鑰匙」，這意味著通訊雙方必須**事先透過某個可信管道**傳遞金鑰——信任的問題根本無法避免。

1976 年 Diffie-Hellman 金鑰交換演算法的突破在於：**兩個從未見過面的陌生人，可以透過公開的通訊頻道建立一個只有彼此知道的秘密**，不需要事先信任任何中間人。這在邏輯上第一次實現了「trustless cryptography（去信任化密碼學）」。

後來 RSA 更進一步：你可以公開一把「公鑰」讓任何人加密訊息給你，但只有你的「私鑰」能解密。這讓陌生人之間的安全交易成為可能，最終催生了電子商務、網路銀行，以及比特幣和以太坊中的數位簽章機制。

簡單說：**沒有公鑰密碼學，就沒有辦法在不認識對方的情況下安全交易，去中心化的金融系統也就無從建立。**

### 自由軟體與開源軟體的差異？各自對以太坊的影響？

**自由軟體（Free Software）由 Richard Stallman 倡導，核心是道德與自由的主張**：使用者有權查看、修改、分享軟體的原始碼，這是一種基本人權，不是商業考量。Stallman 說的「free」是「free speech（言論自由）」，不是「free beer（免費啤酒）」。GPL 授權的設計更具有「傳染性」，任何使用 GPL 程式碼的衍生作品也必須開源。

**開源軟體（Open Source Software）則較偏向實用主義**，強調開放原始碼帶來的實際好處（更多人貢獻、品質更高、可商業運用），不特別強調意識形態。Linus Torvalds 的 Linux 雖然採用 GPL，但開源運動整體上更歡迎商業企業參與。

對以太坊的影響：

-   **自由軟體精神**影響了以太坊的設計理念——協議本身應該是中立、開放、任何人都能參與的，不應被任何公司或政府控制
    
-   **開源運動的實踐模式**影響了以太坊的開發文化——像 Linux 一樣透過社群協作、社會共識來推動協議發展，而非由單一組織主導
    

文章中特別提到 Linux「幾乎為基於社會共識的軟體開發奠定了藍圖」，這正是以太坊治理模式的精神來源。

### 比特幣的限制是什麼，以太坊如何解決？

比特幣的核心限制在於**網路本身過於原始**，無法直接支援複雜的應用程式：

當時開發者嘗試在比特幣網路上建構應用，只能依賴「複雜且可擴展性差的變通方案」，因為比特幣的腳本語言設計上只為了處理轉帳交易，功能非常受限。

Vitalik Buterin 看到這個問題，提出了以太坊作為解決方案——一個**可以支援通用化金融應用的平台**。Gavin Wood 後來將這個設計正式化，2015 年以太坊上線，目標是打造「一個建構無國界、自主數位經濟工具的平台」。

更具體地說，以太坊引入了「智能合約（smart contract）」的概念（Nick Szabo 早在 1994 年就提出），讓協議本身可以用程式碼寫成，自動執行而不需要第三方，把比特幣「點對點電子現金」的概念擴展成「可程式化的去中心化世界電腦」。

“National borders are just speed bumps on the information superhighway.” - Timothy May, Cypherpunk.

> 簡單解釋是「國界對資訊流通來說，根本阻擋不了什麼」，暗喻「政府可以在現實世界用邊境、法律來管控人員和物資的流動，但面對網路上的資訊傳遞，這些管制手段只是小小的不便，無法真正阻擋」- **技術（尤其是密碼學）可以繞過國家權力的管控**，保障個人的隱私與自由。這也是後來比特幣和以太坊「無國界金融」願景的思想根源。

> **「平台層創新」** — 它不是終端產品，而是讓無數其他產品得以被建造出來的**地基**。以太坊本身也是乘載各種應用的地基，沒以太坊，也就沒這些應用。

## Architecture

以太坊 Protocol 由兩個 Client 組成，分別是執行層（EL），以及共識層（CL）

-   EL：接收使用者交易 -> 維護 mempool -> 選交易 -> 執行交易 -> 算出執行後的狀態 -> 組出 execution payload（可被放進區塊的執行內容）
    
-   CL：決定這個 slot 誰能提議 block -> 向 EL 取 payload -> 把 payload 放進 beacon block -> 廣播 block -> 讓其他 validator attestation -> 經過 fork choice / finality 推進鏈
    

流程：

> **EL** 是使用者與以太坊互動的主要入口，通常透過 JSON-RPC API 接收交易請求與查詢。
> 
> 在 EL 內部，交易會先經過基本驗證，進入 mempool；若交易被納入候選區塊，則由 EVM 執行，計算狀態變化，形成 execution payload。EL 也會透過自己的 p2p 網路，與其他 EL 節點同步交易與執行區塊資料。
> 
> **CL** 則不負責執行智能合約，而是負責 PoS 共識流程，包括在每個 slot 中決定誰是 proposer、其他 validator 如何 attestation、如何做 fork choice 與 finality。
> 
> **EL 與 CL 之間透過 Engine API 協作**：CL 在提議或驗證區塊時，會向 EL 取得或驗證 execution payload；EL 則提供執行結果，讓 CL 能把這些執行內容納入 beacon block 並推進全網共識。

![](https://epf.wiki/wiki/protocol/img/protocol-overview.png)

核心理解：

-   EL 決定「交易執行後狀態變成啥」，例如：收交易、跑合約、算狀態
    
-   CL 決定「全網接受哪個區塊與狀態」，例如：誰出塊、大家認哪條鏈、什麼時候 final
    
-   Engine API：讓 CL 與 EL 對接
    

### 一些名詞筆記

剛看完還有好多不知道啥意思，所以記錄一下

-   execution payload：一整份 execution block 的內容物，包含所有要放到 block 中的 txs，以及其他資料，包含 parent hash, fee recipient, state root, logs bloom, prev randao, block number, gas limit, gas used, timestamp, extra data, base fee, block hash, transactions, …。所以總之就是組成 block 的資料。
    
-   fork choice：當節點同時看到多個可能的鏈頭時，決定「現在該跟哪一條鏈」的規則。簡單來說就是：「分叉時，咋選主鏈」，以太坊的 fork choice 核心是 LMD-GHOST，可以說是沿著「獲得最多最新投票權重」的分支往下走，最後找到 head，也就是驗證者投票(attestation) -> 節點會看哪個分支獲得較多支持 -> 把獲得最多支持的鏈當成目前 head。這部分是 CL 搞的。
    
-   attestation：驗證者對某個區塊／某條鏈發出的「簽名背書」或「投票」。

### 從架構的角度去拆解

從架構的角度來看的話，會發現以太坊不純粹是資料庫，也不單純是 VM，而是要同時做到

-   全球狀態機複製
    
-   無需信任的執行程式
    
-   節點可獨立驗證
    
-   在不穩定網路下達成共識
    
-   讓經濟激勵和技術約束一起成立
    

所以這是一個「可驗證計算 + 分散式共識 + 對抗環境」的系統。

為啥要拆成 EL / CL？  
可以發現這兩者性質本質上的不同，其實就是在切責任，讓其更乾淨

這邊的 trade-off 會是

-   好處：責任清楚、client 多樣性可以增加、共識與執行複雜度可拆分
    
-   缺點：系統變雙客戶端、CL/EL協調變複雜、Engine API 成關鍵介面
    

模組化與多樣性，我認為會是以太坊更注重的，所以才會這樣抉擇

哪邊可能會是瓶頸？

-   EL 會不會被垃圾訊息塞爆？  
    EL 的 mempool 是開放的，任何人都能廣播交易，理論上可以被大量低價值交易塞滿（spam 攻擊）。防線是 **Gas 費用機制（Pigovian taxation）**：每筆交易都必須付費，spam 的成本會直接內化給攻擊者。EIP-1559 之後還有 base fee 動態調整，網路壅塞時費用自動上升，讓 spam 在經濟上越來越不合算。但沒有完全免疫，極端低費率的交易仍可能短暫佔用 mempool 空間。
    
-   mempool 會不會不ㄧ致？  
    **本來就會不一致，而且這是預期行為。** 每個節點的 mempool 是本地維護的，透過 P2P gossip 傳播，但網路延遲、節點上線時間不同，導致各節點看到的待處理交易集合天生就不一樣。這不是 bug，協定設計上允許這種差異——最終由 proposer 選出一個 tx set 打包進區塊，所有節點只需對**已上鏈的區塊**達成共識，而不是對 mempool 達成共識。
    
-   network delay 會不會導致短暫分叉？  
    **會，而且這是 fork choice rule 存在的原因。** 網路延遲讓不同節點在同一個 slot 可能收到不同的區塊（或同一 slot 出現兩個 proposer 提案）。以太坊用 **LMD-GHOST** 處理這個問題：選擇「累積 attestation 票重最多的子樹」作為正規鏈，而不是最長鏈。短暫分叉會被收斂，但在 finality 達成前都是「可能被重組」的暫態。
    
-   proposer 選到的 tx set 為什麼每個節點都不同？因為 **mempool 本來就不一致**（見第 2 點），再加上：每個 proposer 有自己的本地 mempool、自己的排序邏輯（通常是 gas price 優先），加上可能有 MEV 策略（如 Flashbots）的介入。協定本身**不規定 proposer 必須選哪些交易**，只規定交易必須合法（nonce、餘額正確等）。所以不同 proposer 在不同時間打包，自然產生不同的 tx set。
    
-   fork choice 為什麼不是 finality？  
    **兩個機制解決不同問題。** Fork choice（LMD-GHOST）是即時的「目前我認為哪條鏈最可能是正確的」，但它可以被重組——如果新的 attestation 累積讓另一條分支更重，鏈頭可以改變。**Finality（Casper FFG）** 才是不可逆的保證：當一個 checkpoint 經過兩輪投票（justify + finalize），要逆轉它必須燒掉至少 **33% 的質押 ETH**，這讓攻擊成本高到幾乎不可能。所以 fork choice 是「活性」機制，finality 是「安全性」機制，兩者分工。
    
-   execution payload 為什麼要每個節點重驗？  
    **信任最小化原則。** 即使 proposer 是誠實的，你也不能只相信他說「這個 payload 合法」——你必須自己跑一遍 EVM 確認每筆交易執行結果、state root 是否正確、gas 是否符合。這是以太坊「無需信任（trustless）」設計的核心：不存在任何可信中介，每個節點對自己看到的狀態負責。若允許節點偷懶不驗，惡意 proposer 就能把任意假狀態塞進鏈裡。
    
-   validator 作惡時靠什麼懲罰？  
    靠 **Slashing（懲罰機制）**，這是 Casper FFG 的關鍵設計。作惡行為主要有兩種：**Double voting**（同一個 slot 簽兩個不同區塊）和 **Surround voting**（用新的 attestation 包圍舊的，試圖重寫歷史）。一旦被偵測到並提交證明，該 validator 的質押 ETH 會被部分或全部銷毀，且被強制退出。這讓作惡的「經濟成本」遠大於潛在收益，形成 cryptoeconomic 安全性。
    
-   CL / EL 脫鉤時怎麼辦？  
    這是個嚴重的情境。CL（共識層）和 EL（執行層）透過 **Engine API** 連接，若 API 中斷或兩者不同步：
    
    -   **CL 正常但 EL 掛掉**：CL 仍可繼續 attest，但無法生產有效的 execution payload，區塊會缺乏有效交易，鏈仍推進但無法執行狀態轉換。
        
    -   **EL 正常但 CL 掛掉**：無法達成共識，鏈停止出塊（liveness failure）。
        
    -   **長期脫鉤**：會觸發 **inactivity leak**，離線的 validator 的質押 ETH 會持續被扣減，直到在線的 validator 佔超過 2/3，讓 finality 能夠恢復。
        
    
    這個設計讓即使部分節點下線，網路也能**最終自我修復**，而不是永久停擺。

## **Architecture：**

以太坊是全球共享状态机，现由 Execution Layer（EL）和 Consensus Layer（CL）双层系统共同组成。

简单来说执行层负责“算”，共识层负责“定”；状态树负责“记”，网络层负责“传”；EVM 负责“跑代码”，账户与交易负责“触发状态变化”。

一个完整节点现在必须运行两个客户端：execution client 与 consensus client。execution client 处理交易、执行 EVM、维护世界状态；consensus client 处理区块传播、验证者职责、分叉选择和最终性。二者通过本地认证 RPC，也就是 Engine API 协同。

**执行层（EL）：**

状态转换引擎，监听新的交易，执行交易，检测状态转换是否合法并维护最新的账户状态数据库。execution client 还负责构造 execution payload，并对收到的新块重新执行交易，以验证块中状态根等执行结果是否正确。

它并不决定哪个块最终胜出，也不直接负责共识投票；它做的是：给定某个区块里的交易列表，按照协议规则算出新的状态。

EL 的核心对象有四个：账户、交易、EVM、状态树。账户是状态承载体，交易是状态转换请求，EVM 是执行语义，状态树是压缩与承诺所有账户状态的加密数据结构。

**共识层（CL）：**

共识层是决定哪一个 execution payload 可以被整个网络接受为 canonical chain 的那一层。Merge 之后，Beacon Chain 成为共识层网络，它负责区块 gossip、验证者管理、区块提议与证明、分叉选择和奖励惩罚。它本身不执行智能合约交易；交易执行仍在 EL 里完成。

在 PoS 之中，时间被切成 **slot（12 秒）** 与 **epoch（32 个 slot）**。每个 slot 随机选一个验证者提议区块，其他验证者进行证明；检查点在获得至少三分之二质押 ETH 支持后可达成 finality。

EL 给出执行结果，CL 赋予这个结果社会共识与经济最终性。

**EL CL 如何配合：Engine API**

在此之前大家已经了解了以太坊是双系统协作，共识客户端在自己网络中接收或提议区块，然后通过本地 RPC 把交易包交给执行客户端，执行客户端执行交易并返回执行结果；当节点轮到产块时，共识层调用执行层生成 block payload。

这意味着以太坊协议架构里最重要的一条边界线就是：

“谁负责执行语义” 与 “谁负责链头选择和最终性” 被制度化分离了。

这种分离提升了模块清晰度，也保留了客户端多样性。

**账户模型：**

两类账户 EOA（外部拥有账户）和 Contract Account（合约账户），两者都能持有 ETH 和代币，也都能与合约交互；区别在于前者由私钥控制，后者由部署到链上的代码控制。

这和比特币的 UTXO 思想不同。以太坊不是通过“花费未花费输出”表达状态，而是维护一棵全局状态树，其中每个账户都有自己的 nonce、balance、storageRoot、codeHash 等信息。这个模型更直接地支持“合约拥有持久状态”和“任意账户间调用”。

**交易和调用：**

交易是从外部世界进入协议的标准入口。EOA 发起交易，可以是简单转账、合约创建、或调用已有合约。执行后会产生日志、收据和状态变更。随着协议演进，交易类型也越来越多：传统交易、访问列表交易、动态费用交易、blob 交易，以及 Pectra 引入的 type 4 / EIP-7702 交易（我已经因为 7702 丢失了上千 U 了，大家多注意小心）

有一个要注意的点：

transaction 是从链外签名后提交到链上的顶层请求，message call 是合约执行过程中内部产生的调用。

前者改变区块历史，后者是 EVM 执行路径的一部分。这个区分是理解 gas 传播、回滚、重入、call stack 的前提。

**EVM（以太坊虚拟机）：**

EVM 是以太坊的执行语义核心。官方文档将它描述为 execution client 内嵌的“计算机”；它读取字节码、操作栈/内存/存储，并对每个 opcode 收取 gas。黄皮书仍然是 EVM 的正式规格来源，而以太坊官方的黄皮书教程也明确指出：虽然以太坊在 2022 年切到了 PoS，但 EVM 本体没有因为 Merge 被重写。

三个特性：

确定性：同一输入必须在所有节点得到同一输出

资源计费：每一个代码都会消耗 gas

全网复制粘贴执行：每个全节点都要重放已接受区块里的执行

**状态树：**

以太坊的状态不是简单数据库快照，而是一个 modified Merkle Patricia Trie。官方文档指出，所有账户通过哈希链接在一起，最终可归约为一个存放在区块链中的根哈希；Merkle Patricia Trie 同时具有确定性和可验证性。

也就是说区块头不需要装下整个状态，只需包含 state\_root。任何节点都能用状态根与 Merkle proof 验证某个账户或某段存储的正确性。轻节点也正是靠区块头与证明来进行有限验证。

这句总结不错：**区块链不是在保存所有真相，而是在保存对真相的加密承诺。**

**区块：**

以太坊区块既有执行层字段，也有共识层时序与验证者语境。执行层块里有 parent\_hash、state\_root、transactions\_root、receipts\_root、gas\_limit、gas\_used、base\_fee\_per\_gas、withdrawals 等字段；而在 PoS 语境下，区块提议发生在 slot 中，由验证者发出。

以太坊的“出块时间”现在被固定节奏化了，slot 为 12 秒，但 slot 可能空缺；块的 gas 目标是 3000 万，上限可到 6000 万，即目标值的两倍，这与 EIP-1559 的弹性区块机制有关。（当然最新的以太坊核心开发者讨论有提到未来可能会再降低）

**网络层 P2P：**

执行客户端和共识客户端各自加入自己的 P2P 网络。官方节点架构页面解释了原因：EL 主要 gossip 交易并维护本地 mempool，CL 主要 gossip 区块并推进共识和链增长。

## **Design rationale：**

以太坊是一条通用链，而不是很多条专用链：白皮书里已经写过：如果很多应用都需要自己的区块链、共识机制和安全预算，那系统会高度重复、开发效率低、互操作性差。以太坊的设计目标是让这些应用共享一个可编程底层。

我目前来看，以太链本身需要做的也就是打好地基，做好基础设施，至于应用专用链，我认为是 L2 的责任，L2 基于以太坊的安全，来做应用来做专有。

**为什么是账户模型，而不是沿用 UTXO？**

以太坊要支持**持久**状态的合约对象。合约不是一次性脚本，而是长期存在、可被反复调用、拥有存储槽的状态实体。账户模型天然更适合表达某个地址拥有余额、代码和存储的世界状态。

从工程角度看，账户模型也让合约调用、内部消息传递和共享状态读写更直接。代价是状态管理更复杂、状态膨胀问题更突出、并行执行更难。后面很多性能与状态树改革，本质上都在给这个选择完善。

**为什么需要 gas：**

汽车跑需要燃油，在以太坊上开发需要 gas。工程计量单位，也可以防止一些循环之类的，来攻击网络（当然你要是有钱，随便写）。gas 的设计不仅是经济学问题，也是虚拟机设计问题、DoS 防御问题和资源配额问题。2016 年后的多次升级都在重新定价 opcode，本质是在修正哪些操作被低估了，可能被攻击者利用。

**为什么 EVM 不通用：**

EVM 的首要目标不是极致性能，而是跨实现一致性。所有客户端都必须在同一笔交易上得出完全一致的结果，因此协议更看重规范明确、行为确定、错误语义一致，而不是某个实现的运行速度。黄皮书作为形式化规格的价值就在这里。

因此 EVM 被设计成一个相对保守、容易验证、全网可重复执行的抽象机器。它确实牺牲了很多传统虚拟机中的高性能特性，但换来的是共识友好性。后来的 EOF、precompiles、gas 重定价、blob 交易分流等改动，也多是在“不破坏共识确定性”的前提下渐进优化。

**为什么状态承诺要做成 MPT；**

因为区块链节点必须能在不相信对方数据库的前提下验证状态。MPT 提供两个关键能力：

一是用单个 root 承诺整套状态；

二是为任意状态项提供可验证证明。

但 MPT 也不是完美结构。相关 EIP 已经明确指出，随着多年运行，磁盘访问而不是哈希计算，逐渐成了瓶颈；这也是后续 Verkle、统一二叉树等路线会不断被讨论的原因。也就是说，MPT 是正确性优先时代的方案，而不是永恒最优方案。

**为什么要做双客户端，不沿用 PoW 单客户端：**

承认了“执行正确” 和 “全网承认哪次执行结果” 是不同问题。因此把这两个问题拆开后，协议升级的空间大得多，比如 blob、withdrawals、账户抽象增强、验证者机制变更，都能分别落到各自层上

**为什么要从 PoW 到 PoS：**

到 PoS 能耗大约下降了 99.95%。。PoS 还带来了新的最终性机制、不同的安全模型，以及对后续扩展路线更友好的结构。攻击者需要考虑大量的经济成本了。

**EIP-1559:**

London 引入 EIP-1559 的核心理由，是老的“纯竞价 gas price 模式”在拥堵时用户体验差、价格发现低效。EIP-1559 在协议中引入按块调整的 base fee，其值会随父块 gas 使用情况上下浮动，并且 base fee 会被销毁。同时区块尺寸具备弹性。可以更平滑应对拥堵。

**为什么引入 blob：**

EIP-4844 的摘要说得很直接：它引入一种携带大量数据的新交易格式，这些数据不能被 EVM 执行直接访问，但其 commitment 可访问，而且其格式与未来完整分片兼容。官方时间线进一步说明，这样做显著降低了 L2 rollup 的数据发布成本。（现在看来似乎已经不太对了）

## **Evolution：**

这个部分不详细看了，直接找 AI 生成了：

**阶段 1：愿景与启动（2013–2015）**

2013 年白皮书发布，2014 年黄皮书发布，2014 年进行了以太销售，2015 年主网 Frontier 上线。这个阶段的关键词是：**提出通用链愿景、给出形式化规格、把最小协议扔进现实世界**。

Frontier 是“技术用户版”的以太坊，Frontier thawing 逐步放开限制，目的是让网络从谨慎启动进入可用状态。这说明以太坊起初就采用了非常工程化的部署策略：先保守启动，再释放能力。

**阶段 2：早期危机与生存性修复（2016）**

2016 年最关键的是三件事：**Homestead**、**DAO fork**、**Tangerine Whistle / Spurious Dragon**。Homestead 让协议具备进一步升级能力；DAO fork 回应了治理与社会共识问题；两次 DoS 响应升级则重定价了许多低估资源消耗的操作。

如果说白皮书定义了以太坊的“理想主义”，那么 2016 年定义了它的“现实主义”：链上代码会出错，协议参数会被攻击，社区必须在不中断系统的前提下做出集体选择。

**阶段 3：为扩展与 PoS 铺路（2017–2020）**

**Byzantium** 和 **Constantinople** 是两个重要里程碑。官方时间线指出，Byzantium 降低了区块奖励、延迟了 difficulty bomb、加入了便于 L2 扩展的密码学能力与静态调用；Constantinople 继续降低区块奖励、优化 EVM 成本，并保证链不会在 PoS 实现前被冻结。

**Istanbul** 则继续优化 EVM 成本、增强 DoS 抵抗力、提升基于 SNARK/STARK 的 L2 扩展性能，并改进跨链互操作能力。可以看出，这一阶段协议的主旋律已经很明显：

一边修补执行层成本模型；

一边为未来的扩展与 PoS 迁移准备基础设施。

2020 年是一个被很多初学者忽视、但非常关键的年份：Beacon Chain 启动，PoS 共识层先以并行链的形式上线，为后来的 Merge 做冷启动准备。官方 Merge 页面明确写到，Beacon Chain 于 2020 年 12 月 1 日作为与 Mainnet 并行运行的独立链创建。

**阶段 4：费用市场改革与 Merge 前夜（2021）**

2021 年的 **London** 升级最重要的变化是引入 EIP-1559。它改变了费用市场，引入 base fee、费用燃烧和弹性区块，让 gas 价格更可预测。

这一阶段的意义在于：以太坊已经不再只处理“能不能运行”的问题，而是在处理“如何在大规模使用下更稳定、更可预测地运行”。手续费机制已经变成协议 UX 的一部分。

**阶段 5：The Merge（2022）**

2022 年 9 月 15 日，The Merge 执行完成。官方页面把它定义为原本的 execution layer Mainnet 与新的 proof-of-stake consensus layer Beacon Chain 的结合；从此 PoW 被正式弃用。

这不是“重启一条新链”，而是保留原有链历史、账户、余额与合约状态，把“决定链头的方式”换掉。官方也特别强调：合并后并没有两条链继续并存，而是只有一条 PoS Ethereum。

从协议学习视角看，Merge 的真正意义有三层：

第一，它把现代以太坊的双层架构永久化；

第二，它把安全基础从算力切换到质押资本与最终性机制；

第三，它打开了后续数据可用性和扩展升级的空间。

**阶段 6：提款、Blob、账户增强（2023–2025）**

2023 年 **Shanghai-Capella（Shapella）** 启用了质押提款，把 Beacon Chain 上的退出与提现能力打通到执行层。这个升级意味着“质押 ETH 可以退出”终于完整进入主网协议语义。

2024 年 **Cancun-Deneb（Dencun）** 引入了 EIP-4844 / Proto-Danksharding。官方说明指出，它显著降低了 L2 rollup 的数据存储成本，通过 blob 让数据可短期发布到主网，从而给 L2 用户带来更低费用。

2025 年 **Prague-Electra（Pectra）** 则把演进重心放到三类主体上：普通用户、L2、质押者与节点运营者。EIP-7702 增强 EOA 的可编程性，EIP-7002 与 EIP-7251 改善验证者资金控制与资本效率。

从这个阶段你应该看出协议演进的方向：

**2015–2020：先活下来；**

**2021–2022：重塑手续费与共识；**

**2023–2025：围绕质押、L2 扩展、账户抽象和运维体验做精细化升级**

### **最后结论：**

1.  以太坊不是“更强的比特币”，而是“通用可编程区块链”。
    
2.  白皮书给愿景，黄皮书给形式化规格，EIP 给持续演进机制。
    
3.  以太坊的核心对象是账户，不是 UTXO。
    
4.  交易是外部入口，message call 是内部执行路径。
    
5.  EVM 是全网共同验证的确定性虚拟机。
    
6.  gas 的本质是资源计量与反垃圾保护，不只是“手续费单位”。
    
7.  世界状态通过 modified Merkle Patricia Trie 承诺到 state root。
    
8.  现代以太坊节点必须运行 execution client 与 consensus client。
    
9.  EL 负责执行与状态，CL 负责提议、证明、分叉选择与最终性。
    
10.  EL 与 CL 通过 Engine API 在本地协作。
     
11.  PoS 里的时间是 slot/epoch，而不是靠 PoW 难度自然涌现。
     
12.  Merge 不是重启新链，而是保留历史、替换共识。
     
13.  finality 是 PoS 时代比“最长链”更强的安全概念。
     
14.  EIP-1559 把手续费市场改成 base fee + tip，并燃烧 base fee。
     
15.  blob 交易是把数据可用性资源与普通 EVM 执行资源分层。
     
16.  2016 年的危机深刻塑造了协议升级文化。
     
17.  Byzantium / Constantinople / Istanbul 是“扩展与 PoS 铺路阶段”。
     
18.  Shapella 让质押提款真正落地。
     
19.  Pectra 标志着协议继续向账户抽象增强和验证者体验优化推进。
     
20.  以太坊协议的本质不是“稳定不变”，而是“在严格兼容性约束下不断演进”。

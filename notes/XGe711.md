今日学习打卡笔记

今天的学习重点主要放在 Execution Layer 中的 Client Architecture（客户端架构概述）上。相比前几天学习的 EVM、Transaction、RLP 这些偏底层执行逻辑的内容，今天更像是在站到更高的视角，去理解整个执行层客户端是如何组织起来并协同工作的。

之前学习 Execution Layer 时，更多关注的是“交易是怎么执行的”，而今天开始思考的是“是谁在执行这些交易”。这个问题的答案就是 EL Client，也就是执行层客户端。像 Geth、Nethermind、Besu、Erigon 这些客户端，本质上都是完整的区块链执行系统，它们负责接收交易、执行区块、维护状态数据库、提供 RPC 接口，并与 Consensus Layer 进行交互。

今天最大的收获，是对客户端内部模块之间的关系有了更清晰的认识。以前总是把这些模块分开理解，现在开始能够把它们串联起来，形成完整的数据流。

首先是 P2P Network 模块，它负责节点之间的通信，包括区块同步、交易传播和节点发现。可以理解为客户端的“信息入口”，没有这个模块，节点就无法与整个网络连接。

然后是 TxPool，也就是交易池。它负责接收并暂存待打包交易。所有用户发起的交易，在真正进入区块之前，都会先进入这里等待。TxPool 会根据 gas fee、nonce 等规则进行排序，为后续区块构建提供候选交易。

接着是 Block Builder 和 Block Execution 模块，它们负责从交易池中选择交易，构建新区块，并逐笔调用 EVM 执行交易，最终更新状态。这部分是真正让区块链“动起来”的核心环节。

而 EVM 则是整个执行系统的计算引擎。无论是转账还是智能合约调用，最终都必须通过 EVM 的 opcode 执行。今天再次强化了一个理解：EVM 不负责“决定做什么”，它只负责“严格执行”。

State Database 则是所有执行结果的最终落点。账户余额、nonce、合约代码、storage 等内容都存储在这里。它通过 Merkle Patricia Trie 保证状态可验证，也保证所有节点最终状态一致。

另外，RPC API Layer 和 Engine API 是今天特别重要的两个接口层。RPC 面向用户和开发者，比如钱包、DApp 调用 eth\_sendTransaction、eth\_call 等接口，都是通过这里进入客户端。Engine API 则面向 Consensus Layer，是 Merge 后新增的重要接口，负责 CL 和 EL 的通信，比如 engine\_newPayload、engine\_forkchoiceUpdated 等。

今天一个非常关键的理解是：Execution Layer Client 并不是一个单独的程序，而是一整套模块协同工作的系统。完整的数据流应该是：

P2P 接收交易  
→ TxPool 暂存交易  
→ Block Builder 构建区块  
→ EVM 执行交易  
→ State DB 更新状态  
→ Engine API 与 CL 同步结果

当这条链路能够在脑海里完整跑通时，对 Execution Layer 的理解就会提升一个层级。

今天的学习让我意识到，之前学的 EVM、Transaction、RLP 更像是在研究“发动机内部结构”，而今天的 Client Architecture 则是在看整辆车是如何运转的。只有把两者结合起来，才能真正理解以太坊执行层。

不过也必须承认，目前的理解仍然偏理论。真正困难的部分其实在源码层面，比如 Geth 中这些模块是如何具体实现的，它们之间的数据是怎么传递的，哪些地方涉及性能优化和状态管理。这些内容才是真正的工程核心。

下一步的学习重点，应该逐渐从“概念架构图”走向“客户端源码阅读”。哪怕只是先看清楚 Geth 中 TxPool、StateDB、EVM 的代码入口，也会比单纯背概念更有价值。

总结一句话：

今天不是在学某一个知识点，而是在建立对整个 Execution Layer Client 的系统认知。只有理解系统，后面的深入学习才有真正的落点。

今天的学习整体是在把以太坊执行层的知识进一步“打通”，不再只是分散地理解几个概念，而是开始把它们串成一条完整的逻辑链。

首先在理解层面，今天最大的进展是对 Execution Layer 的定位更加清晰了。它本质上就是整个以太坊的“执行核心”，所有交易、合约调用、状态变化，最终都要落在这一层完成。相比之前零散地记 EVM、Transaction、RLP，现在开始能把它们看成一个完整系统中的不同环节。

围绕这个核心，今天重点梳理了三个关键部分。

第一个是 EVM。现在可以更明确地理解，EVM 并不是一个“黑盒”，而是一个结构非常清晰的执行模型。它以栈为中心，通过 opcode 一步步执行逻辑，同时用 gas 机制约束资源消耗。stack、memory、storage 三者的分工也更清楚了：一个负责计算，一个负责临时数据，一个负责链上持久状态。理解这一点之后，再看智能合约执行，就不再抽象，而是一个可以拆解的过程。

第二个是 Transaction。今天的重点不再是字段本身，而是生命周期。交易从创建、签名、广播，到进入 txpool，再到被打包执行，最后完成状态更新，这一整条路径已经比较清楚。尤其是一个认知上的提升：交易不是“数据”，而是“触发状态变化的过程”。Execution Layer 的本质，就是不断处理这些过程。

第三个是 RLP 编码。今天的理解比之前更落地了一些。RLP 的关键不在规则本身，而在它解决的问题：让所有节点以完全一致的方式编码数据。只有编码一致，hash 才一致，共识才成立。从这个角度看，RLP 是 Execution Layer 非常底层但又非常关键的一环。它把“结构”变成“字节”，让网络可以正确传递和验证数据。

把这三部分连起来，其实就是一条完整链路：  
用户构造交易 → 交易被 RLP 编码 → 进入网络传播 → 被节点接收 → 进入执行环境（EVM）→ 触发状态变化。

当这条链路能在脑子里跑通时，对 Execution Layer 的理解就不再是碎片，而是系统。

今天最大的收获，其实不是多学了多少内容，而是“连接能力”的提升——开始能把概念之间的关系建立起来。这一点很关键，因为后面的学习（比如客户端架构、Engine API、状态树）都会依赖这种系统化理解。

RLP Serialization 学习笔记（RLP 编码规则）

一、RLP 的基本概念  
RLP（Recursive Length Prefix）是一种用于以太坊的数据编码规则，主要用于将复杂数据结构序列化为字节数组，方便在网络中传输和存储。Execution Layer 中的交易、区块头、账户数据等，都会使用 RLP 编码。

RLP 的设计目标非常简单：  
只负责编码结构，不负责数据类型解释。  
也就是说，RLP 只关心“字节”和“列表”，而不关心整数、字符串或其他高级类型。

二、RLP 支持的数据类型  
RLP 只支持两种基本类型：

第一种是字节数组（String / Byte Array）  
第二种是列表（List）

所有复杂结构最终都可以转换为这两种类型。例如：

整数会被转换为字节数组  
交易是字段列表  
区块头是字段列表  
嵌套结构是列表中包含列表

三、RLP 编码的核心规则  
RLP 编码的核心思想是：  
在数据前加上长度前缀，然后拼接数据内容。

编码时分为以下几种情况：

1.  单字节编码规则  
    如果一个字节的值在 0x00 到 0x7f 之间，那么该字节本身就是编码结果，不需要额外前缀。
    

例如：  
0x05 编码后仍然是 0x05

2.  短字符串编码规则  
    如果字符串长度在 0 到 55 字节之间：
    

编码方式为：  
0x80 + 字符串长度 + 字符串内容

例如字符串长度为 3：

编码前缀 = 0x80 + 3 = 0x83  
最终编码 = 0x83 + 数据

3.  长字符串编码规则  
    如果字符串长度大于 55 字节：
    

编码方式为：

0xb7 + 长度字段长度

-   长度字段
    
-   字符串内容
    

这里长度字段本身也需要用字节表示。

4.  短列表编码规则  
    如果列表总长度小于等于 55 字节：
    

编码方式为：

0xc0 + 列表长度

-   所有元素编码拼接
    

5.  长列表编码规则  
    如果列表长度大于 55 字节：
    

编码方式为：

0xf7 + 长度字段长度

-   长度字段
    
-   列表内容
    

四、RLP 编码示例

示例一：编码字符串 "dog"

"d" = 0x64  
"o" = 0x6f  
"g" = 0x67

字符串长度 = 3

编码前缀 = 0x80 + 3 = 0x83

最终编码：

0x83 64 6f 67

示例二：编码列表 \["cat", "dog"\]

"cat" 编码：

长度 3 → 0x83 63 61 74

"dog" 编码：

长度 3 → 0x83 64 6f 67

拼接内容长度 = 8

列表前缀 = 0xc0 + 8 = 0xc8

最终编码：

0xc8  
83 63 61 74  
83 64 6f 67

五、RLP 在以太坊中的应用

RLP 在 Execution Layer 中使用非常广泛，包括：

交易编码  
区块头编码  
账户状态编码  
网络消息编码  
P2P 通信数据

例如交易在发送前会先进行 RLP 编码，然后再计算 hash。

六、RLP 与交易结构关系

一笔交易在编码时：

会把以下字段组成列表：

nonce  
gas price  
gas limit  
to  
value  
data  
v  
r  
s

然后将整个列表进行 RLP 编码，得到最终字节序列，再进行签名或哈希计算。

七、RLP 的设计特点

第一，结构简单，易于实现  
第二，编码结果紧凑，节省空间  
第三，支持嵌套结构  
第四，不依赖数据类型  
第五，适合区块链确定性编码

RLP 的编码结果在所有节点中必须一致，否则会导致交易 hash 不同，从而破坏共识。

八、RLP 与 ABI 编码的区别

RLP 主要用于：

网络传输  
区块数据  
交易编码

ABI 编码主要用于：

合约函数调用参数  
返回值编码

简单理解：

RLP 用于区块链底层数据结构  
ABI 用于智能合约接口调用

九、学习总结

RLP 是以太坊 Execution Layer 的基础编码方式，用于将数据结构转换为字节序列。  
它只处理两种数据类型：字节数组和列表。  
编码规则基于“长度前缀 + 内容”。  
短数据与长数据使用不同前缀范围。

理解 RLP 的关键是掌握三点：  
第一，字符串与列表两种类型  
第二，长度前缀编码规则  
第三，嵌套结构递归编码

Transaction Anatomy 学习笔记（交易结构与生命周期）

一、交易的基本概念  
Transaction（交易）是用户与区块链交互的最基本单位。在 Execution Layer 中，所有状态变化都由交易触发，例如转账、部署合约、调用合约等。每一笔交易都包含一组固定字段，并遵循严格的生命周期，从创建到最终确认。

简单理解：  
交易 = 用户发起的一次状态变更请求

二、交易的主要类型  
在执行层中常见交易类型包括三种：

第一种是普通转账交易，用于账户之间转移 ETH。  
第二种是合约部署交易，当 to 字段为空时表示创建智能合约。  
第三种是合约调用交易，用于调用已有智能合约函数。

三、交易结构（Transaction Fields）  
一笔完整的交易通常包含以下核心字段：

1.  nonce  
    表示发送账户的交易序号，用于防止重放攻击，同时保证交易执行顺序。每发送一笔交易，nonce 增加 1。
    
2.  gas limit  
    表示交易最多可消耗的 gas 数量，限制执行复杂度。
    
3.  gas price / max fee  
    用户愿意支付的 gas 价格。在 EIP-1559 之后通常包含 maxFeePerGas 与 maxPriorityFeePerGas。
    
4.  to  
    交易接收地址。如果为空，表示部署合约。
    
5.  value  
    转账金额，以 wei 为单位。
    
6.  data  
    交易附带的数据。调用合约时，这里存放函数选择器和参数。
    
7.  chain id  
    用于区分不同链，防止跨链重放攻击。
    
8.  signature（v, r, s）  
    交易签名字段，用于验证交易发送者身份。
    

四、交易生命周期（Transaction Lifecycle）

交易从创建到最终确认通常经历以下阶段：

第一阶段：交易创建  
用户在钱包中构造交易，填写接收地址、金额、gas 等信息，并使用私钥进行签名。

第二阶段：交易广播  
签名后的交易被发送到节点，并通过 P2P 网络传播到其他节点。

第三阶段：进入交易池（TxPool / Mempool）  
节点验证交易合法性，包括签名验证、nonce 检查、余额检查。合法交易进入 txpool 等待打包。

第四阶段：区块打包  
区块构建者从 txpool 中选择交易（通常按 gas 费用排序），将交易加入区块。

第五阶段：交易执行  
Execution Layer 使用 EVM 执行交易，更新状态并计算 gas 消耗。

第六阶段：区块确认  
区块被共识层确认后，交易正式生效并不可逆。

完整流程可以概括为：  
创建 → 签名 → 广播 → 进入 txpool → 打包进区块 → 执行 → 确认

五、交易验证过程  
节点在接收到交易后会执行以下检查：

检查签名是否正确  
检查 nonce 是否匹配  
检查账户余额是否足够支付 value 和 gas  
检查 gas limit 是否合理  
检查交易格式是否正确

只有通过这些验证，交易才会被接受进入 txpool。

六、交易执行结果  
交易执行完成后会产生以下结果：

状态（成功或失败）  
gas used（实际消耗 gas）  
logs（事件日志）  
返回数据（return data）  
收据（receipt）

即使交易执行失败，gas 仍然会被扣除，因为执行过程中已经消耗计算资源。

七、交易排序规则  
区块构建者通常根据交易费用进行排序：

优先打包支付更高 priority fee 的交易  
nonce 必须连续执行  
同一账户交易按 nonce 顺序执行

因此，较高的 gas 费用可以加快交易确认速度。

八、交易与状态变化  
每笔交易都会触发状态转换：

账户余额变化  
nonce 增加  
合约存储变化  
日志生成

Execution Layer 的核心就是不断执行交易并更新状态。

九、学习总结  
Transaction 是 Execution Layer 的输入数据，是状态变化的起点。  
交易由多个字段组成，其中 nonce、gas、to、value、data 是最关键部分。  
交易生命周期从创建开始，经过广播、验证、打包、执行，最终被区块确认。

理解 Transaction Anatomy 的关键在于三点：  
第一，交易字段的作用  
第二，交易从发送到确认的生命周期  
第三，交易执行如何改变区块链状态

EVM 学习笔记（Ethereum Virtual Machine）

一、EVM 的基本概念  
EVM（Ethereum Virtual Machine，以太坊虚拟机）是 Execution Layer 的核心执行环境，负责运行智能合约代码并完成状态更新。可以把它理解为以太坊的“计算机”，所有节点都会运行同一个 EVM，并执行相同的指令，从而保证区块链状态一致。

EVM 的主要职责包括：执行智能合约、处理交易中的数据、计算 gas 消耗、更新账户状态以及返回执行结果。任何部署在以太坊上的智能合约最终都会被编译成字节码，然后由 EVM 逐条执行。

二、EVM 的执行特点  
EVM 是一个确定性执行环境，也就是说在相同输入下，所有节点必须得到相同输出。这一点对区块链共识非常关键。  
它是基于栈（Stack-based）的虚拟机，而不是寄存器型虚拟机，因此所有运算都围绕栈进行。  
EVM 还具备沙盒执行环境的特点，智能合约之间相互隔离，无法直接访问外部系统，从而保证安全性。

三、EVM 的核心组件  
EVM 在执行过程中主要依赖五个核心部分：

1.  Stack（栈）  
    栈是一个后进先出的数据结构，最大深度为 1024。所有算术运算、逻辑运算都从栈中取值，并把结果再压入栈中。例如加法操作会从栈顶取出两个数，相加后再放回栈顶。
    
2.  Memory（内存）  
    Memory 是临时存储区域，在交易执行过程中使用。它是按字节寻址的，可扩展，但执行结束后会被清空，因此不用于永久数据存储。
    
3.  Storage（存储）  
    Storage 是合约的永久存储区域，写入 storage 的数据会被记录在区块链状态中。访问 storage 的 gas 成本较高，因为需要修改链上状态。
    
4.  Program Counter（程序计数器）  
    用于记录当前执行到哪一条指令。每执行一条 opcode，程序计数器都会移动到下一条。
    
5.  Gas Counter（Gas 计数器）  
    记录剩余 gas，每执行一条指令都会消耗 gas，当 gas 用尽时执行会终止并回滚状态。
    

四、EVM 的执行流程  
当一笔交易调用智能合约时，EVM 会按照以下步骤执行：

首先加载合约字节码，然后初始化执行环境，包括栈、内存和 gas。接着 EVM 从第一条 opcode 开始逐条执行。每条指令执行时都会消耗对应的 gas，并可能修改栈、内存或 storage。执行完成后，EVM 返回结果并更新状态。如果执行过程中 gas 不足或发生错误，则回滚状态但消耗的 gas 不退还。

五、Opcode（操作码）  
EVM 通过 opcode 指令运行，常见的指令类型包括：

算术运算指令，如 ADD、SUB、MUL  
比较运算指令，如 LT、GT、EQ  
栈操作指令，如 PUSH、POP、DUP  
内存操作指令，如 MLOAD、MSTORE  
存储操作指令，如 SLOAD、SSTORE  
控制流指令，如 JUMP、JUMPI  
合约调用指令，如 CALL、DELEGATECALL

这些 opcode 构成了 EVM 的基础指令集。

六、Gas 机制在 EVM 中的作用  
Gas 是 EVM 的资源计费机制。每条 opcode 都有固定 gas 消耗，例如简单算术运算消耗较少，而写入 storage 消耗较多。  
Gas 机制的作用包括防止无限循环、限制计算资源、防止恶意攻击以及为矿工或验证者提供激励。  
如果执行过程中 gas 用完，EVM 会立即停止执行，并回滚所有状态修改。

七、EVM 与智能合约执行  
Solidity 等高级语言编写的合约会先被编译成 EVM bytecode，然后部署到区块链。用户调用合约时，本质上就是触发 EVM 执行这段字节码。  
执行过程完全由 EVM 控制，外部无法干预，从而保证合约逻辑不可篡改。

八、EVM 的重要特性  
第一，确定性执行，保证所有节点状态一致。  
第二，图灵完备，理论上可执行任意复杂计算。  
第三，隔离环境，确保安全性。  
第四，gas 计费，控制资源消耗。  
第五，基于栈结构，执行逻辑简单统一。

九、学习总结  
EVM 是 Execution Layer 的核心计算引擎，它负责执行所有交易和智能合约。  
EVM 的运行依赖 Stack、Memory、Storage 三种数据结构，并通过 opcode 指令逐条执行。  
Gas 机制贯穿整个执行过程，保证计算资源可控。

理解 EVM 的关键是抓住三个核心点：  
一是栈式执行模型  
二是 opcode 指令执行  
三是 gas 资源控制

掌握这三点，就基本理解了 EVM 的工作原理。

# Client Architecture 学习笔记

## （Execution Layer 客户端架构概述）

## 一、什么是 Client Architecture

Client Architecture（客户端架构）指的是 **Execution Layer 客户端内部各模块的组成方式及其协作关系**。

Execution Layer 客户端本质上是一个完整的区块链执行系统，负责：

-   接收交易
    
-   执行区块
    
-   维护状态
    
-   提供 RPC 接口
    
-   与 Consensus Layer 通信
    

常见 EL 客户端（仅作为理解背景）包括：

-   Geth
    
-   Nethermind
    
-   Besu
    
-   Erigon
    

虽然实现不同，但模块结构基本一致。

* * *

## 二、EL 客户端整体模块总览

Execution Layer 客户端通常由以下核心模块组成：

```
+--------------------------------------------------+
|                  RPC API Layer                   |
+--------------------------------------------------+
|                  TxPool (Mempool)                |
+--------------------------------------------------+
|                  Block Execution                 |
+--------------------------------------------------+
|                      EVM                         |
+--------------------------------------------------+
|                  State Database                  |
+--------------------------------------------------+
|                    P2P Network                   |
+--------------------------------------------------+
|                Engine API (CL 交互)              |
+--------------------------------------------------+
```

* * *

## 三、核心模块详解

### 1\. P2P Network（点对点网络模块）

作用：

-   与其他节点通信
    
-   同步区块
    
-   同步交易
    
-   广播新区块
    

主要功能：

-   区块同步
    
-   交易传播
    
-   节点发现
    

这是客户端的数据输入来源。

* * *

### 2\. TxPool（交易池 / Mempool）

作用：

-   存储待打包交易
    
-   按 gas price 排序
    
-   提供区块构建候选交易
    

主要流程：

```
接收交易 → 验证 → 放入 txpool → 排序 → 提供给区块构建
```

特点：

-   非共识数据
    
-   每个节点不同
    
-   动态变化
    

* * *

### 3\. Block Builder / Block Execution（区块构建与执行）

作用：

-   从 TxPool 选择交易
    
-   构建区块
    
-   执行交易
    
-   更新状态
    

执行流程：

```
for tx in block:
    执行交易
    更新状态
```

输出：

-   state root
    
-   receipts
    
-   logs
    

* * *

### 4\. EVM（Ethereum Virtual Machine）

作用：

-   执行智能合约代码
    
-   处理 opcode
    
-   计算 gas 消耗
    

EVM 是 Execution Layer 的核心计算引擎。

执行环境包含：

-   Stack
    
-   Memory
    
-   Storage
    
-   Gas counter
    
-   Program counter
    

* * *

### 5\. State Database（状态数据库）

作用：

-   存储账户状态
    
-   存储合约存储数据
    
-   维护 state trie
    

存储内容：

-   balance
    
-   nonce
    
-   code
    
-   storage
    

特点：

-   使用 Merkle Patricia Trie
    
-   可验证
    
-   支持回滚
    

* * *

### 6\. RPC API Layer

作用：

-   向外提供接口
    
-   钱包调用
    
-   DApp 调用
    
-   开发者查询
    

常见接口：

-   eth\_sendTransaction
    
-   eth\_call
    
-   eth\_getBalance
    
-   eth\_getBlockByNumber
    

这是用户与客户端交互入口。

* * *

### 7\. Engine API（与 Consensus Layer 通信）

作用：

-   CL 与 EL 交互
    
-   接收新区块执行请求
    
-   返回 execution payload
    

主要方法：

-   engine\_newPayload
    
-   engine\_forkchoiceUpdated
    
-   engine\_getPayload
    

这是 Merge 后新增的重要模块。

* * *

## 四、模块之间的工作流程

完整数据流：

```
P2P 网络
   ↓
TxPool
   ↓
Block Builder
   ↓
EVM 执行
   ↓
State DB 更新
   ↓
生成 Execution Payload
   ↓
Engine API → Consensus Layer
```

RPC 调用路径：

```
User / Wallet
     ↓
   RPC API
     ↓
   TxPool
```

* * *

## 五、客户端架构核心特点

1.  模块化设计
    
2.  执行与存储分离
    
3.  网络与执行解耦
    
4.  支持并发处理
    
5.  可替换实现
    

* * *

## 六、考试重点

需要重点掌握：

3.  EVM 在架构中的位置
    
4.  State DB 的作用
    
5.  Engine API 的作用
    
6.  模块之间的数据流
    

* * *

## 七、个人学习总结

Execution Layer Client 可以理解为五大核心系统：

1.  网络系统（P2P）
    
2.  交易系统（TxPool）
    
3.  执行系统（EVM）
    
4.  状态系统（State DB）

它们共同完成：

接收交易 → 执行交易 → 更新状态 → 生成区块 → 与 CL 协作

一句话总结：

Execution Layer Client =  
网络模块 + 交易池 + EVM + 状态数据库 + 接口层

# Execution Layer (EL) 学习笔记

## 一、Execution Layer (EL) 基础概念

Execution Layer（执行层，简称 EL）是以太坊架构中负责**交易处理与状态执行**的核心部分。它主要完成以下职责：

1.  **交易处理**
    
    -   接收用户发送的交易（Transaction）
        
    -   验证交易格式与签名
        
    -   按顺序执行交易
        
2.  **状态更新**
    
    -   维护账户状态（Account State）
        
    -   更新余额、nonce、存储等信息
        
    -   生成新的 world state
        
3.  **智能合约执行**
    
    -   运行 EVM（Ethereum Virtual Machine）
        
    -   处理合约调用与部署
        
    -   计算 gas 消耗
        
4.  **区块执行**
    
    -   按区块内交易顺序执行
        
    -   计算新的 state root
        
    -   返回 execution payload
        

简而言之：  
Execution Layer = **交易执行 + 状态更新 + EVM 计算**

* * *

## 二、Execution Layer 与 Consensus Layer 的关系

在以太坊合并（The Merge）之后：

-   **Consensus Layer（CL）**  
    负责区块共识、验证者、出块逻辑
    
-   **Execution Layer（EL）**  
    负责执行交易、维护状态
    

两者通过 **Engine API** 进行通信：

流程：

1.  CL 提议新区块
    
2.  CL 调用 EL 执行 payload
    
3.  EL 返回 execution 结果
    
4.  CL 验证并最终确认区块
    

* * *

## 三、EL Specs（执行层核心规范）

EL Specs 是 Execution Layer 的正式技术规范，定义了执行层必须遵守的规则，包括：

### 1\. Block 执行规范

区块执行流程：

1.  验证父区块存在
    
2.  验证时间戳
    
3.  执行区块内所有交易
    
4.  更新状态树
    
5.  计算 receipts root
    
6.  计算 logs bloom
    
7.  输出 execution payload
    

执行顺序：

```
for tx in block.transactions:
    validate(tx)
    execute(tx)
    update_state()
```

* * *

### 2\. Transaction 执行规则

交易执行包括：

-   nonce 检查
    
-   gas 限制检查
    
-   余额检查
    
-   EVM 执行
    
-   gas 扣除
    
-   状态提交
    

交易执行结果包含：

-   status（成功 / 失败）
    
-   gas used
    
-   logs
    
-   return data
    

* * *

### 3\. 状态管理（State Transition）

EL 的核心就是 **状态转换函数**

State Transition：

```
State' = ApplyTransaction(State, Tx)
```

整个区块执行：

```
State_n = ApplyBlock(State_0, Block)
```

关键点：

-   状态不可变（通过 Merkle Patricia Trie）
    
-   每次执行生成新的 state root
    
-   所有节点可验证一致性
    

* * *

### 4\. Gas 机制

EL Specs 定义了：

-   gas limit
    
-   gas used
    
-   base fee（EIP-1559）
    
-   priority fee
    

Gas 执行顺序：

1.  预扣 gas
    
2.  执行指令消耗 gas
    
3.  未用 gas 返还
    
4.  失败仍扣 gas
    

目的：

-   防止无限循环
    
-   防止网络滥用
    
-   资源定价机制
    

* * *

### 5\. EVM 执行环境

Execution Layer 运行在 EVM 中：

EVM 包含：

-   Stack
    
-   Memory
    
-   Storage
    
-   Program Counter
    
-   Gas Counter
    

执行模型：

```
while gas > 0:
    opcode = fetch()
    execute(opcode)
```

* * *

### 6\. Execution Payload

EL 向 CL 返回的执行结果称为：

Execution Payload

包含：

-   block hash
    
-   parent hash
    
-   state root
    
-   receipts root
    
-   logs bloom
    
-   gas used
    
-   transactions
    
-   withdrawals
    

该 payload 被 CL 用于区块共识验证。

* * *

## 四、Execution Layer 工作流程总结

完整流程：

1.  用户发送交易
    
2.  交易进入 mempool
    
3.  Builder/Proposer 组装区块
    
4.  CL 提议区块
    
5.  EL 执行区块交易
    
6.  生成新的 state root
    
7.  返回 execution payload
    
8.  CL 完成共识
    

* * *

## 五、EL 关键数据结构

### 1\. Account

包含：

-   nonce
    
-   balance
    
-   storageRoot
    
-   codeHash
    

### 2\. Transaction

包含：

-   nonce
    
-   gas limit
    
-   to
    
-   value
    
-   data
    
-   signature
    

### 3\. Block

包含：

-   header
    
-   transactions
    
-   withdrawals
    

* * *

## 六、EL Specs 核心目标

Execution Layer 规范主要解决：

-   所有节点执行一致性
    
-   确保状态可验证
    
-   标准化区块执行
    
-   保证网络安全
    
-   提供 EVM 执行环境
    

* * *

## 七、重点理解（考试/作业常考）

必须掌握：

1.  EL 与 CL 的分工
    
2.  State Transition 概念
    
3.  Transaction 执行流程
    
4.  Gas 机制作用
    
5.  Execution Payload 内容
    
6.  Engine API 作用
    

* * *

## 八、个人学习总结

Execution Layer 是以太坊的“计算引擎”，  
Consensus Layer 是“决策系统”。

EL 负责：

-   执行
    
-   计算
    
-   更新状态
    

EL Specs 则是：

-   定义执行规则
    
-   确保所有客户端行为一致
    
-   保证区块执行可验证
    

理解 EL 的关键就是抓住一句话：

**Execution Layer = 状态转换函数 + EVM 执行环境**

# 📚 打卡笔记：Ethereum The Protocol — Execution Layer (EL) Design Rationale

## 一、EL 的核心定位

Execution Layer 是负责：

-   处理交易
    
-   维护账户状态
    
-   执行智能合约
    
-   产生区块执行结果
    

在合并（The Merge）之后架构变为：

-   CL：Consensus Layer → 负责共识
    
-   EL：Execution Layer → 负责执行
    

👉 设计目标：

1.  保证确定性执行
    
2.  支持复杂智能合约
    
3.  高安全性
    
4.  历史兼容性
    
5.  可扩展性
    

* * *

# 二、账户模型设计（Account Model）

### 设计选择

Ethereum 采用：

-   Account-based model  
    而不是 UTXO（像 Bitcoin）
    

### Design Rationale（设计取舍原因）

优点：

-   更适合复杂合约调用
    
-   状态存储更直观
    
-   支持内部交易
    

代价：

-   状态膨胀问题
    
-   并行执行难度更高
    

👉 核心取舍：  
牺牲并行性 → 换取智能合约灵活性

* * *

# 三、EVM 执行环境

核心组件：  
Ethereum Virtual Machine

### 为什么要设计 EVM？

目标：

-   所有节点执行同一逻辑
    
-   跨平台确定性
    
-   安全沙箱环境
    

### Design Rationale

1.  使用 stack-based VM  
    优点：
    

-   实现简单
    
-   状态可预测  
    缺点：
    
-   执行效率低于 register-based VM
    

👉 取舍：  
牺牲性能 → 换取安全与确定性

* * *

# 四、Gas 机制设计

Gas = 执行成本计量单位

### 为什么必须设计 Gas？

防止：

-   无限循环攻击
    
-   资源滥用
    
-   DOS 攻击
    

### Design Rationale

1.  每个 opcode 定价  
    → 精细资源控制
    
2.  交易预付 gas  
    → 防止恶意交易
    
3.  未用 gas 返还  
    → 提高用户体验
    

👉 关键取舍：  
增加复杂度 → 换取网络安全

* * *

# 五、状态存储（State Storage）

Ethereum 使用：

-   global state
    
-   Merkle Patricia Trie
    

### Design Rationale

优点：

-   支持轻节点验证
    
-   可生成状态证明
    
-   安全性高
    

缺点：

-   数据结构复杂
    
-   IO 开销大
    

👉 核心取舍：  
牺牲性能 → 换取可验证性

* * *

# 六、交易执行顺序（Sequential Execution）

EL 中交易严格顺序执行

### 为什么不并行？

因为：

-   合约之间存在状态依赖
    
-   并行执行会导致非确定性
    

### Design Rationale

优点：

-   简化共识
    
-   保证确定性
    

缺点：

-   TPS 限制
    
-   扩展性受限
    

👉 核心取舍：  
牺牲吞吐量 → 保证一致性

* * *

# 七、合约代码不可变

部署后合约不可修改

### Design Rationale

优点：

-   提高可信度
    
-   防止后门
    
-   去中心化保证
    

缺点：

-   Bug 无法修复
    
-   需要代理模式
    

👉 关键取舍：  
安全可信 > 灵活升级

* * *

# 八、区块结构设计

Execution payload 包含：

-   transactions
    
-   receipts
    
-   logs
    
-   state root
    

### Design Rationale

为什么要 receipts root？  
→ 支持交易执行证明

为什么要 logs？  
→ 支持事件监听

👉 设计重点：  
为 Layer2 与索引服务提供基础

* * *

# 九、EL 与 CL 分离设计（Post-Merge）

合并后：  
CL 提议区块  
EL 执行区块

### Design Rationale

优点：

-   模块化架构
    
-   客户端多样性
    
-   更易升级
    

缺点：

-   复杂度上升
    
-   需要 Engine API
    

👉 核心取舍：  
系统复杂度 ↑ → 可扩展性 ↑

* * *

# 十、整体设计哲学总结

Execution Layer 的核心理念：

1.  确定性优先
    
2.  安全优先
    
3.  可验证优先
    
4.  向后兼容优先
    
5.  去中心化优先
    

牺牲：

-   性能
    
-   TPS
    
-   存储效率
    

换取：

-   全球可信执行环境

# 区块链学习打卡笔记：Execution Layer (EL) 与 EL Specs

📅 学习日期：今日  
📚 学习主题：Execution Layer（执行层）基础 + EL Specs（执行层核心规范）

* * *

## 一、Execution Layer（执行层）基础理解 🧠

Execution Layer（EL）是区块链（尤其是以太坊架构）中的**执行层**，主要负责处理用户交易和智能合约执行。

在当前以太坊模块化架构中，网络被分为两部分：

-   共识层（Consensus Layer，CL）
    
-   执行层（Execution Layer，EL）
    

两者分工明确：

共识层负责：

-   区块排序
    
-   出块规则
    
-   验证者管理
    
-   网络安全
    

执行层负责：

-   交易执行
    
-   状态更新
    
-   Gas 计算
    
-   智能合约运行
    
-   账户余额变化
    

一句话理解：  
👉 Execution Layer 是区块链真正“运行应用逻辑”的地方

* * *

## 二、Execution Layer 主要功能 ⚙️

### 1\. 交易执行

Execution Layer 接收用户提交的交易，例如：

-   转账
    
-   Token 交易
    
-   NFT 铸造
    
-   DeFi 操作
    

EL 负责：

-   验证交易合法性
    
-   执行交易逻辑
    
-   更新链上状态
    

* * *

### 2\. 智能合约执行 🤖

EL 内部包含虚拟机（如 EVM），用于执行智能合约代码。

执行过程：

1.  用户调用合约
    
2.  交易进入执行层
    
3.  虚拟机执行代码
    
4.  状态发生改变
    
5.  结果写入区块
    

* * *

### 3\. 状态管理 📊

执行层维护区块链状态，包括：

-   账户余额
    
-   合约存储数据
    
-   Token 持有情况
    
-   NFT 所有权
    

这些状态随着每笔交易不断更新。

* * *

### 4\. Gas 费用计算 ⛽

Execution Layer 负责：

-   计算每一步执行消耗
    
-   扣除用户 Gas
    
-   防止无限循环
    

Gas 的作用：

-   防止网络滥用
    
-   激励验证者
    
-   限制计算资源
    

* * *

## 三、Execution Layer 组件结构 🧩

Execution Layer 通常包括：

-   交易池（Transaction Pool）
    
-   执行引擎（Execution Engine）
    
-   状态数据库（State Database）
    
-   虚拟机（EVM）
    
-   RPC 接口
    

流程理解：

用户交易  
↓  
交易池  
↓  
执行引擎  
↓  
EVM 执行  
↓  
状态更新  
↓  
打包区块

* * *

## 四、EL Specs（执行层核心规范）理解 📘

EL Specs 是 Execution Layer 的技术规范，定义了：

-   交易格式
    
-   状态转换规则
    
-   Gas 计算方式
    
-   区块结构
    
-   执行结果标准
    

作用：  
👉 确保不同客户端执行结果一致

* * *

## 五、EL Specs 主要内容 🧠

### 1\. 状态转换函数（State Transition Function）

EL Specs 定义了：  
交易输入 → 状态变化 → 新状态

即：

旧状态 + 交易 = 新状态

所有执行层客户端必须遵循同一规则，否则链会分叉。

* * *

### 2\. 交易执行规则

EL Specs 规定：

-   Nonce 必须匹配
    
-   余额必须足够
    
-   Gas 限制检查
    
-   签名验证
    

只有符合规则的交易才会被执行。

* * *

### 3\. Gas 计算规范

EL Specs 明确：

-   每个操作消耗多少 Gas
    
-   存储写入成本
    
-   合约调用费用
    
-   内存扩展费用
    

这样所有节点计算一致。

* * *

### 4\. 区块执行结果

EL Specs 规定执行后必须生成：

-   State Root（状态根）
    
-   Receipt Root（收据根）
    
-   Logs Bloom
    
-   Gas Used
    

这些数据写入区块头。

* * *

## 六、Execution Layer 与钱包、Token、NFT 的关系 🔗

用户通过钱包发起交易  
↓  
交易进入 Execution Layer  
↓  
EL 执行智能合约  
↓  
Token / NFT 状态改变  
↓  
结果写入区块链

因此：

-   钱包 → 发起交易
    
-   Execution Layer → 执行逻辑
    
-   Token / NFT → 被执行对象
    

Execution Layer 是资产变化真正发生的地方。

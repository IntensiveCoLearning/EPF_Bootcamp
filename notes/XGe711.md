# Client Architecture 学习笔记

## （Execution Layer 客户端架构概述）

## 一、什么是 Client Architecture

Client Architecture（客户端架构）指的是 **Execution Layer 客户端内部各模块的组成方式及其协作关系**。

Execution Layer 客户端本质上是一个完整的区块链执行系统，负责：

-   接收交易
    
-   执行区块
    
-   维护状态
    
-   提供 RPC 接口
    
-   与 Consensus Layer 通信
    

常见 EL 客户端（仅作为理解背景）包括：

-   Geth
    
-   Nethermind
    
-   Besu
    
-   Erigon
    

虽然实现不同，但模块结构基本一致。

* * *

## 二、EL 客户端整体模块总览

Execution Layer 客户端通常由以下核心模块组成：

```
+--------------------------------------------------+
|                  RPC API Layer                   |
+--------------------------------------------------+
|                  TxPool (Mempool)                |
+--------------------------------------------------+
|                  Block Execution                 |
+--------------------------------------------------+
|                      EVM                         |
+--------------------------------------------------+
|                  State Database                  |
+--------------------------------------------------+
|                    P2P Network                   |
+--------------------------------------------------+
|                Engine API (CL 交互)              |
+--------------------------------------------------+
```

* * *

## 三、核心模块详解

### 1\. P2P Network（点对点网络模块）

作用：

-   与其他节点通信
    
-   同步区块
    
-   同步交易
    
-   广播新区块
    

主要功能：

-   区块同步
    
-   交易传播
    
-   节点发现
    

这是客户端的数据输入来源。

* * *

### 2\. TxPool（交易池 / Mempool）

作用：

-   存储待打包交易
    
-   按 gas price 排序
    
-   提供区块构建候选交易
    

主要流程：

```
接收交易 → 验证 → 放入 txpool → 排序 → 提供给区块构建
```

特点：

-   非共识数据
    
-   每个节点不同
    
-   动态变化
    

* * *

### 3\. Block Builder / Block Execution（区块构建与执行）

作用：

-   从 TxPool 选择交易
    
-   构建区块
    
-   执行交易
    
-   更新状态
    

执行流程：

```
for tx in block:
    执行交易
    更新状态
```

输出：

-   state root
    
-   receipts
    
-   logs
    

* * *

### 4\. EVM（Ethereum Virtual Machine）

作用：

-   执行智能合约代码
    
-   处理 opcode
    
-   计算 gas 消耗
    

EVM 是 Execution Layer 的核心计算引擎。

执行环境包含：

-   Stack
    
-   Memory
    
-   Storage
    
-   Gas counter
    
-   Program counter
    

* * *

### 5\. State Database（状态数据库）

作用：

-   存储账户状态
    
-   存储合约存储数据
    
-   维护 state trie
    

存储内容：

-   balance
    
-   nonce
    
-   code
    
-   storage
    

特点：

-   使用 Merkle Patricia Trie
    
-   可验证
    
-   支持回滚
    

* * *

### 6\. RPC API Layer

作用：

-   向外提供接口
    
-   钱包调用
    
-   DApp 调用
    
-   开发者查询
    

常见接口：

-   eth\_sendTransaction
    
-   eth\_call
    
-   eth\_getBalance
    
-   eth\_getBlockByNumber
    

这是用户与客户端交互入口。

* * *

### 7\. Engine API（与 Consensus Layer 通信）

作用：

-   CL 与 EL 交互
    
-   接收新区块执行请求
    
-   返回 execution payload
    

主要方法：

-   engine\_newPayload
    
-   engine\_forkchoiceUpdated
    
-   engine\_getPayload
    

这是 Merge 后新增的重要模块。

* * *

## 四、模块之间的工作流程

完整数据流：

```
P2P 网络
   ↓
TxPool
   ↓
Block Builder
   ↓
EVM 执行
   ↓
State DB 更新
   ↓
生成 Execution Payload
   ↓
Engine API → Consensus Layer
```

RPC 调用路径：

```
User / Wallet
     ↓
   RPC API
     ↓
   TxPool
```

* * *

## 五、客户端架构核心特点

1.  模块化设计
    
2.  执行与存储分离
    
3.  网络与执行解耦
    
4.  支持并发处理
    
5.  可替换实现
    

* * *

## 六、考试重点

需要重点掌握：

1.  EL 客户端包含哪些模块
    
2.  TxPool 的作用
    
3.  EVM 在架构中的位置
    
4.  State DB 的作用
    
5.  Engine API 的作用
    
6.  模块之间的数据流
    

* * *

## 七、个人学习总结

Execution Layer Client 可以理解为五大核心系统：

1.  网络系统（P2P）
    
2.  交易系统（TxPool）
    
3.  执行系统（EVM）
    
4.  状态系统（State DB）
    
5.  接口系统（RPC + Engine API）
    

它们共同完成：

接收交易 → 执行交易 → 更新状态 → 生成区块 → 与 CL 协作

一句话总结：

Execution Layer Client =  
网络模块 + 交易池 + EVM + 状态数据库 + 接口层

# Execution Layer (EL) 学习笔记

## 一、Execution Layer (EL) 基础概念

Execution Layer（执行层，简称 EL）是以太坊架构中负责**交易处理与状态执行**的核心部分。它主要完成以下职责：

1.  **交易处理**
    
    -   接收用户发送的交易（Transaction）
        
    -   验证交易格式与签名
        
    -   按顺序执行交易
        
2.  **状态更新**
    
    -   维护账户状态（Account State）
        
    -   更新余额、nonce、存储等信息
        
    -   生成新的 world state
        
3.  **智能合约执行**
    
    -   运行 EVM（Ethereum Virtual Machine）
        
    -   处理合约调用与部署
        
    -   计算 gas 消耗
        
4.  **区块执行**
    
    -   按区块内交易顺序执行
        
    -   计算新的 state root
        
    -   返回 execution payload
        

简而言之：  
Execution Layer = **交易执行 + 状态更新 + EVM 计算**

* * *

## 二、Execution Layer 与 Consensus Layer 的关系

在以太坊合并（The Merge）之后：

-   **Consensus Layer（CL）**  
    负责区块共识、验证者、出块逻辑
    
-   **Execution Layer（EL）**  
    负责执行交易、维护状态
    

两者通过 **Engine API** 进行通信：

流程：

1.  CL 提议新区块
    
2.  CL 调用 EL 执行 payload
    
3.  EL 返回 execution 结果
    
4.  CL 验证并最终确认区块
    

* * *

## 三、EL Specs（执行层核心规范）

EL Specs 是 Execution Layer 的正式技术规范，定义了执行层必须遵守的规则，包括：

### 1\. Block 执行规范

区块执行流程：

1.  验证父区块存在
    
2.  验证时间戳
    
3.  执行区块内所有交易
    
4.  更新状态树
    
5.  计算 receipts root
    
6.  计算 logs bloom
    
7.  输出 execution payload
    

执行顺序：

```
for tx in block.transactions:
    validate(tx)
    execute(tx)
    update_state()
```

* * *

### 2\. Transaction 执行规则

交易执行包括：

-   nonce 检查
    
-   gas 限制检查
    
-   余额检查
    
-   EVM 执行
    
-   gas 扣除
    
-   状态提交
    

交易执行结果包含：

-   status（成功 / 失败）
    
-   gas used
    
-   logs
    
-   return data
    

* * *

### 3\. 状态管理（State Transition）

EL 的核心就是 **状态转换函数**

State Transition：

```
State' = ApplyTransaction(State, Tx)
```

整个区块执行：

```
State_n = ApplyBlock(State_0, Block)
```

关键点：

-   状态不可变（通过 Merkle Patricia Trie）
    
-   每次执行生成新的 state root
    
-   所有节点可验证一致性
    

* * *

### 4\. Gas 机制

EL Specs 定义了：

-   gas limit
    
-   gas used
    
-   base fee（EIP-1559）
    
-   priority fee
    

Gas 执行顺序：

1.  预扣 gas
    
2.  执行指令消耗 gas
    
3.  未用 gas 返还
    
4.  失败仍扣 gas
    

目的：

-   防止无限循环
    
-   防止网络滥用
    
-   资源定价机制
    

* * *

### 5\. EVM 执行环境

Execution Layer 运行在 EVM 中：

EVM 包含：

-   Stack
    
-   Memory
    
-   Storage
    
-   Program Counter
    
-   Gas Counter
    

执行模型：

```
while gas > 0:
    opcode = fetch()
    execute(opcode)
```

* * *

### 6\. Execution Payload

EL 向 CL 返回的执行结果称为：

Execution Payload

包含：

-   block hash
    
-   parent hash
    
-   state root
    
-   receipts root
    
-   logs bloom
    
-   gas used
    
-   transactions
    
-   withdrawals
    

该 payload 被 CL 用于区块共识验证。

* * *

## 四、Execution Layer 工作流程总结

完整流程：

1.  用户发送交易
    
2.  交易进入 mempool
    
3.  Builder/Proposer 组装区块
    
4.  CL 提议区块
    
5.  EL 执行区块交易
    
6.  生成新的 state root
    
7.  返回 execution payload
    
8.  CL 完成共识
    

* * *

## 五、EL 关键数据结构

### 1\. Account

包含：

-   nonce
    
-   balance
    
-   storageRoot
    
-   codeHash
    

### 2\. Transaction

包含：

-   nonce
    
-   gas limit
    
-   to
    
-   value
    
-   data
    
-   signature
    

### 3\. Block

包含：

-   header
    
-   transactions
    
-   withdrawals
    

* * *

## 六、EL Specs 核心目标

Execution Layer 规范主要解决：

-   所有节点执行一致性
    
-   确保状态可验证
    
-   标准化区块执行
    
-   保证网络安全
    
-   提供 EVM 执行环境
    

* * *

## 七、重点理解（考试/作业常考）

必须掌握：

1.  EL 与 CL 的分工
    
2.  State Transition 概念
    
3.  Transaction 执行流程
    
4.  Gas 机制作用
    
5.  Execution Payload 内容
    
6.  Engine API 作用
    

* * *

## 八、个人学习总结

Execution Layer 是以太坊的“计算引擎”，  
Consensus Layer 是“决策系统”。

EL 负责：

-   执行
    
-   计算
    
-   更新状态
    

EL Specs 则是：

-   定义执行规则
    
-   确保所有客户端行为一致
    
-   保证区块执行可验证
    

理解 EL 的关键就是抓住一句话：

**Execution Layer = 状态转换函数 + EVM 执行环境**

# 📚 打卡笔记：Ethereum The Protocol — Execution Layer (EL) Design Rationale

## 一、EL 的核心定位

Execution Layer 是负责：

-   处理交易
    
-   维护账户状态
    
-   执行智能合约
    
-   产生区块执行结果
    

在合并（The Merge）之后架构变为：

-   CL：Consensus Layer → 负责共识
    
-   EL：Execution Layer → 负责执行
    

👉 设计目标：

1.  保证确定性执行
    
2.  支持复杂智能合约
    
3.  高安全性
    
4.  历史兼容性
    
5.  可扩展性
    

* * *

# 二、账户模型设计（Account Model）

### 设计选择

Ethereum 采用：

-   Account-based model  
    而不是 UTXO（像 Bitcoin）
    

### Design Rationale（设计取舍原因）

优点：

-   更适合复杂合约调用
    
-   状态存储更直观
    
-   支持内部交易
    

代价：

-   状态膨胀问题
    
-   并行执行难度更高
    

👉 核心取舍：  
牺牲并行性 → 换取智能合约灵活性

* * *

# 三、EVM 执行环境

核心组件：  
Ethereum Virtual Machine

### 为什么要设计 EVM？

目标：

-   所有节点执行同一逻辑
    
-   跨平台确定性
    
-   安全沙箱环境
    

### Design Rationale

1.  使用 stack-based VM  
    优点：
    

-   实现简单
    
-   状态可预测  
    缺点：
    
-   执行效率低于 register-based VM
    

👉 取舍：  
牺牲性能 → 换取安全与确定性

* * *

# 四、Gas 机制设计

Gas = 执行成本计量单位

### 为什么必须设计 Gas？

防止：

-   无限循环攻击
    
-   资源滥用
    
-   DOS 攻击
    

### Design Rationale

1.  每个 opcode 定价  
    → 精细资源控制
    
2.  交易预付 gas  
    → 防止恶意交易
    
3.  未用 gas 返还  
    → 提高用户体验
    

👉 关键取舍：  
增加复杂度 → 换取网络安全

* * *

# 五、状态存储（State Storage）

Ethereum 使用：

-   global state
    
-   Merkle Patricia Trie
    

### Design Rationale

优点：

-   支持轻节点验证
    
-   可生成状态证明
    
-   安全性高
    

缺点：

-   数据结构复杂
    
-   IO 开销大
    

👉 核心取舍：  
牺牲性能 → 换取可验证性

* * *

# 六、交易执行顺序（Sequential Execution）

EL 中交易严格顺序执行

### 为什么不并行？

因为：

-   合约之间存在状态依赖
    
-   并行执行会导致非确定性
    

### Design Rationale

优点：

-   简化共识
    
-   保证确定性
    

缺点：

-   TPS 限制
    
-   扩展性受限
    

👉 核心取舍：  
牺牲吞吐量 → 保证一致性

* * *

# 七、合约代码不可变

部署后合约不可修改

### Design Rationale

优点：

-   提高可信度
    
-   防止后门
    
-   去中心化保证
    

缺点：

-   Bug 无法修复
    
-   需要代理模式
    

👉 关键取舍：  
安全可信 > 灵活升级

* * *

# 八、区块结构设计

Execution payload 包含：

-   transactions
    
-   receipts
    
-   logs
    
-   state root
    

### Design Rationale

为什么要 receipts root？  
→ 支持交易执行证明

为什么要 logs？  
→ 支持事件监听

👉 设计重点：  
为 Layer2 与索引服务提供基础

* * *

# 九、EL 与 CL 分离设计（Post-Merge）

合并后：  
CL 提议区块  
EL 执行区块

### Design Rationale

优点：

-   模块化架构
    
-   客户端多样性
    
-   更易升级
    

缺点：

-   复杂度上升
    
-   需要 Engine API
    

👉 核心取舍：  
系统复杂度 ↑ → 可扩展性 ↑

* * *

# 十、整体设计哲学总结

Execution Layer 的核心理念：

1.  确定性优先
    
2.  安全优先
    
3.  可验证优先
    
4.  向后兼容优先
    
5.  去中心化优先
    

牺牲：

-   性能
    
-   TPS
    
-   存储效率
    

换取：

-   全球可信执行环境
    

* * *

# 十一、EL 关键设计取舍总表

| 设计点 | 选择 | 放弃 | 原因 |
| --- | --- | --- | --- |
| 账户模型 | Account | UTXO | 合约更灵活 |
| VM | Stack-based | Register | 确定性更强 |

| 合约 | 不可变 | 可修改 | 信任最小化 |
| 架构 | CL + EL | 单层 | 模块化 |

* * *

# 十二、学习复盘

EL 本质目标：

👉 构建一个全球统一计算机  
👉 所有节点执行同一逻辑  
👉 结果完全一致

一句话总结：

Execution Layer =  
“确定性 + 可验证 + 去中心化执行环境”

# 区块链学习打卡笔记：Execution Layer (EL) 与 EL Specs

📅 学习日期：今日  
📚 学习主题：Execution Layer（执行层）基础 + EL Specs（执行层核心规范）

* * *

## 一、Execution Layer（执行层）基础理解 🧠

Execution Layer（EL）是区块链（尤其是以太坊架构）中的**执行层**，主要负责处理用户交易和智能合约执行。

在当前以太坊模块化架构中，网络被分为两部分：

-   共识层（Consensus Layer，CL）
    
-   执行层（Execution Layer，EL）
    

两者分工明确：

共识层负责：

-   区块排序
    
-   出块规则
    
-   验证者管理
    
-   网络安全
    

执行层负责：

-   交易执行
    
-   状态更新
    
-   Gas 计算
    
-   智能合约运行
    
-   账户余额变化
    

一句话理解：  
👉 Execution Layer 是区块链真正“运行应用逻辑”的地方

* * *

## 二、Execution Layer 主要功能 ⚙️

### 1\. 交易执行

Execution Layer 接收用户提交的交易，例如：

-   转账
    
-   Token 交易
    
-   NFT 铸造
    
-   DeFi 操作
    

EL 负责：

-   验证交易合法性
    
-   执行交易逻辑
    
-   更新链上状态
    

* * *

### 2\. 智能合约执行 🤖

EL 内部包含虚拟机（如 EVM），用于执行智能合约代码。

执行过程：

1.  用户调用合约
    
2.  交易进入执行层
    
3.  虚拟机执行代码
    
4.  状态发生改变
    
5.  结果写入区块
    

* * *

### 3\. 状态管理 📊

执行层维护区块链状态，包括：

-   账户余额
    
-   合约存储数据
    
-   Token 持有情况
    
-   NFT 所有权
    

这些状态随着每笔交易不断更新。

* * *

### 4\. Gas 费用计算 ⛽

Execution Layer 负责：

-   计算每一步执行消耗
    
-   扣除用户 Gas
    
-   防止无限循环
    

Gas 的作用：

-   防止网络滥用
    
-   激励验证者
    
-   限制计算资源

## 四、EL Specs（执行层核心规范）理解 📘

EL Specs 是 Execution Layer 的技术规范，定义了：

-   交易格式
    
-   状态转换规则
    
-   Gas 计算方式
    
-   区块结构
    
-   执行结果标准
    

作用：  
👉 确保不同客户端执行结果一致

* * *

## 五、EL Specs 主要内容 🧠

### 1\. 状态转换函数（State Transition Function）

EL Specs 定义了：  
交易输入 → 状态变化 → 新状态

即：

旧状态 + 交易 = 新状态

所有执行层客户端必须遵循同一规则，否则链会分叉。

* * *

### 2\. 交易执行规则

EL Specs 规定：

-   Nonce 必须匹配
    
-   余额必须足够
    
-   Gas 限制检查
    
-   签名验证
    

只有符合规则的交易才会被执行。

* * *

### 3\. Gas 计算规范

EL Specs 明确：

-   每个操作消耗多少 Gas
    
-   存储写入成本
    
-   合约调用费用
    
-   内存扩展费用
    

这样所有节点计算一致。

* * *

### 4\. 区块执行结果

EL Specs 规定执行后必须生成：

-   State Root（状态根）
    
-   Receipt Root（收据根）
    
-   Logs Bloom
    
-   Gas Used
    

这些数据写入区块头。
